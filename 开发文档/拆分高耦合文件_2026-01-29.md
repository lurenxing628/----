# 拆分高耦合文件（2026-01-29 快照）

> 目标：识别当前仓库中“职责过多/依赖过广/改动牵一发动全身”的文件，并给出**可落地、可逐步迁移、兼容优先**的拆分计划。
>
> 说明：本快照仅给出判断与拆分方案，不涉及任何代码改动。

## 0. 判定口径（什么算“过于耦合”）

满足以下任意 2 条，通常就值得拆：

- **职责过多**：一个文件同时承担“HTTP 路由 + 页面拼装 + Excel 导入/导出 + SQL 查询 + 事务/日志 + 业务规则”中的多项。
- **跨层依赖扩散**：路由层直接写大量 SQL/文件 IO；服务层夹带视图拼装；算法层依赖配置服务对象（而不是纯配置快照）。
- **重复实现显著**：多个模块复制粘贴相同的 Excel 上传解析/去重校验/yes-no 标准化等。
- **体量过大**：单文件上千行且持续增长（通常意味着单点修改风险高）。

## 1. 仓库现状概览（哪些已经做得比较解耦）

以下模块当前耦合度可控（不作为本次首要拆分对象）：

- `web/routes/scheduler.py`：已采用“入口聚合 + 子模块注册”的方式拆分（参照用法）。
- `web/routes/reports.py`：路由很薄，主要转调 `core/services/report/ReportEngine`。
- `core/services/report/*`：已按 `queries / calculations / exporters` 分层。
- `core/infrastructure/*`：未发现 core 层依赖 Flask 的问题（`core`/`data` 内无 `from flask ...`）。

## 2. 本次扫描到的高耦合文件清单（按优先级排序）

### P0（优先拆：收益最大、风险可控）

#### 2.1 `web/routes/personnel.py`

**耦合信号**：

- 人员 CRUD、人员-设备关联、个人工作日历页面、以及 3 套 Excel（人员/关联/个人日历）全部在一个文件。
- Excel 路由内包含上传解析、校验、预览 diff、确认落库、模板生成/导出（openpyxl）、审计日志等多个职责。
- 路由层包含一定量的直写 SQL（如聚合关联查询）。

**拆分目标**：维持 `app.py` 里 `from web.routes.personnel import bp as personnel_bp` 不变；URL/endpoint 名称尽量不变。

**建议拆分结构（落地切刀）**：

- `web/routes/personnel_bp.py`：`bp = Blueprint("personnel", __name__)` + 通用显示/规范化（如 status 中文、yes/no 标准化）。
- `web/routes/personnel_pages.py`：人员列表/详情/新增/更新/删除/批量操作/人员-设备关联的增删改。
- `web/routes/personnel_calendar_pages.py`：个人工作日历页面与 upsert。
- `web/routes/personnel_excel_operators.py`：人员基本信息 Excel（page/preview/confirm/template/export）。
- `web/routes/personnel_excel_links.py`：人员设备关联 Excel（page/preview/confirm/template/export）。
- `web/routes/personnel_excel_operator_calendar.py`：人员专属日历 Excel（page/preview/confirm/template/export）。
- `web/routes/personnel.py`：入口聚合文件（`from .personnel_bp import bp` + import 子模块注册路由；对外只暴露 `bp`）。

**迁移步骤（最小风险）**：

1. 先引入 `personnel_bp.py` 与新的 `personnel.py` 聚合入口，让 `bp` 的 import 路径保持不变。
2. 按“页面路由 / Excel 路由”分组，把函数体原样搬迁到子模块（只改 import，不改逻辑）。
3. 最后再做“路由瘦身”（可选）：把 `g.db.execute(...)` 的 SQL 聚合下沉到 repository/service。

#### 2.2 `web/routes/equipment.py`

- Excel 与停机的逻辑块与页面逻辑交织，文件体量大且持续增长。

**建议拆分结构**：

- `web/routes/equipment_bp.py`

- `web/routes/equipment_downtimes.py`：停机计划页面 + 批量停机 + 取消停机。
- `web/routes/equipment_excel_machines.py`：设备 Excel（page/preview/confirm/template/export）。
- `web/routes/equipment_excel_links.py`：设备-人员关联 Excel（page/preview/confirm/template/export）。
- `web/routes/equipment.py`：入口聚合（对外仍导出 `bp`）。

迁移顺序同 personnel：先“拆文件不改逻辑”，再考虑把直写 SQL 下沉。

#### 2.3 `web/routes/process.py`

**建议拆分结构**：

#### 2.4 `web/routes/system.py`

**耦合信号**：

- 同一文件同时覆盖：UI 模式切换、备份/恢复（含文件操作）、插件开关、操作日志查询/删除、排产历史查询等。
- 同时涉及 `current_app` 配置、文件系统、DB 连接管理（含独立 sqlite3 连接）、schema ensure、复杂参数解析等。

**建议拆分结构**（保持 `web/routes/system.py` 仍导出 `bp` 给 `app.py` 注册）：

- `web/routes/system_bp.py`：`bp = Blueprint("system", __name__)`。
- `web/routes/system_ui_mode.py`：`ui_mode_set`（Cookie + SystemConfig）。
- `web/routes/system_backup.py`：`backup_page/backup_create/backup_settings/backup_delete/backup_delete_batch/backup_cleanup/backup_restore`。

- `web/routes/system_logs.py`：`logs_page/logs_settings/logs_delete/logs_delete_batch`。
- `web/routes/system_history.py`：`history_page`。
- `web/routes/system_utils.py`：`_safe_next_url/_parse_dt/_normalize_time_range/_safe_int/_validate_backup_filename` 等纯工具函数。
- `web/routes/system.py`：入口聚合。

**迁移步骤**：

1. 先抽 `system_utils.py`（纯函数，风险低）。
2. 再按子页面拆分路由（backup/logs/history/ui_mode）。
3. 最后统一页面共享的数据获取（如 job_state、system_cfg_snapshot）放到 `system_utils.py` 或 `SystemConfigService` 侧。

## 3. 拆分落地时的“兼容性约束”（建议写进实施 PR 的自检清单）

- **Blueprint 对外不变**：`app.py` 的 import 与 `register_blueprint` 不改；对外仍导出同名 `bp`。
- **URL 与 endpoint 名称不变**：函数名尽量不改；拆到子模块也要确保装饰器仍挂在同一个 `bp` 上。
- **模板引用不变**：`url_for('personnel.xxx')` 等不改，避免牵动模板大范围修改。
- **先搬迁后重构**：第一阶段只做“拆文件”，不做逻辑重写；第二阶段再做 SQL 下沉/工具收敛等优化。
