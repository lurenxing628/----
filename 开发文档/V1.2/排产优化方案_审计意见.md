## 二、代码里发现的实际问题

这部分是在读代码过程中发现的、目前已经存在或方案升级后会踩到的问题。每个都给出了具体位置和建议。

### 2.1 冻结窗口会漏掉"正在进行中"的工序

**在哪里**：`data/repositories/schedule_repo.py` 第 93-99 行

**是什么问题**：冻结窗口的查询 SQL 是这样写的：

```sql
SELECT op_id, machine_id, operator_id, start_time, end_time
FROM Schedule
WHERE version = ?
  AND op_id IN (...)
  AND start_time >= ?    -- 开始时间必须在窗口之内
  AND start_time < ?
```

它只找"开始时间落在冻结窗口范围内"的工序。但有一种很常见的情况被忽略了：一个工序在冻结窗口**之前**就开始了，但它到冻结窗口时间段内**还没做完**，正在进行中。按现在的查询，这个工序不会被冻结，重新排产时可能被打乱。

**举个例子**：假设冻结窗口是 2 月 15 日 ~ 2 月 18 日。一个工序 2 月 14 日开始、2 月 17 日才做完。这个工序不会被冻结，因为它的 `start_time` 是 2 月 14 日，不在窗口范围内。

**建议怎么改**：把查询条件改成找"时间区间和冻结窗口有交集"的工序，大致是 `start_time < 窗口结束 AND end_time > 窗口开始`。这样无论工序是在窗口之前开始的、窗口之中开始的，只要在窗口期间还"活着"，都会被冻结。

### 2.2 auto_assign 排产后会偷偷改输入数据

**在哪里**：`core/services/scheduler/schedule_persistence.py` 第 54-81 行

**是什么问题**：当开启 auto_assign（自动分配资源）并且不是模拟模式时，排产完成后系统会把算法自动分配的设备和人员**写回**到 `BatchOperations` 表里。也就是说，排完一次产之后，原来那些"没指定设备/人员"的工序，现在变成"已指定"了。

**为什么这是问题**：

1. **没法做对比实验**。你想对比两种策略哪个更好，但第一次排产完，数据就变了。第二次排产看到的输入已经不一样了（工序上已经有设备了，不需要自动分配了），两次的结果没法公平比较。
2. **不可逆**。一旦写进去，除非手动清掉，否则后续所有排产都会受影响。

**实际影响**：从 Case03 的测试报告可以看到，greedy 模式（version=1）跑完后再跑 improve 模式（version=2），improve 反而更差（8 个超期 vs 5 个超期）。这可能就和 auto_assign 回写有关——第二次跑的时候输入已经变了。

**建议怎么改**：
- 短期：做对比实验时，用 `simulate=True`（模拟模式不回写），或者每次实验前拷贝一份数据库。
- 长期：加一个配置开关，比如 `auto_assign_persist=yes/no`，让用户自己决定是否要回写。

### 2.3 排序策略"权重混合"在不同日期会给出不同结果

**在哪里**：`core/algorithms/sort_strategies.py` 第 96 行

**是什么问题**：`WeightedStrategy`（权重混合排序策略）里面用了 `today = date.today()` 来计算交期紧迫度。意思是，同样一批订单，今天跑和明天跑，排序结果可能不一样——因为"离交期还有几天"这个数字变了。

```python
def sort(self, batches):
    today = date.today()  # <-- 这里
    def calc_score(batch):
        # ...
        days_left = (batch.due_date - today).days
        due_score = 100 - min(max(days_left, 0), 100)
```

**为什么这是问题**：如果你想复现某次排产结果（比如排查问题或做回归测试），只要不是在同一天跑，结果就可能不同。排查问题时会很困惑。

**建议怎么改**：把 `today` 改成从外部传入，用排产的 `start_dt`（排产起始时间）的日期部分，而不是取当天日期。这样只要 `start_dt` 一样，排序结果就一样。

### 2.4 improve 模式的随机种子导致可复现性差

**在哪里**：`core/services/scheduler/schedule_optimizer.py` 第 327 行

**是什么问题**：improve 模式（多起点搜索 + 局部搜索）的随机数种子是 `version`（排产版本号）：

```python
rnd = random.Random(int(version))
```

每次排产 version 都会递增（1, 2, 3...），所以同样的数据连跑两次，因为 version 不同，随机搜索路径不同，结果也不同。

**这不完全是 bug**，但如果你需要"精确复现"某次排产的 improve 结果（比如客户投诉说排产结果有问题，你想重现），就做不到，因为 version 不可能回退。

**建议怎么改**：在配置或 API 参数里加一个可选的 `random_seed` 字段。正常使用时不填（系统自动用 version），需要复现时填入当时的 version 值即可。

## 三、设计层面的问题与建议

这部分不是 bug，而是方案或架构设计上值得关注的点。

### 3.1 ready_weight 参数：配了但没用

**在哪里**：
- 配置表里有 `ready_weight` 参数
- `core/algorithms/sort_strategies.py` 第 88 行注释明确写了：`V1.1 起，"齐套权重"作为预留字段，不参与当前排序`
- 第 147-152 行的 `StrategyFactory.create` 也跳过了 `ready_weight`

**是什么问题**：用户在界面上能看到这个参数、能填值，但填了也没有任何效果。容易让人误以为"我调了齐套权重但排产结果没变化，是不是有 bug"。

**建议**：要么在 v2 方案中明确这个参数的用途并实现它，要么在 UI 上把它标记为"暂不生效"或干脆隐藏掉。

---

### 3.2 排产结果 JSON 缺少版本标识

**在哪里**：`core/services/scheduler/schedule_summary.py` 第 179-225 行

**是什么问题**：每次排产完成后会生成一个 `result_summary` JSON，保存在 `ScheduleHistory` 表里。但这个 JSON 没有任何版本号字段。

**为什么是问题**：
- 以后 v2 上线后，JSON 的字段肯定会变（新增字段、改字段含义等）。
- 到时候数据库里既有 v1 格式的老数据，又有 v2 格式的新数据，前端或报表代码怎么区分？只能猜。
- v2 方案里提到要做"L2 稳定性基线对比"（比较两次排产结果的差异），如果 JSON 格式不一致，对比代码会很难写。

**建议**：现在就加一个 `"summary_schema_version": "1.0"` 字段。以后每次 JSON 格式有变化就升版本号。代码里根据版本号做兼容处理。这个改动很小，但能省掉以后很多麻烦。

---

### 3.3 目标函数的"拖期"指标口径不统一

**在哪里**：`core/algorithms/evaluation.py` 第 252-253 行

**是什么问题**：

- v1 的 `min_tardiness` 目标函数用的是 `total_tardiness_hours`——就是把所有超期批次的超期小时数加起来，不管优先级高低。
- v2 方案文档里的 L1 层提到的是 `weighted_tardiness_hours`——按优先级加权的拖期，优先级高的批次超期了罚得更重。

这两个指标数值完全不一样。如果 v2 上线后直接切换到加权指标，就没法直接对比"v2 是不是比 v1 好了"。

**建议**：
- 要么 v2 同时保留两个指标，对比时用相同口径。
- 要么现在就在 v1 的 `evaluation.py` 里加上 `weighted_tardiness_hours` 的计算（代码里其实已经有这个字段了，只是 `objective_score` 函数没用它）。

---

### 3.4 efficiency（日历效率因子）在 v2 整数时间轴上怎么办

**是什么问题**：

v1 里日历系统支持设置每天的"效率值"（比如某天效率 0.8，意思是 1 小时的活要干 1.25 小时）。这个功能在 Case01 测试里实际用到了。

v2 方案要把时间轴从 `datetime` 改成 `int64` 的分钟数，用 Numba 加速计算。但方案里没说清楚：在整数分钟的时间轴上，怎么处理 efficiency != 1.0 的情况？

- 如果简单忽略 efficiency，那 v2 和 v1 在有效率变化的日历下排出来的结果会不一样。
- 如果要支持，就需要在整数时间轴上做某种"时间拉伸"，这会让 Numba 加速的实现变得复杂。

**建议**：这个问题应该在 Phase 0（技术验证阶段）就给出明确的技术方案。不能等到 Phase 1 写代码的时候才发现不好处理。可以考虑的思路：
- 把 efficiency 预处理成一个"分钟数映射表"（逻辑分钟 → 实际分钟），在排产前一次性算好。
- 或者把 efficiency 当作工序工时的系数，在准备输入数据时就把工时调整好，而不是在排产过程中动态计算。

---

### 3.5 日历引擎没有缓存——这是 v1 最大的性能瓶颈

**实际数据**：Case03 测试（348 个工序，SGS 派工 + auto_assign），greedy 模式跑了 **17 分钟**（1,019,538 毫秒）。同样的数据量，如果不用 SGS 模式，只需要几秒。

**根本原因在哪里**：`core/services/scheduler/calendar_engine.py` 里的日历查询（`_policy_for_date` 等方法）每次调用都重新计算，没有缓存。在 SGS 派工模式下，每安排一个工序都要对所有候选设备逐个调用 `add_working_hours` 来估算完成时间，每次 `add_working_hours` 又会多次查日历。工序数 × 候选设备数 × 日历查询次数，三者相乘，调用次数爆炸式增长。

**这和 v2 有什么关系**：
- 如果 v2 上线前不修这个问题，v1 在中等规模场景下就已经慢得不可接受。
- v2 方案里用 Numba + int64 时间轴本质上就是为了绕开这个瓶颈，但这是一个大工程。
- 而给日历引擎加缓存是一个**小改动、大收益**的优化。

**建议**：
- **短期（v1 马上可以做）**：给 `_policy_for_date` 加一个 `@lru_cache` 或字典缓存。预估能减少 50-80% 的排产耗时。这个改动可能只需要几十行代码。
- **长期（v2）**：在 Phase 0 的 int64 时间轴方案中，日历信息应该预先展开成数组，避免排产循环中做任何查询。

---

### 3.6 长时间排产会卡死 Web 界面

**是什么问题**：方案全文没有提到排产过程中 Web 前端怎么办。目前系统用 Flask 同步处理请求，排产是在请求处理线程里直接跑的。如果排产需要几分钟甚至十几分钟，用户在浏览器上什么都做不了，甚至可能看到超时错误。

**建议**：
- 把排产改成后台任务（比如用线程池或队列）。
- 前端改成"提交排产任务 → 轮询进度 → 完成后通知"的模式。
- 至少要有一个进度条或"排产中，请稍候"的提示，不能让用户干等着什么反馈都没有。

这个问题在 v1 阶段就已经存在（Case03 的 17 分钟），不是 v2 才需要考虑的。

---

### 3.7 外协合并组的"身份证"没存在工序表里

**在哪里**：`core/services/scheduler/schedule_service.py` 里，外协合并组 `ext_group_id` 是从 `PartOperations`（工艺模板）动态查出来的，`BatchOperations`（批次工序）表本身不存这个字段。

**为什么是问题**：工艺模板是可以修改的。如果创建批次之后有人改了模板里的外协分组，那这个批次在排产时看到的分组关系和创建时不一样。这可能导致：
- 本来应该合并排的外协工序变成分开排了（或反过来）。
- 排产结果和预期不符，但原因很难排查。

**建议**：在创建批次工序的时候就把 `ext_group_id` 快照到 `BatchOperations` 表里，之后排产以 `BatchOperations` 上的值为准，不再回头查模板。

---

### 3.8 物料齐套的"闭环"——方案文档没说清楚

**现状**：系统其实已经有一套完整的物料齐套闭环机制，但方案文档里没有完整描述：

1. `BatchMaterials` 表记录每个批次需要哪些物料，每个物料是否齐套。
2. `BatchMaterialService`（`core/services/material/batch_material_service.py` 第 157-189 行）会自动汇总判断批次齐套状态，并把结果写回 `Batch.ready_status` 和 `Batch.ready_date`。
3. 排产时，如果配置了 `enforce_ready_default=yes`，系统会拦住所有 `ready_status != "yes"` 的批次不让排。
4. 对于已齐套的批次，`ready_date` 会被当作该批次最早可开工时间（`core/algorithms/greedy/scheduler.py` 第 211-224 行）。

**建议**：在方案文档的数据模型或约束章节里补充这个闭环的描述，特别是说清楚：
- `ready_date` 到底是"物料到齐的日期"还是"允许开工的最早日期"？
- `enforce_ready` 和 `ready_date` 是两层不同的约束：前者是"能不能排"，后者是"最早什么时候排"。

## 四、OR-Tools 要不要加

### 结论：不建议加为默认依赖，维持现在的"可选"定位就好

### 理由（都有据可查）

| 事实 | 来源 |
|------|------|
| OR-Tools 官方安装文档只声明/测试了 **Windows 10 x64** | [官方安装页](https://developers.google.com/optimization/install/python/binary_windows) |
| 最新版 `ortools 9.15`（2026-01-14）要求 **Python >= 3.9**，不兼容项目的 Python 3.8 | [PyPI ortools 页面](https://pypi.org/project/ortools/) |
| 旧版 `ortools 9.8`（2023-11）支持 Python 3.8，但额外依赖 `pandas >= 2.0` 等重依赖 | [PyPI ortools 9.8 页面](https://pypi.org/project/ortools/9.8.3296/) |
| OR-Tools 需要 **VS2022 的 VC++ 运行库**，而该运行库最新版已不支持 Win7 | 官方安装文档 |
| wheel 名称只标 `win_amd64`，不标最低 Windows 版本，Win7 能不能跑只能靠实测 | PyPI wheel 命名规范 |

### 现在项目的状况

项目里其实**已经做好了 OR-Tools 的可选接入**，不需要再加什么：

- `core/algorithms/ortools_bottleneck.py`：瓶颈子问题的 CP-SAT 求解，`import` 失败就返回 `None`。
- `core/services/scheduler/schedule_optimizer.py` 第 163-254 行：只有 `ortools_enabled=yes` 才尝试调用，异常了也不影响主流程。
- `plugins/ortools_probe_plugin.py`：探测插件，能告诉用户"OR-Tools 装没装、能不能用"。
- `requirements.txt` 里**没有**把 `ortools` 写死——这就是正确的做法。

### 如果真想试试

按照仓库里 `ORTOOLS_WIN7_SPIKE.md` 的步骤，在 Win7 实机上：
1. 装好 VC++ 运行库（可能需要用旧版本）。
2. 离线安装 `ortools==9.6.2534` 或 `9.8.3296` 的 `cp38-win_amd64` wheel。
3. 跑 `python -c "import ortools; print(ortools.__version__)"` 看能不能导入。
4. 如果能导入，再跑一个 CP-SAT 小例子看看求解是否正常。

如果这四步都过了，可以考虑提供一个"Win10+ 可选加速包"。但**默认安装包里不要带它**。

## 五、方案时间线的建议

### Phase 1 交付物太多了

方案里 Phase 1 要同时交付：int64 时间轴、Numba 加速的 decode、FastDecode / AccurateDecode 双解码器、L0-L4 五层目标函数、Iterated Greedy 算法。这些东西每一项都不简单，放在一个 Phase 里风险太高。

**建议拆成两步**：

- **Phase 1a（核心骨架）**：int64 时间轴 + Numba decode + L0（可行性）+ L1（拖期）+ 基础的 Iterated Greedy（不含 delta_eval）。目标是先跑通，能出结果，性能比 v1 好。
- **Phase 1b（锦上添花）**：FastDecode / AccurateDecode 切换、L2-L4（稳定性/换型/效率）、delta_eval 增量评估。这些是优化项，可以在基础版稳定后再加。

### 整体周期建议

方案里给的时间有点紧。考虑到 efficiency 方案还没定、日历预处理要重新设计、测试回归量大，建议整体预留 **12-16 周**。

### L3（换型最小化）和 L4（效率均衡）暂时没必要

看了现有的测试数据，所有 Case 的 `changeover_count`（换型次数）都是 0。说明目前的数据里没有换型时间（SDST 矩阵全零）。L3 层"最小化换型"目前没有实际意义。

L4 层"效率均衡"（各设备负载均匀）在生产实践中通常不是硬需求，优先级应该低于 L0-L2。

**建议**：初期只实现 **L0 + L1**（可行性 + 拖期最小化），L2（稳定性）在有两个版本可比之后再加，L3 和 L4 等有实际数据需求时再做。

## 六、建议做但还没做的验证实验

以下几个实验建议在 v2 开发之前做掉，都不复杂但能提前发现问题：

### 实验 1：冻结窗口跨边界测试

**目的**：验证第 2.1 节描述的问题是否真的会发生。

**步骤**：
1. 用 Case04（冻结窗口用例）或新造数据。
2. 先排产一次（V1），确保有些工序的开始时间在冻结窗口之前、结束时间在冻结窗口之内。
3. 开启冻结窗口，再排产一次（V2）。
4. 检查那些"跨边界"的工序在 V2 里有没有被保留。

**预期结果**：按现在的代码，跨边界工序不会被冻结（会被重新排），确认后就知道要不要修。

---

### 实验 2：auto_assign 副作用测试

**目的**：验证第 2.2 节描述的问题。

**步骤**：
1. 准备一组"工序没指定设备"的数据。
2. 用**非模拟模式**排产一次，记录结果。
3. **不重置数据库**，用同样的参数再排一次。
4. 对比两次排产的输入（第二次的工序上是否已经有设备了）和输出。

**预期结果**：第二次排产时，之前自动分配的设备已经写死在工序上了，行为和第一次不同。

---

## 七、总结

**最值得立刻动手的三件事**（改动小、收益大）：

1. **给日历引擎加缓存**（3.5 节）—— 预计能把 Case03 从 17 分钟降到几分钟以内。
2. **给 result_summary 加版本号**（3.2 节）—— 几行代码的事，以后能省很多兼容性麻烦。
3. **修冻结窗口的查询条件**（2.1 节）—— SQL 改一下就行。

**v2 方案开工前必须想清楚的一件事**：

- efficiency 因子在 int64 时间轴上怎么表达（3.4 节）。这个不想清楚，Phase 1 开发到一半可能要推倒重来。

**保持现状就好的一件事**：

- OR-Tools 维持"可选依赖"定位，不要加进默认安装包（第四节）。
