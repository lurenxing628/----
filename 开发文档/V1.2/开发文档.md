## 1. V1.2 最终结论与必须落地的变更

### 1.1 总结结论

**V1.2 的核心取舍：默认交付不变，增强能力全部可选。**

在“Win7 单机无网、目标机不装 Python”的前提下：

1. **继续锁定 Python 3.8 + PyInstaller 4.10（Win7 打包机）**
2. **默认依赖仍为 openpyxl-only（可交付性优先）**
3. **pandas/numpy/or-tools 统一按“可选依赖”引入**：通过 `vendor/` 投放 + `plugins/` 开关启用；缺失/失败必须自动降级
4. 排产算法路线升级为：**滚动时域（Rolling Horizon） + 分层计划（粗承诺/细排程） + 改进搜索（LNS/ILS）**  
   - OR-Tools CP-SAT/LNS：定位为 **可选增强**（Win7 兼容风险高，需实机验收）
5. **一人多机（可选）**：默认仍按“一人一机”约束；若启用，则允许同一操作员同一时段并发占用多台设备（上限可配）
6. **门禁/第三资源暂不纳入排产模型**：按你们现状由生产准备保证“可开工集合”；排产只对可开工对象求解（避免给开发增加负担）

### 1.2 V1.2 需要写进文档的新增内容清单

- 依赖与离线投放：
  - wheelhouse 准备 → vendor 投放 → 插件启用/回退 → Win7 验收脚本
- 排产算法路线：
  - 三层火箭（Layer1 数据与规则 / Layer2 粗承诺 / Layer3 细排程）
  - 三段时间栅栏（Frozen / Slushy / Liquid）
  - “少扰动”从口号变成配置与评分项（软约束）
- 配置/接口：
  - `POST /scheduler/config` 新增表单字段（仅配置键扩展，不新增 endpoint）
  - 系统页“插件状态/错误堆栈”作为离线可观测手段

## 3. 技术选型与依赖锁定（V1.2：离线可选依赖）

### 3.1 默认依赖（保持 V1 交付成功率）

默认 `requirements.txt` 不变（openpyxl-only）：

```txt
Flask==2.3.3
Werkzeug==2.3.7
openpyxl==3.0.10
python-dateutil==2.8.2
# PyInstaller==4.10  （仅打包机需要；必须 Win7 环境打包）
```

### 3.2 可选依赖（V1.2 引入，但不进默认 requirements）

#### 3.2.1 pandas/numpy（Excel/统计加速）

建议锁定（Win7 + Python3.8 兼容实践口径）：

```txt
pandas==1.3.5
numpy==1.21.6
openpyxl==3.0.10
```

启用方式：
- 依赖投放到 `vendor/`（见 3.3）后，在系统页启用插件：`plugins/pandas_backend_plugin.py`
- Excel 后端会在 `openpyxl` 与 `pandas` 间自动选择（对外仍只收发 `List[Dict]`）

#### 3.2.2 OR-Tools（可选求解器/可选增强）

结论（必须写死在文档里，避免团队误判）：

- **官方 Windows 安装说明仅声明/测试 Win10 x64**，并要求 VC++ 运行库（参考：`https://developers.google.com/optimization/install/python/binary_windows`）。
- **OR-Tools 最新版已不支持 Python3.8**：`ortools 9.15.6755` Requires Python>=3.9（参考：`https://pypi.org/project/ortools/`）。
- 因此在本项目（Python3.8 + Win7）中，OR-Tools 只能作为 **可选增强**，并必须做 **Win7 实机验收**：
  - 优先候选：`ortools==9.8.3296`（Requires Python>=3.8，提供 `cp38 win_amd64` wheel；参考：`https://pypi.org/project/ortools/9.8.3296/`）
  - 备用候选：`ortools==9.6.2534`（Provides `cp38 win_amd64` wheel；参考：`https://pypi.org/project/ortools/9.6.2534/`）

启用口径（保持“缺失不阻断”）：
- 探测与可观测：`plugins/ortools_probe_plugin.py`
- 已落地的可选用法：瓶颈工种 warm-start（见 `core/algorithms/ortools_bottleneck.py`，配置键 `ortools_enabled/ortools_time_limit_seconds`）
- V1.2 路线图：OR-Tools CP-LNS 仅作为可选增强（通过插件启用，验收通过才上线）

### 3.3 离线投放方式（wheelhouse → vendor）

#### 3.3.1 推荐方式：vendor 投放（目标机无需 Python）

本仓库已具备 vendor 注入：
- 启动时会将交付目录下的 `vendor/` 注入 `sys.path`（`core/plugins/runtime.py`）

推荐目录结构（示例）：

```
dist/排产系统/
  排产系统.exe
  vendor/
    site-packages/
      pandas/
      numpy/
      ortools/
      ...
  plugins/
    pandas_backend_plugin.py
    ortools_probe_plugin.py
```

#### 3.3.2 wheelhouse 准备（在“有网环境”完成）

在有网机器上准备离线包（示例命令，具体以你们 wheelhouse 目录为准）：

```bat
REM 建议新建 wheelhouse 目录
mkdir wheelhouse

REM pandas/numpy（Win_amd64 + cp38）
python -m pip download -d wheelhouse pandas==1.3.5 numpy==1.21.6 openpyxl==3.0.10

REM OR-Tools（候选版本二选一）
python -m pip download -d wheelhouse ortools==9.8.3296
REM 或：
python -m pip download -d wheelhouse ortools==9.6.2534
```

> 说明：最终会下载出一堆依赖 wheel（例如 protobuf 等），这正是离线可复现的关键；不要手工删。

#### 3.3.3 投放与回退（必须可一键回滚）

- 投放：将 `wheelhouse` 用 `pip --target` 安装到 `dist/排产系统/vendor/site-packages/`
- 回退：删除/重命名 `vendor/`，并在系统页关闭插件开关（重启生效）

## 4. 排产算法（V1.2 路线：滚动时域 + 分层计划 +（可选）CP-LNS）

### 4.1 你们要的不是“全局最优”，而是“可承诺 + 可应急 + 少扰动”

结合开发文档背景（Win7 离线、Excel 文化、插单频繁、数据不完美），“一次性全局最优”不现实且不可控。V1.2 定位为：

- **交付可靠**：能回答“接不接插单？影响谁？影响多少？”
- **计划稳定**：不允许每次重算都把短期任务大改
- **可解释**：能说清楚“卡在哪/为什么/怎么解”

### 4.2 三层火箭（不是三段算法）

```
Layer1：数据与规则（把不完美数据变成可计算）
  - 工艺路线解析、内外部工序分流、工时口径、资源关系（人-机）

Layer2：粗粒度能力承诺（秒级~分钟级）
  - 以资源池/桶化负载输出：完工窗口、瓶颈曲线、挤占清单

Layer3：细粒度可执行排程（分钟级）
  - 只对短期窗口做工序级/设备级/人员级排程
  - 支持插单后的局部修补，且尽量少扰动
```

### 4.3 时间栅栏（Frozen / Slushy / Liquid）

V1.2 推荐把滚动窗口拆成三段（比单一冻结窗口更贴合现场）：

- **Frozen（硬冻结）**：未来 F 天（例如 1~3 天）原则上不动  
  - 对应现有配置：`freeze_window_enabled / freeze_window_days`
- **Slushy（软冻结）**：未来 S 天（例如 3~7 天）允许动，但“变更成本”很高  
  - 通过稳定性惩罚实现（见 4.6）
- **Liquid（可优化区）**：未来 H 天（例如 14~30 天）正常优化

### 4.4 约束口径（V1.2 明确纳入/不纳入）

纳入（V1 已有为主）：
- 工序前后约束（批次 seq）
- 设备资源不可重叠
- 人员资源约束（默认一人一机；可选一人多机）
- 日历/停机/外协周期（按现有口径）

暂不纳入（你已确认不做）：
- 门禁/第三资源（由生产准备保证“可开工集合”）

### 4.5 一人多机（可选）——默认仍是一人一机

目的：贴合现场“人盯多机”的事实，但不强迫你一次性做完数据治理。

V1.2 文档口径（建议写入 `ScheduleConfig`）：

```txt
operator_parallel_limit: int (>=1)
  - 1：默认（一个时段一个人只能占用一机）
  - >1：允许同一操作员同一时段并发占用多台设备（上限=operator_parallel_limit）
```

> 说明：本能力只改变“人员资源冲突”的判定，不改变设备侧的单资源约束。

### 4.6 少扰动：稳定性惩罚（软约束，不增加硬规则）

V1.2 建议新增稳定性配置键（默认 0 表示关闭，不影响现有结果）：

```txt
stability_weight: float (>=0)                   # 计划稳定性权重（0=关闭）
stability_slushy_days: int (>=0)                # 软冻结天数（越近惩罚越大）
stability_shift_threshold_minutes: int (>=0)    # 超过阈值才计“开始时间变更”
```

稳定性惩罚建议覆盖三类变更（按权重/阈值可调）：
- 开始时间大幅移动（超过阈值）
- 换机/换人（对现场扰动最大）
- 软冻结区（Slushy）内的移动（惩罚更高）

### 4.7 CP-LNS（可选增强）与“业务邻域”

V1.2 的路线建议是：

1. 先把“少扰动修补”跑顺（即便没有 OR-Tools，也能用启发式邻域 + time_budget 达到可用）
2. 若 OR-Tools 在 Win7 实机验收通过，再把 CP-SAT 用于 **局部子问题**（不是全量建模）

建议至少定义 4 类邻域（文档先写清楚口径与输出，避免后期返工）：

- **关键链邻域**：只动导致超期的那条链及其冲突段
- **瓶颈资源邻域**：只动瓶颈机台/瓶颈人员池在某时间段的连续块
- **冲突簇邻域**：只动同一操作者（或同一设备）冲突集合
- **插单影响邻域**：只动插单相关资源段 + 被挤占任务

### 4.8 插单流程（两步走）

- 第一步（Layer2）：秒级影响评估（接不接单/影响谁/怎么消）
- 第二步（Layer3）：分钟级局部修补（Frozen 不动；Slushy 少动；Liquid 优化）

## 5. 配置键与页面接口（V1.2 增量）

### 5.1 新增配置键（建议写入 ScheduleConfig）

在现有键（`algo_mode/dispatch_mode/dispatch_rule/objective/freeze_window/ortools_*`）基础上，V1.2 增量建议新增：

- `operator_parallel_limit`：int(>=1)，默认 1
- `stability_weight`：float(>=0)，默认 0
- `stability_slushy_days`：int(>=0)，默认 0
- `stability_shift_threshold_minutes`：int(>=0)，默认 0

### 5.2 页面/接口变更点（不新增 endpoint，仅扩展字段）

排产策略配置页：
- 页面：`GET /scheduler/`（`templates/scheduler/batches.html`）
- 保存：`POST /scheduler/config`

V1.2 若落地上述配置键，建议在 `POST /scheduler/config` 的 form 中新增字段（与现有写法一致：flash + redirect）：

- `operator_parallel_limit`
- `stability_weight`
- `stability_slushy_days`
- `stability_shift_threshold_minutes`

系统页插件开关（不变）：
- `GET /system/backup` 查看插件状态与错误堆栈
- 勾选 enabled 后 **需重启**（这点必须写进用户操作说明）

## 7. 风险评估与回退策略（V1.2 重点）

### 7.1 pandas/numpy 风险

- 风险：离线安装失败/依赖不全/体积大
- 缓释：
  - 作为可选依赖，仅在需要时投放
  - 插件启用失败不阻断主流程（系统页可观测）
  - 回退=移除 vendor + 关插件

### 7.2 OR-Tools 风险（Win7 最大风险点）

- 风险：官方只测试 Win10；Win7 可能因 VC++ 运行库/DLL 依赖失败
- 缓释：
  - 永远定位为可选增强（缺失自动降级）
  - 必须有 Win7 实机验收脚本与截图留痕
  - 先 warm-start，后 CP-LNS；逐步上线
