# ADR-0001: Win7 单机共享数据、单活用户运行环境

- **状态**：已采纳
- **日期**：2026-01-10（项目启动），2026-03-18（边界修订）
- **决策者**：项目负责人

## 背景

本系统服务于工厂现场的排产调度，目标机器运行 Windows 7，无网络连接需求。

项目初始假设是“单机单用户”，但现场安装方式已经变化为：

- 由管理员统一安装程序与浏览器运行时。
- 同一台机器上的不同域账户需要轮流使用同一套 APS 数据。
- 不要求多用户并发协作，也不允许两个账户同时操作同一套本地数据。

因此，运行边界需要从“单机单用户”修订为“单机共享数据、仅允许单活用户”。

## 决策

- 目标环境仍锁定 Windows 7 SP1 单机。
- 应用仍以本地 Web 形式运行（Flask + 浏览器），不部署为在线服务。
- 程序采用**机器级共享安装**，允许管理员统一安装后由多个域账户看到入口。
- 业务数据采用**机器级共享数据目录**，不同账户看到的是同一套 APS 数据。
- 系统实行**单活用户保护**：
  - 允许不同账户轮流使用。
  - 禁止两个账户并发打开同一套共享数据。
  - 检测到已有实例运行时，第二个账户必须收到明确提示并被拒绝进入，而不是静默复用。

## 理由

- 工厂现场机器普遍为 Win7，无法升级。
- 现场使用域账户是组织管理要求，但业务上仍是“同机轮流值守”，不是协同排产。
- 共享安装可以消除“管理员装一次、域账户打不开”的交付问题。
- 共享数据可以避免每个账户维护一套独立数据库，减少数据漂移与运维成本。
- 单活用户保护可以在不引入完整多用户架构的前提下，控制 SQLite 与本地运行时的并发风险。

## 被拒绝的替代方案

- 继续坚持严格 per-user 安装与 per-user 数据：安装与运维成本高，无法满足现场管理员统一安装的要求。
- 多用户在线 Web 服务：需要服务器、并发控制、用户管理，复杂度远超当前需求。
- 原生 Windows 桌面应用（tkinter/PyQt）：开发效率低，UI 灵活性差。
- 共享数据但允许并发进入：会与当前 SQLite、本地端口与运行时契约模型冲突，风险不可控。

## 后果

- Python 版本仍锁定 3.8（Win7 支持的最后一个主流版本）。
- 所有依赖仍必须兼容 Win7 + Python 3.8。
- 交付仍通过 PyInstaller 打包，但安装器需要支持机器级共享安装。
- 运行时需要引入共享数据目录、共享运行时契约目录和单活用户锁。
- 文档、安装说明、验收流程都必须以“共享数据 + 单活用户”为准重新定义。
- 系统仍**不支持真正的多用户并发协作**；若未来要支持并发，必须另立 ADR，升级数据与部署模型。
