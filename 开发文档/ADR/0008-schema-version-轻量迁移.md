# ADR-0008: Schema Version 轻量迁移机制

- **状态**：已采纳
- **日期**：2026-01-12（数据库设计阶段）
- **决策者**：项目负责人

## 背景

随着功能迭代，数据库表结构需要演进。需要一种机制支持数据库升级而不丢失用户数据。

## 决策

- 使用 `SchemaVersion` 表记录当前版本号。
- 应用启动时自动检测版本，按顺序执行必要的迁移脚本（`core/infrastructure/migrations/v1.py` ~ `v4.py`）。
- 迁移前自动备份数据库文件。
- 迁移脚本存放在 `core/infrastructure/migrations/` 目录，文件名为 `v{N}.py`。

## 理由

- 轻量级，适合 SQLite 单文件数据库。
- 自动备份降低迁移失败风险。
- 顺序执行确保迁移的可预测性。

## 被拒绝的替代方案

- 无版本管理（每次全量重建）：用户数据会丢失。
- Alembic 等重型迁移框架：对 SQLite + 单机场景过度设计。
- 在 `schema.sql` 中直接修改（不保留旧版本兼容）：已部署的用户无法升级。

## 后果

- 每次 schema 变更必须创建新的迁移脚本（递增版本号）。
- `schema.sql` 保持为"当前最新版本"的完整定义。
- 迁移脚本必须幂等或有版本检查，避免重复执行。
