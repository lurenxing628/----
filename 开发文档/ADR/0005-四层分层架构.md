# ADR-0005: 四层分层架构（Routes / Services / Repositories / DB）

- **状态**：已采纳
- **日期**：2026-01-10（项目启动）
- **决策者**：项目负责人

## 背景

系统包含多个业务模块（人员、设备、工艺、排产、物料、报表），需要合理的代码组织方式确保可维护性。

## 决策

采用四层分层架构：

```
web/routes/          → 参数校验、调用 Service、返回响应
core/services/       → 业务逻辑、事务协调
data/repositories/   → 数据库访问、SQL 执行、返回模型对象
core/models/         → 纯数据结构（from_row / to_dict）
```

辅以基础设施层 `core/infrastructure/`（事务管理、错误处理、数据库迁移）。

## 理由

- 职责分离清晰，每层只关注自己的事。
- 便于测试（Service 可 mock Repository）。
- 降低耦合，修改数据库访问不影响路由层。
- 是 Flask 项目的成熟实践，学习成本低。

## 被拒绝的替代方案

- 单体 Route 文件（业务逻辑写在路由里）：耦合高，难维护，难测试。
- 微服务架构：单机场景过度设计。
- MVC 模式（Model 含业务逻辑）：Model 职责过重，不如 Service 层明确。

## 后果

- 新增功能必须按层创建对应文件（route + service + repository）。
- 层间只能单向调用（route -> service -> repository），禁止反向依赖。
- 单文件超过 500 行时按职责拆分（已有 scheduler 拆分先例）。
