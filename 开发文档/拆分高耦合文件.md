过于耦合的文件（按“优先拆分价值”排序）
core/services/scheduler/schedule_service.py
耦合信号：单个 run_schedule() 同时做参数规范化、读库/拼装算法输入、冻结窗口（含直写 SQL）、停机区间加载、资源池构建（含直写 SQL/技能排序口径）、多起点/局部搜索/可选 OR-Tools、指标计算与目标函数评分、超期判定、结果 JSON 汇总、落库（Schedule + History）+ 状态回写 + 操作日志。
问题本质：把“调度编排（application）/优化策略（algorithm）/数据访问（db）/结果汇总（reporting）/副作用（状态+日志）”压在一个文件/一个方法里。
web/routes/scheduler_pages.py
耦合信号：一个路由文件承载批次列表/详情/批量操作/工序编辑/排产执行/策略配置/效果分析/日历维护；还包含视图模型拼装、趋势图 SVG 生成、直接 SQL 查询（OperatorMachine 联动），并调用服务的“内部方法/内部口径”（如权重归一化走私有实现）。
问题本质：把“HTTP/模板渲染/页面计算/数据查询”混在同一处，导致改 UI 也容易误伤业务。
core/algorithms/greedy.py
耦合信号：文件体量大且职责多（排序策略选择+参数读取、派工模式/规则、seed 去重防御、自动选人/选机、外协组逻辑、停机避让等），并通过 self.config.get(...) 隐式依赖服务层形态的配置对象。
问题本质：算法层不够“纯”，复用/替换算法成本高。
core/infrastructure/errors.py
耦合信号：核心错误码/异常类型与 Flask 的 request/render_template、错误处理器注册绑在一起。
问题本质：核心层对 Web 框架硬依赖，后续做 CLI/批处理/独立服务复用会被拖住。
core/services/report/report_engine.py
耦合信号：同一类里同时做 SQL 汇总、指标计算、并直接用 openpyxl 生成导出；还依赖 CalendarService 的“内部方法”（_policy_for_datetime）。
问题本质：报表“数据生成”和“导出渲染”未分层，且跨模块依赖不稳定。
次级（可后置，但建议列入清理清单）
core/services/scheduler/calendar_service.py：CRUD + 供算法调用的时间推进逻辑同居，且外部在用私有方法。
core/services/scheduler/gantt_service.py：服务层夹带展示/样式拼装（如优先级 class、display 文案），更像 viewmodel。
web/routes/scheduler_utils.py：上传解析/标准化属于通用能力，但放在 routes 下，易被误当“路由层逻辑”。
拆分原则（保证“对外行为不变”）
保留原入口：先把逻辑“搬出去”，原文件/原函数只做薄封装转调；URL、服务方法签名先不动。
把 SQL/查询集中：直写 SQL 要么下沉到 data/repositories/*，要么集中到 core/services/*/queries.py，避免“服务/路由各写一段 SQL”。
算法层只吃纯数据：传入 dataclass/dict 与配置快照，不要传 ConfigService/conn 这类“带 IO 的对象”。
