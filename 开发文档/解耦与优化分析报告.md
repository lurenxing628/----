## 1. 总体评价

项目整体采用了 **四层架构**（Web → Service → Repository → Infrastructure），有明确的架构决策记录（11 个 ADR），有 **13 个架构适应性测试** 保证分层纪律。在同类单用户桌面排产系统中，这已经是相当规范的做法。

**当前架构优势**：
- ✅ 分层清晰，有 fitness test 强制保证路由层不直接访问仓储
- ✅ 无循环依赖（有测试守护）
- ✅ 插件系统设计优秀（隔离性好、可观测性强）
- ✅ Excel 导入三层拆分（通用 → 领域 → 执行器）结构优秀
- ✅ 全局可变状态均有线程锁保护
- ✅ 无 mutable default、无 bare except、无 wildcard import

**主要需要改进的方向**：
- 🔴 调度核心模块存在 **上帝类** 和 **超长方法**（最长 665 行）
- 🟠 Web 路由层存在 **大量重复代码**（59 个路由文件，10302 行）
- 🟡 数据访问层存在 **查询模式重复**
- 🟡 领域模型 **过于贫血**（22 个模型全是纯数据容器）
- 🔵 app.py 与 app_new_ui.py **99% 重复**

## 2. P0 - 核心调度服务（最高优先级）

### 2.1 schedule_service.py — 上帝类（God Class）

**文件**: `core/services/scheduler/schedule_service.py` (617 行)

**问题 1: 依赖 10 个仓储对象**

```python

self.op_repo = BatchOperationRepository(conn, logger=logger)

self.supplier_repo = SupplierRepository(conn, logger=logger)

self.history_repo = ScheduleHistoryRepository(conn, logger=logger)
```

- **影响**: 修改任一仓储都可能影响本类；测试时需要 mock 10 个对象
- **建议**: 将职责拆分——例如把"资源池构建"、"结果持久化"、"输入构建"各自封装为独立服务，ScheduleService 只做流程编排

**问题 2: 单方法 323 行（`_run_schedule_impl`，第 296-617 行）**

此方法包含了：验证 → 资源分配 → 算法配置 → 持久化 → 摘要构建，一个方法做了 5 件事。

- **建议**: 拆分为 `_validate_input()`, `_build_resource_pool()`, `_execute_algorithm()`, `_persist_results()`, `_build_summary()` 等子方法

**问题 3: 延迟导入暴露循环依赖隐患**


- 虽然没有产生真正的循环导入，但这种模式说明模块之间的边界还不够清晰

### 3.4 错误处理模式不一致

59 个路由文件中共有 147 个异常处理块，但处理方式各异：

| 模式 | 数量 | 问题 |
|------|------|------|
| `except ValidationError` → `flash(e.message, "warning")` | 部分文件 | OK |
| `except BusinessError` → `flash(e.message, "warning")` | 部分文件 | OK |
| `except AppError` → `flash(e.message, "error")` | 部分文件 | 与上面不一致 |
| `except Exception:` → 通用消息 | 部分文件 | 用户体验差 |
| `except Exception: pass` | 少数文件 | 静默吞异常 |

**建议**: 引入统一的错误适配器：

```python
class ErrorAdapter:
    @staticmethod
    def to_flash(error: Exception) -> Tuple[str, str]:
        if isinstance(error, ValidationError):
            return error.message, "warning"
        if isinstance(error, AppError):
            return error.message, "error"
        return "系统错误，请稍后重试。", "error"
```

## 5. P3 - 领域模型贫血问题

### 5.1 所有 22 个模型都是纯数据容器

```python
# 典型模型结构（batch_operation.py 第 29-63 行）
@dataclass
class BatchOperation:
    id: Optional[int]
    op_code: str
    batch_id: str
    # ... 10+ 字段

    @classmethod
    def from_row(cls, row) -> "BatchOperation": ...  # 机械映射

    def to_dict(self) -> Dict[str, Any]: ...          # 机械序列化

    # ❌ 没有任何业务方法：validate(), is_overdue(), can_reschedule() 等
```

**影响**:
- 验证逻辑分散在 Service 层各处，无单一可信来源
- 模型无法自我保护不变量
- 丰富领域模型（DDD）模式无法实现

**建议**（可选，非必须）: 对核心模型（Batch, BatchOperation, Schedule）添加关键业务方法：

```python
@dataclass
class Batch:
    # ... 字段 ...

    def is_overdue(self, now: date) -> bool:
        return self.due_date is not None and now > self.due_date

    def validate(self) -> List[str]:
        errors = []
        if not self.batch_id:
            errors.append("批次号不能为空")
        if self.quantity is not None and self.quantity <= 0:
            errors.append("数量必须大于0")
        return errors
```

## 6. P4 - 配置与基础设施改进

### 6.1 错误翻译依赖字符串匹配

**文件**: `data/repositories/base_repo.py` 第 78-113 行

```python
if "UNIQUE constraint failed" in msg:       # ← SQLite 错误消息可能不稳定
    return AppError(ErrorCode.DUPLICATE_ENTRY, ...)
if "FOREIGN KEY constraint failed" in msg:  # ← 跨版本不保证一致
    return AppError(ErrorCode.DB_INTEGRITY_ERROR, ...)
```

**建议**: 可改为匹配 SQLite 错误码（`sqlite3.IntegrityError.args`），更稳定

---

### 6.2 OperationLogger 了解事务语义

**文件**: `core/infrastructure/logging.py` 第 144-191 行

日志组件直接检查是否在事务上下文中（`in_transaction_context()`），自行决定是否 commit/rollback——日志不应该了解事务管理的细节。

**建议**: 让调用方的事务管理器统一处理日志的提交

---

### 6.3 数据库模块职责混合

**文件**: `core/infrastructure/database.py` (200+ 行)

同时负责：连接工厂 + 模式迁移 + Windows 文件锁恢复 + 正则校验。

**建议**: 拆分为 `connection_factory.py`, `migration_runner.py`, `recovery.py`

---

### 6.4 UI 模式系统复杂度偏高

**文件**: `web/ui_mode.py` (370 行)

存在 3 条渲染路径：V1（基础环境）、V2（覆盖环境）、V2 降级回退。每个路由文件都需要导入 `render_ui_template` 替代 Flask 原生 `render_template`。

**建议**: 如果 V2 长期并存，考虑用策略模式简化；如果 V2 是过渡方案，建议尽早收敛到单一 UI
