## 拆分高耦合文件：P1/P2 实施策划（服务层拆分与 Excel 工具收敛）

### 背景

本仓库在 P0 已完成路由层的“入口聚合 + 子模块注册”拆分（`personnel/equipment/process/system`），在**保持 `bp/url/endpoint` 不变**的前提下，显著降低了单文件复杂度。

本文件用于把下一阶段（P1/P2）“怎么拆 / 先拆什么 / 如何迁移 / 回归点是什么”固化到仓库，方便后续直接按图施工与验收留痕。

---

### 总体目标与硬约束

- **目标**
  - **P1**：把服务层内部（`CalendarService` / `GanttService`）按职责拆开，降低耦合与修改风险，提升可测试性。
  - **P2**：把路由层重复的 Excel 工具函数收敛到单一模块，消除复制粘贴型耦合。

- **硬约束（必须满足）**
  - **对外 API 不变**：`CalendarService` / `GanttService` 的公开方法签名与语义不变（调用方无需改）。
  - **入口文件稳定**：外部 import 路径不变（例如 `from core.services.scheduler import CalendarService` 仍可用）。
  - **行为优先**：第一轮只做“搬迁 + 结构化”，不引入新业务规则；若必须修复 bug，需最小化并补回归用例。

## P1：服务层内部拆分

### 1) `CalendarService`（建议：按 `admin / engine` 拆）

#### 现状问题（典型高耦合信号）

- 同一文件内混杂：
  - **管理/CRUD**：WorkCalendar、OperatorCalendar 的增删改查、导入/覆盖语义。
  - **排产引擎能力**：工作时间推进、效率读取、班次窗口计算、优先级差异（普通/急件）。
  - **配置读取**：如假期默认效率等。

#### 建议文件结构

- `core/services/scheduler/calendar_service.py`
  - 保留 `CalendarService` 作为**对外 façade**（兼容层）
  - 仅保留：公开方法定义 + 少量编排/注释 + 向子模块委托
- `core/services/scheduler/calendar_admin.py`
  - WorkCalendar/OperatorCalendar 的 CRUD、列表、导入落库（含 tx/no_tx 的边界）
  - 日历记录的标准化：`day_type` 统一口径、日期/时间字符串规范化等
- `core/services/scheduler/calendar_engine.py`
  - `adjust_to_working_time` / `add_working_hours` / `get_efficiency` 等“时间推进/可用性判定”
  - `DayPolicy` / “优先级允许”/ “班次窗口”相关逻辑（若存在）

#### 拆分边界（落地判据）

- **凡是会直接写库/读库的 CRUD** → `calendar_admin.py`
- **凡是给排产算法用的“工作时间计算/推进”** → `calendar_engine.py`
- `calendar_service.py` 不再承载大段实现，避免再次膨胀

#### 迁移步骤（推荐顺序）

1. **新增子模块文件**：先创建 `calendar_admin.py` / `calendar_engine.py`，不动现有逻辑。
2. **搬迁实现**：
   - 先搬迁“纯函数/工具函数”（更容易验证）。
   - 再搬迁与 DB 相关的 CRUD（注意 tx/no_tx 语义保持一致）。
3. **在 `calendar_service.py` 中改为委托**：
   - 保持 `CalendarService.<method>` 的签名与异常行为一致。
   - 内部只调用 `calendar_admin.*` / `calendar_engine.*`。
4. **兼容 import**：确保 `core/services/scheduler/__init__.py` 导出未变（若有）。

#### 最小回归点（建议）

- **脚本/自动化**
  - `python tests/smoke_phase0_phase1.py`
  - `python tests/run_complex_case_and_export_gantt.py`（涉及时间推进/效率）
- **页面冒烟**
  - `排产调度 -> 工作日历配置`
  - `人员管理 -> 人员详情 -> 个人工作日历`
  - 若启用人员专属日历：验证“个人覆盖全局”的表现符合预期（至少不报错）

### 2) `GanttService`（建议：按 `range / tasks / critical_chain / week_plan` 拆）

#### 现状问题（典型高耦合信号）

同一文件内混杂：
- 时间范围解析（周起止、字符串解析/纠错）
- 甘特 tasks 构建（Schedule join、字段拼装、颜色/分组等）
- 关键链识别（依赖关系、主路径推导）
- 周计划导出（按天拆分、字段映射、Excel 输出）

#### 建议文件结构

- `core/services/scheduler/gantt_service.py`
  - 保留 `GanttService` façade（对外 API 不变）
- `core/services/scheduler/gantt_range.py`
  - `WeekRange` / 周范围计算、解析与标准化
- `core/services/scheduler/gantt_tasks.py`
  - tasks 构建与数据准备（对 repo 层的调用集中在此或由 service 编排）
- `core/services/scheduler/gantt_critical_chain.py`
  - 关键链识别算法（尽量纯函数化，便于单测）
- `core/services/scheduler/gantt_week_plan.py`
  - 周计划拆分/格式化/导出（Excel 输出/字段映射）

#### 迁移步骤（推荐顺序）

1. **先拆 range（低风险）**：把周范围解析/计算从 `gantt_service.py` 抽到 `gantt_range.py`。
2. **再拆 tasks（中风险）**：把“Schedule 数据读取 + tasks dict 拼装”抽到 `gantt_tasks.py`。
3. **再拆 critical_chain（中风险）**：保持输入/输出结构不变，集中到 `gantt_critical_chain.py`。
4. **最后拆 week_plan（中风险）**：把周计划拆分与导出抽到 `gantt_week_plan.py`。
5. **`GanttService` 只做编排**：方法签名不变、返回结构不变。

#### 最小回归点（建议）

- 页面：
  - `排产调度 -> 甘特图`（至少能打开、能切版本、能渲染 tasks）
  - `排产调度 -> 周计划`（能查询、能导出）
- 脚本：
  - `python tests/run_complex_case_and_export_gantt.py`

## P2：Excel 路由工具收敛

### 现状（横向重复点）

目前多个路由模块都存在重复工具函数（名称可能不同但语义类似）：
- `_parse_mode`（字符串 -> `ImportMode`）
- `_read_uploaded_xlsx`（上传文件 -> rows）
- `_ensure_unique_ids`（Excel 主键去重）

这类复制粘贴型耦合会导致：
- 修一处 bug 但其它模块漏修
- Excel 行为不一致（同样的输入在不同页面报错不同）

### 建议落点

新增统一模块（推荐放在路由层，避免服务层引入 Flask 依赖）：
- `web/routes/excel_utils.py`

建议函数清单（第一期最小集）：
- `parse_import_mode(value: str) -> ImportMode`
- `read_uploaded_xlsx(file_storage) -> List[Dict[str, Any]]`
- `ensure_unique_ids(rows, id_column: str) -> None`

### 迁移步骤（推荐顺序）

1. **先新增 `excel_utils.py`**：实现与现有逻辑保持一致（包括异常类型与错误文案）。
2. **小步替换**（按模块逐个替换，避免大爆炸）：
   - `scheduler_*` → `personnel_*` → `equipment_*` → `process_*` → `excel_demo`
3. **回归**：每替换 1 个模块，跑一遍对应页面的 Excel 预览/确认流程（至少走到预览）。
4. **清理冗余**：确认无引用后再删除各模块内的旧函数，避免双实现长期并存。

### 风险点与规避

- **循环依赖**：`excel_utils.py` 不应 import 具体 bp 模块；应保持纯工具属性。
- **异常类型**：尽量继续抛 `ValidationError/AppError`（与现有错误处理一致）。
- **编码/临时文件**：统一临时文件生命周期管理，确保 Windows 下文件句柄及时关闭。

## P3：算法层与迁移模块拆分（路线图/下一阶段）

> 说明：P3 不要求与 P1/P2 同步实施；本节仅用于把后续“高耦合热点”的拆分方向固化到仓库，便于后续直接按图施工与验收留痕。

### 1) 贪心算法 `core/algorithms/greedy/scheduler.py`

#### 现状问题（典型高耦合信号）

- 同一文件内混杂：
  - dispatch_mode（`batch_order/sgs`）两套流程
  - 派工规则解析与评分（DispatchRule / build_dispatch_key / scoring）
  - 资源占用/时间推进/停机避让（timeline）
  - seed_results 兼容、冻结窗口、自动选人/选机等多处分支

#### 建议拆分结构（保持对外入口稳定）

- `core/algorithms/greedy/scheduler.py`
  - 保留 `GreedyScheduler` 作为对外 façade（`schedule()` 签名与返回结构不变）
  - 仅负责：参数解析、组装输入、调用子模块、汇总 warnings/errors
- `core/algorithms/greedy/dispatch/batch_order.py`
  - 传统 batch_order 流程（按批次顺序全局排序工序）
- `core/algorithms/greedy/dispatch/sgs.py`
  - Serial SGS（eligible set）流程
- `core/algorithms/greedy/timeline.py`
  - 资源占用/冲突检测/空闲窗口搜索（machine/operator）
- `core/algorithms/greedy/scoring.py`
  - dispatch key/评分相关（slack/cr/atc 等规则）

#### 迁移顺序（推荐）

1. 先抽 `timeline.py`（可复用，且能显著降低主流程复杂度）。
2. 再把两套 dispatch 流程拆到 `dispatch/batch_order.py` 与 `dispatch/sgs.py`（减少条件分支交织）。
3. 最后抽 `scoring.py`（与 dispatch_rule 强相关，抽出后更便于单测/调参）。

### 2) DB 迁移模块 `core/infrastructure/database.py`

#### 现状问题

- `CURRENT_SCHEMA_VERSION` 增长会导致 `_run_migration()` 与各版本迁移函数持续膨胀。
- 迁移逻辑与 schema 初始化/备份/回滚等编排逻辑同文件耦合。

#### 建议拆分结构

- `core/infrastructure/database.py`
  - 保留对外入口：`ensure_schema/get_connection` 等（调用方不改）
  - `_run_migration()` 改为“分发器”，只负责按版本调用对应模块
- `core/infrastructure/migrations/`
  - `v1.py` / `v2.py` / `v3.py` …：每个版本一个模块，暴露 `run(conn, logger=None)`
  - （可选）`__init__.py` 统一注册/导出

#### 风险点与规避

- **幂等与可重入**：迁移函数需保持可重复执行；失败时由现有“迁移前强制备份 + 回滚”兜底。
- **导入路径**：迁移模块不得反向 import 业务层，避免循环依赖。
- **Windows 文件句柄**：迁移失败回滚时要确保连接关闭（避免占用 db 文件）。

#### 最小回归点（建议）

- `python tests/smoke_phase0_phase1.py`（schema 初始化 + 迁移备份）
- `python tests/regression_migrate_backup_dir_none_creates_backup.py`
- 涉及算法再补跑：`python tests/run_complex_case_and_export_gantt.py`
