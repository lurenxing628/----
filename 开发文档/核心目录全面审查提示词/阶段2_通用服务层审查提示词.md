# 阶段2：通用服务层审查提示词

> 适用范围：`core/services/common/`
>
> 使用方式：将下方整段提示词直接发给审查模型；如果阶段1已经有共性发现，可作为前置背景一并附上。
>
> 本版已补充：三轮审查、脚本测试、引用链深挖、函数与变量级分析、根因定位、静默回退与过度防御专项要求。

## 提示词正文

```text
你现在是资深后端架构审查专家，请对 APS 项目的阶段2范围执行一次只审不改的深度 review。

【审查范围】
- core/services/common/

【本阶段目标】
把通用服务层当作整个系统的数据入口与清洗底座来审查，重点识别以下问题：
1. BUG 与逻辑缺陷：导入链路断裂、校验失真、行号错位、严格模式失效、预览与确认语义不一致
2. 架构问题：通用层承载业务特例、职责分散、同类规则重复实现
3. 错误处理：导入失败是否可定位到文件、工作表、行、列、字段
4. 性能隐患：大文件一次性读入、重复扫描、重复构造中间结构、无界缓存
5. 安全风险：日志泄露敏感数据、单元格公式内容误用、文件名与路径处理不当
6. 可维护性：模板定义分散、别名规则难以追踪、归一化函数隐式改值
7. 测试覆盖：多格式输入、脏数据、边界值、异常路径是否被覆盖
8. 防御性代码风险：兼容模式失控、默认值静默改义、过度清洗、过度兜底

【三轮审查机制】
本阶段必须按三轮执行，不能只做文本推理：

第一轮：静态结构与引用链审查
1. 先梳理导入主链路：文件读取 → 模板匹配 → 行解析 → 字段归一化 → 校验 → 预览 → 确认 → 写入 → 审计。
2. 对关键函数做引用链追踪，至少追到“路由或服务入口 → 归一化函数 → 校验函数 → 执行器 → 落库或返回结果”。
3. 对高风险点必须下钻到函数参数、局部变量、中间结构、错误列表、警告列表、状态字段级别。
4. 特别标出 `safe_*`、`normalize_*`、`validator_*`、`template_*`、`or 默认值`、宽泛捕获与空集合兜底。

第二轮：脚本测试与证据验证
1. 必须结合现有测试、定向脚本或最小复现脚本验证关键结论。
2. 高风险问题尽量用“现有测试复跑”或“最小输入文件脚本”验证，不允许只凭猜测下结论。
3. 若没有现成测试，请给出最小复现输入，例如表头组合、空白行、大小写混合值、非法日期、公式单元格等。
4. 记录执行证据：测试或脚本名称、输入形态、预期行为、实际结果、是否复现。

第三轮：根因归并与修复定位
1. 对每个已确认问题说明根因落在模板定义、后端读取、归一化、校验、执行器、审计还是日志层。
2. 区分“根因”“传播路径”“表面现象”“被兼容分支掩盖后的假象”。
3. 修复建议必须指出应收口到哪一层，避免继续把规则散落到多处。

【新增专项：静默回退与过度防御】
本阶段必须单独检查以下模式：
1. 把非法值偷偷改成默认值、空值、0、空字符串、合法枚举。
2. 严格模式失败后，悄悄改走兼容模式或宽松路径。
3. 行级错误、字段级错误被吞掉，最后只剩下模糊提示。
4. 为了“尽量导入成功”而过度防御，结果把数据质量问题带入后续链路。
5. 预览、确认、正式导入三阶段口径不一致，造成伪成功或伪失败。

【执行原则】
1. 先梳理导入主链路，再分层检查校验、模板、后端适配、归一化工具，不要把这些文件割裂开看。
2. 所有发现必须给出证据：文件路径 + 行号 + 输入场景 + 错误后果。
3. 任何把非法值偷偷改成默认值、空值、合法值的逻辑，都要重点关注。
4. 如果某问题只在特定表头、大小写、空白行或混合类型下触发，要把触发条件写清楚。
5. 如需追到调用方验证语义，可以少量查看上游，但本阶段以通用服务层为主。
6. 对关键变量要回答：原始值是什么、归一化后变成什么、在哪一步被兜底、最终如何影响落库或返回。
7. 发现“为了避免报错而继续执行”的路径时，要重点分析它是否导致 BUG 被隐藏。

【建议审查顺序】
第一批：`excel_service.py`、`excel_import_executor.py`、`excel_validators.py`、`excel_templates.py`
第二批：`openpyxl_backend.py`、`pandas_backend.py`、`tabular_backend.py`
第三批：`enum_normalizers.py`、`normalize.py`、`datetime_normalize.py`、`number_utils.py`
第四批：`excel_audit.py`、`safe_logging.py` 以及其余辅助文件

【脚本测试优先项】
优先搜索并运行与本阶段直接相关的现有测试；例如但不限于：
- `tests/regression_excel_import_executor_status_gate.py`
- `tests/regression_excel_import_result_semantics.py`
- `tests/regression_excel_failure_semantics_contracts.py`
- `tests/regression_excel_validators_yesno_mixed_case.py`
- `tests/regression_excel_normalizers_mixed_case.py`
- `tests/regression_excel_preview_confirm_baseline_guard.py`
- `tests/regression_excel_preview_confirm_extra_state_guard.py`
- `tests/regression_batch_excel_import_strict_mode_hardfail_atomic.py`
- `tests/test_excel_import_hardening.py`
- `tests/test_excel_normalizers_contract.py`
如现有测试不足，请补最小复现脚本验证表头别名、混合大小写、空白行、公式单元格、严格与兼容模式切换、部分失败回滚等场景。

【必须重点检查的内容】

一、导入主链路
1. `excel_service.py` 与 `excel_import_executor.py`
   - 预览、确认、正式写入三个阶段的状态语义是否一致
   - 原子性是否可靠，部分成功是否会污染数据库或缓存状态
   - 严格模式与兼容模式切换时，错误是否被正确保留和上抛
   - 行级错误、表级错误、文件级错误是否能区分并准确回传
   - 失败后是否正确回滚，是否存在已落库但界面显示失败的风险
   - 是否存在宽泛兜底让失败导入看起来只是“部分警告”的路径

二、校验与模板
1. `excel_validators.py`
   - 必填、唯一、枚举、日期、数值、引用存在性校验是否完整
   - 空字符串、全空白、大小写混合、是或否变体、数值字符串等边界是否处理一致
   - 校验失败消息是否可直接指导修复，而不是笼统报错
   - 多字段联动规则是否存在漏检或错判
   - 是否存在校验失败后被下游再次兜底为合法值的路径

2. `excel_templates.py`
   - 模板字段定义、别名映射、表头顺序依赖是否可靠
   - 同一字段在不同模板中的命名与语义是否一致
   - 模板变更后是否容易造成历史文件静默兼容失败
   - 是否存在模板定义与实际导入逻辑分离，导致改一处漏一处的问题
   - 模板匹配失败时是否错误回退到近似模板

三、后端适配层
1. `openpyxl_backend.py`、`pandas_backend.py`、`tabular_backend.py`
   - 不同后端对空值、日期、数值、公式单元格的解释是否一致
   - 后端切换时是否会改变错误语义或字段类型
   - 大文件读取是否存在整表一次性进内存的问题
   - 异常处理是否完整，是否把底层错误包装成模糊提示
   - 读取失败时是否悄悄回退到其他后端并改变行为

四、归一化与通用工具
1. `enum_normalizers.py`
   - 大小写、别名、历史兼容值是否统一收敛到同一结果
   - 无法识别的值是否会被静默降级成默认枚举
   - 是否存在“为了兼容历史数据”而无限扩张映射表的风险

2. `normalize.py`、`datetime_normalize.py`、`number_utils.py`
   - 文本清洗是否误删合法内容
   - 日期解析是否会受本地格式、时分秒、省略年份等情况影响
   - 数值解析失败时是否错误返回 0、空字符串或其他默认值
   - 边界值如负数、极大值、非有限数值、科学计数法是否被妥善处理
   - 是否存在过度防御式解析，让上层永远看不到真实错误输入

五、审计与安全日志
1. `excel_audit.py`
   - 审计信息是否完整记录关键输入、操作者、结果、失败原因
   - 审计记录是否可能与真实落库结果不一致
   - 失败时是否只记摘要而不保留足够复盘信息

2. `safe_logging.py`
   - 敏感字段是否脱敏
   - 超长内容是否截断且保留定位信息
   - 是否存在因为过度脱敏导致排障信息不足的问题
   - 是否存在日志包装把高价值错误细节全部抹平的情况

【产出要求】
请输出一份结构化 Markdown 审查报告，严格包含以下部分：
```
