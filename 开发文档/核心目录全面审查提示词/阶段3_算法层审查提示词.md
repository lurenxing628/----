# 阶段3：算法层审查提示词

> 适用范围：`core/algorithms/`
>
> 使用方式：将下方整段提示词直接发给审查模型；建议把阶段2中与数据清洗、枚举归一化相关的发现作为背景附上，以便识别输入前提与算法假设的断裂点。
>
> 本版已补充：三轮审查、脚本测试、引用链深挖、函数与变量级分析、根因定位、静默回退与过度防御专项要求。

## 提示词正文

```text
你现在是资深调度算法与后端架构审查专家，请对 APS 项目的阶段3范围执行一次只审不改的深度 review。

【审查范围】
- core/algorithms/
- 重点包含 core/algorithms/greedy/

【本阶段目标】
验证算法层在正确性、稳定性、边界处理、可解释性方面是否可靠，特别关注以下问题：
1. BUG 与逻辑缺陷：排产顺序错误、资源分配错误、违反约束、重复排产、漏排、死循环、去重失效
2. 架构问题：规则散落、同类判断重复、算法前提分散在多个文件里
3. 错误处理：不可排场景、异常输入、非有限数值、缺失资源时是否有可观测结果
4. 性能隐患：高复杂度循环、重复排序、重复扫描候选集、关键链路无剪枝
5. 安全与鲁棒性：异常输入导致崩溃、时间解析异常、外协与停机数据错配
6. 可维护性：策略命名混乱、打分公式分散、隐式默认权重过多
7. 测试覆盖：排序规则、资源池为空、停机冲突、跨天、冻结、非有限数值等边界是否充分覆盖
8. 防御性代码风险：不可排场景被静默打低分、非法值被默认权重吸收、失败结果被包装成可用结果

【三轮审查机制】
本阶段必须按三轮执行，不能只看代码表面：

第一轮：静态结构与引用链审查
1. 先建立算法总图：输入对象、约束、排序、分派、评分、评估、输出。
2. 对关键函数做引用链追踪，至少追到“输入构建来源 → 关键参数 → 中间得分变量 → 最终排产结果或评估结果”。
3. 对高风险点必须深入到函数参数、局部变量、权重、优先级、时间变量、状态变量级别。
4. 特别标记：默认权重、无穷值处理、空候选集处理、评分降级、兜底排序、不可排惩罚、兼容旧值映射。

第二轮：脚本测试与证据验证
1. 必须结合现有回归测试、最小排产数据脚本、定向验证脚本确认关键结论。
2. 对“静默产出错误排产结果”的怀疑，优先通过脚本或测试验证，而不是只做抽象推理。
3. 尽量给出最小触发场景：最少批次、最少资源、最少约束、最少停机记录。
4. 记录测试或脚本名称、输入、预期排产行为、实际行为、是否复现。

第三轮：根因归并与修复定位
1. 对已确认问题回溯根因，明确问题源于输入假设、排序规则、评分函数、资源过滤、时间推进、配置映射还是结果评估。
2. 区分“根因”“传播路径”“错误排产表现”“被兜底掩盖后的假象”。
3. 修复建议必须指出落点：某个规则函数、某个打分变量、某个排序策略、某个时间解析路径，而不是泛泛而谈。

【新增专项：静默回退与过度防御】
本阶段必须单独盘点以下模式：
1. 输入非法或缺失时，被悄悄转成默认权重、默认时间、默认排序策略。
2. 资源不可用、候选集为空、评分不可算时，被兜底成“低优先级但继续排”。
3. 异常被吞掉后继续产出排产结果，但结果质量已失真。
4. 非有限数值、坏时间戳、坏交期被安全解析后，实际改变了排产语义。
5. 为了“算法尽量可运行”而加入的过度防御逻辑，是否掩盖了真正应被抛出的 BUG。

【执行原则】
1. 先建立算法总图，再进入主循环和规则函数，不要从细枝末节开始。
2. 优先审查调度主循环和规则函数，再看辅助模块。
3. 对任何可疑逻辑，都尽量给出最小触发场景，说明为什么会得出错误结果。
4. 如果算法行为依赖默认值、兜底值、兼容分支，必须检查这些分支是否改变业务语义。
5. 不要只找会抛异常的问题；静默产生错误排产结果的问题优先级更高。
6. 对关键变量要回答：初始值是什么、被谁改写、何时被降级或兜底、最终影响哪个排序或评分结果。
7. 如果某条结论没有脚本或测试验证，只能标记为“高怀疑待验证”，不能直接下结论。

【建议审查顺序】
第一批：`types.py`、`value_domains.py`、`priority_constants.py`、`dispatch_rules.py`、`sort_strategies.py`
第二批：`greedy/scheduler.py`、`greedy/dispatch/sgs.py`、`greedy/dispatch/batch_order.py`
第三批：`greedy/auto_assign.py`、`greedy/seed.py`、`greedy/date_parsers.py`、`greedy/schedule_params.py`、`greedy/config_adapter.py`
第四批：`greedy/downtime.py`、`greedy/external_groups.py`、`evaluation.py`、`ortools_bottleneck.py`

【脚本测试优先项】
优先搜索并运行与本阶段直接相关的现有测试；例如但不限于：
- `tests/regression_greedy_date_parsers.py`
- `tests/regression_greedy_scheduler_algo_stats_auto_assign.py`
- `tests/regression_greedy_scheduler_algo_stats_seed_counts.py`
- `tests/regression_auto_assign_empty_resource_pool.py`
- `tests/regression_sgs_penalize_nonfinite_proc_hours.py`
- `tests/regression_sgs_scoring_fallback_unscorable.py`
- `tests/regression_dispatch_rules_nonfinite_proc_hours_safe.py`
- `tests/regression_sort_strategy_case_insensitive.py`
- `tests/regression_ortools_budget_guard_skip_when_no_time.py`
- `tests/regression_seed_results_dedup.py`
- `tests/test_greedy_scheduler_base_date.py`
- `tests/test_value_domains_consistency.py`
如现有测试不足，请补最小复现脚本验证空资源池、坏时间、非有限数值、冻结约束、停机冲突、外协默认天数等场景。

【必须重点检查的内容】

一、规则与基础类型
1. `dispatch_rules.py`、`sort_strategies.py`
   - 相同优先级下是否有稳定且可解释的次序规则
   - 大小写、空值、未知策略名时是否会静默落到默认策略
   - 规则之间是否存在重复实现或彼此矛盾
   - 默认策略是否让真实配置错误无法显露

2. `types.py`、`value_domains.py`、`priority_constants.py`
   - 类型与常量定义是否与实际算法使用保持一致
   - 权重、优先级、枚举取值是否存在隐式假设
   - 默认常量是否可能改变排产结果却没有被显式记录
   - 是否存在安全常量或默认域把坏输入直接吸收掉

二、贪心调度核心
1. `greedy/scheduler.py`
   - 主循环是否保证终止
   - 每轮状态推进是否正确，是否可能重复消费同一工序或跳过工序
   - 资源不可用、约束冲突、空候选集时是否有正确分支
   - 冻结工单、就绪条件、批次顺序等约束是否真正生效
   - 不可排工序是否被错误地留在结果中却没有暴露异常

2. `greedy/dispatch/sgs.py`
   - 串行生成方案实现是否满足资源占用、时间推进、约束检查的基本语义
   - 设备、人员、外协之间的选择是否存在错配或漏判
   - 延期惩罚、不可排惩罚、缺失资源惩罚是否一致且可解释
   - 无法评分时是否被过度兜底为可评分结果

3. `greedy/dispatch/batch_order.py`
   - 批次排序是否可重复、稳定、符合业务优先级
   - 日期、优先级、交期为空或异常时是否会扭曲排序结果
   - 是否存在为了保持可排序而静默丢弃异常信息的路径

三、辅助算法模块
1. `greedy/auto_assign.py`
   - 空资源池、部分资源池、资源技能不匹配时是否正确处理
   - 自动分配是否会越权选择不允许的设备或人员
   - 找不到可行资源时是否会错误返回伪成功结果
   - 是否存在过度兼容导致错误资源被错误纳入候选集

2. `greedy/seed.py`
   - 种子结果是否去重可靠
   - 历史结果、重复工序、坏时间戳是否会污染新结果
   - 无效历史是否被静默保留并影响新排产

3. `greedy/date_parsers.py`
   - 日期字符串解析是否鲁棒
   - 单数字月份日期、缺少年份、混合格式、非法值是否安全处理
   - 是否存在过度防御性解析把非法时间伪装成合法时间

4. `greedy/schedule_params.py`、`greedy/config_adapter.py`
   - 配置项映射是否完整
   - 默认值、空值、非法值处理是否改变算法语义
   - 配置异常是否被吞掉并回退为默认参数

5. `greedy/downtime.py`、`greedy/external_groups.py`
   - 停机时间裁剪、重叠、非法时间段是否正确跳过或报错
   - 外协工序时长、默认天数、资源可用性映射是否正确
   - 无效停机和外协数据是否被静默忽略并影响排产质量

四、评估与求解器集成
1. `evaluation.py`
   - 指标计算是否与排产目标一致
   - 逾期、利用率、加权指标、不可排统计是否准确
   - 非有限数值、空结果、部分结果时是否有安全处理
   - 安全处理是否掩盖了上游错误排产

2. `ortools_bottleneck.py`
   - 求解器不可用、超时、输入非法时是否优雅失败
   - 是否保留足够日志和警告，而不是静默退回其他路径
   - 求解结果与贪心结果切换时，统计口径是否一致
   - 是否存在异常后悄悄退回贪心且不暴露失败事实的情况
```
