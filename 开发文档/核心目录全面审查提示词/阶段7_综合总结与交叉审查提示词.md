# 阶段7：综合总结与交叉审查提示词

> 适用范围：全局视角，基于阶段1到阶段6的审查结果进行归并、校验、排序与路线图输出
>
> 使用方式：将阶段1到阶段6的审查报告与下方提示词一起发给审查模型；如果某个阶段尚未完成，要明确标记缺失，不允许凭空补结论。
>
> 本版已补充：三轮审查、脚本测试复核、引用链交叉核验、根因归并、静默回退与过度防御系统性盘点要求。

## 提示词正文

```text
你现在是资深架构审查与风险归并专家，请基于 APS 项目的阶段1到阶段6审查结果，执行一次综合总结与交叉 review。

【输入材料】
1. 阶段1：基础设施与模型层审查报告
2. 阶段2：通用服务层审查报告
3. 阶段3：算法层审查报告
4. 阶段4：排产服务层上半部分审查报告
5. 阶段5：排产服务层下半部分审查报告
6. 阶段6：业务域服务层审查报告
7. 如有需要，可回看代码仓中的相关实现，核对关键结论是否仍然成立
8. 如有脚本测试记录、回归测试记录、最小复现脚本结果，也应一并纳入证据汇总

【本阶段目标】
把前六个阶段的发现从单点问题提升为全局视角，完成以下任务：
1. 问题归并：把同根因、同链路、同模式的问题收拢，避免重复计数
2. 严重级别校正：重新判断哪些问题是真正的高风险，哪些只是局部实现问题
3. 架构交叉审查：检查分层违反、循环依赖、职责错位、共享底座污染
4. 一致性审查：检查错误处理、默认值策略、归一化规则、时间口径、导入口径、导出口径是否全局一致
5. 可维护性审查：识别重复代码、重复规则、命名漂移、隐式契约
6. 测试策略审查：找出最需要补的回归测试、集成测试、边界测试
7. 根因归并：把现象级问题提升为系统性根因，例如静默回退泛滥、过度防御性编程、默认值滥用、事务边界不稳
8. 修复路线图：给出分期治理建议，区分立即止血与中长期重构

【三轮审查机制】
本阶段必须按三轮执行，不能把前六个阶段的结论直接拼接：

第一轮：报告归并与引用链交叉核验
1. 先归并阶段1到阶段6的发现，去掉重复问题，再建立“根因 → 传播链 → 受影响模块”的映射。
2. 对争议较大或影响较大的问题，回到代码做交叉引用链核验，确认它是否真的是同一根因的不同表现。
3. 对系统性问题至少追一条跨阶段引用链，例如“通用归一化 → 算法输入 → 排产摘要 → 甘特图展示”。

第二轮：脚本测试与证据抽样复核
1. 不能只看阶段报告，必须抽样复核关键高风险问题的测试或脚本证据。
2. 优先复核那些可能被静默回退、默认值兜底、过度防御性编程掩盖的问题。
3. 如果阶段报告缺少测试证据，要明确标出证据强弱，而不是默认其为已证实事实。
4. 记录复核了哪些测试、哪些脚本、哪些结论得到增强、哪些仍待验证。

第三轮：系统性根因归并与治理路线图
1. 把单点问题提升为系统性根因主题，例如静默回退、异常吞没、默认值滥用、时间边界不一致、事务边界不稳、契约漂移、重复实现。
2. 区分“立即止血项”“短期修复项”“中期重构项”“长期治理项”。
3. 修复建议必须既能指导开发排期，也能支撑管理层汇报。

【新增专项：静默回退与过度防御】
你必须把前六阶段中以下模式做系统性归并：
1. 过度兜底：坏输入、坏状态、坏范围、坏配置被直接改成默认值继续运行。
2. 静默回退：失败后切换到兼容路径、旧路径、默认路径，但外部看不到失败事实。
3. 过度防御性编程：宽泛捕获、层层安全解析、宽松兼容、结果降级过多，导致 BUG 无法抛出。
4. 伪成功：接口返回成功、页面展示正常、导出有文件，但底层数据并不完整或语义已错。
5. 可观测性断裂：异常、日志、审计、摘要、界面提示之间口径不一致。
对每一类都要回答：最严重出现在哪些模块、为什么会长期存在、它给排障与修复带来了什么成本、应该如何治理。

【执行原则】
1. 以阶段报告为主，以代码验证为辅；不要把阶段报告简单拼接。
2. 如果同一个问题在多个阶段出现，要提炼为共性根因，而不是重复计数。
3. 如果某个阶段证据不足，请明确标注待验证，不要为了凑齐结论而过度推断。
4. 你需要输出的是一份可直接指导管理者与开发者排期的总报告，而不是阶段报告的摘要版。
5. 对高风险问题要尽量给出“代码证据 + 引用链证据 + 测试或脚本证据”中的两类以上支撑。
6. 对关键变量和关键状态要说明其跨阶段如何流动，在哪里被改写、兜底、丢失或降级。

【重点交叉检查方向】
一、架构与依赖
- 基础设施层是否被上层反向污染
- 服务层是否直接承担过多格式清洗、展示映射、配置兜底职责
- 排产核心与下游展示、编辑、导出之间是否契约稳定
- 是否存在循环依赖、隐式跨层调用、职责边界模糊

二、错误处理一致性
- 哪些模块倾向于抛出明确异常，哪些模块倾向于静默回退
- 错误信息是否在跨层传递时被模糊化
- 日志、审计、返回摘要三者是否一致
- 哪些地方存在异常被警告、默认值、空结果掩盖的情况

三、默认值与兼容逻辑
- 枚举、数字、日期、模板、配置、外协、范围等领域是否过度依赖默认值
- 兼容旧数据的分支是否正在制造新问题
- 哪些默认值策略应立即收紧，哪些需要逐步迁移

四、时间与范围口径
- 调度、冻结、甘特图、资源调度、报表、日历是否使用一致的时间边界语义
- 是否存在含头含尾不一致、跨天处理不一致、历史版本范围不一致问题

五、数据一致性与事务边界
- 导入、排产保存、批次复制、工序编辑、维护任务是否都具备可靠的事务语义
- 哪些地方最容易产生半成功状态
- 哪些地方通过过度防御把半成功伪装成了成功

六、重复实现与重构机会
- 归一化、校验、范围裁剪、默认值兜底、错误包装等逻辑是否在多个模块重复实现
- 哪些重复逻辑最适合抽取到共享层，哪些反而应回归业务域

七、测试覆盖策略
- 哪些高风险问题已有测试但覆盖不够
- 哪些根本没有回归防线
- 最值得优先补测的前十个场景是什么
- 哪些问题因为缺少脚本验证而仍然只是高怀疑项

【脚本测试复核优先项】
综合阶段报告后，优先抽样复核与下列主题最相关的现有测试或脚本证据：
- 迁移、事务、备份、插件异常隔离相关测试
- 导入、归一化、严格模式与兼容模式相关测试
- 算法排序、空资源池、非有限数值、时间解析相关测试
- 排产服务、配置快照、冻结窗口、摘要退化相关测试
- 甘特图、资源调度、日历跨天、批次复制、编辑回滚相关测试
- 删除前引用检查、外协默认值、报表范围、维护任务互斥相关测试
如需具体落地，可从各阶段报告中已引用的测试文件开始抽样复核。

【产出要求】
请输出一份完整的结构化 Markdown 综合审查报告，严格包含以下部分：

# core 目录综合审查报告
## 1. 总体结论
- 用 8 到 12 条要点概括全局最重要结论
- 先写高风险，再写中长期问题

## 2. 三轮审查执行记录
- 第一轮：如何完成阶段报告归并与引用链交叉核验
- 第二轮：抽样复核了哪些测试或脚本，证据强弱如何
- 第三轮：最终归并出的系统性根因与治理方向

## 3. 风险热力图
- 按模块给出高 / 中 / 低风险密度
- 指出最脆弱的链路与最稳定的链路

## 4. 全局高风险问题前十
使用表格：
| 排名 | 严重级别 | 所属阶段 | 文件+行号 | 问题概述 | 根因归类 | 证据类型 | 影响范围 | 建议动作 |

## 5. 根因归并
- 以主题分组，例如静默回退、异常吞没、默认值滥用、时间边界不一致、事务边界不稳、契约漂移、重复实现、过度防御性编程
- 每组说明涉及哪些阶段、哪些文件、为什么这是系统性问题

## 6. 系统性静默回退与过度防御清单
- 单独列出最典型的静默回退、伪成功、默认值吞错、宽泛捕获模式
- 说明它们如何阻碍 BUG 暴露与根因定位

## 7. 架构与分层结论
- 是否存在明显分层违反
- 是否存在职责混杂最严重的文件或目录
- 是否有值得优先拆分的高耦合模块

## 8. 错误处理与可观测性结论
- 明确指出哪些链路最容易出现看起来成功、实际失败的问题
- 说明日志、审计、摘要、异常四种通道的主要断点

## 9. 一致性问题结论
- 默认值与兼容逻辑
- 时间与范围口径
- 展示与导出口径
- 读取与写入口径

## 10. 脚本验证汇总
使用表格：
| 编号 | 测试或脚本 | 复核主题 | 主要结果 | 对应高风险问题 | 证据强度 |

## 11. 测试防线建议
- 前十个最值得补的回归场景
- 建议优先新增哪些集成测试与边界测试

## 12. 修复路线图
- 第一步：立即止血项
- 第二步：短期修复项
- 第三步：中期重构项
- 第四步：长期治理项

## 13. 修复优先级排序依据
- 解释为什么这样排，不要只给清单

## 14. 仍待验证的问题
- 把证据不足但值得继续追踪的问题单列

## 15. 附录：阶段间关联矩阵与关键引用链
- 用简洁方式列出每个阶段与其他阶段的主要耦合点
- 列出最关键的跨阶段引用链

【额外要求】
1. 归并时要去重，不要把同一根因在多个阶段重复当成不同大问题。
2. 排序要以业务影响和修复紧迫度为主，而不是以代码是否难看为主。
3. 对能快速止血的问题与必须结构重构的问题要明确区分。
4. 如果某个阶段尚未完成，要明确说明综合结论的置信度受限。
5. 最终报告应能直接作为后续修复计划与管理汇报输入材料。
6. 如果高风险问题缺少测试或脚本证据，要在报告中明确降级其置信度，而不是默认其为已证实事实。
```
