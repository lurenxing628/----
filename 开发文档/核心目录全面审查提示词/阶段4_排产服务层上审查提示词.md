# 阶段4：排产服务层（上）审查提示词

> 适用范围：`core/services/scheduler/` 中的调度核心子集
>
> 建议重点文件：`schedule_service.py`、`schedule_input_builder.py`、`schedule_persistence.py`、`schedule_summary.py`、`schedule_optimizer.py`、`schedule_optimizer_steps.py`、`schedule_template_lookup.py`、`schedule_history_query_service.py`、`config_service.py`、`config_snapshot.py`、`config_presets.py`、`config_validator.py`、`freeze_window.py`
>
> 使用方式：将下方整段提示词直接发给审查模型；建议附上阶段3的算法发现，以便核对服务层是否正确承接算法层假设。
>
> 本版已补充：三轮审查、脚本测试、引用链深挖、函数与变量级分析、根因定位、静默回退与过度防御专项要求。

## 提示词正文

```text
你现在是资深排产业务与后端架构审查专家，请对 APS 项目的阶段4范围执行一次只审不改的深度 review。

【审查范围】
- core/services/scheduler/ 中的调度核心子集
- 重点关注：schedule_service、输入构建、优化、持久化、摘要、配置、冻结窗口

【本阶段目标】
验证排产服务层上半部分是否把请求、配置、算法调用、结果保存、历史记录、摘要输出串成一条一致且可追踪的主链路。重点识别：
1. BUG 与逻辑缺陷：输入构建错误、状态切换错误、重复保存、空结果误成功、历史记录缺失、摘要失真
2. 架构问题：服务层职责混杂、配置逻辑散落、持久化与计算交织
3. 错误处理：算法失败、输入不完整、配置非法、持久化失败时是否有一致结果
4. 性能隐患：重复构建大对象、重复查询配置、重复汇总统计
5. 安全与数据一致性：事务边界错误、快照与实际运行配置不一致、冻结窗口误放行
6. 可维护性：配置来源不透明、模板查找规则隐式、摘要字段语义不统一
7. 测试覆盖：空可排集合、部分可排、优化失败、保存失败、配置非法、冻结边界等场景是否被覆盖
8. 防御性代码风险：异常被摘要掩盖、空结果被包装成成功、配置非法被默认值吞掉、冻结边界问题被宽松处理

【三轮审查机制】
本阶段必须按三轮执行，不能只做入口扫读：

第一轮：静态结构与引用链审查
1. 从排产入口开始，顺着主链路做引用链追踪：请求参数 → 输入构建 → 算法调用 → 优化 → 持久化 → 摘要 → 历史记录。
2. 对关键函数深入到参数、局部变量、状态字段、警告列表、错误列表、配置快照、动作类型级别。
3. 对高风险路径至少追到“上游输入点、当前处理点、下游持久化或返回点”。
4. 特别标记：`safe_*`、`summary_*`、`config_*`、`snapshot_*`、`fallback_*`、`or 默认值`、宽泛捕获、空结果短路。

第二轮：脚本测试与证据验证
1. 必须结合现有回归测试、定向服务测试或最小复现脚本验证关键结论。
2. 对“看起来排产成功但结果已不一致”的问题，必须尽量用脚本或测试给出证据。
3. 尽量覆盖空可排集合、全部冻结、配置非法、优化失败、持久化中断、摘要退化等场景。
4. 记录测试或脚本名称、前置数据、预期结果、实际结果、是否复现。

第三轮：根因归并与修复定位
1. 对已确认问题回溯根因，明确落在入口校验、输入构建、配置映射、优化编排、持久化、摘要、历史记录还是冻结判断。
2. 区分“根因”“传播路径”“对外表现”“被兜底掩盖后的假象”。
3. 修复建议必须指出修复收口点，避免在多个层次重复打补丁。

【新增专项：静默回退与过度防御】
本阶段必须单独盘点以下模式：
1. 输入缺失或配置非法后，被静默回退到默认配置继续排产。
2. 算法失败、优化失败、保存失败后，被包装成警告、摘要提示或空结果成功。
3. 冻结窗口、模板查找、历史版本选择采用宽松兼容，导致真实问题未暴露。
4. 为了“不影响主流程”而过度吞错，结果制造数据不一致或伪成功。
5. 日志、摘要、历史记录、返回值四者口径不一致，导致外部看不到真实失败。

【执行原则】
1. 从排产入口开始，顺着主链路往下看：参数进入 → 输入构建 → 算法或优化 → 持久化 → 摘要或历史。
2. 不要把各服务当独立工具函数看，必须验证它们串联后的最终语义。
3. 对所有可能导致看起来排产成功、实际结果不完整或不一致的路径提高警惕。
4. 如发现跨到路由层或仓储层才能确认的语义，可做少量外延验证，但主报告聚焦本阶段范围。
5. 对关键变量要回答：它从哪里来、如何被标准化、何时被默认值覆盖、最终影响哪个对外结果。
6. 如果没有脚本或测试证据，涉及高风险伪成功的结论必须标记为“高怀疑待验证”。

【建议审查顺序】

第二批：`schedule_optimizer.py`、`schedule_optimizer_steps.py`、`freeze_window.py`

【必须重点检查的内容】

一、排产入口与输入构建
1. `schedule_service.py`
   - 排产入口是否正确校验前置条件
   - 无可排对象、全部冻结、缺失资源、配置非法时是否给出正确结果
   - 算法失败、优化失败、保存失败三类异常是否被正确区分
   - 是否存在异常被吞掉后返回部分成功或伪成功的路径
   - 是否存在为了继续流程而把错误改写成警告的逻辑

2. `schedule_input_builder.py`
   - 输入数据是否完整、字段语义是否与算法层契约一致
   - 资源、日历、停机、外协、冻结信息是否全部纳入
   - 数值解析、默认值、空值处理是否会改变排产语义
   - 是否存在读侧归一化与写侧存储不一致的问题
   - 是否有安全解析把坏输入静默变成可排输入

二、优化与冻结
1. `schedule_optimizer.py`、`schedule_optimizer_steps.py`
   - 优化开关、预算、步骤编排是否正确
   - 某一步失败时是否会保留坏状态或错误统计
   - 优化前后结果口径是否一致
   - 统计信息是否真实反映优化过程，而不是只反映最终结果
   - 步骤失败后是否被静默跳过并继续产出结果

2. `freeze_window.py`
   - 冻结区间边界是否正确，是否存在含头不含尾或时分秒错位问题
   - 已冻结工序是否可能被误改动
   - 冻结与可重排、插单、继续排产等语义是否冲突
   - 是否通过宽松兜底掩盖了冻结边界错误

三、结果保存与历史
1. `schedule_persistence.py`
   - 持久化是否具有事务完整性
   - 部分写入失败时是否会残留半成品排产结果
   - 历史记录、当前结果、摘要之间是否一致
   - 版本号、批次号、动作类型等关键字段是否正确维护
   - 是否存在保存失败后仍返回成功摘要的伪成功路径

2. `schedule_history_query_service.py`
   - 历史读取是否可能遗漏、乱序、取错版本
   - 空历史、异常历史、旧版本历史是否有健壮处理
   - 异常历史是否被静默过滤，导致问题被掩盖

四、摘要与模板
1. `schedule_summary.py`
   - 逾期、不可排、警告、统计字段是否准确
   - 空结果、部分结果、大结果集时是否安全
   - 摘要是否丢失关键失败上下文，导致前端只能看到模糊信息
   - 为了摘要稳定而做的兜底是否掩盖了真实失败

2. `schedule_template_lookup.py`
   - 模板查找优先级、兼容逻辑、默认模板回退是否清晰
   - 查找失败时是否正确暴露问题，而不是默默落到错误模板
   - 是否存在过度兼容导致模板错误长期不暴露

五、配置链路
1. `config_service.py`、`config_snapshot.py`、`config_presets.py`、`config_validator.py`
   - 配置来源、覆盖顺序、默认值、快照保存是否一致
   - 非法配置是否会在早期被拒绝
   - 快照是否真实反映本次排产使用的配置
   - 预设应用后是否会遗漏字段、混入旧值、产生半更新状态
   - 是否存在配置非法时静默回退到默认值并继续运行的路径
```
