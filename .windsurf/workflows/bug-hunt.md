description: GPT 单模型 Bug 三轮深审：候选问题发现、证伪核实、根因链闭合、同根因扩搜与最终修复计划收敛
---
当用户提到“查 Bug / 排查 bug / 深挖根因 / 先别改先找问题 / 这个是不是 bug / 帮我把问题挖透 / 先出修复 plan”时，使用本 workflow。

默认按 `GPT-5.4 XHigh Thinking Fast` 的工作方式执行本 workflow；若用户当前会话使用其他模型，以当前会话模型为准，但仍严格遵守以下证据与止损规则。

## 核心硬规则

1. 任何“发现项”都不能直接当作缺陷结论，必须先判断它是：
   - `确认缺陷`
   - `高概率缺陷`
   - `证据不足`
   - `确认属于有意实现`
   - `需要用户确认业务意图`

2. 每一轮都必须先复核上一轮结论，不允许无条件继承。

3. 现有找 Bug 的 skills / workflows 输出只能作为：
   - 线索来源
   - 辅证来源
   - 优先级参考
   绝不能直接当作“确认缺陷”的实锤。

4. 多个 skills / workflows 同时提示同一问题，只能提高优先级，不能提高证据等级。

5. 最终修复计划只允许纳入：
   - 有源码级证据支持的项
   - 或有测试失败 / 运行异常 / 日志 / 契约冲突支持的项
   `证据不足` 与 `相似但未证实问题` 只能进入观察项或后续排查项。

6. 若某轮发现前面轮次的核心假设不成立，立即终止当前链路，回到 Phase 0 重新定义问题。

7. 如果连续两轮都无法把某项从 `证据不足` 升级，则停止对该项继续深挖，转为待用户确认或后续观察。

8. 单个问题默认最多深挖 2~3 层调用链；除非用户明确要求或已有强证据，不要把局部问题扩展成全仓审计。

## Phase 0：定义问题、范围与证据门槛

1. 先澄清问题定义。
   - 用户看到的现象是什么。
   - 用户期望的正确行为是什么。
   - 是否有复现线索、报错信息、输入条件、涉及页面或接口。

2. 建立本轮调查范围。
   - 指出入口文件、核心函数、关键变量、可能影响的模块。
   - 明确本轮是否只做分析、是否允许修改代码、是否需要运行验证。

3. 明确“判定为 bug 的最小证据标准”。
   - 至少满足其一：复现异常、测试失败、源码逻辑矛盾、数据流/调用链闭合后明确冲突、与既定业务规则或接口契约冲突。

4. 先做严重度分级。
   - `P0`：数据损坏 / 安全 / 系统不可用。
   - `P1`：核心功能异常。
   - `P2`：非核心功能异常 / 可降级接受。
   - `P3`：体验、文案、样式问题。
   - 默认只对 `P0/P1` 走完整深审流程；`P2/P3` 视成本决定是否缩短流程。

## Phase 1：候选问题发现（只发现，不定案）

1. 从用户现象和入口代码开始，先列候选问题清单。
   - 每个候选项都必须附：文件路径、函数/路由/变量位置、为什么像 bug、可能的反解释。

2. 可以使用现有 bug-finding skills / workflows 作为线索来源。
   - 例如改后检查、深度 review、架构审计、漂移检测、全量自测、真实路由重跑等。
   - 这些输出只能标记为：`skill 提示的候选问题`。

3. 不要急着横向扩搜全仓。
   - 先确认主问题是否真实存在。
   - 除非已经看到明确的同根因模式，否则不要把“相似写法”直接算入问题列表。

4. Phase 1 输出必须包含。
   - 候选问题列表。
   - 每项的严重度。
   - 每项的线索来源。
   - 每项的可能反解释。
   - 下一轮优先核实的项目。

## Phase 2：证伪上一轮并做独立核实

1. 逐条复核 Phase 1 的候选项。
   - 核对源码。
   - 核对调用方与被调方。
   - 核对关键变量的来源、默认值、空值分支、异常分支与兜底逻辑。

2. 严禁用“另一个 skill 也这么说”来替代核实。
   - 这类信息最多只能说明优先级高，不能当作最终证据。

3. 必须主动找反证。
   - 什么业务约束成立时，它其实是有意实现。
   - 什么上游前提成立时，这个风险不会发生。
   - 什么调用链证据一出现，就足以推翻当前怀疑。

4. 如果具备条件，优先补运行层验证。
   - 查看现有测试是否覆盖相关路径。
   - 能跑现成测试就先跑。
   - 能做最小复现脚本或最小输入验证就先验证。
   - 若当前任务明确不允许运行，则说明这一层验证被跳过。

5. Phase 2 结束后，每项必须升级为以下之一。
   - `确认缺陷`
   - `高概率缺陷`
   - `证据不足`
   - `确认属于有意实现`
   - `需要用户确认业务意图`

## Phase 3：根因链闭合

1. 只对仍保留的 `确认缺陷` 或 `高概率缺陷` 继续深挖。

2. 深挖的目标不是“找更多问题”，而是闭合根因链。
   - 上游入口从哪里传入。
   - 中间变量如何流转、转换、兜底、覆盖。
   - 下游最终影响到哪里。
   - 是否涉及事务、异常传播、提前 return、静默跳过。

3. 每轮结束都要沉淀结构化证据摘要。
   - `文件路径`
   - `行号或函数名`
   - `关键代码片段`
   - `当前结论`
   - `仍未解决的问题`
   不要只依赖对话上下文自然保留。

4. 如果在本阶段发现 Phase 0 的问题定义不成立。
   - 立即回到 Phase 0。
   - 明确说明：为什么原假设被推翻、现在哪个假设更合理。

## Phase 4：同根因扩搜（后置）

1. 只有在主问题根因基本闭合后，才允许搜索同类问题。

2. 只围绕“已确认根因模式”搜索。
   - 相同判断逻辑。
   - 相同变量处理方式。
   - 相同调用链断层。
   - 相同兜底或异常吞掉模式。

3. 此阶段最适合再次参考现有 bug-finding skills / workflows。
   - 但它们仍然只能提供扩搜线索，不能独立定案。

4. 输出时必须严格区分。
   - `同根因已确认问题`
   - `相似但未证实问题`
   - `仅来自 skill 提示的线索`
   - `已排除的相似实现`

5. 如果三轮结论高度一致但核心证据仍然薄弱，要主动提示：
   - 可能存在模型偏见放大。
   - 需要回到源码或运行验证重新确认。

## Phase 5：最终计划收敛

1. 最终只保留可落到证据的结论。

2. 输出必须至少包含。
   - 确认缺陷列表。
   - 每个缺陷的根因。
   - 影响文件、函数、调用方、关键变量。
   - 是否存在同根因批量修复机会。
   - `需要用户确认业务意图` 的问题。
   - 观察项 / 证据不足项。

3. 最终方案必须分成两档。
   - `最小修复方案`：只修确认缺陷，控制回归风险。
   - `扩展修复方案`：包含同根因批量修复，并明确回归风险更高。

4. 验证方案必须单列。
   - 单元测试 / 回归测试 / 页面验证 / 最小复现用例。
   - 哪些问题已经通过运行验证确认。
   - 哪些仍然只停留在源码级推断。

## 固定输出字段

每个问题项至少输出以下字段：

- `线索来源`
- `独立证据`
- `证据等级`：线索 / 辅证 / 实锤
- `当前结论状态`
- `是否已排除有意实现`
- `严重度`
- `是否存在同类问题`
- `需要用户确认的业务判断`
- `下一轮必查项`

若本轮某字段没有新增，允许写 `N/A - 本轮无新增`，不要为填模板而制造噪音。
