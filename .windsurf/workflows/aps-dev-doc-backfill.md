---
description: APS 开发文档回填：基于 git diff 与当前实现，对开发文档逐文件逐章节核对并回写事实
---
当用户提到“回填开发文档 / 更新开发文档 / 对齐开发文档 / 文档同步 / 接口清单更新 / 速查表更新”时，使用本 workflow。

1. 先建立变更文件清单，这是强制步骤。
   - 读取 `git status`。
   - 读取 `git diff`（必要时含 `--staged`）。
   - 生成表格：`文件 → 变更摘要 → 变更类型（DB/路由/模板/算法/配置/测试/其他）`。
   - 每个变更文件都必须在后续回填映射表中出现，即使结论是“对文档无影响”。

2. 并行扫描事实来源。
   - 变更面扫描：从 diff 归类模块、行为变化、用户可感知变化。
   - 接口与页面事实：扫描 `app.py`、`web/routes/*.py`、`templates/`、`web_new_test/templates/`。
   - 数据库事实：扫描 `schema.sql`、`core/models/*`、`data/repositories/*`。
   - 排产与日历事实：扫描 `core/services/scheduler/*`、`core/algorithms/*`。

3. 生成“回填映射表”。
   - 把每个变更文件映射到目标文档和章节。
   - 至少指出影响哪份文档、哪一节，以及如何验证文档是否与代码一致。

4. 逐文档逐章节核对与回填。
   - 优先处理最容易误导用户的内容：接口/页面/参数/文案。
   - 再处理速查表里的关键约束、枚举、模板、ScheduleConfig、DB 字段字典。
   - 再处理综合开发文档与实现计划表。
   - 已失效内容要明确标注“已废弃/仅留档”，不要假装仍有效。

5. 回填后自检。
   - URL、参数、默认值、枚举、表字段必须能在代码或 `schema.sql` 找到证据。
   - 关键规则要能追溯到具体文件。
   - 所有回填内容必须有事实来源，不靠猜测。

6. 向用户输出结果。
   - 修改了哪些文档。
   - 每份文档修了哪些章节。
   - 哪些问题仍需用户确认。
   - 如需要，提示后续再跑 `aps-post-change-check` 或 `aps-drift-detect`。
