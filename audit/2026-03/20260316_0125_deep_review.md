## 二、审查结论

### 2.1 总结论

- **未发现会直接导致当前排产主流程错误落库、崩溃或大面积错误统计的 P0/P1 缺陷。**
- **发现 1 个已证实的口径偏差族问题**：超期判定边界在算法层和摘要层不一致。
- **发现 2 个已证实的搜索空间/可解释性缺口**：`improve` 不跨 `dispatch_mode` 探索；`result_summary` 缺完整配置快照与 `best_score` 语义说明。
- **确认 2 条先前“疑似问题”其实是设计决策，不应误判为 BUG**：`min_tardiness` 不使用 `weighted_tardiness_hours`；`ready_weight` 为预留字段。
- **确认边界与回退机制总体清晰且基本属实**，但观测性仍不够，导致“算法为什么这样选”需要反推。

## 三、必须明确的发现

### F1 超期判定边界不一致

- 位置：
  - `core/algorithms/evaluation.py:115-118`
  - `core/services/scheduler/schedule_summary.py:65`
  - `core/algorithms/dispatch_rules.py:25-29`
- 事实：
  - `evaluation.py` 用 `due_end = 23:59:59`，判定条件是 `fin > due_end`
  - `schedule_summary.py` 用 `finish.date() > due_date`
  - 最小化实验构造 `finish_time = 2026-01-01 23:59:59.500000`
  - 结果：`compute_metrics_overdue_count=1`，`summary_overdue_items_count=0`
- 判断：
  - 这是**已证实的口径偏差**
  - 影响非常窄，只会出现在“带亚秒精度的完工时间”窗口内
  - 但它是一个**同类问题族**：`evaluation.py` 与 `dispatch_rules.py` 一致，`schedule_summary.py` 单独偏离
- 建议：
  - 统一三处口径
  - 若坚持 v1 兼容，优先让 `schedule_summary.py` 对齐算法层
  - 若准备面向 v2 演进，则统一改成 `due_exclusive = due_date + 1 day 00:00`

### F2 `min_tardiness` 不使用 `weighted_tardiness_hours`

- 位置：
  - `core/algorithms/evaluation.py:243-258`
  - `开发文档/V1.2/排产优化_目标与验收标准.md:88-93`
- 事实：
  - 代码当前 `min_tardiness` 仍按 `total_tardiness_hours` 择优
  - 文档明确写了“v1.x 的 `min_tardiness` 仍使用 `total_tardiness_hours`”
  - 合成案例与最小化实验都证明：如果改用加权拖期，择优结果会变
- 判断：
  - 这不是 BUG
  - 这是 **v1 设计决策与 v2 目标态差异**
  - 但它会带来业务上可感知的结果差异，不能再把它表述成“实现遗漏”
- 建议：
  - 保留 `min_tardiness` 现状不动
  - 新增 `min_weighted_tardiness` 作为新 objective，避免破坏现有行为
  - UI / 报告要明确“当前目标函数口径”

### F3 `improve` 的 multi-start 不跨 `dispatch_mode` 探索

- 位置：
  - `core/services/scheduler/schedule_optimizer.py:341-347`
  - `core/services/scheduler/schedule_optimizer_steps.py:173-186`
- 事实：
  - 当配置为 `batch_order` 时，multi-start 只轮询策略，不切到 `sgs`
  - 当配置为 `sgs` 时，multi-start 只在 `sgs` 内轮询 `slack/cr/atc`
  - 最小化实验结果：
    - `batch_order_case.unique_dispatch_modes=["batch_order"]`
    - `sgs_case.unique_dispatch_modes=["sgs"]`
    - `sgs_case.unique_dispatch_rules=["atc","cr","slack"]`
- 判断：
  - 这是**已证实的优化空间**
  - 不是逻辑错误，但搜索空间被人为锁窄
  - 与 F5（局部搜索只扰动 batch order）属于同一类问题：`improve` 实际只在局部维度搜索
- 建议：
  - 若要改善解质量，优先扩展 multi-start 外层的 `dispatch_mode` 维度
  - 这是低侵入改造，不影响 `greedy` 路径

### F4 `result_summary` 的观测性不足

- 位置：
  - `core/services/scheduler/schedule_summary.py:220-267`
- 事实：
  - 当前会落盘 `best_score`、`metrics`、`best_batch_order`
  - 但不会落盘完整 `config_snapshot`
  - 也不会说明 `best_score` 每一位对应什么含义
- 判断：
  - 这不是算法错误
  - 这是**已证实的可解释性/可复现性缺口**
  - 也是这次深审为何必须额外写审计脚本的主要原因
- 建议：
  - 新增 `algo.config_snapshot`
  - 新增 `algo.best_score_schema`
  - 保持原字段不删，避免下游兼容问题

### F5 SGS 评分对资源空隙的估算偏保守

- 位置：
  - `core/algorithms/greedy/downtime.py:7-17`
  - `core/algorithms/greedy/dispatch/sgs.py:230-235`
- 事实：
  - `get_resource_available()` 只取某资源所有占用区间的最大结束时间
  - 评分阶段因此看不到中间空隙
  - 实际排程阶段仍会用 `find_overlap_shift_end()` 扫描重叠区间，因此不会排出冲突计划
- 判断：
  - 这不是 correctness bug
  - 这是**评分启发式偏保守**
- 建议：
  - 若后续要继续优化 `sgs`，这是很适合优先试验的点
  - 只改评分路径，副作用较低

## 五、边界与回退机制审查

### 5.1 边界总体是否明确

结论：**总体明确，且比初看时更清楚。**

明确体现在：

- `ScheduleService.run_schedule()` 用 `_RUN_SCHEDULE_LOCK` 防并发，`finally` 保证释放
- `build_freeze_window_seed()` 把“冻结窗口失败”降级为 warning，不直接破坏主流程
- `schedule_optimizer._run_ortools_warmstart()` 明确是 fail-open，可选项失败不阻断
- `optimize_schedule()` 在理论异常路径上仍有 `best is None` 单次回退
- `dispatch_sgs()` 对评分异常使用 penalty key，而不是静默跳过
- `dispatch_rules._safe_positive()` 对 NaN/Inf/<=0 做回退，避免 `ATC/CR` 极值污染
- `schedule_summary._compute_downtime_degradation()` 会把停机约束降级显式写入结果摘要

### 5.2 回退机制是否属实

结论：**属实。**

本轮运行中未出现任何与代码声明相矛盾的情况：

- OR-Tools 未启用时未影响主流程
- `smoke_phase7` 覆盖的 merged 外协、冻结窗口、improve 留痕都按文档工作
- `improve` 在固定 `version` 下可做到 bit-exact 一致，说明“版本号当随机种子”的设计确实成立

### 5.3 是否还有同类问题

有，主要分三类：

1. **时间口径类**
   - F1/F3 同类：`evaluation.py`、`dispatch_rules.py`、`schedule_summary.py` 对“交期截止/是否超期”的表达不完全一致

2. **搜索空间锁窄类**
   - F3/F5 同类：`improve` 只探索了 order 和 rule 的局部子集，没有探索 mode 切换

3. **观测性不足类**
   - F4 同类：配置快照缺失、`best_score` 无 schema、`improvement_trace` 和 `attempts` 语义需要靠阅读源码理解

## 六、已确认正确的既有意见

以下意见本轮核验后仍成立：

- `CalendarEngine` 是当前主要性能热点，尤其在 SGS + auto-assign 场景
- `SGS` 的异常兜底与 `seed_results` 去重/冻结逻辑已较为稳固
- `improve` 当前是“增量改进器”，不是广义搜索框架
- v1 与 v2 文档必须分开看，不能把目标态文档直接当现状实现清单

## 七、建议优先级

### 建议改（P2）

- [ ] 统一超期判定口径，至少先让 `schedule_summary.py` 与 `evaluation.py`/`dispatch_rules.py` 对齐
- [ ] 在 `result_summary` 增加 `config_snapshot` 与 `best_score_schema`
- [ ] 扩展 `improve` 的 multi-start，让其可探索 `batch_order ↔ sgs`
- [ ] 在 `sgs` 评分路径试验“空隙可见”的资源可用估算

### 可排期优化（P3）

- [ ] 在 `CalendarEngine.add_working_hours()` 路径做更强的按日缓存/预取
- [ ] 让 `local_search` 低概率扰动 `dispatch_rule`
- [ ] 对 `ext_group_id=""` 一类脏数据写 warning，而不是继续沉默降级

### 不建议当作 BUG 处理

- [x] `min_tardiness` 不使用 `weighted_tardiness_hours`
- [x] `ready_weight` 暂不生效
- [x] seed 缺单边资源时的部分冻结

## 八、最终判断

当前排产算法并不是“有明显错漏、随时会排坏”的状态；相反，它的**主流程稳定性、回退机制和回归覆盖已经比较扎实**。这次深审真正确认下来的，主要是：

1. 有一个**已证实但影响很窄**的时间口径偏差（超期边界）。
2. 有两个**会实质影响优化效果和排障效率**的工程性问题（`dispatch_mode` 搜索空间锁死、`result_summary` 观测性不足）。
3. 有两条**容易被误判成 BUG 的设计决策**（`min_tardiness` 口径、`ready_weight` 预留），它们现在应被归档，而不是继续当成 defect 追着修。

如果后续只允许做一轮“小步快跑”的改善，我建议顺序是：

1. 先补 `result_summary` 的 `config_snapshot + best_score_schema`
2. 再统一超期判定口径
3. 最后扩 `improve` 的 `dispatch_mode` 搜索维度

这三步都不需要更换算法框架，但能明显提升“可解释性 + 一致性 + 改善空间”。
