## 0. 先看结论：这份修订版和原报告的关系

一句话总结：

> **原报告的主方向是对的，热点判断也是对的，足以指导第一轮整改；但如果把它当成“精确基线台账”或“端到端已证实的用户不可见问题清单”，还需要再补一层证据和结构化沉淀。**

### 0.1 哪些结论我维持不变

以下判断，经源码复核后，我认为**总体成立**：

1. `core/` 不是“少量兜底”的代码，而是一个**系统性容错优先**的代码库；
2. 真正高风险的静默回退，主要集中在：
   - 排产主链；
   - 资源约束链；
   - 路由/外协周期链；
   - 参数/配置默认化链；
3. 基础设施层大量回退更像**保护性回退**，不能直接和“静默改业务结果”混算；
4. 最值得优先治理的，不是“会崩”的代码，而是“**还能继续跑、还能给结果、但结果口径悄悄变了**”的代码。

### 0.2 哪些结论我建议收紧或修正

以下判断，我建议在修订版中**改成更审慎的说法**：

1. **`68 个方法 / 178 个实现点` 目前更像“高可信人工统计结果”，还不是“可复跑、可验收的精确基线台账”。**
   - 原因：缺少 commit 锚点、扫描脚本、原始命中清单和 machine-readable inventory。

2. **“这些问题用户默认看不到”这个判断，方向对，但证据链还应再分层。**
   - 源码显示 warning/logger 的确不等于用户一定能看到；
   - 但从已读代码和测试可确认，`downtime / freeze_window` 这类降级字段已经存在一部分 `summary/result_summary -> analysis 页面` 的贯通链，不应简单写成“完全不可见”。

3. **部分风险分级还可以再细化。**
   - 例如 `dispatch_sgs` 里某些 `pass` 非常危险，但更像“启发式负荷状态漂移”，和“默认 1 天直接落库”“停机约束整体失效”相比，语义层级仍有差别。

4. **部分 normalizer 在主表里被归成“完全静默”，但源码实际是“空值默认 + 未知值透传/抛错”的混合行为。**
   - 例如 `normalize_supplier_status` / `normalize_op_type_category` / `normalize_skill_level` / `MachineService._validate_machine_fields`；
   - 修订版应避免把它们解读成“任何非法值都会被吞掉”。

### 0.3 本轮新增的几个重要判断

本轮补充复核后，我认为原报告还应增加下面几类判断：

1. `schedule_summary.py` 处在“算法 warning -> 结果摘要 -> 页面认知”的关键链路上，原报告对它展开不够；
2. 目前的统计方法对 `except Exception` 抓得很好，但对**非异常型静默回退**覆盖不够，例如：
   - `if not x: x = default`
   - `raw or default`
   - 条件分支里的静默 `continue`
   - 单条坏记录被直接丢弃但不触发异常
3. `config_snapshot -> schedule_params` 存在**双层默认化链**，坏值有机会在更早层就被“洗白”；
4. `route_parser` 的“默认 1 天”风险，不只来自“确实无供应商”，还可能来自**命名匹配未命中**或 `_build_supplier_map()` 中更早层的异常转义；
5. `_build_supplier_map()` 除了“取工种失败 -> 跳过映射”之外，还存在 `float(getattr(supplier, "default_days", 1.0) or 1.0)` 这一非异常型回退，会把 falsy `default_days`（尤其 `0.0`）收敛为 `1.0`；这也是本轮回填后 strict count 从 `177` 修订为 `178` 的原因。
6. 从回归测试命名和已读测试可以确认：**部分 fallback 已经不是单纯技术债，而是现有兼容/可用性契约的一部分**。

## 2. 一页风险地图：先看哪里最危险

### 2.1 宽口径和严格口径的关系

原报告最大的优点之一，是没有把 `except Exception` 数量直接等同于“有业务风险的静默回退数”。

它做了两层区分：

1. **宽口径**：先把候选点扫出来；
2. **严格口径**：再剔除纯 close/pass、二次 logger 失败、纯 rethrow 包装等低价值噪音。

这个方法论是对的。

### 2.2 从阅读和整改角度，更适合看的四类风险

| 类别 | 典型行为 | 业务影响 | 当前建议 |
|---|---|---|---|
| 保护性回退 | 迁移失败回滚、恢复失败尽量回滚、插件失败不阻断主程序 | 主要保护系统存活 | 保留，但补结构化结果码/状态 |
| 兼容性默认化 | 空值回默认、非法枚举回默认、读侧 normalizer 自愈 | 兼容旧数据和脏输入，但会洗白坏值 | 区分读侧/写侧，给 strict mode |
| 业务语义改写 | 外协周期默认 1 天、未知工种默认 external、状态缺省默认 active | 直接改变业务口径 | 高优先级治理 |
| 约束降级 | 忽略停机、关闭 auto-assign、冻结窗口失效 | 结果继续产出，但约束边界变了 | 高优先级治理，并补结构化 degrade 字段 |

### 2.3 按风险族看的优先级

原报告的四大类族仍然成立：

| 类族 | 方法数 | 实现点数 | 我的复核判断 |
|---|---:|---:|---|
| 输入 / 标准化族 | 28 | 47 | 影响广，但多数是系统风格问题，宜做“契约化 + 分层治理” |
| 算法调度族 | 14 | 69 | 仍是最危险的一类；但内部还需细分“硬约束失效”与“启发式退化” |
| 基础设施 / 插件族 | 16 | 45 | 大量是保护性回退，不应与业务口径改写混算 |
| 业务与展示非阻断族 | 10 | 17 | 方法数不多，但若落库或影响约束，风险很高 |

## 5. 对原报告的几个关键修订意见

## 5.1 这是一份“高质量研判报告”，还不是“精确盘点台账”

原报告已经足够用于：

- 找热点；
- 排优先级；
- 开第一轮整改；
- 给团队建立一致认知。

但它还不够作为终版基线台账，原因主要有四个：

1. 没有 commit / revision 锚点；
2. 没有把扫描规则与原始命中集保留下来；
3. 没有 machine-readable inventory；
4. 非异常型静默分支仍可能漏检。

### 我建议补的台账字段

后续如果要把这份报告转成长期维护资产，建议单独产出 `json/csv` 台账，至少包含：

- `id`
- `file`
- `function`
- `line_start`
- `line_end`
- `trigger`
- `fallback_action`
- `visibility`
- `scope`
- `persisted`
- `constraint_impact`
- `risk_level`
- `intent`（compat / resilience / debt）
- `evidence`

---

## 5.2 报告对“异常驱动型回退”抓得很好，但对“纯分支驱动型静默回退”还不够完整

原报告主要按：

- `except Exception`
- `except:`
- 回退/默认/跳过等关键词

去抓风险点，这对大多数核心热点是有效的。

但它容易漏掉下面这种代码：

- `if not x: x = default`
- `value = raw or default`
- `if st and et and et > st: append(...)`，否则静默不入结果
- 单条坏记录 `continue`，但不抛异常、不写 warning

### 一个典型例子

在 `resource_pool_builder.py` 里，停机记录只有在 `st/et` 都合法且 `et > st` 时才入 `intervals`。如果某条停机记录本身坏掉：

- 不一定抛异常；
- 不一定进入 warning；
- 但那条停机记录会被静默丢掉。

这类问题不一定会落进当前报告的“广义异常点”统计。

## 5.3 `schedule_summary.py` 应纳入主叙述，而不是只停留在边角

这是我认为原报告最值得补的一处。

原报告已经注意到：

- `schedule_service.py` 的 warning 合并可能失败；
- `summary` 不等于用户可见；
- logger 也不等于用户可见。

但还缺少一环：

> **`schedule_summary.py` 本身就在“算法输出 -> 结果摘要 -> 页面认知”的主链上。**

而我核到两个实际点：

1. `summary.warnings.append(msg)` 自己可能失败且被吞；
2. `_extract_freeze_warnings()` 任意异常后返回空数组。

所以，修订版应该把它明确列入“观测链风险点”。

---

## 5.4 “用户是否看得见”要分层说，不宜一刀切

原报告的直觉是对的：warning/logger 并不天然可见。

但本轮我补读了两个关键位置：

### 代码证据

- `core/services/scheduler/schedule_service.py:425-443`
  - 会把 `algo_warnings` 合并进 `summary.warnings`；
- `core/services/scheduler/schedule_service.py:500-509`
  - `summary.warnings` 会进入 `result_summary` 的 `summary` 部分。

### 回归测试证据

- `tests/regression_scheduler_analysis_observability.py`
  - 明确验证了 `downtime_avoid.degraded` 和 `freeze_window.degraded` 可以保留在 `selected_summary` 中；
  - 并验证调度分析页会展示：
    - “停机避让约束已降级”
    - “冻结窗口约束已降级”
    - 对应的降级原因文本。

### 因此更准确的说法应是

> **并不是所有降级都完全不可见；当前至少已有一部分 `downtime / freeze_window` 相关降级，能通过 `summary/result_summary -> analysis 页面` 被看到。**

但同时也要保留原报告的警惕：

- 这条可见链并不是对所有 fallback 都成立；
- `warnings` 自己仍可能丢；
- 很多点仍只停留在 logger；
- 业务用户是否会进入分析页查看，和“系统具备该字段”是两回事。

所以最终建议写法是：

> **当前系统已经存在部分结构化降级可观测链，但它覆盖不完整、稳定性也不足，不能把“有 logger / 有 warning / 某个页面可展示”简单等同于“用户一定知道这次结果已经退化”。**

## 5.5 `config_snapshot -> schedule_params` 的双层默认化链，需要单独强调

原报告已注意到：

- `config_snapshot.build_schedule_config_snapshot()` 会对 choice/int/float/yes-no 做默认化；
- `resolve_schedule_params()` 会继续对 `dispatch_mode/dispatch_rule/start_dt/end_date/auto_assign_enabled` 做默认化。

我建议修订版把这个点单独拉出来，因为这不是一个方法问题，而是一个**链路问题**：

> **坏值可能在更早层就已经变成“合法值”了，后面很多 warning 根本不会触发。**

这意味着：

- 用户看到的不是“非法配置被拒绝”；
- 而是“系统给出了一份看起来合法的配置快照”；
- 最后调度也能继续跑，但用户未必知道中间发生了默认化。

这正是“容错优先系统”的典型代价。

不过这里还有一个容易混淆的边界：从主服务入口看，`ScheduleService._run_schedule_impl()` 已在更早层对 `start_dt/end_date` 做显式 `ValidationError` 校验。因此 `resolve_schedule_params()` 对这两个字段的默认化风险，更多体现在算法 API / 非主入口调用，而不是所有 Web 调用路径都同样可达。

---

## 5.6 `route_parser` 的风险，不只是“默认 1 天”，还包括“默认 1 天很容易被误触发”

`route_parser.py` 这条链，修订版应补两层理解：

1. **表面层**：找不到供应商映射 -> `default_days = 1.0`
2. **更深层**：供应商映射未命中，未必是真没有配置，也可能是：
   - 名称匹配没命中；
   - `_build_supplier_map()` 底层异常导致映射被跳过；
   - 工种命名存在空格/别名/大小写/全半角等差异。

所以这个点的风险，不只是结果有多坏，还在于：

> **它很可能比表面看上去更容易触发。**

另一个相邻风险也值得补一句：

- `_build_supplier_map()` 里 `float(getattr(supplier, "default_days", 1.0) or 1.0)` 会把 falsy `default_days`（尤其 `0.0`）收敛成 `1.0`；
- 这说明上层“默认 1 天”不只来自“无映射”，也可能来自“映射存在但周期值在中间层被洗白”；
- 同时，这条表达式自身没有 `try`，因此整条链仍是“部分转义成默认、部分又可能直接硬失败”的混合行为。

这类混合行为更应该整链治理，而不是只盯 `parse()` 的 `default_days = 1.0`。

## 5.7 需要把“这是兼容性契约，还是技术债”明确拆开

从已读测试可以看到，部分 fallback 已经被写进回归测试，说明它们不是单纯“代码作者偷懒留下来的异常吞并”，而是现有行为契约的一部分。例如：

- `tests/regression_sgs_scoring_fallback_unscorable.py`
  - 明确验证 SGS 在“候选都不可估算”时，仍应通过 fallback key 维持可比性，而不是无条件退化到 `candidates[0]`；
- `tests/regression_seed_results_drop_duplicate_op_id_and_bad_time.py`
  - 明确验证坏 seed 会被去重/忽略，并且 warning 应进入 `summary.warnings`；
- `tests/regression_auto_assign_empty_resource_pool.py`
  - 明确验证 `resource_pool={}` 时的语义，不应简单等同于“未提供 resource_pool”；
- `tests/regression_scheduler_analysis_observability.py`
  - 明确验证停机/冻结窗口降级信息可以进分析页展示。

这说明后续治理不能只分“坏代码/好代码”，更需要分：

| 类型 | 含义 | 处理建议 |
|---|---|---|
| `contract_keep_observable` | 兼容性/可用性契约，应保留 | 增强结构化可观测 |
| `contract_keep_strict_option` | 默认兼容，但需要 strict mode | 增加兼容/严格双模式 |
| `debt_fix_now` | 明显不该继续静默 | 优先整改 |
| `debt_fix_later` | 有风险，但暂不应打断当前业务 | 补指标、补台账、排期改造 |

## 10. 最终判断（修订版）

### 10.1 对原报告的总体评价

> **原报告是一份质量很高的风险识别报告。**

它最大的价值是：

- 没把所有 `except` 一股脑等同于坏代码；
- 成功找到了真正高风险的热点区域；
- 成功把“保护性回退”和“业务语义回退”区分开了；
- 已经足够指导第一轮整改。

### 10.2 我对它的保留意见

> **它还不是一份足够精确、可复跑、可持续维护的终版台账。**

主要还差：

- 可复跑 inventory；
- commit 锚点；
- 非异常型静默回退补扫；
- `schedule_summary.py` 等观测链路补充；
- UI/API 端到端证据矩阵。

### 10.3 我对这套代码的最终判断

一句话：

> **这不是一个“少量兜底”的代码库，而是一个“容错优先、保护可用性优先”的代码库。**

它的优点是：

- 不容易一处坏就整次调度全挂；
- 对旧数据、脏数据、可选依赖、桌面式部署环境友好；
- 数据库迁移/恢复链有明显的保护性设计。

它的代价是：

- 错误常常不会以 crash 暴露，而会以“结果悄悄变差”暴露；
- 一些默认值不是低风险 UI 默认，而是直接影响排产语义和业务口径；
- 观测链路还不够稳定，导致“有 warning”不等于“问题会被知道”。

### 10.4 修订后的优先级结论

如果只按“最值得先改什么”排序，我的建议是：

1. **先堵住默认 `1.0` 天**
   - `RouteParser.parse`
   - `ExternalGroupService._apply_separate_mode`
2. **再堵住停机/冻结/资源池的约束降级**
   - `load_machine_downtimes`
   - `extend_downtime_map_for_resource_pool`
   - `build_resource_pool`
   - `_safe_load_schedule_map`
3. **给 SGS / auto-assign / dispatch key 补计数器和结构化 degrade 字段**
4. **把参数/配置默认化改成 strict/compatible 双模式**
5. **最后再做 broad except 缩窄**

## 13. 附录 C：运行时隐藏问题核实长表（初版原始留痕）

### C.1 核实方法与判定标准

- 本节不是按关键词复扫，而是对主表中的重点方法逐个打开源码，沿主调用链确认“异常发生后到底会不会被看到”。
- 本次重点复核的源码包括：`dispatch/sgs.py`、`schedule_params.py`、`seed.py`、`auto_assign.py`、`scheduler.py`、`resource_pool_builder.py`、`freeze_window.py`、`route_parser.py`、`external_group_service.py`、`part_service.py`、`gantt_tasks.py`、`gantt_service.py`、`_helpers.py`、`enum_normalizers.py`、`config_snapshot.py`、`calendar.py`、`calendar_admin.py`、`calendar_engine.py`、`dispatch_rules.py`、`schedule_service.py`，并对 `database.py` / `backup.py` / `transaction.py` / `logging.py` 做了边界复核。
- 这里把“运行时隐藏问题”定义为：
  1. 异常/坏数据进入后，没有继续向上抛；
  2. 调用方拿到的仍是“成功结果”或一个看似正常的默认值；
  3. 回退已经改变了 **业务口径 / 排产顺序 / 约束是否生效 / 持久化值**；
  4. 即使有 `warning` 或 `logger`，默认 UI/API 仍不足以把它提升为明确失败。
- 不纳入本节主清单的情形：
  - 纯资源释放 `close()/pass`
  - 二次 logger 失败
  - 数据库迁移/恢复的保护性回滚
  - 可选插件/OR-Tools 的显式降级
  - maintenance 清理任务的 best-effort 跳过

### C.2 复核后的总判断

- 原报告“68 个方法 / 177 个实现点”的**扫描与归类总体成立**；
- 不过本轮补充复核额外识别出 `RouteParser._build_supplier_map()` 中 1 处此前未计入的非异常型 fallback，因此当前修订版 strict count 已回填为 **68 个方法 / 178 个实现点**；
- 但从“会不会在运行时把 BUG 吃掉、让问题不显现”的角度收窄后，真正需要拉红线的主体，主要集中在 **排产主链、资源约束链、工艺解析链、外协周期写入链**；
- 其中最危险的不是 crash，而是：
  - **静默改口径**：默认 `1.0` 天、默认 `batch_order/slack`、默认 `internal/active/no`
  - **静默关约束**：忽略停机、关闭 auto-assign、冻结窗口失效
  - **静默丢观测**：warning 丢失、JSON 坏掉返回空、页面构建异常返回空数组

| 级别 | 已核实问题项 | 判定口径 |
|---|---:|---|
| 致命 | 3 | 会直接写入错误业务值，或让关键约束整体失效，且默认调用链缺少稳定失败信号 |

#### C.4.1 需要特别补充的两个源码细节

1. **`RouteParser._build_supplier_map()` 还会放大第 2 条致命问题**
   - `core/services/process/route_parser.py:286-291`
   - 这里 `self.op_types_repo.get(...)` 异常后直接把该 supplier 映射跳过；
   - 上层 `parse()` 再遇到该工种时，就会走“无供应商 -> 默认 1.0 天”的分支；
   - 同时，`float(getattr(supplier, "default_days", 1.0) or 1.0)` 还会把 falsy `default_days`（尤其 `0.0`）收敛成 `1.0`；
   - 也就是说，**底层映射构建异常会被转义成表面上的“供应商缺省”**，而部分“已存在映射但周期值异常/被洗白”的情形，也会在中间层被折叠成同样的 `1.0` 天结果。

2. **`GreedyScheduler.schedule()` 的 warning 不是天然可靠可见**
   - `resolve_schedule_params()`、`load_machine_downtimes()`、`build_resource_pool()` 等虽然会产出 `warnings`；
   - 但这些 warning 还要经过 `summary` 合并、`result_summary` 生成、前端展示三层链路；
   - 它们并不等于“用户一定能看到的错误”。

### C.6 已复核为“保护性回退”，不建议直接当作运行时隐藏 BUG 主项

下面这些点我看过源码后，仍然认为**更像保护系统存活的显式降级**，而不是“把 BUG 吃掉不让人发现”的主体：

1. **数据库迁移 / 备份 / 恢复链**
   - `core/infrastructure/database.py::_migrate_with_backup`
   - `core/infrastructure/backup.py::BackupManager.backup/restore`
   - 这些路径虽然大量 `try/except`，但核心语义是：
     - 尽量回滚
     - 结果对象明确失败
     - 日志明确记录
   - 它们更像“故障保护”，不是“静默改业务结果”。

2. **事务管理链**
   - `core/infrastructure/transaction.py::TransactionManager.transaction`
   - 这里绝大多数关键异常都会 `logger.error(...)` 后 `raise`，并没有把业务错误吞掉。

3. **操作日志落库**
   - `core/infrastructure/logging.py::OperationLogger.log`
   - 它的问题在于“审计可观测性可能变差”，但不会直接改变业务数据正确性；
   - 更适合归入“监控/留痕补强”，而不是“隐藏业务 BUG”。

4. **插件与 OR-Tools 可选能力**
   - `core/plugins/manager.py::PluginManager.load_from_base_dir`
   - `core/services/scheduler/schedule_optimizer_steps.py::_run_ortools_warmstart`
   - 都属于可选能力失败后主程序继续，且已有日志/状态，不属于最危险的隐藏问题主体。

5. **maintenance 清理任务**
   - `cleanup_task.py` / `backup_task.py`
   - 主要是后台 housekeeping；
   - 有 best-effort 跳过，但不直接改排产业务语义。

### C.7 结合源码后的最终优先级建议

如果只按“最可能在运行时把 BUG 吃掉、让结果悄悄变形”的标准排整改优先级，我建议顺序改成下面这样：

1. **先堵住“默认 1 天”**
   - `RouteParser.parse`
   - `ExternalGroupService._apply_separate_mode`
   - 原因：这是最典型的业务语义静默改写，而且会进入持久化数据。

2. **再堵住“停机约束整体/部分失效”**
   - `load_machine_downtimes`
   - `extend_downtime_map_for_resource_pool`
   - 原因：一旦失效，排产结果表面正常，实际已越过关键约束。

3. **给 SGS 增加可观测计数器，去掉关键 `pass`**
   - `dispatch_sgs`
   - `build_dispatch_key`
   - `auto_assign_internal_resources`
   - 原因：这里是“结果悄悄变差”的最大黑洞。

4. **把参数默认化改成 strict/compatible 双模式**
   - `resolve_schedule_params`
   - `build_schedule_config_snapshot`
   - 原因：当前配置/输入错误太容易被“合法默认值”掩盖。

5. **把 warning 链路做成稳定结构化字段**
   - `schedule_service._run_schedule_impl`
   - 原因：如果 warning 自己都可能丢，前面所有“弱可观测降级”都会继续恶化成“用户无感知”。

### C.8 一句话复核结论

> **原报告里列出的 68 个方法并不都等于“隐藏 BUG”，但其中确实有一批会在运行时把错误吞掉，并把“异常状态”伪装成“正常默认值”。真正最危险的，不是会报错的代码，而是那些还能继续跑、还能给出结果、但结果口径已经悄悄变了的代码。**
