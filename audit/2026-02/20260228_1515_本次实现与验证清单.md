## 3) 关键行为变化点（审查时重点看）

### 3.1 Conformance“排产落库+留痕”检查位置调整（属于测试/验收口径修复）

- 变化：
  - 原先 `tests/generate_conformance_report.py` 在 `schedule_service.py` 里用字符串匹配找 `action="simulate" if simulate else "schedule"`。
  - 由于排产落库+留痕实际在 `schedule_persistence.py`（职责拆分后更合理），因此把检查改为读 `schedule_persistence.py`。
- 风险：
  - 若未来又迁移留痕位置，对标脚本仍需同步更新，否则会出现“假阴性”。
- 覆盖：
  - `python tests/generate_conformance_report.py` 已跑过并通过。

### 3.2 scheduler 内 `yes/no` 判断收敛为 `to_yes_no + YesNo`

涉及文件：`freeze_window.py`、`resource_pool_builder.py`、`schedule_summary.py`、`schedule_service.py`、`schedule_optimizer_steps.py` 等。

- 变化：
  - 从 `== "yes"` 变为 `to_yes_no(x) == YesNo.YES.value`。
- 影响：
  - 行为更“容错”（接受 `true/1/on/y` 等）。对既有 `"yes"/"no"` 数据保持兼容。
- 风险：
  - 若某些字段历史上用非标准值表达“否”，容错可能把其误判为 yes/no（需结合 `to_yes_no` 的真值集合审查）。
- 覆盖：
  - scheduler 相关回归（冻结窗口/预算/策略解析）已跑过。

### 3.3 状态字段从裸字符串改为枚举 `.value`（等价字符串）

涉及文件：`schedule_persistence.py`、`batch_copy.py`、`batch_template_ops.py`、`batch_excel_import.py`。

- 变化：
  - `"scheduled"` → `BatchOperationStatus.SCHEDULED.value`；`"pending"` → `BatchStatus/BatchOperationStatus.*.value` 等。
- 影响：
  - 存储到 DB 的仍是同样的字符串值；减少散落字符串导致的不一致风险。
- 覆盖：
  - `smoke_phase6.py` / 相关 regression 已验证。

### 3.4 `except Exception: pass` 改为 `except Exception: ...`（仅为满足门禁，语义不变）

涉及文件：`resource_pool_builder.py`、`schedule_summary.py`。

- 变化：
  - 将静默吞异常的 `pass` 改为 `...`，避免被 fitness 门禁识别为新增违反。
- 影响：
  - 仍然是吞异常（等价），不会改变控制流；仅满足“新增违反必须修复”的门禁规则。
