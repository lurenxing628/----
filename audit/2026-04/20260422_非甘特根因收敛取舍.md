## 事实摘要

1. version 合同当前至少有三类：latest-fallback、strict-int、strict-positive-int。
2. AppError 全局 handler 除 413 外几乎统一返回 400，同时大量 route 会本地吞错。
3. config hidden repair 主漏洞已堵住，但 readonly truth-chain 仍可能对 drifted baseline 报 exact。
4. warning 文本主链已可见；week-plan/history 主要缺 rendered HTML guard，结构化 warning provenance 仍未进入稳定 display contract。

## 结论与取舍

### 必须改（P1）

- [ ] config readonly truth-chain 误报 exact
  - 证据：共享 readonly 链不做 baseline diff，而相关 diff helper 只在写链使用
  - 建议：优先在 `get_preset_display_state` / `_build_current_config_state` 补真实 diff，并让 `_readonly_active_preset_state`、bootstrap 复用结果
  - 成本：M
  - 回归风险：中

- [ ] AppError HTTP 语义失真
  - 证据：全局 400 压平 + route 本地吞错双层并存
  - 建议：先补集中 `ErrorCode -> HTTP` 映射，再梳理 route 本地翻译白名单
  - 成本：M
  - 回归风险：中

- [ ] version 合同 owner 不清
  - 证据：shared helper family 与两类 strict 合同并存，但命名和边界不清
  - 建议：显式化三类 helper/owner；保留 resource_dispatch service 不变量
  - 成本：S-M
  - 回归风险：低到中

- [ ] warning_pipeline 结构化 provenance 未进入稳定 display contract
  - 证据：warning 文本已可见，但 `summary_merge_error` 等字段不上屏
  - 建议：补共享 display 字段，并优先给 week-plan/history 增加 rendered HTML guard

当前不建议按现状合入。

原因：

- config truth-chain 和 AppError 语义这两条都还处于“会给出错误事实”级别，而不是单纯文案欠账

- warning 链路已不再是 blocker，但适合同批补齐守卫，降低下轮返工
