# SP04 请求级容器与仓储束交接说明

- 日期：2026-04-11
- 批次：SP04
- 范围：`web/bootstrap/request_services.py`、`core/services/scheduler/repository_bundle.py`、`web/bootstrap/factory.py`、`web/ui_mode.py`、两批调度相关路由、两处自定义测试工厂、直接装配门禁与 `_repos` 漂移门禁。

## 一、已稳定的装配入口

### 1. 请求级入口

- 本批目标路径现在统一通过 `g.services` 取服务。
- `factory.py:_open_db()` 的挂载时点已经固定：
  1. 白名单 / 维护窗口短路先判定；
  2. 主路径建立 `g.db`、`g.op_logger`；
  3. 维护任务调用点执行完毕；
  4. 显式挂载 `g.services = RequestServices(...)`。
- 维护窗口短路路径不会挂载 `g.services`；`web/ui_mode.py` 也不再回退到直接构造 `SystemConfigService`。

### 2. `ScheduleService` 构造边界

- 公开签名保持不变：`ScheduleService(conn, logger=None, op_logger=None)`。
- 内部仓储创建已收口到 `core/services/scheduler/repository_bundle.py`。
- `ScheduleService.__init__` 只新增私有 `_repos` 聚合持有，不新增公开参数。
- 外部消费面继续锁定为：
  - `svc.batch_repo`
  - `svc.op_repo`
  - `svc.part_op_repo`
  - `svc.group_repo`
  - `svc.machine_repo`
  - `svc.operator_repo`
  - `svc.operator_machine_repo`
  - `svc.supplier_repo`
  - `svc.schedule_repo`
  - `svc.history_repo`
- 本批未引入 `svc._repos.<repo>` / `svc.repos.<repo>` 新消费口。

### 3. `RequestServices` 冻结属性清单

当前公开属性固定为：

- `schedule_service`
- `batch_service`
- `config_service`
- `schedule_history_query_service`
- `machine_service`
- `operator_service`
- `supplier_service`
- `operator_machine_query_service`
- `material_service`
- `batch_material_service`
- `gantt_service`
- `part_service`
- `part_operation_query_service`
- `excel_service`
- `system_config_service`
- `system_job_state_query_service`
- `operation_log_service`

说明：

- 除 `excel_service` 外，统一复用 `db + app_logger + op_logger`。
- `excel_service` 只在首次访问时调用 `get_excel_backend()`；本批没有新增第二套 Excel 工厂机制。
- `system_config_service` 只注入 `logger`，不伪造 `op_logger`。
- 容器使用 `cached_property` 惰性构造；构造异常不缓存，后续访问允许重试。
- 仅服务构造阶段的 `AttributeError` 会被转成带属性名上下文的 `RuntimeError`，其余异常直接上抛。
- 后续若新增请求级服务属性，需同步更新 `REQUEST_SERVICES_PUBLIC_ATTRS`、对应 `cached_property` 与相关契约回归；若新增排产仓储，需同步更新 `ScheduleRepositoryBundle` 字段、`build_schedule_repository_bundle(...)`、`ScheduleService.__init__` 代理赋值与相关契约/门禁。
