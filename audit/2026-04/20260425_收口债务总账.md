# 2026-04-25 收口债务总账

## 这份档案为什么存在

这轮修复已经持续很久，牵到排产配置、甘特图、错误提示、页面展示、检查脚本、自检报告和审计材料。当前工作区已经不是几个孤立小改动，而是上百个文件互相牵连。继续边修边扩大范围，只会让“到底什么时候算结束”越来越不清楚。

所以本档案只做一件事：把这轮能收住的成果、必须挡住提交的问题、后续债务和结束标准写清楚。以后看这轮收口，先看本文，再去翻分散的旧审查文件。

## 这轮已经解决了什么

1. 检查链不再轻易相信脏工作区里的旧报告。没有干净状态和重新生成的证明，就不能把旧结果当成本轮通过。
2. 甘特图关键链不可用时，不再继续显示像是关键链仍然有效的高亮、箭头和说明。
3. 页面错误提示做了收敛。用户看到的是中文提示，不再直接看到内部字段名、内部原因或调试信息。
4. 版本选择、无历史数据、模拟排产结果等页面提示更一致，避免同一份历史在不同页面说法不一样。
5. 旧的失败报告、临时探针和不能代表当前状态的证据，已经从正式证明边界里剥离出来，避免以后误读。
6. 子代理和人工审查发现的问题，不再只留在聊天记录里，而是落到审计档案和机器台账各自该去的位置。

## 本轮必须挡住提交的问题

这些问题不是“以后慢慢优化”，而是本轮提交前必须有明确处理或明确标注的事：

1. staged-only 边界不完整：一部分修复只在 unstaged，直接提交会漏掉半截状态展示和用户提示修复。
2. 干净证明还没有拿到：当前只能说明某些定向回归通过，不能宣称质量门禁和完整自检已经在干净工作区通过。
3. 页面状态展示存在半改风险：部分页面已经使用组合后的中文状态，部分页面仍可能直接显示原始状态码。
4. 未知降级代码可能露出内部名字：如果后端带出调试用代码，页面和接口不能把原始内部名直接给用户看。
5. HTML 错误页不能展示机器键：字段名、清理键、调试细节可以留在诊断结构里，但 HTML 页面只应展示中文字段说明。

本轮已用最小改动补住第 3、4、5 点；第 1、2 点要靠边界整理、暂存确认和干净证明命令来完成，不能靠口头说明跳过。

## 本轮到哪里为止

本轮只继续处理会挡住收口的问题：暂存边界、干净证明、内部信息外露、状态展示半改、债务归档。

本轮不继续做这些事：不强行合并所有旧解析函数，不整理全部文案常量，不拆所有大文件，不改变路由注册兼容方式，不重排历史测试体系。

到以下条件满足时，本轮就应该停下：

1. `git status --short` 能说明准备提交的边界是完整的。
2. `git diff --check` 和 `git diff --cached --check` 没有空白错误。
3. `python scripts/sync_debt_ledger.py check` 通过，机器台账和实际扫描一致。
4. 本轮新增的内部信息外露、HTML 错误页、状态展示回归通过。
5. 如果要宣称完整通过，必须在本地 checkpoint commit 后跑干净工作区质量门禁和完整自检。
