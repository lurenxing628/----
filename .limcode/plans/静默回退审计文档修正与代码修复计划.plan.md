## 2. 代码修复方案

### Phase 1：堵住"默认 1 天"和"约束降级"（P1 高危）

#### 1.1 `RouteParser.parse()` — 无供应商映射时默认 1 天

- **文件**: `core/services/process/route_parser.py` L212-219
- **现状**: 外部工序找不到供应商映射时 `op.default_days = 1.0` 且无 warning
- **修复**:
  1. 在 else 分支增加 `warnings.append(f"工种'{op_type_name}'未找到供应商配置，外协周期已按默认 1 天处理")`
  2. 这样 `ParseResult.status` 会自动从 `SUCCESS` 降为 `PARTIAL`，让上层感知到
- **验证**: 补充回归测试确认 warning 存在且 status 为 PARTIAL

#### 1.2 `RouteParser._build_supplier_map()` — falsy default_days 被收敛为 1.0

- **文件**: `core/services/process/route_parser.py` L295
- **现状**: `float(getattr(supplier, "default_days", 1.0) or 1.0)` 会把 `0`/`0.0`/`None`/`""` 全部收敛为 1.0
- **修复**:
  1. 拆开逻辑：先取值、再判断，区分"属性不存在"和"属性值为 0/None"
  2. `default_days = 0.0` 时应记 warning 而非静默改为 1.0
  3. 对 `op_types_repo.get()` 异常记 warning（当前被完全吞掉）
- **验证**: 测试供应商 default_days=0 时是否保留 0 或给出明确提示

#### 1.3 `ExternalGroupService._apply_separate_mode()` — per_op_days 非法时默认 1 天落库

- **文件**: `core/services/process/external_group_service.py` L126-132
- **现状**: `dv is None or dv <= 0` → `dv = 1.0` → `op_repo.update()`，完全静默
- **修复**:
  1. 记录 warning/audit trail：`logger.warning(f"零件{part_no}工序{op.seq}外协周期未填/非法({d})，已按默认1天处理")`
  2. 考虑是否应该返回一个 "defaulted_ext_days" 计数器给调用方
  3. 如果 `per_op_days` 中有该 key 但值非法（而不是 key 缺失），应该更严格处理
- **验证**: 回归测试确认 logger 有输出、计数器可用

#### 1.4 `load_machine_downtimes()` — 整批停机约束失效

- **文件**: `core/services/scheduler/resource_pool_builder.py` L118-160
- **现状**: 外层 try 包住整段逻辑，任一步异常 → `downtime_map = {}`
- **修复**:
  1. 将逻辑拆为"逐机"try：单台设备停机加载失败不应清空所有设备的停机数据
  2. 在 meta 中补充 `downtime_partial_fail_machines: List[str]` 
  3. 保留现有 `meta["downtime_load_ok"]` / `meta["downtime_load_error"]` 接口
- **验证**: 测试一台机异常时其他机停机数据仍保留

#### 1.5 `extend_downtime_map_for_resource_pool()` — 候选设备停机扩展失败

- **文件**: `core/services/scheduler/resource_pool_builder.py` L243-280
- **现状**: 整段 try，失败后保留旧 map
- **修复**: 与 1.4 同理，改为逐机 try，单台失败记 meta 但不影响其他候选设备
- **验证**: 测试部分候选扩展失败时其余候选仍有停机数据

#### 1.6 `_safe_load_schedule_map()` — 冻结窗口降级

- **文件**: `core/services/scheduler/freeze_window.py` L37-54
- **现状**: 加载上一版本排程失败 → `return {}`，等价于不冻结
- **修复**:
  1. 保持当前降级逻辑（冻结窗口失败时继续排产是合理的）
  2. 但需要确保 warning 稳定进入 `result_summary`：在 meta 中补 `freeze_window_degraded: True`

## 3. 修复原则与约束

### 3.1 不破坏现有兼容性契约

已有回归测试（如 `regression_sgs_scoring_fallback_unscorable.py`、`regression_seed_results_drop_duplicate_op_id_and_bad_time.py` 等）确认的行为契约不能被破坏。所有修改必须通过现有回归测试套件。

### 3.2 可观测性优先于 fail-fast

在 Phase 1-3 完成前，不建议直接删除任何 `except` 或改为 fail-fast，因为当前的 warning/summary 链路尚不稳定，贸然 fail-fast 可能导致排产在生产环境直接不可用。

### 3.3 每个 Phase 独立可验收

每个 Phase 完成后应能独立通过完整回归测试和冒烟测试，不依赖后续 Phase。
