# 2026-04-05 strict 旁路收口修复 plan

> **执行方式**：优先使用 `subagent-driven-development`；如果当前环境不适合子代理或用户要求当前会话直接执行，则使用 `executing-plans`。

**目标**：让 strict 语义在 direct config、weighted getter、preset apply 三条公开支持的旁路入口上闭合，移除剩余静默修补，同时不重新打开已经收口的 UI 模式与工艺路线基线逻辑。

**总体做法**：先用失败用例锁住“显式空白不是缺失”“getter 读取失败不是默认值”“坏 preset 不能先修补后应用”这三条行为契约；再把配置读取改成缺失 / 空白 / 读取失败分离；最后只跑受影响的定向回归，不扩大战线。

**涉及技术 / 模块**：`core/algorithms/greedy/config_adapter.py`、`core/algorithms/greedy/schedule_params.py`、`core/services/scheduler/config_validator.py`、`core/services/scheduler/config_presets.py`、`tests/regression_scheduler_strict_mode_dispatch_flags.py`、`tests/regression_schedule_params_read_failure_visible.py`、`tests/regression_config_validator_preset_degradation.py`、`tests/regression_scheduler_apply_preset_reject_invalid_numeric.py`。

## 验收标准

1. `resolve_schedule_params()` 的 direct config 路径不再把显式空白 `sort_strategy`、`dispatch_mode`、`dispatch_rule`、`auto_assign_enabled` 冒充成缺失。
2. weighted 的 object / getter 配置读取失败不再静默落回 `0.4 / 0.5`；strict 下失败，非 strict 下可见。
3. `apply_preset()` 不再接受 blank dispatch / yes-no preset 并通过“规范化/修补”继续应用。
4. missing 字段的既有默认语义保持不变，不误伤合法旧数据。
5. 整个修复不引入新的全局兼容层、总开关或重复校验链。

## 非目标

1. 本轮不重构所有 `cfg_get()` 调用点。
2. 本轮不把所有配置字段一次性迁入新的统一读取框架。
3. 本轮不修改 UI 模式、Excel 导入基线、运行时入口等已收口模块。
4. 本轮不改模板或用户手册，除非实测发现 `scheduler_config` 路由无法正确展示 `ValidationError`。
