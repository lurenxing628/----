# 修复第四轮审查 5 项低级别问题（修订版 v3）

> 基于真实引用链、调用前守卫、相关测试与计数链路的二次深挖后修订。

## 概述

5 项修复仍然属于局部改动，不涉及公开函数签名、返回结构或算法语义变更。当前代码已经完成原始 4 项修复，但二次核验确认还存在 2 组仅影响测试覆盖度的低风险补强点：

1. `accumulate_busy_hours` 的坏参测试只锁定了 `start_time`，尚未锁定对称的 `end_time` 分支；
2. 零工时与效率回退测试已锁定“区间内部后移”“`None` / `0.0` / 异常上浮”，但尚未锁定“区间起点不后移”与“负值效率回退”。

因此本版 plan 从“只剩统一回归”调整为“补两组测试后，再统一回归”。

---

## 二次深挖核验结论（基于真实引用链）

### 1. `_build_order` 的真实消费点已核实

`_build_order` 只被两条链路消费：

1. `core/services/scheduler/schedule_optimizer_steps.py::_get_cached_multi_start_order()` 通过 `build_order` 回调在 `_run_multi_start()` 中取排序结果并做单轮本地缓存；
2. `core/services/scheduler/schedule_optimizer.py::optimize_schedule()` 收口阶段用它回填 `best_order`。

`due_date` / `ready_date` / `created_at` 的排序前解析已经完全收口在 `core/algorithms/greedy/scheduler.py::build_batch_sort_inputs()` 及其 `_parse_due_date_for_sort()`、`_parse_ready_date_for_sort()`、`_parse_created_at_for_sort()` 三个助手里。因此删除旧的重复解析闭包仍然是等价删除，不存在隐藏的第三条排序输入路径。

### 2. `accumulate_busy_hours()` 的真实调用链已核实

`accumulate_busy_hours()` 当前只有 3 处调用：

1. `core/algorithms/greedy/scheduler.py` 的 `seed_results` 预热路径；
2. `core/algorithms/greedy/dispatch/batch_order.py` 的正式派工路径；
3. `core/algorithms/greedy/dispatch/sgs.py` 的正式派工路径。

其中：

- `scheduler.py` 在调用前已同时检查 `sr.start_time` / `sr.end_time` 非空，且显式检查二者都是 `datetime`；
- `batch_order.py` 与 `sgs.py` 两处虽然只做 truthy 检查，但它们消费的是调度过程内部生成的 `ScheduleResult`，上游来源要么是 `_schedule_internal()`，要么是外协排程构造出来的 `ScheduleResult`，时间字段本身就是 `datetime`。

因此这里增加类型断言的意义是把共享助手的契约显式化，而不是修补一个已经确认存在的调用点 BUG。基于这个事实，补测应聚焦“共享助手自己的双参数契约”，而不是再去扩写 `seed_results` 入口用例。

### 3. `find_overlap_shift_end()` 的真实使用点已核实

`find_overlap_shift_end()` 只被两处直接调用：

1. `core/algorithms/greedy/downtime.py::find_earliest_available_start()`；
2. `core/algorithms/greedy/internal_slot.py::estimate_internal_slot()`。

当前新增测试已经锁定“零工时落在占用区间内部时仍应后移”的语义，但尚未锁定同一半开区间规则下的另一个关键边界：当 `start == end == segment_start` 时，由于 `s >= end` 提前成立，应当“不后移”。这个缺口需要补齐。

### 4. `_resolve_efficiency()` 的传播链已核实

`_resolve_efficiency()` 只在 `core/algorithms/greedy/internal_slot.py::estimate_internal_slot()` 内部使用。`efficiency_fallback_used` 这个标记随后只在 `core/algorithms/greedy/scheduler.py::_schedule_internal()` 的正式排产路径里计入 `internal_efficiency_fallback_count`。

现有测试已经覆盖：

- `get_efficiency()` 返回 `None`；
- 返回 `0.0`；
- 抛异常直接上浮；
- `float("inf")` 场景下正式计数与评分探测路径的隔离。

但实现条件是 `eff <= 0`，因此负值效率（如 `-1.0`）与 `0.0` 属于同一回退分支；目前没有独立测试锁住这个语义，需要补齐。

## 同事 review 意见处置

### ① 脚本式回归在 `pytest` 下“跑不到断言” — **不成立，无需修改**

同事认为 `regression_downtime_overlap_skips_invalid_segments.py` 等脚本放进 `pytest` 命令会跑不到断言。但 `tests/conftest.py` 的 `pytest_collect_file` + `RegressionMainFile` + `RegressionMainItem` 自定义收集器会识别“文件名匹配 `regression_*.py`、包含 `def main(`、且不含 `def test_`”的脚本，并在 `runtest()` 里直接调用 `main()`。因此 `python -m pytest tests/regression_xxx.py` 确实会执行 `main()` 内的全部断言。此担忧为误报，验证命令不需要改。

### ② 修复 2 缺少坏参直测 — **接受，补齐为对称契约测试**

同事指出当前测试只覆盖了 `update_machine_last_state` 的 `TypeError` 分支，没有直测 `accumulate_busy_hours` 的新断言。这个方向成立；二次深挖后进一步确认：`accumulate_busy_hours` 对 `start_time` 与 `end_time` 的约束是完全对称的，因此补测不应只测一半，而应把两个参数的坏参路径都锁住。

### ③ 零工时 + 占用/停机重叠语义 — **接受，并补齐边界两侧**

同事指出零工时工序落在已占用区间内时，`find_overlap_shift_end` 仍会将其推后。经代码核验：当 `start == end` 且 `segment_start < start < segment_end` 时，`end <= s or start >= e` 都不成立，会被判定为重叠并后移。这条语义必须保留测试。

二次深挖后进一步确认：同一半开区间规则下，`start == end == segment_start` 时会因为 `s >= end` 提前成立而不后移。因此本版 plan 不仅保留“区间内部后移”用例，还追加“区间起点不后移”用例，避免只锁一侧。

### ④ 修复 1 验证覆盖度 — **覆盖真实链路，仍不额外补 `due_date` / `ready_date` 排序回归**

`test_algorithm_date_boundary_split.py` 确实走到了真实 `optimize_schedule()` → `_build_order()` → `build_batch_sort_inputs()` → `StrategyFactory.sort()` 链路；`test_optimizer_build_order_once_per_strategy.py` 则锁定了 `_run_multi_start()` 内部的本地缓存语义。二次深挖后确认 `_build_order()` 没有隐藏调用点，因此“仅删除等价重复解析，暂不额外补优化器级 `due_date` / `ready_date` 排序回归”的结论继续成立。

### ⑤ 二次深挖后新增的低风险补强项 — **接受，补两处，不扩散到业务代码**

本轮新增的补强都只涉及测试，不新增业务逻辑：

1. 在修复 2 中把 `end_time` 的对称坏参路径补齐；
2. 在修复 5 中补齐“零工时区间起点不后移”与“负值效率回退”两个语义锁点。

这两处都不要求改动生产代码，只要求把现有实现的真实契约写进测试。

## 修复 1+3：消除 `_build_order` 双重解析 + 延迟导入

**文件**：`core/services/scheduler/schedule_optimizer.py`

**做法**：
1. 删除 `_build_order()` 周边历史遗留的 `_parse_due_date` / `_parse_ready_date` 重复解析闭包；
2. 删除 `_build_order()` 内部对 `batch_for_sort` 的二次覆盖循环；
3. 保持 `_build_order()` 只做一件事：调用 `build_batch_sort_inputs()` 生成排序输入，再交给 `StrategyFactory.create(...).sort(...)`。

删除后 `_build_order()` 应保持为：

```python
def _build_order(strategy0: SortStrategy, params: Dict[str, Any]) -> List[str]:
    batch_for_sort = build_batch_sort_inputs(
        normalized_batches_for_sort,
        strict_mode=bool(strict_mode),
        strategy=strategy0,
    )
    sorter0 = StrategyFactory.create(strategy0, **(params or {}))
    return [item.batch_id for item in sorter0.sort(batch_for_sort, base_date=start_dt.date())]
```

**验证说明**：
- `test_algorithm_date_boundary_split.py` 覆盖真实 `optimize_schedule()` → `_build_order()` → `build_batch_sort_inputs()` 链路；
- `test_optimizer_build_order_once_per_strategy.py` 覆盖 `_run_multi_start()` 经 `_get_cached_multi_start_order()` 消费 `build_order` 回调的缓存语义；
- 二次深挖已确认 `_build_order()` 不存在隐藏第三调用点，因此这里仍按“等价删除”处理。

**验证**：`python -m pytest tests/test_algorithm_date_boundary_split.py tests/test_optimizer_build_order_once_per_strategy.py -q`
