# 算法层统一估算与派工重构 实施 plan

> **执行方式**：优先使用 `subagent-driven-development`；如果当前环境不适合子代理或用户要求当前会话直接执行，则使用 `executing-plans`。

**目标**：在不改变服务主链公开函数签名、返回结构和落库口径的前提下，统一内部工序估算，实现 `dispatch_sgs()` 内部工序评分估算与真实执行估算共用同一估算器，并收口时间线、运行期状态、算法层日期边界与优化器重复工作；同时明确保留 `GreedyScheduler.schedule()` 的当前 raw 输入兼容边界，不在本轮里静默收窄。

**总体做法**：以 `ScheduleService._run_schedule_impl()` 到 `GreedyScheduler.schedule()` 的真实主链为准绳，先把 `scheduler.py` 与 `auto_assign.py` 的内部工序估算收口到 `internal_slot.py`，并在这里一并消除固定 `200` 次避让硬上限。随后拆分 `dispatch_sgs()`，把其内部评分显式升级到与执行路径同一套估算器，同时把“工时合法性”和“资源可得性”拆成两段式判别，避免评分语义漂移。最后完成时间线有序化、运行期状态收口、算法层日期来源收口与优化器微优化；`schedule_summary.py` 与报表层日期函数本轮不扩散处理。

**涉及技术 / 模块**：`core/services/scheduler/schedule_service.py`、`schedule_input_collector.py`、`schedule_orchestrator.py`、`schedule_optimizer.py`、`schedule_optimizer_steps.py`、`core/services/scheduler/schedule_input_builder.py`、`core/services/common/strict_parse.py`、`core/algorithms/greedy/scheduler.py`、`core/algorithms/greedy/dispatch/sgs.py`、`core/algorithms/greedy/dispatch/batch_order.py`、`core/algorithms/greedy/auto_assign.py`、`core/algorithms/greedy/downtime.py`、`core/algorithms/greedy/date_parsers.py`、`core/algorithms/evaluation.py`、`core/algorithms/dispatch_rules.py`、`core/algorithms/ortools_bottleneck.py`、相关回归测试。

6. 任务 5 想真正吃到 `build_order()` 缓存收益，还依赖任务 4 先把 `GreedyScheduler.schedule()` 的“`batch_order_override` 全覆盖时跳过默认排序”快速路径落地；否则外层虽缓存了 `order`，每次调度内部仍会重复跑一轮默认排序与日期解析。

## 实施约束

- 不新增第二套排产链路，不引入新的平行算法模式，不预先引入“快速估算器 / 精确估算器”双模式。
- 不新增广域 `except Exception: pass`，不新增静默回退。真正的程序错误必须显式暴露；只有业务上明确允许的“不可估算候选”才允许构造降级评分键。
- 新引入需要 `pytest` 夹具、`monkeypatch`、参数化或细粒度调用参数断言的用例，一律使用 `tests/test_*.py` 命名；`tests/regression_*.py` 仅用于现有 `main()` 风格回归，不得在同一文件里混入 `def test_...`。
- 若需要补现有 `tests/regression_*.py` 的调用参数断言，继续保持 `main()` 风格，并用 `unittest.mock`、显式包装或辅助计数替代 `monkeypatch` 夹具；不得为了图省事把 `tests/conftest.py` 的收集规则一并改掉。
- `internal_slot.py` 必须以当前 `_schedule_internal()` / `auto_assign_internal_resources()` 这套“同时检查三条时间线并反复避让”的算法为基准，不能退回 `find_earliest_available_start()` 的顺序链式近似。
- `estimate_internal_slot()` 只接受调用方已确认的非空 `machine_id`、`operator_id`；缺资源判断、自动分配探测与“无候选”分支都必须在调用前完成，估算器本身不再承担缺资源判别。
- 服务主链热路径直接消费 `OpForScheduleAlgo.setup_hours`、`unit_hours` 这两个已规范化字段；不得在 `scheduler.py`、`auto_assign.py`、`sgs.py` 再对这两个字段调用 `parse_required_float()`。但为了保留 `GreedyScheduler.schedule()` 的 raw 输入兼容边界，本轮只在“贪心执行 / 自动分配 / `dispatch_sgs()` 内部评分”这三条内部工序链路里，把内部总工时读取统一收口到 `validate_internal_hours(op, batch)`；`core/algorithms/ortools_bottleneck.py` 的 warm-start 瓶颈聚合公式继续作为范围外独立实现保留，不纳入本轮完成判定。并且该入口对已规范化 `float` 值必须保持透明，不得额外引入第二层清洗分支。
- `validate_internal_hours()` 必须显式保留当前直接调用入口的 `getattr(..., 0) or 0` 兼容语义：`setup_hours`、`unit_hours`、`quantity` 中，`True` 继续沿用当前 direct-call 执行语义参与 `float(...)` 转换（即 `True -> 1.0`）；除 `True` 外，凡能安全求值为假值的原始输入（包括缺字段、`None`、真正空串 `""`、`False`、空容器等）都先按 `0.0` 参与统一公式；仅含空白但非空的字符串（如 `"   "`）不在兼容边界内，必须继续作为坏值走 `float(...)` / `ValueError`；若布尔求值本身抛异常，则按非预期程序错误直接上浮。该函数只允许把**标量数值转换失败**、可定位 `repr` 的坏值、以及 `total_hours_base` 非有限 / 负值重写为 `ValueError`；对象属性实现异常、访问期程序错误等非预期异常必须直接上浮，不得被统一业务化。其中“缺字段兼容”仅指属性本身不存在；若属性已定义但取值过程抛异常（包括 `property`、描述符或自定义 `__getattribute__` / `__getattr__` 内部异常），不得按缺字段吞掉并压成 `0.0`。若实现上无法无歧义地区分“缺字段”与访问期 `AttributeError`，以保守暴露为准，必须直接上浮；并且不得调用 `strict_parse.py` 中的 `_parse_finite_float()`，也不得复用 `parse_required_float()`，以免把 direct-call 兼容语义意外收紧。
- 固定 `200` 次硬上限必须删除；任务 1 只允许使用基于三条时间线有效片段总数的显式上界，并带保底下限（例如 `max(N_machine + N_operator + N_downtime + 1, 10)`），而且必须由 `estimate_internal_slot()` 在**每次调用入口**按当前 `machine_timeline`、`operator_timeline` 与停机片段状态现场计算。若仍越过这条可证明上界，应视为实现错误直接抛 `RuntimeError`；不得把这类错误折叠为 `abort_after_hit` 或任何业务失败标志。
- `runtime_state.py` 中的共享状态更新函数不包广域 `try/except`；调用方也不再为了简单 `dict` 加法与最近结束时间更新保留吞异常分支。
- 共享运行期状态更新要么在 `results.append(...)`、`batch_progress` 推进与 `scheduled_count` 累加之前执行，要么实现为对当前已守卫输入无异常的纯字典更新；不得只删除内层吞异常而保留“结果已写入、状态更新抛错、外层再记失败”的部分提交窗口。
- `runtime_state.py` 的 seed 预热模式必须显式保留“`op_type_name` 为空时不推进 `last_end_by_machine` / `last_op_type_by_machine`”的现有语义；派工成功模式继续保持“`last_end_by_machine` 条件更新、`last_op_type_by_machine` 仅在非空时直接更新”。
- seed 预热路径在调用共享状态更新前，必须显式确认 `start_time`、`end_time` 已是 `datetime`（或由共享函数入口做等价的轻量类型断言）；不得只凭 truthy 守卫继续做时间差计算。
- `occupy_resource()` 继续留在 `scheduler.py` 的 seed 预热与 `_schedule_internal()` 成功落点处；`internal_slot.py` 只负责估算，`runtime_state.py` 只负责运行期统计，不搬运资源时间线占用职责。
- `estimate_internal_slot()` 对 `machine_timeline`、`operator_timeline` 与 `machine_downtimes` 保持只读；不得在估算器内部原地排序、追加、删除或改写传入片段。任何一次性规范化只能发生在 `GreedyScheduler.schedule()` 等调用方入口，不能借估算过程悄悄修改输入。
- `dispatch_sgs()` 的内部评分顺序必须固定为：**先 `validate_internal_hours()` 判定工时是否合法，再做资源补全 / `probe_only=True` 自动分配探测，最后才调用 `estimate_internal_slot()`**。不得让 `probe_only` 返回的 `None` 同时承担“工时坏值”和“缺资源 / 无候选”两类根因。
- `probe_only=True` 的自动分配探测不得保留双层广域异常吞没；返回形状错误、调用契约错误或内部程序异常都必须直接暴露，`chosen is None` 才是唯一允许的“缺资源且无法补全”正常分支。
- `estimate_internal_slot()` 中由 `abort_after` 命中的早停，必须仅通过 `abort_after_hit=True` 表示“当前候选无法优于已知 best_end”；它与 `blocked_by_window` 以及避让上界越界是三类不同语义，不得混用。
- `abort_after` 的比较语义必须与当前 `auto_assign.py` 保持一致：只在 `adjust_to_working_time()` 之后检查，首次进入和每轮避让后的再次调整各检查一次；比较对象固定为 `earliest`，比较符号固定为严格大于 `>`，不得改成比较 `end`，也不得改成 `>=`。
- 任务 4 统一算法层纯日期函数与算法入口消费的日期字段边界：`due_date`、`ready_date`、`created_at` 都要显式声明 strict / non-strict 包装；其中 `created_at` 必须通过 `core/services/common/strict_parse.py` 中共享的严格日期时间解析器收口，且只在 FIFO 相关排序路径真正消费该字段时才允许 strict 硬失败。
- strict 日期时间解析必须显式复刻当前非 strict 兼容矩阵：接受 `datetime`、`date`（按当日 `00:00:00` 归一）、`YYYY-MM-DD`、`YYYY-MM-DD HH:MM`、`YYYY-MM-DD HH:MM:SS`，并在解析前统一 `/`、`T`、全角 `：`；不得在 FIFO 相关路径额外收紧已有可接受输入。
- `GreedyScheduler.schedule()` 中若 `batch_order_override` 经规范化 / 去重 / 过滤后已完整覆盖全部规范化批次，必须直接以 override 生成 `batch_order` 并跳过默认 `sorter.sort(...)`、`BatchForSort` 构造以及一切仅供默认排序消费的字段解析；因此该路径不得再因仅供默认排序使用的 `due_date` / `created_at` 坏值提前 strict 失败，但 `ready_date` 仍因 `batch_progress` 初始化继续按真实消费路径校验。只有 override 未覆盖全部批次、需要按策略补齐漏批时，才允许构造默认排序输入；其中 `due_date` 继续沿当前策略的默认排序语义参与，`created_at` 仅在 FIFO 路径真正消费。
- `optimize_schedule()` 中与 `created_at` 严格边界有关的启用条件必须收紧到**单次实际构造排序器的当前策略级**，而不是停在整轮 `keys` 粒度；即使本轮 `keys` 含 `fifo`，非 FIFO 的当前 `build_order()` 尝试也不得提前 strict 解析 `created_at`。实现上可以把 strict 包装延迟到 `_build_order(strat, params0)` 或等价的当前策略级构造点，而不是在整轮共用的 `batch_for_sort` 阶段提前解析。
- `schedule_optimizer.py` 构造 `batch_for_sort` / `order` 时，`BatchForSort.batch_id` 必须直接复用 `GreedyScheduler.schedule()` 入口同一共享规范化函数（或等价的唯一事实源实现），与其完整保持一致（含字典键优先、字段回退、空值处理与冲突处理）；不得各自手写“看起来等价”的清洗逻辑，也不得直接信任未清洗的 `b.batch_id` 原始字段。
- `evaluation.py` 中 `_parse_due_date_state()` 允许保留“坏值标记”包装，但内部解析必须委托共享 `parse_date()`，不得保留第二套 `_parse_due_date()` 实现。
- `estimate_internal_slot()` 只通过 `efficiency_fallback_used` 暴露效率回退事实；`internal_efficiency_fallback_count` 只能由正式 `_schedule_internal()` 路径根据该标记累加，评分与 `probe_only` 探测路径不得污染该计数。
- 效率边界语义必须显式写死：`get_efficiency()` 返回 `None`、`0.0` / 其他 `<=0` 值、或非有限值时，统一按业务坏值回退到 `1.0` 并置 `efficiency_fallback_used=True`；返回有限正值时按实际值使用，其中 `>1.0` 继续允许缩短工时；`get_efficiency()` 直接抛异常视为程序错误上浮，不得回退。
- `ready_date` 下界调整在 strict / non-strict 两种模式下都不再在日历异常时静默回退到 `dt0`；一旦 `adjust_to_working_time()` 对已验证合法的 `ready_date` 抛错，允许整次排产沿现有调用链失败上浮，不做当前批次降级续排。
- 验证算法层效率边界时，测试必须直接注入替身日历或替身 `get_efficiency()`；不得经由 `CalendarEngine.get_efficiency()` 间接验证 `None`、`0.0` / 其他 `<=0` 值、非有限值或异常路径，否则会被引擎层抄底抹平。
- 本轮不扩到 `core/services/scheduler/schedule_summary.py` 与 `core/services/report/calculations.py` 的日期函数；若用户后续要做全链路日期单一事实源，另开后续 plan。
- `downtime.py` 的有序插入必须兼容 Python 3.8；若使用 `bisect.insort`，只能依赖 `(start, end)` 元组自然顺序，不能使用 `key=` 参数。
- 任务 3 引入“有序时间线”前提时，不得只依赖服务层停机排序保证；为保持 `GreedyScheduler.schedule()` 的 direct-call 兼容边界，必须在 `GreedyScheduler.schedule()` 入口对外部传入的 `machine_downtimes` 做一次性轻量 `defensive copy + sort` 规范化，再让 `internal_slot.py`、`auto_assign.py`、`dispatch_sgs.py` 依赖有序前提；不得把排序放进 `estimate_internal_slot()` 这类高频入口重复执行。
- 性能对照只按同一命令“预热 1 次 + 记录 3 次”后的中位数比较；单次墙钟波动只作为人工复核提示，不作为直接硬暂停条件。
- `tests/smoke_phase10_sgs_auto_assign.py` 以功能回归与端到端主链健康核验为主；其耗时只作为人工复核提示，不单独构成性能硬暂停。真正的 `30%` 性能硬门槛由 `tests/benchmark_sgs_large_resource_pool.py` 的中位数对照承担。
- `tests/smoke_phase10_sgs_auto_assign.py` 当前每次运行都会覆盖固定 `report.md`；因此“预热 1 次 + 记录 3 次”阶段必须在每次运行后立即把 `总耗时` 与关键摘要摘录或复制到 `before/after` 留痕文件，不能依赖脚本自动保留历史版本。
- `radon` 复杂度命令属于环境前提；执行前需先确认本地可 `import radon`。若环境缺失，只能先补环境或把复杂度复核标记为待补，不能把复杂度结果写成已完成。`ruff` 同理：仅在环境前提满足且实际执行失败时，才构成暂停条件；环境缺失只记录为待补，不视为本轮实现失败。
- 本轮“无静默回退”验收不覆盖 `scheduler.py` 入口 `_norm_id()` 与 `auto_assign.py` 中 `pair_rank` 的历史兼容兜底；除非实施中主动把它们显式纳入并补回归，否则它们只能按范围外已知残留记录，不能在收尾口径里写成“已清零”。
- 完成判定中的“无静默回退”只覆盖本轮触达的算法热路径与优化器路径；`schedule_params.py` 的历史遗留静默吞错若未纳入实现，不得被写成“已清零”。

## 文件职责映射

### 新建文件
- `core/algorithms/greedy/internal_slot.py`
  - 职责：内部工序可行时段估算的单一事实源。
  - 只做：给定 `op / batch / machine_id / operator_id / base_time / prev_end / timelines / downtime / calendar`，计算 `start_time`、`end_time`、`total_hours`、`changeover_penalty`、`blocked_by_window`、`abort_after_hit`、`efficiency_fallback_used`。

  - 额外导出：`validate_internal_hours(op, batch) -> float`，统一做 raw 输入安全的内部总工时预检；它既服务服务主链，也服务 direct `GreedyScheduler.schedule()` 调用，并且对已规范化 `float` 值保持透明。
  - 不做：字符串清洗、计数器更新、错误文案拼接、资源时间线占用、结果落库。
- `core/algorithms/greedy/dispatch/runtime_state.py`
  - 职责：统一维护派工成功后的运行期状态更新。
  - 只做：累计忙时、更新设备最近工种 / 最近结束时间，并显式区分 seed 预热与正式派工两种状态语义；共享入口只接受已守卫的 `datetime` 起止时间。
