## 背景

本文件是在以下两份既有计划基础上形成的合并增补版：

- `.limcode/plans/静默回退与过度防御审查问题修复计划_执行版.plan.md`
- `.limcode/plans/静默回退与过度防御审查问题修复计划.plan.md`

上一版共同项合并稿只收了重复项，但遗漏了用户明确关心的三类内容：

1. **静默降级**：页面继续可用，但真实状态被隐藏；
2. **过度兜底**：已经有明确输入契约的代码，仍然用宽泛回退把问题吃掉；
3. **防御性编程**：需要区分“确实应该保留的防御”和“已经确认多余的防御”，不能一刀切。

因此，本版不再只收“共同项”，而是把两份计划与三轮复核后的结论一起收口成一份可执行文档：

- 既写**本轮必须修复**的静默降级与过度兜底问题；
- 也写**本轮保持现状**的合理防御项；
- 同时保留执行顺序、回归清单、人工验收与收口标准。

---

## 本版使用原则

1. 先修会**误导用户、掩盖真实状态、吞掉关键信号**的问题。
2. 对“防御性编程”不做情绪化清理，只改三轮复核后**确认成立**的过度防御项。
3. 对已经确认有合理边界意义的防御代码，本轮**明确保留**，不顺手重构。
4. 不混入无关样式微调、命名整理、格式化噪音。
5. 原始两份计划文件继续保留；本文件只是统一执行口径，不回写覆盖原文件。

## 修复 1：请求级日志链断裂（`SF1` / `request-scope-app-logger-missing`）

### 目标
让请求链中所有通过 `logger=getattr(g, 'app_logger', None)` 构造的服务，不再拿到 `None`。

### 涉及文件
- `web/bootstrap/factory.py`

### 统一改法
在 `web/bootstrap/factory.py::_open_db()` 中，静态资源与健康检查短路逻辑之后、`if 'db' not in g:` 之前，独立补一行：

```python
        g.app_logger = current_app.logger
        if 'db' not in g:
            g.db = get_connection(app.config['DATABASE_PATH'])
            g.op_logger = OperationLogger(g.db, logger=current_app.logger)
```

### 边界约束
- 不要把 `g.app_logger` 塞进 `if 'db' not in g:` 分支。
- 不要这一轮大面积把所有 `getattr(g, 'app_logger', None)` 改成 `current_app.logger`。
- 不要顺手重构所有服务构造函数。

### 新增回归
- `tests/regression_request_scope_app_logger_binding.py`

建议断言：
1. 构建应用并进入真实请求上下文。
2. 执行 `app.preprocess_request()` 后：
   - `g.app_logger is current_app.logger`
   - `g.db` 已存在
   - `g.op_logger` 已存在
3. 至少走一条依赖 `logger=getattr(g, 'app_logger', None)` 的路由链，确认传入服务的 `logger` 不再为 `None`。

## 修复 6：删除确认成立的过度兜底（`OD3`）

def _count_internal_ops(ops: List[Any]) -> int:

## 保持现状 / 接受风险说明

### `OD4-REV`
- `launcher.py` 中只有 `str(cfg_log_dir).strip()` 这一类个别点位可确认多余；
- 其余 `int()` 与日志相关保护仍有边界意义；
- 本轮不做批量清理，避免把启动链路风险引进来。

### `OD5-REV`
- `plugins.py` 同时管理数据库连接、插件加载、日志落库三个故障域；
- 可精简的主要是个别对日志调用的包裹；
- 整体结构不宜粗暴折成一个顶层 `try`。

### `OD6-REV`
- `request` 与 `current_app` 是 `LocalProxy`；
- 上下文异常时会抛 `RuntimeError`，而不是 `AttributeError`；
- 因此 `getattr(obj, attr, default)` 的默认值机制并不能覆盖这类失败；
- 相关 `try/except` 本轮必须保留。

### `SF4`
- `process_excel_routes.py` 的确认阶段理论上仍有部分失败风险；
- 但当前基线签名机制已经显著降低发生概率；
- 本轮不把事务语义改造混进来，只在计划中留痕。

### `SF5`
- `ui_mode.py` 的多个回退分支属于“页面可用性优先”的有意降级；
- 关键降级路径已有 warning 或一次性提示覆盖；
- 本轮不削弱这些保活逻辑。
