## 二、统一执行规则

### 1. 每个任务都按同一顺序推进
1. 先确认本任务涉及的文件、模块和已有测试护栏。
2. 先补或整理本任务的定向验证；如果已有回归测试，就先跑一次记录当前基线。
3. 只做本任务定义范围内的最小实现，不顺手扩写无关重构。
4. 先跑本任务定向回归，再跑受影响的连带回归。
5. 回归通过后，再更新 `开发文档/技术债务治理台账.md`、相关文档和白名单。
6. 如果本任务引入了新目录规则、新门禁规则或新入口规则，必须同步更新测试与文档，不能只改代码。

### 2. 每个任务的变更粒度要求
1. 一次提交只解决一个任务或一个任务中的一个明确子批次。
2. 同一批次里允许同时改代码、测试、文档，但必须围绕同一条治理目标。
3. 不把“目录迁移”“类型收紧”“行为改造”“模板收口”混在同一小批次里一起完成。

### 3. 每个任务都必须写清的东西
1. 具体文件路径。
2. 新建文件负责什么，旧文件保留什么。
3. 哪些函数、对象、数据结构要被搬动或收口。
4. 要跑哪些测试。
5. 更新哪些文档或治理台账。

### 4. 全局禁止项
1. 禁止出现 `TODO`、`TBD`、`后面再补`、`视情况处理`、`类似前面`、`自行判断` 这一类占位语。
2. 禁止新增临时兼容桥、签名探测、`TypeError` 后重试、静默兜底解析。
3. 禁止新增 `except Exception` 后直接 `pass`、`return None`、`return ""`、`return False`、`return []`、`return {}`、`return 0`、`return ()`、`continue` 这类不可观测静默回退；确需保留的历史点必须进入治理台账、专项门禁与退出条件。
4. 禁止新增整页复制模板、第二套降级事实源、第二套路由注册壳、第二套静态资源版本规则。
5. 禁止使用本仓库尚未声明依赖支持的验证参数。
6. 禁止在根目录继续增加新的 `tests/regression_*.py`、`smoke_*.py`、`run_*.py`、`check_*.py`、`verify_*.py` 文件。

### 5. 每个批次结束时的固定收尾动作
1. 更新 `开发文档/技术债务治理台账.md` 的状态、负责人、退出条件。
2. 如白名单已不再需要，立即从架构门禁测试文件中移除；任务 9 迁移前对应 `tests/test_architecture_fitness.py`，迁移后对应 `tests/architecture/test_architecture_fitness.py`。
3. 如公开入口、模板、脚本初始化协议有变化，立即更新 `README.md`、`开发文档/README.md` 或 `开发文档/系统速查表.md`。
4. 在 `evidence/README.md` 或阶段留痕中记录本批次跑过的关键验证入口。

## 三、关键结构决策

### 1. 路由分组采用“非同名子包”策略
不创建 `web/routes/scheduler/`、`web/routes/process/`、`web/routes/personnel/`、`web/routes/equipment/`、`web/routes/system/` 这类会与现有同名文件冲突的目录。

统一采用：
- `web/routes/domains/scheduler/`
- `web/routes/domains/process/`
- `web/routes/domains/personnel/`
- `web/routes/domains/equipment/`
- `web/routes/domains/system/`

现有 `web/routes/scheduler.py`、`process.py`、`personnel.py`、`equipment.py`、`system.py` 仅保留为薄门面或注册壳，转发到 `web/routes/domains/**`，不再承载真实页面逻辑。

### 2. 调度服务目录采用子包化，新文件统一进入对应子包
统一重组到：
- `core/services/scheduler/config/`
- `core/services/scheduler/run/`
- `core/services/scheduler/batch/`
- `core/services/scheduler/dispatch/`
- `core/services/scheduler/gantt/`
- `core/services/scheduler/summary/`
- `core/services/scheduler/calendar/`

因此：
- 配置事实源文件统一落在 `core/services/scheduler/config/config_field_spec.py`
- 运行上下文文件统一落在 `core/services/scheduler/run/run_command.py`、`core/services/scheduler/run/run_context.py`
- 根层只保留少量门面和明确对外入口，不再混放新文件。

### 3. 降级状态只允许一个事实源
必须复用：
- `core/services/common/degradation.py`
- `core/services/scheduler/summary/schedule_summary_degradation.py`

如果后续新增 `core/services/scheduler/summary/degradation_report.py`，它只能是汇总装配层，不得定义新的事件模型、事件码来源或第二套统计对象。

### 4. Excel 页面统一采用“页面流程 + 领域钩子”
允许新增：
- `web/pageflows/excel_import_flow.py`
- `web/pageflows/excel_import_hooks.py`

不引入宽泛的 `ExcelImportController`。公共部分只抽“上传 → 预览 → 确认”骨架；各页面特有的模式限制、基线校验、结果渲染、导出逻辑仍保留在领域钩子中。

### 5. 前端初始化采用 DOM 协议，不再增加新的全局总注册表
统一采用：
- `static/js/page_boot.js`
- `data-page`
- `data-*`

不新增 `window.__APS_PAGES__`。现有 `window.__APS_COMMON__`、`window.__APS_GANTT__` 只允许收口，不允许继续扩张为更大的全局命名空间。
完成判定必须直接验证：`static/js/page_boot.js` 已被 `templates/base.html` 或等价公共模板接入、目标页面根节点真实产出 `data-page`，且新增页面初始化不再通过扩张 `window.__APS_COMMON__` / `window.__APS_GANTT__` 落地；若旧入口短期保留，必须在治理台账登记“已迁 / 未迁页面”清单。

### 6. 测试分层采用“架构 / 回归 / 冒烟 / 端到端 / 支撑”口径
统一采用：
- `tests/architecture/`
- `tests/regression/`
- `tests/smoke/`
- `tests/e2e/`
- `tests/support/`

原因：当前仓库已经以 `regression_*`、`smoke_*`、`run_*`、`check_*`、`verify_*` 为主要历史形态，先按执行语义分层，迁移成本最低，也最利于逐步收口根目录平铺文件。

### 7. 算法类型约束优先使用标准库能力
算法层如需引入 `Protocol`，优先使用 Python 3.8 标准库 `typing.Protocol` 与 `typing.runtime_checkable`；不把未声明的运行时依赖硬塞进生产代码。
`runtime_checkable` 只允许用于算法入口或边界适配层的运行时检查；主调度循环、评分循环、资源分配等热路径中禁止反复执行 `isinstance(..., Protocol)` 判定。

### 任务 1：建立工程基线、治理台账与统一质量门禁

**对应问题**
- `D3` / `F-开发依赖不完整-缺乏覆盖率工具`
- `D4` / `F-缺乏自动化持续集成流水线`
- `C7` / `arch-whitelist-governance`
- `E1` / `docs-root-readme-gap`

**本任务完成后系统状态**
- 仓库根有清晰入口文档。
- 质量检查有统一入口，不再本地一套、托管一套、文档一套。
- 白名单从“长期挂着”变成“有退出条件、有负责人”的治理台账。

**文件职责**
- `README.md`：仓库根入口，负责启动、测试、目录导航、文档导航、交付边界。
- `开发文档/README.md`：开发文档入口，负责把速查表、阶段留痕、版本方案和专题文档串成稳定索引。
- `audit/README.md`：审计入口，负责把人工审计结论按时间或主题收口成可导航索引。
- `scripts/run_quality_gate.py`：质量门禁总入口，串联静态检查、架构门禁（含 `web/bootstrap/` 基础门禁）、重点回归、文档一致性检查。
- `.github/workflows/quality.yml`：托管环境入口，只调用同一门禁脚本，不另写第二套逻辑。
- `开发文档/技术债务治理台账.md`：白名单、接受风险、批次完成状态、退出条件总台账。
- `requirements-dev.txt`：本地与托管环境的真实开发依赖声明。
- `.pre-commit-config.yaml`：本地提交钩子入口，必须复用统一的 `ruff` 调用方式与版本区间断言。
- `pyproject.toml`：测试目录递归门禁、最小 `pytest` 采集边界与 `ruff` 文件级例外的单一事实源。
- `tests/test_architecture_fitness.py`：架构门禁、治理台账存在性约束，并把 `web/bootstrap/` 与 `web/ui_mode.py` 纳入扫描范围或单文件专项门禁。
- `web/bootstrap/plugins.py`、`launcher.py`、`entrypoint.py`、`security.py`、`runtime_probe.py`：启动链静默回退清单与首批收口对象。
- `web/bootstrap/paths.py`：运行根目录解析入口，`runtime_base_dir()` 的静默回退必须纳入启动链治理台账与专项门禁。
- `tests/regression_runtime_probe_resolution.py`、`tests/test_win7_launcher_runtime_paths.py`、`tests/regression_startup_host_portfile.py`、`tests/regression_plugin_bootstrap_config_failure_visible.py`、`tests/regression_runtime_stop_cli.py`：任务 1 的启动链专项回归护栏，分别覆盖运行时探测、启动契约与运行路径、插件装配可观测性、停机链路。
