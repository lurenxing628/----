## 修复 1：SF1 —— 请求上下文中独立注入 `g.app_logger`

### 文件
`web/bootstrap/factory.py`

### 位置
`_open_db` 函数中，维护窗口检查之后、`if "db" not in g:` 之前。

### 真实问题链路

当前仓库中，路由层存在约 48 处 `getattr(g, "app_logger", None)` 的使用点。其典型调用链为：

- 路由层从 `g.app_logger` 取日志对象
- 服务层构造函数继续透传该对象
- 仓储层与安全日志辅助函数再继续使用该对象
- 当该值为 `None` 时，数据库错误日志、降级告警日志等会直接不落盘

因此，这不是单纯的风格问题，而是实际的可观测性断链。

### 修改方式

将 `g.app_logger = app.logger` 作为独立赋值，不再依赖数据库连接是否首次创建。

### 目标代码
```python
        g.app_logger = app.logger
        if "db" not in g:
            g.db = get_connection(app.config["DATABASE_PATH"])
            g.op_logger = OperationLogger(g.db, logger=app.logger)
```

### 说明

- `g.app_logger` 的注入职责，与 `g.db`/`g.op_logger` 的初始化职责拆开处理
- 不改变 `g.op_logger` 的现有初始化逻辑
- 不使用临时 `print` 作为验证手段

### 验证方法

1. 全局搜索 `getattr(g, "app_logger", None)`，确认现有调用点仍为既有规模
2. 补一条定点回归（建议文件名：`tests/regression_request_context_app_logger_injection.py`）：
   - 构造应用
   - 进入请求上下文
   - 调用 `app.preprocess_request()`
   - 断言 `g.app_logger is app.logger`
3. 结合现有页面或路由冒烟，确认未引入请求前置回归

## 修复 2：OD2 —— 资源池告警合并改为“归一化 + 追加”

### 文件
`core/services/scheduler/schedule_service.py`

### 位置
资源池构建之后、停机区间扩展之前的告警合并段。

### 真实前置条件

1. 在同一函数前文，`algo_warnings` 已先做过一次空值归一化
2. `build_resource_pool(...)` 的返回签名明确为 `(resource_pool, warnings_list)`，第二项是字符串列表
3. 现有相关回归已经覆盖“算法告警透传到结果摘要”的主链路

因此，这里的三层嵌套降级没有实际收益。

### 修改方式

把原来的三层嵌套改为“先归一化，再追加”：

```python
        pool_warnings = list(pool_warnings or [])
        if pool_warnings:
            algo_warnings.extend(pool_warnings)
```

### 本轮边界

- **只修改资源池告警并入 `algo_warnings` 的这一小段逻辑**
- **不动** 后面 `summary.warnings` 的摘要合并兜底逻辑；那一层面对的是更宽输入面，且已有现成回归覆盖

### 验证方法

1. 运行 `python tests/regression_schedule_summary_algo_warnings_union.py`
2. 运行 `python tests/regression_schedule_service_passes_algo_stats_to_summary.py`
3. 如本地允许，再补一条排产链路验证，确认资源池告警仍能出现在结果摘要中

## 修复 3：OD3 —— 删除 `_count_internal_ops` 的多余 `try/except`

### 文件
`web/routes/scheduler_batch_detail.py`

### 位置
第一个辅助函数 `_count_internal_ops`

### 真实依据

该函数的输入不是任意未约束对象，而是来自批次详情页主链路：

- `batch_detail()` 调用 `ScheduleService.list_batch_operations()`
- 服务层再调用 `operation_edit_service.list_batch_operations()`
- 仓储层 `BatchOperationRepository.list_by_batch()` 返回的是 `BatchOperation` 对象列表
- `BatchOperation.from_row()` 已在模型构造时把 `source` 归一化为小写字符串

因此，`_count_internal_ops` 面对的是已经标准化过的批次工序对象列表，整块 `try/except` 没有必要继续保留。

### 修改方式

去掉外围 `try/except`，保留原有计数逻辑：

```python
def _count_internal_ops(ops: List[Any]) -> int:
    cnt = 0
    for op in ops or []:
        src = (getattr(op, "source", "") or "").strip().lower()
        if src == SourceType.INTERNAL.value:
            cnt += 1
    return cnt
```

### 验证方法

1. **不再**把 `tests/regression_lazy_select_orphan_option.py` 作为本项主验证，因为该脚本不经过 `_count_internal_ops` 所在链路
2. 运行批次详情页面链路冒烟，优先使用：`python tests/smoke_web_phase0_6.py`
3. 若本地只做定点验证，则补一个仅覆盖批次详情页读取链路的小回归，至少验证 `GET /scheduler/batches/<batch_id>` 可正常渲染
