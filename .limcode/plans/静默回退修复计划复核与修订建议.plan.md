# 静默回退修复计划复核与修订建议

## 0. 复核结论

当前计划的**方向基本正确**，尤其是：

- 先修文档再动代码；
- 优先处理 `默认 1 天`、停机/冻结/资源池降级；
- 先补可观测性，再考虑 strict mode；
- 不把基础设施保护性回退和业务语义回退混在一起。

但如果把当前计划直接当作实施蓝图，存在 4 类明显问题：

1. **基线修订过早**：有些新增条目还没完成范围裁决，不宜直接把 `68/179` 改写成 `71/183`；
2. **覆盖不完整**：至少漏掉了 `PartService._save_template_no_tx`、`build_resource_pool`、`optimize_schedule`、`SupplierService` 等相关链路；
3. **若干任务的技术落点不成立或过于模糊**：例如“把 warning 合并失败写入 `result_summary_obj`”时对象尚未生成；`build_dispatch_key` 加参数会扩大公共接口面；`_safe_load_schedule_map` 增 meta 与现有摘要链重复；
4. **没有把调用链、schema、测试、UI/API 透传一起纳入**：容易出现“底层改了，但结果摘要/分析页/路由层没有接住”的半修状态。

## 1.2 Route / Supplier / External Group 链：当前计划覆盖不够

## 已验证的调用链

### 链 1：route 解析 -> 模板持久化

- `core/services/process/route_parser.py:277-296`
  - supplier 映射里：`float(getattr(supplier, "default_days", 1.0) or 1.0)`

### 链 2：供应商写侧 -> route 读侧

- `core/services/process/supplier_service.py:53-81`
  - `_validate_fields()`：`self._normalize_float(default_days, default=1.0)`
  - 非数字会直接变成 `1.0`

- `core/services/process/supplier_service.py:166-167`
  - `update()`：`updates["default_days"] = float(sdays or 1.0)`
- `core/services/process/supplier_excel_import_service.py:72-81`
  - Excel 导入反而是**严格的**：非法/<=0 会直接 `ValidationError`

### 结论

当前计划只覆盖了：

- `RouteParser.parse()`
- `RouteParser._build_supplier_map()`
- `ExternalGroupService._apply_separate_mode()`

但**漏掉了两个关键写侧/持久化点**：

1. `PartService._save_template_no_tx()`
2. `SupplierService._validate_fields()` + `create()` + `update()`

这会导致：

- 就算修了 `_build_supplier_map()`，`_save_template_no_tx()` 仍会把 `0.0` 洗回 `1.0`；
- 就算修了 route 解析链，供应商服务 API 写侧仍可能继续把坏 `default_days` 静默改成 `1.0`；
- Excel 导入是 strict，但 service API 是 permissive，**同一领域出现双重契约**。

### 对当前计划的修改建议

#### 新增 Phase 1 条目

- **P1-route-3**：`PartService._save_template_no_tx()` 的 `float(op.default_days or 1.0)` 拆解
  - compatible mode：保留现有写入策略，但显式 warning / audit
  - strict mode：后续拒绝写入无效 external default_days
- **P1-route-4**：`SupplierService` 写侧默认化链复核与收敛
  - `default_days` 非数字当前会静默归一成 `1.0`
  - 先补测试，再决定 compatible/strict 策略

#### 同时修改 P1-2 的目标表述

当前计划写成：

> `default_days = 0.0` 时应记 warning 而非静默改为 1.0

这句话还不够准确。应改为：

> **先区分“缺失值 / 非法值 / falsy 但合法业务值 / legacy 0 值”这四种来源，再决定 compatible mode 是否仍写 1.0、strict mode 是否拒绝。**

否则容易在未做契约确认的情况下，把原本可运行的链路直接改成不可排产链路。

## 1.4 warning / result_summary 链：当前计划的技术落点有明显问题

## 已验证的事实

- `schedule_service.py:425-443`
  - `algo_warnings` 合并到 `summary.warnings`
  - 最后一个 except 只打 logger
- `schedule_summary.py:396-398`
  - `freeze_warnings = _extract_freeze_warnings(summary)`
- `schedule_summary.py:455`
  - `warnings = list(all_warnings or [])`

### 当前计划的问题

#### 问题 E："把 warning 合并失败写进 `result_summary_obj`" 这一落点当前不存在

因为：

- merge failure 发生在 `build_result_summary()` 调用之前；
- 失败发生时，`result_summary_obj` 还没有生成；
- 因此当前计划这个动作在技术上无处安放。

### 更稳的修订建议

不要再依赖“必须先把 `algo_warnings` 并进 `summary.warnings`”。改成：

#### 方案建议：`build_result_summary()` 增加独立的 `algo_warnings` 输入

- `schedule_service._run_schedule_impl()`：
  - 继续尝试 merge（保持兼容）
  - 但无论 merge 是否成功，都把原始 `algo_warnings` 作为**独立参数**传给 `build_result_summary()`
- `build_result_summary()`：
  - `all_warnings = union(summary.warnings, algo_warnings)`
  - `freeze_warnings` 从这个 union 里提取
  - 若 merge 失败，可额外写一个结构化标记：
    - `algo.warning_merge_failed = True`
    - `algo.warning_merge_failed_count = len(algo_warnings)`

这样可以直接解决：

- 冻结窗口 warning 丢失
- 停机 warning 丢失
- 结果摘要与 analysis 页失真

### 对当前计划的修改建议

- **删除** “最终 except 分支应将 algo_warnings 数量和失败原因写入 result_summary_obj” 这句
- **改为新增任务**：
  - `build_result_summary(..., algo_warnings: Optional[List[str]] = None, merge_status: Optional[Dict[str, Any]] = None)`
  - 由 `build_result_summary()` 自己做 union 和结构化留痕

## 1.5 调度核心统计项：当前计划缺 carrier 设计，且有接口面扩大风险

### 当前计划的问题

#### 问题 F：`build_dispatch_key()` 加 `fallback_counter` 参数，不是最优方案

- `build_dispatch_key()` 是直接对外暴露的纯函数；
- 已有测试直接 import 它（如 `regression_dispatch_rules_nonfinite_proc_hours_safe.py`）；
- 给它加计数器参数，会扩大公共接口面，且所有直接调用都要考虑兼容。

#### 更优建议

- 不改 `build_dispatch_key()` 签名；
- 在 `dispatch_sgs()` 调用前后按分支记录：
  - `dispatch_key_proc_hours_fallback_count`
  - `dispatch_key_avg_proc_hours_fallback_count`

#### 问题 G：`dispatch_sgs()` / `auto_assign_internal_resources()` / `normalize_seed_results()` 的统计结果，没有明确 carrier

当前计划写法：

- `scheduler.meta`
- `score_stats`
- `return value or scheduler.meta`

但源码里：

- `GreedyScheduler` 没有 `meta` 字段；
- `schedule()` 返回值是固定四元组；
- `OptimizationOutcome` 也没有 `algo_stats` 字段。

#### 更优建议

优先采用**私有、最小扩展**的 carrier：

1. `GreedyScheduler.schedule()` 开头初始化 `self._last_algo_stats = {}`
2. `dispatch_sgs()` / `normalize_seed_results()` / `_auto_assign_internal_resources()` 写入这个私有字典
3. `schedule_optimizer_steps.py` 在每次 `scheduler.schedule()` 后，把 `scheduler._last_algo_stats` 拷贝到：
   - `attempts[*]["algo_stats"]`（如需 per-attempt）
   - `best["algo_stats"]`
4. `OptimizationOutcome` 新增可选字段 `algo_stats: Dict[str, Any] = field(default_factory=dict)`
5. `build_result_summary()` 再把它落到 `result_summary_obj["algo"]["fallback_counts"]`

这样：

- 不破坏 `GreedyScheduler.schedule()` 公开返回值；
- 只在服务层/优化层增加可选结构化字段；
- 和当前 analysis/result_summary 体系兼容。

### 对当前计划的修改建议

- **删除**：`build_dispatch_key() fallback_counter 参数`
- **删除**：`scheduler.meta`
- **改写** Phase 2 为：
  - `GreedyScheduler._last_algo_stats` 方案
  - `OptimizationOutcome.algo_stats` 方案
  - `result_summary_obj.algo.fallback_counts` 作为统一对外出口

#### 问题 K：三处都说“改成 logger”，但并不都适合 logger-first

- `_config_snapshot_dict()` 没有 `svc/logger` 参数；
- `_extract_freeze_warnings()` 更适合改成“逐项安全转换”，而不是一旦异常就整段丢掉；
- `_build_overdue_items()` 因为有 `svc`，最适合补 logger。

### 更优建议

#### `_config_snapshot_dict()`

不要直接说“记 warning”。改成：

- `return (snapshot_dict, issues)` 或
- 在 `build_result_summary()` 内部封装：
  - `config_snapshot`
  - `config_snapshot_incomplete`
  - `config_snapshot_issue_count`
  - `config_snapshot_issue_keys_sample`

#### `_extract_freeze_warnings()`

改成局部容错：

- `all_warnings` 不是 list 时先 `list(...)` 防御；
- 单个 `str(w)` 失败时跳过该项并计数；
- 不要因为一个坏 warning 把整个 `freeze_warnings` 清空。

#### `_build_overdue_items()`

- 这里补 logger 是合理的；
- 同时可加一个返回值 side flag，如 `invalid_due_warning_emit_failed`
