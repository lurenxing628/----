# 架构解耦整改实施计划

## 修订说明

本版已吸收对当前 plan 的三轮审核意见，重点修正如下：

1. **P4.1 与现状对齐**：`core/services/process/unit_excel/parser.py` 已存在 `Worksheet` 类型守卫，本项不再作为待实施代码改动，而转为“已完成、待状态同步”的收口项。
2. **P1 范围修正**：Excel 确认导入重复控制流的实际覆盖范围是 **12 个 confirm 路由**，除原计划中的 11 个文件外，还应纳入 `web/routes/excel_demo.py`，否则重复无法真正收口。
3. **P1.2 设计降噪**：不再设计一个试图包办页面回退的“大而全骨架函数”，而改为抽取“确认载荷解析、基线守卫、错误行收集与文案拼装、导入统计提取”四类稳定公共能力；各路由仍保留自己的页面回显函数与页面参数组装。
4. **P2 收口边界下探**：整改目标不只覆盖 `build_result_summary()` 与 `_build_result_summary_obj()`，还应继续下探到 `_algo_dict()`，否则参数面只会从一层转移到下一层。
5. **P2 截断策略建模修正**：`apply_summary_size_guard()` 当前是 **9 级渐进截断梯度**，不能用单一 `SizeTruncationPolicy` 对象表达，必须改成“梯度列表/层级策略”。
6. **P3 运行根目录语义保护**：共享入口必须保留 `runtime_base_dir(anchor_file=__file__)` 的现有行为；若直接把 `__file__` 固化到 `web/bootstrap/entrypoint.py`，源码运行时的根目录会从仓库根漂移到 `web/bootstrap/`。
7. **P5 补齐遗漏整改项**：三轮复核提出的“第二份 review 证据颗粒度偏粗”尚未进入原 plan，本版新增 P5.3 进行补证与文档回写。

### 阶段 P1：Excel 导入路由重复控制流收敛（U-1）

**目标**：将分散在 12 个 confirm 路由中的重复确认导入控制流收敛为稳定公共能力，同时保留各页面自己的回显与渲染职责。

**现状分析**：

当前稳定重复的 5 段控制流为：
1. `raw_rows_json` / `preview_baseline` 缺失校验 → 抛出 `ValidationError`
2. 预览数据 `json.loads()` + `rows not list` 保护（部分文件已用 `_parse_preview_rows_json()`，部分仍内联）
3. `preview_baseline_matches()` 不通过后的 `flash()` + 页面回退
4. 错误行筛选 + 错误示例拼装 + `flash()` + 页面回退
5. 导入结果统计提取（`new_count` / `update_count` / `skip_count` / `error_count`）+ `log_excel_import()` + `flash_import_result()`

### 阶段 P2：调度摘要上下文参数面收敛（U-2）

**目标**：将 `build_result_summary()`、`_build_result_summary_obj()`、`_algo_dict()` 的散装参数链收敛为具名上下文与具名状态对象，同时保持过渡期兼容。

**现状分析**：
- `build_result_summary()` 当前接收超宽关键字参数；
- `_build_result_summary_obj()` 继续向下传递大量参数；
- `_algo_dict()` 仍是下一层的宽接口；
- `apply_summary_size_guard()` 依赖 9 级硬编码渐进截断梯度；
- 当前实际主调用点在 `core/services/scheduler/schedule_orchestrator.py`，不是 `schedule_service.py`。

#### P2.1 新建 `schedule_summary_types.py`

在 `core/services/scheduler/` 下新增 `schedule_summary_types.py`，定义以下具名对象：

- `SummaryBuildContext`
- `RuntimeState`
- `WarningState`
- `FreezeState`
- `FallbackState`
- `AlgorithmSummaryState`（用于 `_algo_dict()` 的输入收敛）

`SummaryBuildContext` 负责承载外部输入；其余状态对象承载中间计算结果，避免继续用裸 `dict` 在模块间传递。

#### P2.2 重构 `build_result_summary()` 为“上下文优先、关键字兼容”

建议签名：

```python
def build_result_summary(
    svc,
    *,
    ctx: Optional[SummaryBuildContext] = None,
    **kwargs,
) -> Tuple[...]:
    if ctx is None:
        ctx = SummaryBuildContext(**kwargs)
    ...
```

要求：
- 兼容现有按关键字直接调用的测试与调用方；
- 新代码优先走 `ctx=`；
- 兼容层只做过渡，不作为长期主路径。

#### P2.3 将收敛继续下推到 `_build_result_summary_obj()` 与 `_algo_dict()`

仅让 `build_result_summary()` 接收 `ctx` 还不够，还需继续向下收敛：

- `_build_result_summary_obj()` 改为接收 `ctx + 状态对象`
- `_algo_dict()` 改为接收 `AlgorithmSummaryState`
- `_build_runtime_state()`、`_build_warning_state()`、`_build_freeze_state()`、`_build_fallback_state()` 的返回值由裸字典改为对应数据类

这样才能真正减少跨模块参数面，而不是把参数宽度平移到下一层。

#### P2.4 将“单一策略对象”改为“梯度策略列表”

`apply_summary_size_guard()` 当前语义是“逐级收缩直到体积达标”，因此应抽为分层策略，而不是单个固定阈值对象：

```python
@dataclass(frozen=True)
class TruncationTier:
    trace_limit: int
    warning_limit: int
    attempt_limit: int
    best_order_limit: Optional[int] = None
    selected_ids_limit: Optional[int] = None
    overdue_items_limit: Optional[int] = None


DEFAULT_TRUNCATION_TIERS: Tuple[TruncationTier, ...] = (
    TruncationTier(80, 50, 12),
    TruncationTier(20, 20, 12),
    ...
)
```

要求默认梯度与当前 9 级行为完全一致，仅改变组织方式，不改变对外语义。

#### P2.5 更新调用方与回归测试

- 更新 `core/services/scheduler/schedule_orchestrator.py` 的主调用点
- 更新直接调用 `build_result_summary()` 的回归测试（当前可搜索到 16 个直接调用点）

### 阶段 P3：应用入口双文件合并（U-3）

**目标**：消除 `app.py` 与 `app_new_ui.py` 的近乎逐行复制，同时保证运行根目录、启动契约和打包入口行为不变。

**现状分析**：

两文件的业务差异仅在：
1. `create_app_core()` 的 `ui_mode` 默认值不同；
2. 运行时契约里 `ui_mode` 的回退值不同；
3. 注释存在轻微差异。

但它们都依赖 `runtime_base_dir(anchor_file=__file__)` 取得源码运行根目录，因此共享入口设计必须保留“由壳文件传入锚点文件路径”的语义。

#### P3.1 新建共享入口模块 `web/bootstrap/entrypoint.py`

建议提取：

```python
def create_app_with_mode(ui_mode: str = "default") -> Flask:
    ...


def configure_runtime_contract(
    app: Flask,
    runtime_dir: str,
    host: str,
    port: int,
    owner: str,
    *,
    default_ui_mode: str,
) -> None:
    ...


def app_main(
    ui_mode: str = "default",
    *,
    anchor_file: str,
    argv=None,
) -> int:
    runtime_dir = runtime_base_dir(anchor_file=anchor_file)
    ...
```

关键约束：
- `anchor_file` 必须由 `app.py` / `app_new_ui.py` 传入各自的 `__file__`；
- 不允许让共享模块内部直接把 `__file__` 固定为 `entrypoint.py` 自己；
- 运行时契约中的 `ui_mode` 回退值仍由壳层传入。

#### P3.2 将 `app.py` 与 `app_new_ui.py` 缩为薄壳

目标是保留：
- `create_app()`
- `if __name__ != "__main__": app = create_app()`
- `if __name__ == "__main__": raise SystemExit(app_main(..., anchor_file=__file__))`

示例：

```python
from web.bootstrap.entrypoint import app_main, create_app_with_mode


def create_app():
    return create_app_with_mode("default")


if __name__ != "__main__":
    app = create_app()


if __name__ == "__main__":
    raise SystemExit(app_main("default", anchor_file=__file__))
```
