# 分析页补显坏交期未排批次与冻结摘要实现计划

## 背景
当前排产摘要与历史里，坏交期批次数、未排批次数、冻结窗口状态等数据已经有稳定输出，但分析页还没有把这些信息完整展示出来：

1. **坏交期批次数 / 未排批次数** 已进入摘要与历史字段，但分析页现有卡片只展示超期、拖期、工期、换型、利用率等指标，导致页面说明与真实界面对不上。
2. **冻结窗口信息** 已写入摘要，分析页目前只在降级时展示黄色提示，缺少结构化摘要，用户无法直接看出本次是否启用、是否生效、冻结了多少工序、涉及多少批次。

这次实现只做**分析页读侧补显**，优先复用现有摘要合同，不反向扩大排产核心链改动范围。

---

## 目标

### 目标一：补两张指标卡片
在分析页"版本概览"区补上：
- 坏交期批次数
- 未排批次数

要求：
- 与现有卡片风格一致；
- 有当前版本数值；
- 若存在上一版可比数据，则显示对比差值；
- 老摘要缺字段时不报错，按"无数据/默认隐藏差值"处理。

### 目标二：补冻结状态摘要
在分析页"版本概览"区增加一块冻结状态摘要，至少展示：
- 是否启用冻结窗口；
- 锁定天数；
- 当前状态（未启用 / 已生效 / 已降级）；
- 是否实际生效；
- 冻结工序数；
- 冻结批次数；
- 降级原因（如有）；
- 示例批次（如有且数量不大）。

要求：
- 现有黄色降级提示保留；
- 新摘要负责说明"发生了什么"，降级提示负责强调"需要关注"；
- 老摘要缺字段时页面仍能安全渲染。

---

## 代码风格约束

本次实现必须遵守以下风格要求，贯穿展示态、模板和测试：

1. **不做过度兜底**
   - 不要对每个字典访问都套 `try/except`；
   - 不要在已知字段类型正确的前提下再做 `isinstance` 防御；
   - 摘要结构已有明确合同，按合同读取即可。

2. **不做静默回退**
   - 字段缺失时直接标记为"无数据"展示，不要偷偷用 0 补位让用户误以为真实值；
   - 对比差值只在**两端字段都真实存在**时才计算，缺字段时不显示差值而不是显示成 0。

3. **不做多余的防御性编程**
   - 展示态函数的入参由 `build_analysis_context` 统一提供，上游已做过解析，下游不必重复校验；
   - 模板里不要用冗长的 `.get(...) or {} | .get(...) or 0` 链式调用，由展示态统一整理好字段后直接渲染。

4. **代码要简洁优雅**
   - 函数职责单一，一个函数做一件事；
   - 展示态函数命名直接表达意图；
   - 模板代码保持可读，结构化数据在展示态层就准备好，模板只做"取值和渲染"。

5. **符合设计意图，适合用户使用**
   - 新增卡片和冻结摘要是为了让用户快速掌握排产质量全貌，不是为了应付边界条件；
   - 显示内容应当清晰、有信息量，不要显示冗余的技术性字段；
   - 未启用冻结窗口时直接显示"未启用"，不要显示一堆 0 值字段。

- `core/algorithms/evaluation.py`
  - 指标对象已包含坏交期与未排批次统计

### 页面当前缺口

- `web/viewmodels/scheduler_analysis_vm.py`
  - 目前主要提取常规指标与尝试过程
  - 尚无"分析页展示态"来统一整理这两类新增信息

### 新 UI 模板确认
- `web_new_test/templates/scheduler/` 下**没有**分析页模板，不需要同步修改。

## 实现原则

1. **先补展示，不改核心口径**
   - 只要现有摘要字段够用，就不回头改排产摘要生成链。
2. **读侧集中整理**
   - 展示态函数统一整理新增字段并完成类型转换、状态映射、缺字段标记，模板只做渲染。
3. **老记录安全渲染**
   - 老版本摘要没有这些字段时，页面不报错，也不误显示成 0。
4. **本次不扩范围**
   - 不顺手追加趋势图；
   - 不顺手调整历史页；
   - 不顺手改说明书；
   - 先把分析页事实补齐。

## 拟改动范围

### 一、分析页展示态补齐
**目标文件**：`web/viewmodels/scheduler_analysis_vm.py`

**计划改动**：

1. 新增两个展示态整理函数：
   - `build_extra_cards(selected_summary, selected_metrics, prev_metrics)` → 产出坏交期/未排批次的展示结构
   - `build_freeze_display(selected_summary)` → 产出冻结状态摘要展示结构

2. **坏交期/未排批次卡片取值口径**：
   - 优先取 `selected_metrics`（即 `algo.metrics`）中的 `invalid_due_count` / `unscheduled_batch_count`；
   - 若 `selected_metrics` 为 None 或字段不存在，回退到 `selected_summary` 顶层同名字段；
   - 若仍不存在，返回 `None` 表示"无数据"，模板端不渲染该卡片。
   - ⚠️ **两层来源口径说明**：`algo.metrics` 的值来自最优方案的 `ScheduleMetrics.to_dict()`，顶层 `invalid_due_count` 来自 `_metric_int(best_metrics, ...)` 合并逻辑。正常场景两者一致，`best_metrics` 为 None 的退化场景下顶层仍可能有值。

3. **对比差值处理**（落实审核 R2-F1）：
   - 用 `key in prev_metrics` 判断上一版字段**是否存在**，而非 `prev_metrics.get(key, 0)`；
   - 字段不存在时，差值设为 `None`，模板端不显示差值行；
   - 字段存在且值为 0 时，正常计算并显示差值 0。

4. **冻结摘要展示对象**统一整理为结构化字典：
   ```python
   {
       "enabled": bool,          # ⚠️ 从 "yes"/"no" 字符串转为布尔（落实审核 R2-F2）
       "days": int,
       "state": str,             # "disabled" / "active" / "degraded"
       "state_label": str,       # "未启用" / "已生效" / "已降级"
       "applied": bool,
       "frozen_op_count": int,
       "frozen_batch_count": int,
       "sample_batches": list,   # 截断到前 5 个，超过则附带总数
       "sample_total": int,      # frozen_batch_count 原值，用于"及其他 N 个"
       "degraded": bool,
       "degradation_reason": str | None,
   }
   ```
   - `enabled` 字段：用 `str(fw.get('enabled', '')).lower() == 'yes'` 转为布尔
   - 状态映射表：`{'disabled': '未启用', 'active': '已生效', 'degraded': '已降级'}`
   - `frozen_batch_ids_sample` 截断到前 5 个展示，超过 5 个时模板显示"及其他 N 个…"
   - 若 `algo.freeze_window` 不存在，返回 `None`，模板不渲染冻结摘要区

5. 在 `build_analysis_context` 中调用上述两个函数，返回值新增 `extra_cards` 和 `freeze_display` 字段。

### 二、分析页模板补显
**目标文件**：`templates/scheduler/analysis.html`

**计划改动**：

1. **两张新卡片**：
   - 插入 `stat-grid` 内部，紧跟"超期批次数"卡片之后
   - "坏交期批次数"用 `type-info`（蓝灰色边框）
   - "未排批次数"用默认样式（无 type 修饰）
   - 卡片由 `extra_cards` 上下文驱动，字段为 `None` 时整张卡片不渲染
   - 对比差值由展示态整理好，为 `None` 时差值行不渲染

2. **冻结状态摘要**：
   - 放在现有降级提示（第 116-125 行黄色 flash）之后
   - 由 `freeze_display` 上下文驱动，为 `None` 时不渲染
   - 未启用时：显示"冻结窗口：未启用"，不显示一堆 0 值字段
   - 已生效时：显示启用状态、锁定天数、冻结工序数、冻结批次数、示例批次
   - 已降级时：显示以上内容 + 降级原因，与上方黄色提示互相印证而不矛盾
   - 样式采用与配置快照同风格的信息区（`details` 或紧凑列表），不另加卡片

3. **新卡片在 `selected_metrics` 为 None 时的行为**：
   - 现有 `stat-grid` 受 `{% if selected_metrics %}` 守护
   - 新卡片的数据来自 `extra_cards`（可能从顶层字段回退取得），因此新卡片放在 `stat-grid` 之后、单独用 `{% if extra_cards %}` 守护，不依赖 `selected_metrics`

### 三、路由改动
**目标文件**：`web/routes/scheduler_analysis.py`

**预期**：不需要改。路由用 `**ctx` 透传全部上下文，只要在 `build_analysis_context` 中新增返回字段即可。

## 回归计划

### 一、补测试数据（落实审核 R3-F2）
**文件**：`tests/regression_scheduler_analysis_observability.py`

必须先补测试数据，而不仅仅加断言：

1. `make_metrics()` 新增参数并输出：
   - `invalid_due_count`（默认 0）
   - `unscheduled_batch_count`（默认 0）

2. `make_new_summary()` 补充：
   - 顶层 `invalid_due_count` 和 `unscheduled_batch_count`
   - `freeze_window` 补全字段：`freeze_state`、`freeze_applied`、`freeze_degradation_codes`
   - `make_metrics(overdue_count=1)` 改为 `make_metrics(overdue_count=1, invalid_due_count=2, unscheduled_batch_count=3)`

### 二、新断言
在新摘要场景下验证页面包含：
1. "坏交期批次数" 卡片及数值

3. 冻结状态摘要中的中文状态标签（如"已降级"）
4. 冻结工序数 / 冻结批次数

### 三、旧摘要兼容断言
旧摘要场景下验证：
1. 页面不报错，能正常渲染
2. 不出现坏交期 / 未排批次卡片（因为旧 `make_metrics()` 不含这些字段）
3. 不出现冻结摘要区（因为旧摘要无 `freeze_window`）

### 四、读侧回退断言
构造一个特殊摘要：`algo.metrics` 中无 `invalid_due_count`，但顶层有 `invalid_due_count`，验证页面仍能正确展示当前值且不显示差值。

## 验收标准

1. 分析页能稳定展示坏交期批次数与未排批次数两张卡片。
2. 若上一版存在可比值，两张卡片能展示差值；若无可比值，不出现报错或误导性差值（不把"无数据"显示为 0）。
3. 分析页能稳定展示冻结状态摘要，至少覆盖"未启用 / 已生效 / 已降级"三种状态。
4. 冻结窗口已降级时，现有黄色提示仍保留，且与摘要信息一致。
5. 老版本摘要、裁剪摘要、字段缺失摘要都能安全渲染。
6. 回归用例能覆盖：正常展示、降级展示、旧摘要兼容、读侧回退。
7. 代码简洁优雅，不含多余的 try/except、不含静默回退、不含冗余的 isinstance 链。
