# 排产空执行修复计划

## 1. 审核结论

结合 `.limcode/review/排产空执行修复实施清单.md` 与当前代码实现，结论为：**当前实现尚未完成实施清单中的关键收口，需继续修复。**

### 1.1 已确认的差异

1. `core/services/scheduler/schedule_service.py`
   - 在 `reschedulable_operations` / `reschedulable_op_ids` 计算后，**没有第一次空短路**。
   - `allocate_next_version()` 仍在冻结过滤前执行，说明**版本号分配过早**。
   - `algo_ops_to_schedule` 计算后，**没有第二次空短路**。
   - 这意味着以下空场景仍可能继续流入算法、摘要与持久化：
     - 没有任何可重排工序
     - 有可重排工序，但被冻结后全部不可执行

2. `core/services/scheduler/schedule_persistence.py`
   - `persist_schedule(...)` 当前**没有“可执行排产动作”显式防线**。
   - 该函数即使 `schedule_rows` 为空，仍会继续写 `ScheduleHistory`，并在事务后写操作日志。
   - 这与“空场景不建版本、不写历史、不写操作日志”的口径不一致。

3. `web/routes/scheduler_run.py` 与 `web/routes/scheduler_week_plan.py`
   - 两处路由都已统一捕获 `AppError`，主逻辑**暂不需要改代码**。
   - 但当前仍缺少针对空执行场景的页面级回归，尚未证明不会误入成功提示分支。

4. `tests/`
   - 现有 `regression_schedule_service_reschedulable_contract.py` 仅覆盖“终态工序不进入可重排集合”的契约。
   - 现有 `regression_schedule_persistence_reschedulable_contract.py` 仅覆盖“持久化只处理可重排工序”的契约。
   - **尚未覆盖**以下关键风险：
     - `reschedulable_operations == []` 时直接拒绝
     - `algo_ops_to_schedule == []` 时直接拒绝
     - `persist_schedule(...)` 拒绝空执行留痕
     - 正式排产页面不误报成功
     - 模拟排产页面不误报成功

## 2. 修复目标

本次修复采用 review 中已确认的最小改动策略：

- 空场景不分配新版本号
- 空场景不写 `Schedule`
- 空场景不写 `ScheduleHistory`
- 空场景不写排产操作日志
- 正式排产与模拟排产统一通过 `ValidationError` 返回“本次未执行”
- 本阶段**不引入新的 `result_status`**

## 3. 具体实施步骤

### 步骤一：修复服务层第一次空短路

目标文件：`core/services/scheduler/schedule_service.py`

实施内容：

1. 在 `reschedulable_operations` 与 `reschedulable_op_ids` 计算完成后立即判断是否为空。
2. 若为空：
   - 正式排产抛 `ValidationError("所选批次没有可重排工序，本次未执行排产。", field="排产")`
   - 模拟排产抛 `ValidationError("所选批次没有可重排工序，本次未执行模拟排产。", field="排产")`
3. 保证该判断发生在以下调用之前：
   - `build_algo_operations(...)`
   - `build_freeze_window_seed(...)`
   - `allocate_next_version()`

重点验收：
- 空可重排场景不触发算法
- 不触发持久化
- 不占用新版本号

### 步骤二：后移版本号分配

目标文件：`core/services/scheduler/schedule_service.py`

实施内容：

1. 保留 `latest = self.history_repo.get_latest_version()` 与 `prev_version = int(latest or 0)` 的读取位置，用于冻结窗口读取上一版本。
2. 将 `with self.tx_manager.transaction(): version = int(self.history_repo.allocate_next_version())` 后移到：
   - `build_freeze_window_seed(...)` 之后
   - `algo_ops_to_schedule` 计算之后
   - 第二次空短路之后
   - `optimize_schedule(...)` 之前

重点验收：
- 任一空场景都不会调用 `allocate_next_version()`
- 正常场景版本号行为保持不变

### 步骤三：补齐冻结后全空短路

目标文件：`core/services/scheduler/schedule_service.py`

实施内容：

1. 在 `algo_ops_to_schedule = [...]` 之后立即判断是否为空。
2. 若为空：
   - 正式排产抛 `ValidationError("冻结窗口内无可调整工序，本次未执行排产。", field="排产")`
   - 模拟排产抛 `ValidationError("冻结窗口内无可调整工序，本次未执行模拟排产。", field="排产")`
3. 保证该判断发生在：
   - `allocate_next_version()` 之前
   - `optimize_schedule(...)` 之前

重点验收：
- 冻结全空场景不调算法
- 不建空版本
- 不写历史与日志

### 步骤四：给持久化层增加保险丝

目标文件：`core/services/scheduler/schedule_persistence.py`

实施内容：

1. 为 `persist_schedule(...)` 增加显式参数，例如：`has_actionable_schedule: bool`。
2. 在函数入口增加防御判断：
   - 若 `has_actionable_schedule` 为 `False`，抛 `ValidationError("本次没有实际可执行排产任务，禁止写入排产历史。", field="排产")`
3. 在 `core/services/scheduler/schedule_service.py` 的正常调用路径中显式传入 `has_actionable_schedule=True`。
4. 同步更新所有直接调用 `persist_schedule(...)` 的测试与桩函数，使其参数契约一致。

当前已确认的影响点：
- 真实调用：`core/services/scheduler/schedule_service.py`
- 直接测试调用：`tests/regression_schedule_persistence_reschedulable_contract.py`
- 桩函数接收：
  - `tests/regression_schedule_service_reschedulable_contract.py`
  - `tests/regression_schedule_service_missing_resource_source_case_insensitive.py`

重点验收：
- 即使未来服务层绕过前置判断，持久化层也无法写出空历史

## 4. 实施顺序

1. 先改 `schedule_service.py`：第一次空短路、版本号后移、第二次空短路
2. 再改 `schedule_persistence.py`：增加 `has_actionable_schedule` 保险丝
3. 再补服务层、持久化层回归
4. 最后补页面回归与数据库级验收回归
5. 提交前复查：
   - 冻结全空是否仍会生成纯复制版本
   - 页面是否还存在乐观成功提示
   - `ScheduleHistory` 是否仍可能出现空版本

## 5. 风险与注意事项

1. `persist_schedule(...)` 参数变更会影响现有测试桩；需要一次性同步更新，避免出现参数不匹配。
2. 页面层主逻辑虽无需改动，但若测试发现空场景仍进入成功提示，才再回头补路由分支。
3. 本次不要半途引入新状态值，否则会扩散到历史页、分析页、甘特图、报表等展示层。

## 6. 验收标准

- `reschedulable_operations == []` 时直接拒绝且不建版本
- `algo_ops_to_schedule == []` 时直接拒绝且不建版本
- 空执行场景不写 `Schedule`
- 空执行场景不写 `ScheduleHistory`
- 空执行场景不写排产操作日志
- 正式排产页面不再误报成功
- 模拟排产页面不再误报成功
- 新增回归全部通过
- 既有契约回归继续通过：
  - `tests/regression_schedule_service_reschedulable_contract.py`
  - `tests/regression_schedule_persistence_reschedulable_contract.py`
  - `tests/regression_schedule_service_missing_resource_source_case_insensitive.py`
