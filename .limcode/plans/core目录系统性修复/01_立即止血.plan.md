# core目录系统性修复：01_立即止血

## 这一段要解决什么问题
这一段只做一件事：**先把最危险、最会悄悄改业务语义的地方停下来。**

当前最急的不是代码是不是优雅，而是这些链路还在持续制造坏结果：
- 空白数字被偷偷改成 `0`、`0.0`、`1.0`；
- 用户没填供应商默认周期，页面链路却先替他补成一天；
- 用户清空外协周期，编辑链又替他写回一天；
- 持久化层明明有空行动保险丝，主链却可能把它当成“已通过”；
- 已有批次改了图号，工序却不重建；
- 展示层把坏时间行丢了，再包装成“当前没任务”；
- 页面知道自己降级了，导出却还装成完整事实。

如果这一段不先止血，后面再去统一严格口径、统一退化可见性、统一结构边界，只是在继续出血的基础上补制度。

## 改后硬口径

### 1. 新写路径的硬口径
| 场景 | 旧行为 | 新行为 | 对外表现 |
| --- | --- | --- | --- |
| `parse_finite_*` 遇到空白 | 可能回 `0/0.0` | `allow_none=False` 直接失败 | 调用方拿到明确错误，不得自行补回默认值 |
| 供应商 `default_days` 为空或非法 | 路由层可能先补 `"1"`，服务层再兼容 | 直接失败 | 页面仍走现有提交流程，但提示必须明确说明“默认周期不能为空/不合法” |
| 外协工序 `ext_days` 为空 | 编辑链写回 `1.0` | 直接失败 | 页面或接口错误明确指向外协周期字段 |
| 改图号但不重建工序 | 允许继续更新批次头 | 直接拒绝 | 返回稳定错误说明，提示开启自动生成工序后再试 |

### 2. 排产主链的硬口径
| 场景 | 旧行为 | 新行为 |
| --- | --- | --- |
| 无可重排工序 | 上游已有短路 | 保持现有短路，但与后面两类空结果明确区分 |
| 全部被冻结 | 上游已有短路 | 保持现有短路，但不与“优化器无有效结果”混用 |
| 优化器返回结果但无有效落库行 | 可能继续写版本、历史与摘要 | 直接阻断落库，并使用稳定原因标识 `no_actionable_schedule_rows` |

### 3. 展示链与导出链的硬口径
| 场景 | 旧行为 | 新行为 |
| --- | --- | --- |
| 坏时间行被过滤 | 只影响最终行数 | 必须写出 `bad_time_row_skipped` 事件与计数 |
| 全部行都被过滤 | 像普通空结果 | 返回“退化空结果”，同时给出 `empty_reason` |
| 导出与页面口径 | 页面与导出各说各话 | 导出摘要直接复用同一退化事件与计数 |

## 对外返回与页面口径

### 1. 甘特图合同新增字段
**传播链**：`gantt_tasks.py -> gantt_service.py -> gantt_contract.py -> web/routes/scheduler_gantt.py`

在不删除现有字段的前提下，给甘特图合同新增：
- `degraded: bool`
- `degradation_events: list[dict]`
- `degradation_counters: dict[str, int]`
- `empty_reason: str | None`

约束：
- 保持现有 `task_count == len(tasks)` 约定不变；
- 若过滤了坏时间行，则 `task_count` 仍按最终 `tasks` 计算，但必须在 `degradation_counters.bad_time_row_skipped` 中反映被过滤数量；
- 若全部任务都被过滤，则 `task_count == 0`，但 `degraded` 必须为 `true`，`empty_reason` 必须是稳定值，例如 `all_rows_filtered_by_invalid_time`。

### 2. 周计划数据新增字段
**传播链**：`gantt_week_plan.py -> gantt_service.py -> web/routes/scheduler_week_plan.py`

在现有周计划 `data` 包装层新增：
- `degraded`
- `degradation_events`
- `degradation_counters`
- `empty_reason`

约束：
- 页面预览表格仍可复用现有 `rows` 字段；
- 但页面或导出必须能区分“本周确实没任务”和“有任务，但都因坏时间被过滤掉了”。

### 3. 资源排班返回与导出新增字段
**传播链**：`resource_dispatch_rows.py -> resource_dispatch_support.py -> resource_dispatch_service.py -> resource_dispatch_excel.py -> web/routes/scheduler_resource_dispatch.py`

统一在 `payload.summary` 或等价摘要位置新增：
- `degraded`
- `degradation_events`
- `degradation_counters`
- `empty_reason`

约束：
- `empty_message` 不再只根据最终行数判断；
- 若全部数据因坏时间被过滤，`empty_message` 必须明确说明这是退化空结果；
- 导出摘要页必须包含与页面相同的退化计数，至少覆盖：
  - `bad_time_row_skipped`
  - 超期标记退化
  - 推断日历使用中（该项允许在下一段进一步补强，但本段需预留透传位）

## 实施内容

### 1. 修复 `parse_finite_float/int` 的空白语义
**目标文件**：`core/services/common/number_utils.py`

**要做的事**：
- `allow_none=False` 时，空白或 `None` 直接报错；
- `allow_none=True` 时，才允许返回 `None`；
- 逐一核查当前 `allow_none=False` 的调用点，不允许通过本地 `or 0`、`or 1` 把旧行为补回来；
- 若某调用点确实需要兼容旧数据，迁到 `compat_parse.py`，不要继续藏在 `number_utils.py` 里。

### 2. 禁止供应商默认周期静默写 `1.0`
**目标文件**：
- `core/services/process/supplier_service.py`
- `web/routes/process_suppliers.py`

**要做的事**：
- 删除路由层对 `default_days` 的 `or "1"`；
- `SupplierService._normalize_default_days()` 不再承担写入链兼容职责；
- `create()` / `update()` 只接受写入必填语义；
- 历史脏值兼容仅读路径保留，移到后续读侧兼容链。

### 3. 禁止外协周期编辑静默写 `1.0`
**目标文件**：`core/services/scheduler/operation_edit_service.py`

**要做的事**：
- `update_external_operation()` 中删除 `dv = 1.0` 兜底；
- 外协工序编辑时，`ext_days` 为空或非法直接失败；
- 外协工序与内部工序走不同字段语义，避免继续混用。

### 4. 恢复空行动保险丝
**目标文件**：
- `core/services/scheduler/schedule_service.py`
- `core/services/scheduler/schedule_persistence.py`

**要做的事**：
- 在 `_run_schedule_impl()` 落库前真实计算是否存在有效可落库排程行；
- 传给 `persist_schedule()` 的 `has_actionable_schedule` 不再硬编码；
- 明确区分三类空结果：
  - `no_reschedulable_operations`
  - `all_operations_frozen`
  - `no_actionable_schedule_rows`
- 只有真正有可落库排程行时，才允许写历史、版本和摘要。

### 5. 拒绝“改图号但不重建工序”
**目标文件**：`core/services/scheduler/batch_excel_import.py`

**要做的事**：
- 在 `_apply_row_no_tx()` 中，若目标是更新已有批次且 `part_no` 变化、同时 `auto_generate_ops=False`，直接拒绝；
- 错误提示必须明确告诉用户：开启自动生成工序后再试；
- 本段只做前置拒绝，不顺手做事务重构。

### 6. 展示层坏时间过滤统一改成 `BuildOutcome`
**目标文件**：
- `core/services/scheduler/gantt_tasks.py`
- `core/services/scheduler/gantt_week_plan.py`
- `core/services/scheduler/resource_dispatch_rows.py`
- 联动透传：`gantt_service.py`、`gantt_contract.py`、`resource_dispatch_service.py`、`resource_dispatch_support.py`

**要做的事**：
- `_build_one_task()`、`build_week_plan_rows()`、`normalize_dispatch_row()` 等函数不再返回裸对象或 `None` 语义；
- 统一改成 `BuildOutcome`；
- 过滤坏时间行时统一记录 `bad_time_row_skipped`；
- 全部过滤时必须产生退化空结果，而不是普通空结果；
- 路由层只负责透传，不再自己拼第二套过滤说明。

### 7. 资源排班导出复用统一退化事件与计数
**目标文件**：`core/services/scheduler/resource_dispatch_excel.py`

**要做的事**：
- `_summary_pairs()` 与 `_write_summary_sheet()` 不再自己推断降级；
- 直接消费页面链已经形成的统一退化事件与计数；
- 导出摘要至少要保留：
  - 坏时间行过滤数量；
  - 超期标记退化说明；
  - 当前是否是退化空结果。
