# core目录系统性修复：00_总纲与共享合同底座

## 这份 plan 解决什么问题
当前 `core/` 里的大问题，不是只有某一个函数写错，而是同一类坏习惯在不同层不断重复：
- 写入链碰到空白、脏值、坏值，喜欢先补默认值再继续；
- 读取链碰到历史脏数据，喜欢直接兼容掉，但不把兼容事实说清楚；
- 展示链拿到退化结果后，又继续包装成“普通空结果”或“看起来正常”；
- 每个模块都各写一套 `_safe_*`、`_cfg_*`、`warning` 拼装，结果同一种坏值在不同位置有不同命运。

如果不先把共用底座立起来，后面每修一段都会重新长出局部兜底、局部口径、局部告警，最后还是会回到“问题修一点、分支多一圈”的状态。

这份 plan 的职责不是立刻把全部业务修完，而是先把后续四段都会复用的五类基础件立住：
1. 字段语义矩阵；
2. 严格解析入口；
3. 兼容读取入口；
4. 统一退化事件；
5. 统一结果封套。

## 字段语义矩阵：先把边界写死
**目标文件**：`core/services/common/value_policies.py`

### 首批必须入表的字段
| 字段 | 所属链路 | 写入语义 | 读取语义 | 严格模式 | 兼容读取 | 首批原因码 |
| --- | --- | --- | --- | --- | --- | --- |
| `default_days` | 供应商默认周期 | 写入必填 | 可兼容历史脏值 | 空白/非法直接失败 | 仅历史读取可回退 | `invalid_number` |
| `ext_days` | 外协工序周期 | 外协写入必填，非外协不适用 | 可兼容历史脏值 | 空白/非法直接失败 | 非严格读取可回退并留痕 | `legacy_external_days_defaulted` |
| `setup_hours` / `unit_hours` | 排产输入 | 写入必填 | 可兼容历史脏值 | 空白/非法直接失败 | 读侧兼容必须留痕 | `invalid_number` |
| `priority_weight` / `due_weight` / `ready_weight` | 配置/优化器/算法参数 | 写入必填 | 配置快照可兼容旧值 | 空白/非法直接失败 | 允许读旧值但必须产生事件 | `invalid_number` |
| `freeze_window_days` | 冻结窗口配置 | 写入必填 | 配置快照可兼容旧值 | 空白/非法直接失败 | 非严格读取允许降级 | `freeze_seed_unavailable` |
| `due_date` | 批次交期 | 写入可空但格式必须合法 | 旧数据可兼容 | 非法格式直接失败 | 非严格读取标记为坏交期 | `invalid_due_date` |
| `start_time` / `end_time` | 展示/导出读侧 | 不在本段定义写入链 | 读侧可兼容但必须外显 | 不适用 | 仅读侧过滤并记事件 | `bad_time_row_skipped` |

### 建议最小模型
```python
@dataclass(frozen=True)
class FieldPolicy:
    field: str
    write_mode: str
    read_mode: str
    value_kind: str
    strict_reason_code: str
    compat_reason_code: str | None = None
    notes: str | None = None
```

### 硬约束
- 关键字段的语义先入表，再允许写实现。
- 不允许调用方继续用 `or 0`、`or 1`、`or "1"`、`if not value: default` 自行定义关键业务字段语义。
- 本矩阵是后续 `01` 到 `04` 四段的唯一事实来源；若要改口径，先改矩阵，再改消费点。

## 严格解析层：只做“合法值或报错”
**目标文件**：`core/services/common/strict_parse.py`

### 建议首批函数
```python
def parse_required_float(value, *, field: str, min_value: float | None = None) -> float: ...
def parse_optional_float(value, *, field: str, min_value: float | None = None) -> float | None: ...
def parse_required_int(value, *, field: str, min_value: int | None = None) -> int: ...
def parse_optional_int(value, *, field: str, min_value: int | None = None) -> int | None: ...
def parse_required_date(value, *, field: str): ...
def parse_optional_date(value, *, field: str): ...
```

### 改后口径
- 严格解析函数只负责两件事：
  1. 返回合法值；
  2. 抛出 `ValidationError`。
- 不负责：
  - 记日志；
  - 记计数；
  - 追加 warning；
  - 自动回默认值；
  - 自动把错误变成 `None`。

### 迁移约束
- `number_utils.py` 中现有 `parse_finite_float()` / `parse_finite_int()` 可以暂留，但只能做薄门面转调 `strict_parse.py`。
- 后续任何新模块不得再新增局部 `_safe_float()`、`_cfg_float()`、`_get_float()` 实现。

---

## 兼容读取层：只处理历史脏数据
**目标文件**：`core/services/common/compat_parse.py`

### 适用范围
只允许用于以下场景：
- 历史配置快照读取；
- 旧排程结果读取；
- 旧模板或旧工序记录读取；
- 展示链读取旧数据；
- 导出链读取旧数据。

### 明确禁止的范围
- 新建、编辑、导入确认、批量更新等写入链不得复用兼容读取层。
- 任何会把兼容值再写回数据库的路径，必须经过单独审批，不得默认放行。

### 建议最小行为
1. 先调用严格解析；
2. 失败后查询 `value_policies.py`；
3. 按字段策略决定：
   - 回退为 `None`；
   - 回退为指定兼容值；
   - 直接阻断；
4. 只要发生兼容回退，必须生成 `DegradationEvent`。

## 统一退化事件：以后所有退化都走同一管道
**目标文件**：`core/services/common/degradation.py`

### 建议最小模型
```python
@dataclass(frozen=True)
class DegradationEvent:
    code: str
    scope: str
    field: str | None
    message: str
    count: int = 1
    sample: str | None = None

class DegradationCollector:
    def add(...): ...
    def extend(...): ...
    def to_list(...): ...
    def to_counters(...): ...
```

### 首批必须预留的原因码
- `blank_required`
- `invalid_number`
- `invalid_due_date`
- `bad_time_row_skipped`
- `calendar_load_failed`
- `legacy_external_days_defaulted`
- `freeze_seed_unavailable`
- `template_missing`
- `external_group_missing`

### 统一口径
- `code`：稳定、可统计、可测试。
- `scope`：至少能区分 `config`、`schedule_input`、`freeze_window`、`gantt`、`resource_dispatch`、`backup` 等链路。
- `field`：能定位到字段时必须带字段名。
- `message`：给人看的短句，不在不同模块各写一版。
- `count`：用于合并计数，不让页面或摘要自己再数一遍。
- `sample`：仅保存少量样本，用于排障。

### 硬约束
- 不允许新增“只在日志里写一句话”的退化处理。
- 不允许新增“只返回一个 warning 字符串”的退化接口。
- 摘要、页面、历史、导出都必须优先消费统一事件，而不是各自从异常文本里猜。

## 统一结果封套：先从读侧构建器收口
**目标文件**：`core/services/common/build_outcome.py`

### 建议最小模型
```python
@dataclass
class BuildOutcome(Generic[T]):
    value: T
    events: list[DegradationEvent] = field(default_factory=list)
    counters: dict[str, int] = field(default_factory=dict)
    empty_reason: str | None = None
```

### 第一批强制接入范围
- `core/services/scheduler/gantt_tasks.py`
  - `build_calendar_days()`
  - `_build_one_task()`
  - `build_tasks()`
- `core/services/scheduler/gantt_week_plan.py`
  - `build_week_plan_rows()`
- `core/services/scheduler/resource_dispatch_rows.py`
  - `normalize_dispatch_row()`
  - `build_dispatch_detail_rows()`
  - `build_dispatch_tasks()`
  - `build_dispatch_calendar_matrix()`

### 改后口径
- `value`：真正的结果值。
- `events`：统一退化事件列表。
- `counters`：供摘要、页面、导出复用的计数字段。
- `empty_reason`：区分“天然为空”和“退化后为空”。

### 硬约束
- 禁止继续新增 `(items, skipped_count)`、`(data, error_message)`、`(rows, warning_text)` 这类裸返回。
- `BuildOutcome` 只先落在读侧构建器，不在本段强行改写所有领域对象。

## 改后统一口径与交付物

### 1. 写入链统一口径
- 写入链只能用 `strict_parse.py`；
- 写入链不得使用 `compat_parse.py`；
- 写入链若需要外显错误，由调用方接住 `ValidationError` 后转成页面提示或接口错误。

### 2. 读取链统一口径
- 读取链若必须兼容历史脏值，只能走 `compat_parse.py`；
- 只要发生兼容回退，就必须带上 `DegradationEvent`；
- 页面、摘要、导出、历史只消费事件与计数，不再自己拼第二套语义。

### 3. 旧门面保留规则
- 允许保留：`number_utils.py` 的两个门面函数，前提是内部完全转调新入口；
- 不允许保留：新的局部 `_safe_*`、`_cfg_*`、`_get_*` 解析分支。

### 4. 本段必须产出的文件
- `core/services/common/value_policies.py`
- `core/services/common/strict_parse.py`
- `core/services/common/compat_parse.py`
