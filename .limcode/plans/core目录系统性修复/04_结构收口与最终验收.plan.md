# core目录系统性修复：04_结构收口与最终验收

## 这一段要解决什么问题
前面三段解决的是语义：
- 不再继续偷偷改业务值；
- 严格合同真正生效；
- 退化事实能够被看见。

这一段要解决的是：**为什么这些问题会反复长出来。**

如果只修行为、不收结构根因，问题很容易重新冒出来：
- 模板自动补建还会在事务外先写副作用；
- 全局日历确认链还在自己维护一套导入循环和计数；
- 导入链依然丢原始源行号，空白行一压缩，错误定位就会漂；
- `ScheduleService` 还太胖，什么都往里塞；
- 归一化规则在 `common`、`route`、`service` 多处复制，`SkillLevel` 这类值域口径也容易漂；
- 插件层仍在反向依赖服务层和仓储层；
- 报表导出继续默认全量内存构建，规模一上来就重新变成稳定性风险。

所以这段不是“再补几个判断”，而是：**把最容易重新长出兜底、漂移和重复规则的结构根因收住。**

## 本段拆分为 7 个可施工子批次

### 子批次 A：事务边界止漏
**对应 TODO**：`#p04-1`、`#p04-9`

#### 目标
把模板自动补建从“事务外可能先写副作用”改成“事务外只读预检、事务内统一写入”。

#### 实施锚点
- `BatchService._load_template_ops_with_fallback()`
- `BatchService.create_batch_from_template()`
- `BatchService.create_batch_from_template_no_tx()`

#### 具体动作
1. 将 `_load_template_ops_with_fallback()` 拆成两步：
   - 只读预检：只判断模板是否存在、是否需要自动补建；
   - 事务内执行：真正补模板、建批、建工序放进同一事务。
2. `create_batch_from_template()` 中，事务外只允许做：
   - 参数归一；
   - 零件是否存在校验；
   - 模板是否缺失的只读判断。
3. 一旦需要自动补建模板，必须和批次创建、工序重建放进同一事务。
4. 若自动补建失败，必须保证：
   - 模板库无残留半副作用；
   - 批次头无残留；
   - 批次工序无残留。

#### 交付物
- 新增或拆分事务内辅助函数，例如：
  - `_probe_template_ops()`
  - `_ensure_template_ops_in_tx()`
- `create_batch_from_template()` 保持对外入口不变。

### 子批次 D：`ScheduleService` 拆委托但不改入口
**对应 TODO**：`#p04-3`、`#p04-11`、`#p04-12`

#### 目标
把 `_run_schedule_impl()` 从“大总管方法”拆成薄门面 + 明确委托模块。

#### 目标模块边界
| 新模块 | 主要职责 | 当前来源 |
| --- | --- | --- |
| `schedule_input_collector.py` | 请求归一、批次/工序读取、可重排筛选、冻结输入、停机与资源池输入准备 | `schedule_service.py` 前半段 |
| `schedule_orchestrator.py` | 调优化器、版本号分配、warning 合并、组织优化结果 | `schedule_service.py` 中段 |
| `schedule_persistence.py` | 结果落库与状态更新 | 现有文件继续复用 |
| `schedule_summary.py` | 摘要拼装 | 现有文件继续复用 |

#### 具体动作
1. 保持 `run_schedule()` / `_run_schedule_impl()` 对外方法名和返回结构不变。
2. 把输入收集逻辑抽到 `schedule_input_collector.py`，返回明确的结果对象，例如：
   - `ScheduleRunInput`
3. 把优化编排逻辑抽到 `schedule_orchestrator.py`，返回明确的结果对象，例如：
   - `ScheduleOrchestrationOutcome`
4. `ScheduleService._run_schedule_impl()` 最终只保留：
   - 参数校验；
   - 委托收集输入；
   - 委托优化编排；
   - 调摘要与落库；
   - 组装返回值。
5. 退化事件沿统一通道透传，不允许拆分后重新散落到各处拼摘要。

#### 额外守门要求
- 本子批次结束时，`_run_schedule_impl()` 的逻辑层级明显下降；
- 新模块要有独立回归，而不是只靠原有大链路回归兜住。

## 对外合同与兼容边界

### 1. 导入链
- 新增 `source_row_num` 等元数据可以加，但不得偷换现有 `row` 的旧含义。
- 公共执行器返回的统计字段名保持兼容，新增元数据优先作为增量字段接入。

### 2. 排产入口
- `ScheduleService.run_schedule()` / `_run_schedule_impl()` 的参数与返回主结构保持兼容。
- 内部模块拆分后，返回值字段名和主含义不得改变。

### 3. 插件状态
- 允许新增冲突字段与 telemetry 字段；
- 不得删除现有 `statuses`、`registry`、`plugins_dir` 等字段。

### 4. 报表导出
- 对外导出入口保持原路由不变；
- 仅新增模式选择和明确拒绝策略，不偷换文件命名与基础格式。

7. 报表导出是否已具备规模阈值与模式分流？
8. 前三段的治理成果是否通过最终回归得到锁定？

---

## 风险控制
1. `ScheduleService` 拆分只改内部委托，不改主要对外调用入口。
2. 导入链补源行号元数据时，优先新增字段，不删除旧字段。
3. 公共执行器收口和归一化矩阵统一都要优先复用既有合同，不顺手再造新接口。
4. 插件层依赖方向纠偏只做“注入读取接口 + 能力冲突显式策略”，不重写插件框架。
5. 报表规模治理先加阈值和模式分流，不强行补不成熟后台框架。
6. 最终验收必须以摘要、代码、回归三方对照为准，不能只看某一侧通过。

---

## 完成标准
1. 模板自动补建与建批副作用已进入同一事务边界。
2. 全局日历确认链已收口到公共执行器，不再旁路。
3. 导入链已具备源行号元数据，空白行压缩后不再造成错误定位漂移。
4. `ScheduleService` 已完成内部拆委托，但主要对外入口保持兼容。
5. 归一化规则与 `SkillLevel` 等值域口径已收口为统一来源。
6. 插件层不再反向依赖服务 / 仓储，capability 覆盖也有显式冲突策略。
7. 报表导出已具备规模阈值与模式分流策略，不再默认无上限全量内存构建。
8. 全链路最终验收给出明确结论，并对剩余债务做清晰留档。
