# core目录系统性修复：02_严格合同与硬约束纠偏

## 这一段要解决什么问题
如果说上一段是在止血，这一段要解决的是：**严格模式到底算不算严格，硬约束到底算不算硬约束。**

当前最别扭的地方在于：
- 配置快照、输入构建、优化器读取、算法参数解析都说自己支持严格；
- 但真遇到空白、非法、非有限值时，很多地方还是会偷偷补默认值继续跑；
- 模板缺失、外部组缺失会静默清空语义；
- 冻结窗口名义上是硬约束，失败时却可能继续放行，摘要还继续说它生效了；
- `active_preset` 这个元信息看起来像“当前生效配置”，实际上却可能落后于真实配置。

这一段的任务，就是把这些“看起来很严、实际上漏风”的边界全部收紧：
1. 重复解析入口只保留一套；
2. `strict_mode` 从配置、输入、算法入口到排序链都用同一套合同；
3. 硬约束失败时不再假装成功；
4. 配置元信息不再误导人。

## 严格模式统一口径

### 1. 首批必须统一的字段
| 字段 | 严格模式 | 非严格模式 | 退化事件 |
| --- | --- | --- | --- |
| `priority_weight` / `due_weight` / `ready_weight` | 空白、非法、非有限值直接失败 | 允许兼容读取，但必须留痕 | `invalid_number` |
| `freeze_window_days` | 空白、非法、非有限值直接失败 | 允许兼容读取，但冻结状态必须变为退化 | `freeze_seed_unavailable` |
| `setup_hours` / `unit_hours` | 空白、非法、非有限值直接失败 | 允许兼容读取，但必须留痕 | `invalid_number` |
| `ext_days` | 空白、非法、非有限值直接失败 | 仅历史读取链允许兼容，且必须留痕 | `legacy_external_days_defaulted` |
| `due_date` | 非法格式直接失败 | 可继续运行，但必须标记为坏交期 | `invalid_due_date` |
| `dispatch_mode` / `dispatch_rule` / `auto_assign_enabled` | 非法值直接失败 | 按当前兼容规则修正，但必须保留已存在行为 | 对应已有 warning 或新事件 |
| 模板 / 外部组缺失 | 直接失败 | 允许继续，但必须显式说明语义已退化 | `template_missing` / `external_group_missing` |

### 2. 严格模式统一原则
- 同一种坏值在不同层只能有一种命运；
- 严格模式下不允许“这一层报错，下一层又偷偷补默认值继续跑”；
- 非严格模式允许兼容，但兼容必须通过统一退化事件对外可见。

## 改后口径与稳定字段

### 1. 冻结窗口状态字段
冻结窗口不再只靠“warning 里提到没有”，统一使用以下字段：
- `freeze_state: "disabled" | "active" | "degraded"`
- `freeze_applied: bool`
- `freeze_degradation_codes: list[str]`

约束：
- `freeze_state == "active"` 时，`hard_constraints` 才允许包含 `freeze_window`；
- `freeze_state == "degraded"` 时，摘要、历史、页面至少两处能看到“冻结未生效”；
- 严格模式下若冻结种子构建失败，直接阻断排产，不进入继续编排。

### 2. 模板 / 外部组合并语义字段
输入构建与摘要至少要能透传：
- `merge_context_degraded: bool`
- `merge_context_events: list[dict]`

约束：
- 模板缺失或外部组缺失时，不允许再只是把 `ext_group_id` / `merge_mode` / `total_days` 清空然后继续假装正常；
- 非严格模式下可以继续，但必须把“组合并信息已退化”透传到摘要；
- 严格模式下直接失败。

### 3. 预设元数据字段
`active_preset` 之外，增加或统一输出：
- `active_preset_reason: str | None`

统一规则：
- 只有当当前生效配置与目标预设快照完全一致、且应用过程中没有修补或降级，`active_preset` 才允许保留预设名；
- 任何手工 `set_*()` 调用成功后，都必须切到 `custom`；
- 应用预设过程中一旦发生修补、回退、降级或自动规范化，也必须切到 `custom`，并给出 `active_preset_reason`。

## 实施内容

### 1. 删除本地重复数值解析实现
**目标文件**：
- `core/services/scheduler/config_snapshot.py`
- `core/services/scheduler/config_validator.py`
- `core/services/scheduler/schedule_optimizer.py`
- `core/services/scheduler/schedule_optimizer_steps.py`
- `core/algorithms/greedy/schedule_params.py`
- `core/services/scheduler/schedule_input_builder.py`

**要做的事**：
- 删除或收口局部 `_safe_float()`、`_cfg_float()`、`_get_float()`、`_get_int()` 变体；
- 严格场景统一调用 `strict_parse.py`；
- 非严格读取统一调用 `compat_parse.py`；
- 若局部函数因为历史调用点太多需要暂留，只能保留薄门面，不得继续自带默认化逻辑。

### 2. 输入构建按字段矩阵贯穿严格语义
**目标文件**：`core/services/scheduler/schedule_input_builder.py`

**要做的事**：
- `build_algo_operations()` 显式接收严格模式开关和退化收集器；
- `setup_hours`、`unit_hours`、`ext_days`、组合并相关字段先查字段矩阵，再决定失败、返回 `None` 或兼容读取；
- 不再返回零散中文 warning 字符串，统一产出结构化退化事件；
- 输出对象优先改为 `BuildOutcome[list[OpForScheduleAlgo]]` 或等价结果封套。

### 3. `SGS` 批内预排序与评分链统一走公共数值合同
**目标文件**：`core/algorithms/greedy/dispatch/sgs.py`

**要做的事**：
- `dispatch_sgs()` 中涉及优先级、交期、序号、外协周期、工时的排序键与评分值，不再直接吃原始值；
- 严格模式下，一旦出现不可比较或非有限值，直接失败；
- 非严格模式下，不可安全评分的批次只能进入稳定退化分支，例如“排序尾部 + 记录退化事件”，不能依赖解释器隐式容错；
- 不允许 `SGS` 再新增局部 `_safe_*`、`_score_*` 数值兜底分支。

### 4. 冻结窗口改为失败关闭
**目标文件**：
- `core/services/scheduler/freeze_window.py`
- `core/services/scheduler/schedule_summary.py`

**要做的事**：
- 冻结种子加载失败、时间非法、前缀缺失时统一产生 `freeze_seed_unavailable` 事件；
- 严格模式下直接阻断排产；
- 非严格模式下允许继续，但 `freeze_state` 必须改为 `degraded`；
- `schedule_summary._hard_constraints()` 只在 `freeze_state == "active"` 时保留 `freeze_window`；
- 摘要、历史、页面至少两处要能看到“冻结未生效”。

### 5. 模板 / 外部组缺失不再静默降级
**目标文件**：
- `core/services/scheduler/schedule_template_lookup.py`
- `core/services/scheduler/schedule_input_builder.py`

**要做的事**：
- 模板缺失记录 `template_missing`；
- 外部组缺失记录 `external_group_missing`；
- 输入构建结果带上 `merge_context_degraded` 与事件；
- 严格模式下直接失败；
- 非严格模式下允许继续，但摘要必须显示“组合并语义已退化”。

### 6. `active_preset` 与真实配置同步
**目标文件**：
- `core/services/scheduler/config_service.py`
- `core/services/scheduler/config_presets.py`
- `core/services/scheduler/config_validator.py`

**要做的事**：
- 所有 `set_*()` 写接口成功后统一切到 `custom`；
- `apply_preset()` 后比较“目标预设快照”与“最终写入快照”，若不完全一致则切到 `custom`；
- 预设应用过程若发生修补、回退、自动规范化，必须输出 `active_preset_reason`；
- 只有真正未修补、未降级、与预设完全一致时，`active_preset` 才能保留原预设名。

## 实施顺序
1. 先收口配置快照、预设归一、优化器与算法参数里的重复解析入口。
2. 再改输入构建，让严格语义真正贯穿到算法入口。
3. 接着改 `SGS` 预排序与评分链，避免算法内部重新长出数值兜底。
4. 然后修冻结窗口失败关闭与摘要硬约束口径。
5. 再把模板 / 外部组缺失纳入结构化退化事件。
6. 最后修 `active_preset` 与真实配置的一致性。
