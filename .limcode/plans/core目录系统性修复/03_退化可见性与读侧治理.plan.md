# core目录系统性修复：03_退化可见性与读侧治理

## 这一段要解决什么问题
前两段解决的是：不要再制造新的坏语义，也不要让严格模式继续漏风。

这一段要解决的是：**历史脏数据和退化路径就算暂时不能一次性清干净，也绝不能继续装作没发生。**

当前很多问题不是没处理，而是“处理得太安静”：
- 坏交期会被当成没交期；
- 未排上的批次在指标里直接消失；
- 历史外协周期还可能默认成一天，但没人知道用了这条兼容；
- 甘特日历加载失败时，前端会用推断日历顶上，用户看不出来；
- 恢复链只是复制了文件，还没完成结构校验，就已经像成功一样返回；
- 系统配置脏值被钳制后，看起来像本来就合法；
- `SystemJobState.last_run_time` 坏掉后，维护任务可能被误当成“从未运行”；
- 人机关联详情把脏关联归一成正常展示，用户不知道底层已经脏了；
- 单元产品转换过程里的默认补齐继续发生，但没有诊断出口；
- 预热失败、插件启动留痕失败，常常只在日志里知道。

这一段的任务，就是把这些“静默兼容”改成“允许兼容，但必须可见、可计数、可追溯”。

2. 历史兼容回退必须有统一原因码、统一计数、统一外显。
3. 甘特日历失败、恢复链复制未校验、系统配置脏值、维护任务脏时间、预热失败、插件留痕失败都要有对外可见的退化事实。

5. 页面、摘要、历史、导出尽量共用同一套退化信息源。

## 改后对外口径

### 1. 排产摘要与历史必须新增的稳定字段
优先在 `schedule_summary.py` 结果中统一新增或稳定以下字段：
- `invalid_due_count`
- `invalid_due_batch_ids_sample`
- `unscheduled_batch_count`
- `legacy_external_days_defaulted_count`
- `degradation_events`
- `degradation_counters`

约束：
- `invalid_due_count` 与 `unscheduled_batch_count` 必须既能进摘要，也能随历史写回；
- `legacy_external_days_defaulted_count` 只能代表历史兼容链使用次数，不能被误解为正常业务值；
- 页面展示和历史查看页应复用同一组字段，不再各自重新数一遍。

### 2. 恢复链返回码口径
`RestoreResult.code` 至少要稳定区分：
- `copied_pending_verify`
- `verified`
- `failed`

约束：
- 只有 `verified` 才允许被页面映射成“恢复成功”；
- `copied_pending_verify` 必须明确提示“文件已复制，结构仍待校验”；
- `backup_restore()` 路由的提示文案和操作日志必须跟 `code` 对齐。

### 3. 系统配置脏值口径
`SystemConfigSnapshot` 至少新增：
- `dirty_fields: list[str]`
- `dirty_reasons: dict[str, str]`

约束：
- 数值被钳制、回退、修正时，必须记录字段名；
- 页面提示从 `dirty_fields` / `dirty_reasons` 取数，不再靠说明文字猜；
- 兼容读取的结果不得伪装成“本来就合法”。

### 4. 维护任务时间脏值口径
维护任务状态至少新增：
- `last_run_state: "valid" | "missing" | "invalid"`
- `last_run_raw: str | None`

约束：
- `last_run_state == "invalid"` 时，不得再并入“从未运行”；
- 页面、任务结果或操作日志至少一处要明确显示“最后执行时间损坏”；
- 保留原始坏值文本，便于后续修数。

### 5. 人机关联与转换脚本诊断口径
- 人机关联查询结果至少新增：
  - `dirty_fields`
  - `dirty_reasons`
- 单元产品转换结果至少新增：
  - `diagnostics.events`
  - `diagnostics.counters`
  - `diagnostics.samples`

约束：
- 人机关联详情页必须能显示“当前展示包含兼容归一结果”；
- 转换脚本标准输出必须区分“普通成功”和“带退化成功”。

### 6. 预热失败与插件留痕失败口径
统一使用结构化退化事件，不再只靠日志：
- `ortools_warmstart_failed`
- `plugin_bootstrap_telemetry_failed`
- `plugin_capability_conflict_detected`（若上一段已接入冲突状态，当前段负责外显）

## 不做的事
这一段不做：
- 事务边界收口；
- 公共执行器收口；
- `ScheduleService` 大拆分；
- 归一化矩阵统一；
- 插件 capability 冲突策略本身的底层实现。

这一段只聚焦一件事：**退化事实必须被看见。**

---

## 完成标准
1. 摘要与历史中能看到坏交期数量、未排批次数量和历史兼容回退计数。
2. 甘特日历失败、恢复链复制未校验、系统配置脏值、维护任务脏时间、预热失败、插件留痕失败都可对外可见。
3. 人机关联读侧归一和单元产品转换默认补齐不再静默伪装成普通成功结果。
4. 页面、摘要、历史、导出之间不再出现大口径断裂。
5. 回归测试能够锁住上述可见性要求。

---
