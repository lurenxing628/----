## 当前代码核实结论
已按当前工作区代码逐项核实，两份 review 中下列问题仍然成立：

1. **恢复链仍不原子**：`BackupManager.restore()` 的自动回滚只覆盖“复制阶段异常”；`web/routes/system_backup.py` 在 `copied_pending_verify` 后执行 `ensure_schema()`，失败时只提示错误，不回滚主库。
2. **甘特页仍吞掉通用退化事实**：`static/js/gantt_boot.js` 仍只识别 `calendar_load_failed`；`static/js/gantt_render.js` 在无任务时仍统一显示“暂无排程数据”。
3. **插件配置读取失败仍可能静默退回默认**：`bootstrap_plugins()` 只对留痕失败追加退化；数据库连接失败、仓储构造失败、单插件配置读取失败仍只写日志或直接回 `None`。
4. **人机关联空白历史值仍无 dirty 标记**：`OperatorMachineQueryService._normalize_row()` 仅在原值非空时标记，空白 `skill_level` / `is_primary` 会被静默归一为 `normal` / `no`。
5. **排产摘要顶层退化事件仍未成为统一主事实源**：`schedule_summary.py` 顶层 `degradation_events/degradation_counters` 目前未并入冻结窗口、停机避让、资源池退化；分析页仍优先读取嵌套 `algo.freeze_window`。
6. **`parse_finite_float()` 默认值 BUG 仍在**：类型重载默认 `allow_none=False`，实现默认却仍是 `allow_none=True`。
7. **`field_parse.py` 仍有三个未收口点**：
   - `collector=None` 时创建临时收集器，事件会丢失；
   - 最小值违反仍转入 `parse_compat_float()` / `parse_compat_int()`，事件语义仍落成泛化的 `invalid_number`；
   - 最小值违反路径会重复解析同一值。
8. **`config_validator.py` 非严格路径仍双调用**：当前实现仍靠“一次严格解析 + 一次非严格解析”表达“非法数字报错、下界越界兼容回退”，意图不透明。
9. **`BuildOutcome` 计数重叠风险仍在**：`__post_init__()` 仍允许 `events` 导出的原因码与外部 `counters` 同名后双重累加。
10. **`field_parse.py` 仍缺专门回归**：`tests/**/*field_parse*.py` 当前为空。
11. **局部解析函数守门仍缺失**：`tests/test_architecture_fitness.py` 仍没有“禁止新增局部 `_safe_*` / `_cfg_*` / `_get_*`”的守门；当前存量至少包括：
    - `core/services/scheduler/batch_service.py:_safe_float`
    - `core/services/scheduler/_sched_utils.py:_safe_int`
    - `core/algorithms/greedy/scheduler.py:_safe_int`
    - `core/services/scheduler/config_validator.py:_get_float/_get_int`
    - `core/services/scheduler/config_snapshot.py:_get_float/_get_int`
    - `core/services/scheduler/schedule_optimizer_steps.py:_cfg_float/_cfg_int`
12. **`STABLE_DEGRADATION_CODES` 仍是死声明**：当前只定义未校验，也没有测试保证实际使用原因码受控。
13. **展示构建文件仍保留冗长拆散导入**：`gantt_tasks.py`、`gantt_week_plan.py`、`resource_dispatch_rows.py` 仍是多段独立导入。

同时也确认了几条“做了半截但还没闭环”的现状，这些必须作为修复设计前提，而不是另起一套：
- `copied_pending_verify` / `verified` 返回码已存在，但“校验失败后的补偿”尚未接上；
- 周计划页已经能消费 `bad_time_row_skipped` 与 `all_rows_filtered_by_invalid_time`，甘特页应直接复用同一事实口径；
- 插件留痕失败已经能进 `degradation_events`，配置读取失败应沿同一外显通道补齐，而不是再造第二套提示结构。

## 统一修复原则
1. **只有 `verified` 才算恢复成功**。复制成功、校验失败、回滚成功、回滚失败必须是不同稳定状态。
2. **退化事实只有一个主事实入口**。顶层 `degradation_events/degradation_counters` 承担主事实，嵌套结构保留为明细，但不再承担主口径。
3. **兼容可以保留，但不能静默**。凡是默认化、兼容归一、过滤、降级，都必须留下结构化事实，并由页面或摘要消费。
4. **对编程错误优先快速失败**。像 `BuildOutcome` 计数重叠、共享解析函数漏传收集器这类问题，不继续靠“默默兼容”掩盖。
5. **不新增局部解析器**。需要过渡包装时，只允许做薄转调并显式登记；不得再长出自带默认化语义的新 `_safe_*` / `_cfg_*` / `_get_*`。
6. **只做与当前缺口直接相关的最小改动**。私有 helper 可以抽，但不为当前缺口再造新的公共层、第二份诊断对象或多层回退分支。

### 批次 B：共享底座合同精修（类型安全 + 共享解析 + 结果封套）
**覆盖问题**：
- `parse_finite_float()` 默认值不一致
- `field_parse` 最小值违反语义不精确
- `field_parse` 在 `collector=None` 时事件丢失
- 多层解析链重复解析
- `config_validator` 双调用意图不透明
- `BuildOutcome` 计数重叠风险
- `field_parse.py` 缺专门回归

**目标文件**：
- `core/services/common/number_utils.py`
- `core/services/common/field_parse.py`
- `core/services/common/compat_parse.py`
- `core/services/common/build_outcome.py`
- `core/services/scheduler/config_validator.py`
- `tests/regression_number_utils_facade_delegates_strict_parse.py`
- 新增 `tests/regression_field_parse_contract.py`（或同等职责文件）
- `tests/regression_build_outcome_contract.py`

**具体动作**：
1. **修正 `parse_finite_float()` 的实现默认值**
   - 让运行时默认值与重载声明一致，默认行为必须与 `allow_none=False` 对齐。
   - 补一条直接断言默认行为的回归，不只测“是否转调”。
2. **把 `field_parse` 的收集器改成必传合同**
   - 先按当前全仓调用点核实所有现有调用都已显式传入 `collector`；
   - 随后取消 `collector=None` 的临时收集器回退，杜绝“事件生成了但立刻丢失”。
3. **最小值违反改为本层直接处理，不再回流到兼容解析重跑**
   - `field_parse` 在“值可解析，但低于下界”时直接做最小值违反处理：
     - 使用更精确的稳定原因码（建议统一为 `number_below_minimum`，由 `field` 区分具体字段）；
     - 直接产出一致消息；
     - 直接返回兼容回退值；
     - 不再把同一个原值重新送进 `parse_compat_*()` 重跑。
   - 这样同时解决“语义不精确”和“重复解析”两个问题。
4. **把 `config_validator` 的特殊合同写清楚并收成单次判定路径**
   - 目标合同保持不变：**非法数字直接报错，仅下界越界允许兼容回退**。
   - 优先方案是：先做一次严格数值解析，显式区分“非法数字”和“数值合法但低于下界”；仅在后者路径产出统一退化事件并回退。
   - 不再保留“两次 `parse_field_*()` 调用 + 丢弃第一次返回值”的隐式写法；若确需保留局部薄包装，也只能负责本地语义命名，不再夹带第二层解析行为。
5. **`BuildOutcome` 对计数重叠改为快速失败**
   - 当 `events` 导出的原因码与外部 `counters` 同名时，直接视为调用方合同错误；
   - 运行时抛出清晰异常，不再继续静默双倍累加；
   - 回归测试直接锁住“重叠即失败”的合同。
6. **补齐 `field_parse` 专门回归**
   - 至少覆盖：
     - 严格模式直接报错；
     - 非严格模式兼容回退留痕；
     - 最小值违反产生精确原因码；
     - 结果不重复解析、不重复计数；
     - 必传收集器合同。

**完成标准**：
- 共享解析层不再存在“类型声明一套、运行时另一套”的情况；
- 任何兼容回退都不会因为漏传收集器而静默丢事件；
- `config_validator` 的非严格语义可一眼读懂；
- `BuildOutcome` 遇到编程错误时快速失败，不再悄悄累错计数。

## 验收清单
本 plan 完成时，至少满足以下验收条件：
1. `ensure_schema()` 失败后，恢复链能稳定区分“已回滚”和“回滚失败”，且不会写成功日志。
2. 甘特页对 `calendar_load_failed`、`bad_time_row_skipped`、`all_rows_filtered_by_invalid_time` 都有明确前端表现。
3. 插件配置读取失败不再只存在于日志里；页面能看到“当前按默认开关运行”的结构化事实。
4. 顶层 `degradation_events/degradation_counters` 能覆盖冻结窗口、停机避让、资源池退化；分析页优先消费统一事实源，但仍保留必要的嵌套明细展示。
5. 空白 `skill_level/is_primary` 历史值会留下 dirty 标记并出现在页面提示中。
6. `parse_finite_float()` 默认行为与类型声明一致。
7. `field_parse` 有专门回归，且不再隐式丢事件、不再用泛化原因码描述下界越界。
8. `BuildOutcome` 不再默许事件计数与外部计数重叠，而是直接快速失败。
9. 架构适应度新增局部解析函数守门；稳定原因码有实际测试约束。
