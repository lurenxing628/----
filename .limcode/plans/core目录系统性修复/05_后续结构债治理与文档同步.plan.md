## 统一执行节奏（所有批次必须遵守）

1. **先做现状诊断**：记录当前文件行数、目标函数位置、显式登记状态，以及直接引用链，不再先删白名单再猜测是否需要重构。
2. **先跑基线门禁**：在改动前跑与本批次直接相关的架构适应度与回归，并对目标热点函数补一次 `radon` 实测留痕，确认当前基线稳定。
3. **先迁最确定的职责边界，再测量是否继续拆分**：先做能够明确降低耦合的移动，再根据文件行数与复杂度复测决定是否新增模块；不得为了预设文件结构而提前搬运代码。
4. **债务真实消除后再删显式登记**：只有在文件行数 / 圈复杂度实际降到门禁以内后，才允许修改 `known_oversize` 或 `known_violations`；若未达标，必须继续拆分或重新开 review，不得靠新增白名单或调整门禁口径收口。
5. **禁止新增静默回退与广义异常兜底**：
   - 新增代码不得使用 `except Exception`、`except:` 或“捕获后直接返回默认值”作为正常控制分支；
   - 写路径的校验失败必须显式抛错，不得以默认值补齐后继续落库；
   - 读侧摘要只允许显式呈现已有退化事实，不得为了“字段完整”偷偷补 `0` / `[]` / `{}` 掩盖数据问题。
6. **禁止新增泛化杂物间文件**：
   - 不得新增 `payload`、`support`、`helpers_v2` 这类职责模糊的命名；
   - 若必须新增文件，文件名必须直接表达职责，并且只承接一个明确语义域。
7. **每批结束都做文档与快检收口**：同步受影响文档、跑快检、补一轮 review，并把剩余债务清单写清楚。

10. 拆分后 `schedule_summary.py` 仍是唯一的公开入口：`schedule_orchestrator.py`、`schedule_service.py` 等外部调用者仍只 `from .schedule_summary import build_result_summary`。
