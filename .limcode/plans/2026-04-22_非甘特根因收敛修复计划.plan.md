# 非甘特根因收敛修复计划

> 执行方式：优先使用 `subagent-driven-development`；如当前会话直接执行，则按 `executing-plans` 拆分并逐步落地。

## 目标

- 收口 config readonly truth-chain，消除 `current_config_state` 的 exact 假阳性
- 建立统一的 `ErrorCode -> HTTP` 语义映射，并把 route 本地翻译收敛到白名单
- 把 version 三类合同显式化，阻止 shared helper 继续外溢
- 补齐 warning/provenance 的显示与 rendered HTML 守卫

## 总体原则

1. 不做“大一统重写”，优先修最小 owner
2. 先补会失败的合同测试，再改生产逻辑
3. page/export/bulk 的既有 UX 契约保留，data/API 优先统一
4. provenance 与 warning 分两条线处理，不再混成一个问题

## 执行顺序建议

1. Phase 1
2. Phase 2
3. Phase 3
4. Phase 4

原因：

- Phase 1 和 Phase 2 直接影响 correctness 与 HTTP 语义
- Phase 3 是合同显式化，能降低后续再漂移
- Phase 4 主要是可观测性与守卫补强，建议同批但后置

## 验收标准

- config 页面与 batches 页面不再对 drifted baseline 报 exact
- 未捕获 `AppError` 不再统一压平成 400
- version 三类合同有明确 owner、明确测试、明确命名
- week-plan/history 对 warning/provenance 至少有 rendered HTML guard
- 新增守卫已进入 quality gate 白名单
