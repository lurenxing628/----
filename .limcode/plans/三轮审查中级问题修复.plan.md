# 三轮审查中级问题修复

## 来源
`.limcode/review/core目录系统性修复-00至02段-三轮实现审查.md` 中 4 个中级问题。

## 本次修订说明

在执行前，已按源码做过一轮交叉核对，确认原始 plan 需要做 3 处修订后再实施：

1. `config_validator._get_int()` 改用 `parse_field_int()` 时，必须保留当前“最小值钳制”语义，不能退回运行时默认值。
2. `schedule_service` 的告警合并逻辑可以简化为一行 `summary.warnings.extend(algo_warnings)`，但不能顺手删除 `warning_merge_status` 这条既有摘要合同；`schedule_summary.py` 仍在消费它，现有回归也在校验它。
3. 展示层共享常量不仅在 `gantt_tasks.py` / `gantt_week_plan.py` / `resource_dispatch_rows.py` 三处存在，`resource_dispatch_support.py` 也消费同一空结果原因码；此外 `resource_dispatch_rows.py` 的提示文案是“排班行”，不能在抽取时被悄悄改成“排程行”。

## 问题1：config_validator 仍保留独立回退逻辑

### 现状
`core/services/scheduler/config_validator.py` 的 `_get_float()` / `_get_int()` 在非严格模式下仍自行处理最小值违规与默认回退，不走 `field_parse.py`，也不会把坏值回退事件注入 `ScheduleConfigSnapshot.degradation_events` / `degradation_counters`。

### 改后口径
- `_get_float()` / `_get_int()` 全部改为调用 `parse_field_float()` / `parse_field_int()`。
- `normalize_preset_snapshot()` 顶层创建一个 `DegradationCollector`，传给所有数值字段解析入口。
- 非严格模式下，整数项仍保持原有“最小值钳制”语义：
  - `ortools_time_limit_seconds < 1` → 归一化为 `1`
  - `time_budget_seconds < 1` → 归一化为 `1`
  - `freeze_window_days < 0` → 归一化为 `0`
- 归一化完成后，把 `collector` 中的退化事件注入返回的 `ScheduleConfigSnapshot`。

### 具体改动

**文件：`core/services/scheduler/config_validator.py`**

1. 新增导入：
   - `DegradationCollector`
   - `degradation_events_to_dicts`
   - `parse_field_float`
   - `parse_field_int`
2. 删除 `parse_required_float` / `parse_required_int` 的直接导入。
3. 在 `normalize_preset_snapshot()` 顶层新增 `collector = DegradationCollector()`。
4. `_get_float()` 改为统一走 `parse_field_float()`，保持：
   - 空值仍直接回退为 `default`
   - 其他坏值在非严格模式下通过 `field_parse` 产出退化事件
5. `_get_int()` 改为统一走 `parse_field_int()`，并显式传入：
   - `min_value=min_v`
   - `min_violation_fallback=int(min_v)`
   以保留当前最小值钳制口径，而不是退回 `default`。
6. 返回的 `ScheduleConfigSnapshot(...)` 新增：
   - `degradation_events=tuple(degradation_events_to_dicts(collector.to_list()))`
   - `degradation_counters=collector.to_counters()`

### 额外回归要求
新增一条**直测 `normalize_preset_snapshot()`** 的回归，至少覆盖：
- 非严格模式下整数最小值违规仍然钳到最小值
- 非严格模式下坏值回退会产生 `degradation_events`
- `degradation_counters` 中能看到对应原因码

### 不做的事
- 不改变选择型字段（`sort_strategy` / `dispatch_mode` / `dispatch_rule` / `algo_mode` / `objective`）的现有回退逻辑。
- 不改 `_yesno()` 的布尔回退逻辑。

## 问题2：schedule_service 告警合并逻辑过于防御

### 现状
`core/services/scheduler/schedule_service.py` 当前用多层 `try/except` 包裹 `summary.warnings.extend(algo_warnings)`，其中还伴随 `warning_merge_status` 的失败分支回写。

### 交叉核对结果
- `summary` 的合同来自 `core/algorithms/types.py` 中的 `ScheduleSummary`，其 `warnings` 明确声明为 `List[str]`。
- 但 `warning_merge_status` **不能整体删除**：`core/services/scheduler/schedule_summary.py` 仍会消费它，并输出到 `result_summary_obj["algo"]["warning_pipeline"]`；现有回归也覆盖了这条结构化摘要合同。

### 改后口径
- 合并逻辑简化为：
  ```python
  if algo_warnings:
      summary.warnings.extend(algo_warnings)
  ```
- 删除多层 `try/except` 和兜底回退。
- **保留** `warning_merge_status` 这条结构化摘要输入，但固定表达为“已尝试、未失败、无错误”：
  ```python
  warning_merge_status = {
      "summary_merge_attempted": bool(algo_warnings),
      "summary_merge_failed": False,
      "summary_merge_error": None,
  }
  ```
- `schedule_summary.py` 不做结构删改，继续沿用现有 `warning_pipeline` 输出口径。

### 具体改动

**文件：`core/services/scheduler/schedule_service.py`**

1. 删除当前 `summary.warnings.extend(...)` 周围的多层异常吞并逻辑。
2. 保留 `warning_merge_status` 字典，但改为固定成功态。
3. 将告警合并简化为一行 `summary.warnings.extend(algo_warnings)`。
4. 保持向 `build_result_summary()` 透传：
   - `algo_warnings`
   - `warning_merge_status`

### 不做的事
- 不修改 `ScheduleSummary` 的字段定义。
- 不删 `schedule_summary.py` 中的 `warning_pipeline` 结构。
- 不新增额外防御性 `list()` 转换。

## 问题4：BuildOutcome 计数合并可能重复累加

### 现状
`core/services/common/build_outcome.py` 的 `__post_init__` 会把 `events` 导出的计数与外部 `counters` 叠加合并；如果未来调用方同时传入重叠键，会出现双倍计数。

### 改后口径
不改行为，只补充注释，明确合并规则与误用风险。

### 具体改动

**文件：`core/services/common/build_outcome.py`**

1. 在 `__post_init__()` 上方添加注释，说明：
   - `events` 导出的原因码计数会先汇总
   - 外部 `counters` 会在此基础上继续叠加
   - 若两边键重叠，会出现双倍计数
2. 在 `from_collector()` 的文档字符串中明确：
   - `counters` 仅应用于补充 `collector` 事件之外的额外统计
   - 若和 `collector` 事件原因码重叠，最终会叠加

### 不做的事
- 不改变 `__post_init__()` 当前合并逻辑。
- 不增加运行时去重检查。
