# 补充两条针对性回归测试消除假覆盖

## 背景

两份独立深度审查（`第三轮未提交修改深度审查.md`、`20260405_未提交修改三轮深度审查.md`）一致指出同样的两个测试盲区：

1. **SGS 探测计数假覆盖**（两份报告均发现）：`regression_sgs_scoring_fallback_unscorable.py` 的 probe_keys 断言构造了两个外协工序，SGS 评分阶段走的是外协估算分支（`sgs.py:138`），永远不会进入内部工序自动分配探测路径（`sgs.py:227-250`）。6 个 `probe_keys` 断言以零值平凡通过——即使删掉 `probe_only` 修复，测试仍然通过。
2. **工作日历严格数字缺覆盖**（第二份报告显式发现，第一份隐含确认）：`scheduler_excel_calendar.py` 的 preview/confirm 两处 `validate_row` 闭包已改用 `parse_finite_float`，但现有测试均未直接喂入 `True/NaN/Inf` 来验证该修复点，且两份重复闭包的同步性无回归保障。

两个问题的共性：**实现本身正确，但缺少能真正锁住修复的回归测试**。本次只补测试，不改业务代码。

## 任务 1：补充 SGS + 内部工序 + 自动分配探测的回归测试

### 目标

在 `tests/regression_sgs_scoring_fallback_unscorable.py` 中新增一个测试函数，构造能真正命中 `_auto_assign_internal_resources(..., probe_only=True)` 的场景，断言评分探测不污染 `fallback_counts`。

### 触发条件分析

要命中 `sgs.py:230` 的 `auto_assign_enabled and resource_pool is not None` 分支，工序必须满足：
- `source` 为 `"internal"`（跳过外协分支 → 进入 `sgs.py:227 else`）
- `machine_id` 或 `operator_id` 为空（触发自动分配尝试）
- 调用方传入 `auto_assign_enabled=True` + `resource_pool` 非 None

同时为了让自动分配在探测阶段产生计数（如果 `probe_only` 不生效的话），需要让 `auto_assign_internal_resources` 走到某个 `_count(...)` 调用。最简单的方式：`op_type_id` 为空 → 命中 `auto_assign.py:49` 的 `auto_assign_missing_op_type_id_count`。

### 实现方案

在现有 `main()` 函数末尾，新增 `_test_probe_only_internal_ops()` 调用：

```python
def _test_probe_only_internal_ops():
    """验证 SGS 评分阶段的自动分配探测不污染 fallback_counts。"""
    # 构造：两个内部工序，缺 machine_id/operator_id/op_type_id，
    # 开启自动分配 + resource_pool 非空 → 评分阶段走 probe_only=True
    batch = SimpleNamespace(
        batch_id="B_INT", priority="normal", due_date="2026-01-10",
        ready_status="yes", ready_date=None, created_at=None, quantity=1,
    )
    op = SimpleNamespace(
        id=10, op_code="OP_INT", batch_id="B_INT", seq=1,
        source="internal",
        machine_id="", operator_id="",  # 缺资源 → 触发自动分配
        setup_hours=1.0, unit_hours=0.5,
        op_type_id=None,  # 缺 op_type → auto_assign 走 missing_op_type_id 分支
        op_type_name=None, supplier_id=None,
        ext_days=None, ext_group_id=None, ext_merge_mode=None, ext_group_total_days=None,
    )

    sched = GreedyScheduler(calendar_service=_StubCalendarService())
    results, summary, _strategy, used_params = sched.schedule(
        operations=[op],
        batches={"B_INT": batch},
        start_dt=datetime(2026, 1, 1, 8, 0, 0),
        dispatch_mode="sgs",
        dispatch_rule="slack",
        auto_assign_enabled=True,
        resource_pool={},  # 非 None → 触发自动分配
    )

    algo_stats = getattr(sched, "_last_algo_stats", {}) or {}
    fallback_counts = algo_stats.get("fallback_counts", {}) or {}

    # 核心断言：评分阶段 probe_only=True → auto_assign 的计数不应出现
    assert fallback_counts.get("auto_assign_missing_op_type_id_count", 0) == 0, (
        f"评分探测污染了 auto_assign_missing_op_type_id_count="
        f"{fallback_counts.get('auto_assign_missing_op_type_id_count', 0)}，"
        "probe_only=True 时不应记数"
    )

    # 同时验证正式排产阶段的自动分配（非 probe_only）确实记了数
    # SGS 评分后选中候选 → 进入正式排产 → _schedule_internal →
    # auto_assign_enabled + resource_pool → 调 auto_assign（probe_only=False）→ 因 op_type_id 缺失而失败记数
    assert int(fallback_counts.get("internal_auto_assign_attempt_count", 0)) >= 1, (
        f"正式排产阶段应尝试自动分配：{fallback_counts!r}"
    )
    assert int(fallback_counts.get("internal_auto_assign_failed_count", 0)) >= 1, (
        f"正式排产阶段自动分配应失败（缺 op_type_id）：{fallback_counts!r}"
    )
```

### 为什么这个测试能真正覆盖

- 若删掉 `probe_only` 透传（`sgs.py:249`），评分阶段调用 `auto_assign` 时 `probe_only` 默认为 `False`，`_count` 就是 `increment_counter` → `auto_assign_missing_op_type_id_count` 被评分阶段累加 → 断言失败
- 若 `probe_only` 正常工作，评分阶段的 `_count` 是空操作 → 该计数只来自正式排产阶段（`_schedule_internal` 中的调用，那里不传 `probe_only`）

### 需要注意

- `_StubCalendarService` 需要支持 `operator_id` 关键字参数（现有桩的 `adjust_to_working_time` 和 `add_working_hours` 缺少 `operator_id`），与 `regression_greedy_scheduler_algo_stats_auto_assign.py` 中的桩签名保持一致

## 任务 2：补充工作日历 Excel 路由严格数字回归测试

### 目标

新增 `tests/regression_scheduler_excel_calendar_strict_numeric.py`，直接走 `/scheduler/excel/calendar/preview` 路由，喂入包含 `True`、`NaN`、`Inf` 的"可用工时"和"效率"单元格，断言 preview 返回的结果中这些行被标记为错误。

### 实现方案

复用现有 `regression_scheduler_excel_calendar_uses_executor.py` 的测试基础设施模式（创建临时库 + 通过 `app.test_client()` 发 HTTP 请求），但只关注 preview 阶段的校验行为：

```python
def main():
    # ... 标准项目根查找 + 临时库创建 + app 构造（同现有模式） ...

    # 构造含非法数值的 Excel
    invalid_rows = [
        {"日期": "2026-05-01", "类型": "workday", "可用工时": True, "效率": 1.0, ...},
        {"日期": "2026-05-02", "类型": "workday", "可用工时": 8, "效率": float("nan"), ...},
        {"日期": "2026-05-03", "类型": "workday", "可用工时": float("inf"), "效率": 1.0, ...},
        {"日期": "2026-05-04", "类型": "workday", "可用工时": 8, "效率": 1.0, ...},  # 正常行
    ]

    preview_resp = client.post("/scheduler/excel/calendar/preview", ...)
    # 断言：HTTP 200（preview 页面正常渲染）
    # 断言：前 3 行被标记为错误（包含"必须是数字"或"必须是有限数字"）
    # 断言：第 4 行为正常行
```

### 关键设计决策

- **只测 preview，不测 confirm**：preview 和 confirm 的 `validate_row` 逻辑完全相同（都调用 `parse_finite_float`），preview 是第一道防线。若 preview 能正确拒绝，confirm 阶段根本收不到含非法值的行。测一处即可锁住修复点，避免测试与实现过度耦合。
- **通过 HTML 内容断言错误**：preview 响应页面中，每一行的错误信息会渲染在 HTML 里。解析 `raw_rows_json` 中的 `__status` 或页面上的错误文案来确认拒绝行为。
- **不引入额外依赖**：只用 `openpyxl`（已是项目依赖）构造测试 Excel。

### 边界说明

- `True`/`False` → `isinstance(value, bool)` → 被 `_parse_finite_float` 拒绝（"必须是数字"）
- `float("nan")` / `float("inf")` → `math.isfinite = False` → 被 `_parse_finite_float` 拒绝（"必须是有限数字"）
- 注意：openpyxl 写入 `float("nan")` / `float("inf")` 后再读回的行为需要在测试中确认（可能被序列化为字符串 `"nan"`/`"inf"`，但 `float("nan")` 仍可被 `float()` 解析 → `math.isfinite` 仍能拒绝）
- `True`/`False` 在 openpyxl 中是合法的单元格值（布尔类型），读回后仍为 `bool` 类型

- **不合并两份 `validate_row` 闭包**：属于代码重复但非 BUG，不在本次范围内（若后续重构可单独处理）
