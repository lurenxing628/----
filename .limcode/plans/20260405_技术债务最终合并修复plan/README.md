# APS 技术债务最终合并修复 plan（拆分索引）

> 说明：本目录承接原主 plan 的拆分结果。为兼顾原入口稳定与拆分后文档集中管理，执行者应先看本索引，再进入对应子 plan 开工。各子 plan 都写明了背景、前后承接关系、只做什么 / 不做什么、完成判定与最小验证命令，避免阶段无法连续。

## 一、背景与拆分原因

原始总 plan 已覆盖本轮 review 确认的问题，但单文件同时承担了路线图、目录迁移清单、执行步骤、验证命令、问题覆盖矩阵与 review 吸收闭环六种职责，导致执行者难以在 5 分钟内定位“当前批次到底先做什么、不能做什么、做完交给谁”。

本次拆分遵循三条原则：
1. **索引与执行分离**：本文件只保留边界、里程碑、结构决策、硬约束与交付口径；细节下沉到子 plan 与附录。
2. **路径迁移与行为改造分离**：先做 `SP05` 目录骨架和路径门禁；`SP05` 完成后，已迁组统一按迁移后真实实现路径执行，延后组在 `SP08/SP09` 触达前仍以根层稳定入口为准。凡涉及路径迁移，必须把真实实现迁移、导入改写、根层兼容薄门面回填与路径敏感门禁更新放进同一原子步骤，避免中间态断裂。
3. **持续反兜底**：拆分后任何子 plan 都不得靠新的静默回退、隐藏 BUG 的空值返回、签名探测、过度防御性包装去维持“假通过”；`SP05` / 附录 C 明示的根层兼容薄门面仅允许零逻辑单层转发，不属于可扩张兼容桥。

## 二、固定边界

1. `pyproject.toml` 中 `requires-python = ">=3.8"`、`target-version = "py38"` 视为当前交付边界。
2. Win7 / Python 3.8 / `Chrome109` / 现有安装包链路 / 现有数据库迁移链路不得被本轮整改破坏。
3. Win7 与 Python 3.8 的长期升级风险继续视为接受风险，只留痕，不并入本轮主动整改。
4. 本轮只做“可回归、可验证、可分批落地”的治理，不做脱离当前交付边界的大规模重写。
5. 任何未在子 plan 中明确允许的新总抽象、新全局注册表、第二事实源、临时兼容桥，默认禁止；仅 `SP05` / 附录 C 明示的根层兼容薄门面可以存在，且必须零逻辑、单层转发、可回收。

## 三、关键结构决策

1. **请求级装配**：统一通过 `g.services` 承接请求级服务取用；维护任务继续直接构造，不倒挂到容器里。
2. **目录重组**：`core/services/scheduler/` 先落 `config/run/summary` 骨架；`web/routes/` 先把 scheduler 域全部 route 实现收口到 `domains/scheduler`，根层 `scheduler.py` 保留一级门面并只导向二级聚合链；其余服务分组与非 scheduler 域 route 按 `SP08` / `SP09` 触达时同步推进。
3. **配置单一事实源**：调度配置规则统一收敛到注册表；页面层至少对 `algo_mode/objective/dispatch_mode/dispatch_rule` 四个核心字段的选项值来源与后端保持一致。
4. **主链边界**：排产主链统一收口到稳定调用面（当前以 `ScheduleRunInput`、`ScheduleOrchestrationOutcome`、`SummaryBuildContext` 为准）、状态语义模块与稳定编排结果；禁止继续靠兼容桥、`getattr(...)` 猜对象形态、双路径猜测或过度防御包装掩盖契约缺口；`SP05` 明示保留的根层兼容薄门面只允许单层转发。
5. **页面协议**：Excel 页面走“页面流程 + 领域钩子”；前端启动采用 `page_boot.js + data-page` 试点落地；`_safe_next_url` 统一收口到 `web/routes/navigation_utils.py`；旧全局入口只允许收口，不允许继续扩张。
6. **测试与文档**：测试最终落到分层目录；文档、证据、审计三条轨道分开维护，附录只做导航，不做执行依据。
7. **反静默回退**：本轮所有子 plan 都继承统一禁令——不得新增静默回退、隐藏 BUG 的空值返回、过度防御性编程，不得把真实契约问题伪装成“兼容成功”。

- **主链串行**：`SP07` 内部固定为 A（输入边界 / 稳定调用面 / 状态语义 / 缓存归属）→ B（编排 / 持久化 / 摘要 / 门面收短）；A 未通过前禁止开始 B。
- **算法与页面并行条件**：若为单人执行，建议先 `SP08` 后 `SP09`；若多人并行，需确保不同时改同一门禁文件、路径白名单、兼容薄门面回收矩阵或同一迁移映射清单。
- **`web/ui_mode.py` 边界**：启动链相关守卫归 `SP03`；渲染期桥接、`ui_v2_static` 与版本化 `url_for` 归 `SP09`。
- **测试路径切换时点**：`SP10` 完成前，前序子 plan 继续沿当前根层测试路径执行；`SP10` 完成后再统一切换到分层路径。只有 `SP01/SP02` 创建出来的新目录允许承接后续新增专项回归。

## 六、全局禁止项

1. 禁止新增静默回退、隐藏 BUG 的空值返回、吞异常后继续跑的假成功路径。
2. 禁止新增签名探测、`TypeError` 后重试、第二事实源、过度防御性包装；除 `SP05` / 附录 C 明示的零逻辑兼容薄门面外，禁止引入临时兼容桥。
3. 禁止新增整页复制模板、根目录平铺测试文件、未声明依赖的验证参数或新的全局注册表。
4. 禁止为了“看起来更整洁”而做脱离本轮边界的大规模重命名、整仓统一风格重写或无验证的大搬迁。

## 七、统一执行规则

1. 每份子 plan 都必须写清**背景、前置条件、只做什么、不做什么、承接给谁**，避免阶段断裂。
2. 每个子 plan 先补定向验证，再做最小实现；凡涉及路径迁移的任务，必须把“真实实现迁移 + 导入改写 + 根层兼容薄门面 + 路径敏感门禁/白名单更新”放进同一原子提交；回归通过后再更新治理台账、路径映射和证据留痕。
3. 任一子 plan 如需保留历史残项，必须明确绑定后续子 plan，不允许出现“先白名单、后面再看”。
4. 附录只做导航和汇总，不替代子 plan 正文的执行依据；每个子 plan 必须保留自己的最小验证命令。
5. 涉及路径迁移、白名单、门禁、治理台账、兼容薄门面的改动，必须同步更新对应文档与回归，不允许代码、测试、文档三者脱节。
6. 所有子 plan 都继承反静默回退与反过度兜底要求；`SP05` 允许的根层兼容薄门面仅限零逻辑单层转发，任何超出这一范围的“额外包一层”都必须先说明为什么不是在隐藏 BUG。

## 八、横向治理约束

1. 每完成一个子 plan，都要更新 `开发文档/技术债务治理台账.md` 与必要的阶段留痕。
2. `tests/test_architecture_fitness.py`（迁移后为 `tests/architecture/test_architecture_fitness.py`）、`tests/test_source_merge_mode_constants.py`、`tools/quality_gate_shared.py`、`tools/quality_gate_operations.py`、`tools/quality_gate_scan.py`、`tools/quality_gate_support.py`、治理台账与路径映射附录视为**串行修改文件**；多人并行时不得同时改同一份。
3. `SP08` 与 `SP09` 如并行，只能在互不争用的工作区上并行，提交粒度必须分开，避免一份提交里混入算法、甘特、页面协议和模板桥接多类改动。
4. 任何子 plan 如果发现原 plan 仍有遗漏，不回填到本索引正文扩写，而是补到对应子 plan 或附录中，保持索引稳定。

## 九、最终验收口径

1. 当前 review 已确认、且不依赖环境升级的问题，全部在 10 份子 plan 中有主归属，在附录 B 中有覆盖映射。
2. 所有执行细节已下沉到同名目录，执行者可在 5 分钟内定位当前批次应看的子 plan。
3. `SP05` 之后，已迁组不存在“旧路径、新路径并行猜测”的执行歧义；延后组在触达子 plan 前仍以根层稳定入口为准，并已明确首提交迁移规则；如根层旧路径仍保留，也已被限定为兼容薄门面而非新改动入口。
4. 拆分后没有弱化任何反静默回退、反过度兜底、反第二事实源约束；只是把细节按职责移动到了更合适的文件。
5. 后续实施以本索引 + 对应子 plan 为准；附录用于查长清单、查映射、查命令、查吸收闭环，不单独作为执行底稿。
