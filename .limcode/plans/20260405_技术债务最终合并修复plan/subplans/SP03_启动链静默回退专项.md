# SP03：启动链静默回退专项

## 背景

SP02 已建立门禁与台账，但没有实际清理启动链中的静默回退。原总 plan 中这部分工作量很大，且 `factory.py`、`ui_mode.py` 处在应用生命周期最敏感的入口位置，`runtime_probe.py`、`paths.py` 又分别位于运行时探测 / 路径解析边界，如果不单独拉出来，极易和后续请求级装配、目录迁移、排产主链改造互相污染。

因此，SP03 只处理**启动链、请求入口层与运行时探测 / 路径解析层**的静默回退治理，且按 A / B / C 三层优先级推进，避免把 Win7 打包兼容路径和真正高风险热点混为一谈。这个分层不是为了“少做”，而是为了避免把环境兼容接受风险和真实业务入口缺陷放进同一锅里，最后谁也落不稳。

本子 plan 延续全局约束：
- 不允许新增“更隐蔽的回退”去代替旧回退；
- 不允许用过度防御性包装掩盖真实契约问题；
- 保留历史点时必须是“可观测保留”，不能是“看起来没报错”的假成功；
- `web/ui_mode.py` 在本子 plan 中只处理 `scope_tag=startup_guard` 的启动链守卫，`scope_tag=render_bridge` 的渲染期桥接明确留给 `SP09`。
- `web/ui_mode.py` 的 `scope_tag` 按职责域划分，不等同于纯执行时机；`_log_warning()` 属于启动守卫 / 渲染桥接共用底层助手，默认不在 `SP03` 单独改动函数本体，除非同步覆盖两侧回归。
- 门禁分类仍沿用现有四类（`silent_swallow` / `silent_default_fallback` / `observable_degrade` / `cleanup_best_effort`）；“日志自身安全网”通过处置说明、台账 `notes` 或 `accepted_risks` 表达，不在 `SP03` 扩展第五类。

## 前置条件

- `SP01` 与 `SP02` 已完成。
- 治理台账已能登记 `web/bootstrap/` 白名单与接受风险。
- 启动链专项回归已接入统一门禁入口。
- 执行者理解 A / B / C 三层优先级，不得一上来把 `launcher.py` 拖进逐点清零。

4. 对 C 类热点复核现有落账并补齐接受风险：
   - `web/bootstrap/launcher.py` 的绝大多数异常吞没点
5. 将本批处置结果写回治理台账与启动链回归说明。
6. 向 `SP04` 明确交付“已稳定的启动链基础语义”，避免容器接入时再次踩到静默回退残项。

### 不做什么
1. 不改请求级服务容器，不引入 `g.services`。
2. 不改 `ScheduleService`、排产主链、配置规则或目录结构。
3. 不处理 `web/ui_mode.py` 的 `scope_tag=render_bridge` 符号级条目，以及 `web/bootstrap/static_versioning.py` 的版本化 `url_for` 桥接逻辑；这些按边界归 `SP09`。但 `init_ui_mode()` 作为 `startup_guard` 符号，其内部 5 个 except（含 V2 静态目录告警失败、V2 环境创建失败、V1/V2 globals 注入失败）仍需在本批给出处置决定，不得用“模板叠加环境留给 `SP09`”跳过。
4. 不对 `launcher.py` 做大规模逐点清零；只做接受风险登记与必要的可观测说明。

## 核心红线（执行前先看）

- 不得把 `_open_db` 中 `request.path` 异常改成直接 `return` 或其他短路拒绝；必须按“请求白名单异常链”保留主路径，并把最坏影响链与配套观测写入处置表。
- `_read_ui_mode_from_db()` 的“无数据库连接 → `missing=True` → 回退默认”语义不得改变；维护窗口错误页在 `g.db` 建立前依赖这条守卫合同。
- `get_ui_mode()` / `_read_ui_mode_from_db()` 虽归 `startup_guard`，实际按请求频率执行；补观测必须限频 / 去重，并严格区分启动期 `app.logger` 与请求期 `_log_warning()`。
- 新增观测不得再引入新的 `pass`、空值回退或日志黑洞；无有效 logger 的早期路径必须直接退到标准错误。
- 任何从“静默”改为“可观测”的点都必须有直接断言日志 / 退化信号的专项回归；退出备份专项回归不能替代请求钩子专项回归。

## 执行顺序（细化版）

## 阶段 0：先按 A / B / C 锁定清单，不直接开改
1. 以 `SP02` 的扫描基线为准，把启动链热点拆成 A / B / C 三层清单。
2. 对每层清单写清：
   - 本批是否必须动代码
   - 目标处置类型：清零 / 可观测降级 / 可观测保留 / 清理型尽力而为
   - 预期验证命令
   - 允许保留的原因
   - 是否涉及共用底层助手 / 日志自身安全网
   - A 类文件当前实际 except 总数，以及“组”与“条”的对应关系（至少对齐到 `symbol / except_ordinal`）
3. 明确 `launcher.py` 默认归 C 类，除非出现阻断性 BUG，否则不升级为 A 类。
4. `web/bootstrap/static_versioning.py` 先做边界裁剪：默认按总 plan 交给 `SP09`，并同步把台账 `owner / batch` 从 `SP03` 改绑到 `SP09`；若阶段 0 复核后认定只有 `install_versioned_url_for()` 的启动期注入点需要留在本批，则只升格该条，不得把整文件含混拉回 `SP03`。

**处置类型最小示例**：
- 清零：把真正改变语义且无观测的 `except Exception: pass` 改成显式失败、显式分支或显式拒绝。
- 可观测降级：保留既有降级结果，但要有稳定日志、结构化退化信息或等价可追踪信号。
- 可观测保留：探测型函数继续按可空返回契约返回 `None` / 空结果，但必须在阶段 0 说明这属于“未命中”而不是假成功。
- 清理型尽力而为：`close()`、`remove()`、`shutdown()` 一类收尾动作允许低噪声保留，不把收尾异常升级成新的请求失败。

3. 对重建后的主条目，使用 `set-entry-fields` / `upsert-risk` 回填 `owner`、`batch`、`notes`、`accepted_risks` 等人工字段；仅在条目标识不变时才允许补跑 `refresh-auto-fields`。
