# SP06：配置单一事实源

## 背景

`SP05` 已把本轮后续整改的路径口径固定下来，`SP06` 才能在不再纠结“改旧路径还是改新路径”的前提下，集中处理调度配置规则分散的问题。原总 plan 的任务 4 指向的不是一次普通整理，而是把默认值、合法值、文本规范化、预设同步、算法消费与页面选项来源，收口到**字段注册表 + 统一字段门面 + `value_policies` 分层协作**的结构中，避免同一字段在多个文件里各维护一套“看起来差不多”的规则。

本子 plan 既承接 `SP05`，又直接为 `SP07` 铺路：如果这里没有先把配置语义稳定下来，`SP07` 在收口输入、上下文、编排和持久化边界时，就还得继续猜测配置默认值、枚举集合和文本规范化规则，主链边界不会真正收紧。

本子 plan 继续执行以下治理口径：
- 不新增静默回退；
- 不把配置失败继续藏进空值链或默认链；
- 不保留算法层、摘要层、页面层的第二事实源；
- 不用页面专用字段表、兼容桥或临时过渡结构维持假统一。

## 前置条件

- `SP05` 已完成，后续只认迁移后路径。
- `SP04` 已完成，`ConfigService` 的请求级取用方式已稳定。
- 本子 plan 以 `.limcode` 为事实源；修订目标是配置规则收口，不改 `SP07` 负责的主链输入、编排和持久化边界。
- 执行者理解：`SP07` 只能消费 `SP06` 交出的稳定配置契约，不能与本子 plan 交叉改同一批配置规则。
- 页面侧选项值的最小统一范围固定为以下 4 个核心字段：
  - `algo_mode`
  - `objective`
  - `dispatch_mode`
  - `dispatch_rule`
- `freeze_window_*` 不做整页重构，但页面标签与提示语必须随字段元数据统一，避免继续和父 plan 冲突。

## 范围

### 只做什么
1. 冻结真实配置键清单、规则归属表与第二事实源清单。
2. 新建 `core/services/scheduler/config/config_field_spec.py`，建立唯一字段注册表，并明确它与既有 `core/services/common/value_policies.py` 的分层协作关系：registry 负责字段登记、页面元数据和字段级门面，`value_policies.py` 继续负责数值读写 / 兼容 policy。
3. 先让 `config_snapshot.py`、`config_validator.py` 改走注册表。
4. 再让 `config_service.py`、`config_presets.py` 改走注册表，并把对外投影口径统一下来。
5. 第三批清理 `config_adapter.py`、`scheduler.py`、`schedule_params.py`、`schedule_optimizer.py`、`schedule_optimizer_steps.py`、`freeze_window.py`、`resource_pool_builder.py`、`schedule_input_collector.py`、`schedule_persistence.py`、`schedule_summary_freeze.py`、`schedule_summary_assembly.py`、`schedule_summary_degradation.py` 中的本地配置读取与第二规则源。
6. 最后才统一 V1 / V2 配置页中 4 个核心字段的选项值来源，并把 `freeze_window_*` 的标签 / 提示语接到统一字段元数据。
7. 在宣布 `objective` 注册完成前，必须窄修 `objective_score()`，显式支持所有公开合法 objective；不再选择阻断或移除已公开 objective。
8. 向 `SP07` 交接稳定配置契约：字段登记口、规范化口与合法值来源都已固定。

### 不做什么
1. 不在本子 plan 内大改 `scheduler_config.py` 的手册页结构、整页文案或全部字段展示方式。
2. 不改排产主链输入、编排、持久化边界；本批只允许清理这些链路里直接读取调度配置默认值的第二事实源。
3. 不改页面流程、脚本协议、静态资源版本化或页面分层。
4. 不把 `freeze_window_*` 扩大成页面侧全量重做范围；它们应纳入后端注册表语义，页面侧只收口标签 / 提示语来源，不改布局和交互结构。
5. 不新建页面专用字段表，不在模板或路由里再维护一套第二字段清单。
6. 不通过新的兼容桥、空值保底或额外默认链维持表面可运行。
7. 不复制 `value_policies.py` 已经承载的数值读写 / 兼容策略，不让 `config_field_spec.py` 变成第二政策源。
8. 不把 objective 行为修复扩大成算法重构；这里只做同批窄修，且不得阻断或移除已公开 objective。

### 阶段 1：先建立字段注册表与统一文本规范化入口
1. 新建 `core/services/scheduler/config/config_field_spec.py`，定位为“调度配置字段注册表”。
2. 每个字段至少登记：
   - 键名；
   - 字段类型；
   - 默认值；
   - 是否允许空白；
   - 数值上下限；
   - 枚举集合；
   - 是否需要大小写规范化；
   - 如确有页面消费需求，对应的统一字段元数据入口。
3. 注册表只承载字段名、默认值、枚举、大小写规范化、页面元数据入口和 scheduler 专属字段语义。
4. 已由 `core/services/common/value_policies.py` 承载的数值读写 / 兼容策略，必须通过 `get_field_policy(key)` 引用或投影；不得在 `config_field_spec.py` 中复制第二套 policy。
5. `priority_weight`、`due_weight`、`ready_weight` 的“权重和为 1”属于跨字段规则例外，不要求经 `coerce_config_field()` 单字段独立完成；它可以继续由 service 层统一归一化，但不得扩散成新的字段级事实源。
6. 至少固定以下 registry API，避免后续实现自行发明入口：
   - `get_field_spec(key)`
   - `list_config_fields()`
   - `default_for(key)`
   - `choices_for(key)`
   - `normalize_text_field(key, value)`
   - `coerce_config_field(key, value, *, strict_mode, source)`
   - `page_metadata_for(keys)`
7. `coerce_config_field()` 是字段级规范化、类型转换、范围校验与 strict / relaxed 行为分流的唯一门面；不得在算法层、摘要层或页面层重新实现 `_cfg_float()`、`_cfg_int()`、yes/no 解析、枚举 default 解析或范围钳制。
8. `page_metadata_for(keys)` 的最小投影结构固定为：
   - `key`
   - `label`
   - `hint`
   - `unit`
   - `choices: [{"value": value, "label": label}]`
   没有选项的字段返回空 `choices`，但模板不得基于空 `choices` 再造本地 option 回退。
9. 如需统一文本规范化能力，必须放在配置层共享门面中，由算法层复用；禁止把算法层 `_norm_text()` 平移成新的本地闭包。
10. 注册表只负责字段规则与字段元数据，不承担读库逻辑，不在这里引入第二套快照流程。

**硬停点**：
- 注册表结构、字段命名和规范化入口未稳定前，不允许改 `config_snapshot.py`、`config_validator.py`。
- `coerce_config_field()` 的 strict / relaxed 行为、错误类型和 degradation 记录口径未固定前，不允许迁移算法侧或摘要侧读取。
- `page_metadata_for(keys)` 的投影结构与路由变量名未固定前，不允许开始页面统一。
- 禁止在注册表之外再新建“页面专用字段元数据表”或“算法专用字段白名单表”。
- 数值字段与 `value_policies.py` 的关系未写清前，不允许开始快照 / 校验器迁移。

**本阶段完成后要交代**：
- 注册表覆盖了哪些字段；
- 统一文本规范化入口落在哪里；
- `coerce_config_field()` 如何处理缺失、空白、非法枚举、非法数值和数值下限；
- `page_metadata_for(keys)` 的 route 变量名与模板消费 shape；

### 阶段 2：先改快照与校验器，冻结后端单一事实源
1. 修改 `config_snapshot.py`，把默认值、空白处理、大小写规范化全部切到注册表。
2. 修改 `config_validator.py`，把合法值、数值范围、严格空值规则全部切到注册表。

## 2026-04-16 修订补强边界

> 本节用于修正文案与实现状态边界；如与前文冲突，以本节为准。除明确写成“已确认边界”的条目外，其余均视为目标态或待收口项，不得据此宣称仓库当前已完全落地。

### 已确认边界与允许例外

1. `config_field_spec.py` 与 `core/services/common/value_policies.py` 的关系是“分层协作”而不是“二选一单源”：registry 负责字段登记、页面元数据和字段级门面，`value_policies.py` 继续负责数值读写 / 兼容 policy。
2. `priority_weight`、`due_weight`、`ready_weight` 的“权重和为 1”属于跨字段规则例外；它可以继续保留在 service 层统一归一化，不要求逐字段经 `coerce_config_field()` 独立完成。
3. `ScheduleConfigSnapshot` 仍是运行链和 `SP07` 交接对象；`auto_assign_persist` 应作为隐藏字段收口到 registry / snapshot / persistence 契约，不进入配置页和摘要展示。
4. `web/viewmodels/scheduler_analysis_vm.py` 中的 objective fallback 只允许作为旧 summary 兼容桥，不构成新的页面 / 摘要事实源；新 summary 必须优先使用 `comparison_metric` / `best_score_schema`。
5. `holiday_default_efficiency` 在页面 / 日历链路上的兼容读取属于显式可见退化，不计入静默回退违例；配置写入与 Excel 导入仍按 `strict` 语义处理。
6. `objective_score()`、summary schema、页面标签与字段 metadata 必须对齐全部公开 objective；这是一条收口目标，不代表当前代码已经没有残留旧块或第二事实源。

### 当前真实状态（2026-04-16 审核）

- `SP06` 的主设计方向成立，但当前仓库仍处于“行为大体可用、结构债未收干净”的状态，文档不再把它表述为“已全部落地”。
