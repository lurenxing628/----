# SP04：请求级容器与仓储束

> 2026-04-15 收口补充：`scheduler_analysis.py`、`system_history.py` 与 system 侧 `system_backup.py`、`system_logs.py`、`system_plugins.py`、`system_ui_mode.py`、`system_utils.py` 已被接受为对项目有利的扩围，并正式并入 SP04 的 request-services / gate 范围。正文中仍保留的“若本批不切容器则入账”表述仅代表阶段 0 基线记录，不再作为回退依据。

## 背景

原总 plan 的任务 2 同时承载了两类强关联工作：一类是请求级服务容器，把路由层从 `Service(g.db, ...)` 直接装配中解放出来；另一类是 `ScheduleService` 内部的仓储束，把 10 个仓储实例的内部创建集中化。两者的回归面高度重叠，都围绕请求上下文、`ScheduleService` 构造边界与路由取用方式展开，因此合并成一个子 plan 最稳。

SP04 的目标不是“做一套新容器框架”，而是**统一取用入口、稳定旧公开签名、显式收口直接装配残余**。这份子 plan 也是 `SP05` 的直接前置：只有装配入口、构造边界和容器缺失守卫稳定下来，后面的目录迁移才不会把“路径问题”和“装配问题”搅在一起。

其中仓储束的目标不是强推 `svc.repos.<repo>` 新消费方式，也不是新增一个对外稳定的聚合入口，而是把 `ScheduleService` 的 10 个仓储依赖收口为私有单点持有边界（计划命名为 `_repos`），为 `SP05` 的路径迁移提供更集中的内部实现边界；`_repos` 仅作内部组织方式，不对外描述为稳定接缝或新公开入口。

本子 plan 同样继承主索引中的硬约束，禁止靠新的兼容桥、第二套容器、隐藏 BUG 的静默回退或过度防御性包装去糊住装配问题；`excel_service` 只允许继承 `get_excel_backend()` 现有 provider 缺失 / 调用失败回退语义，不得在本批再新增新的静默回退。

### 阶段 0.5：冻结容器接口与例外规则
1. 先把上文属性清单写成 `RequestServices` 的公开接口，不在编码时临时改名或补属性。
2. `RequestServices.__init__` 必须显式接收 `db`、`app_logger`、`op_logger`（以及阶段 0.5 冻结的少量例外依赖），并在实例上保存为私有字段；容器属性统一使用 `functools.cached_property` 实现惰性构造与单请求缓存；`cached_property` 只读取 `self._db`、`self._app_logger`、`self._op_logger`，不在属性内部直接读取 `g`；只有构造成功才写入缓存，构造抛异常时不写缓存属性，下次访问允许重试；`RequestServices` 本体不得定义 `__getattr__`，也不得通过等价动态属性分发包装做兼容桥、懒转发、动态补路由或观测包装；未冻结属性访问只能由常规属性查找失败产生标准 `AttributeError`；各 `cached_property` 方法内若服务构造抛出 `AttributeError`，必须立刻转换为带属性名上下文的 `RuntimeError(... ) from exc`，避免被 Python 属性查找语义误解为容器属性缺失；`RequestServices` 不得使用 `__slots__` 或其他会移除实例 `__dict__` 的实现方式，以满足 Python 3.8 下 `cached_property` 的缓存前提。
3. 逐项确认构造规则：
   - 默认统一使用构造时注入的 `db`、`app_logger`、`op_logger`；工厂挂载时分别来自 `g.db`、`g.app_logger`、`g.op_logger`；
   - `BatchService` 不传自定义 `template_resolver`；
