# SP10：测试分层与文档收口

## 背景

前 9 个子 plan 主要在收紧代码边界、目录结构、页面协议和运行时行为。如果 `SP10` 不把测试目录、支撑脚本、质量门禁入口、文档入口链、速查表校验、证据目录与审计目录一起收口，仓库很快就会回到“代码路径已经切走，但测试还指旧位置、文档还写旧入口、证据与审计各写一遍”的状态。

`SP10` 同时承接原总 plan 的任务 9 与任务 10，但它不是“把历史遗留全部搬完”。它做的是**带明确切换时点的增量式收口**：
- 只迁本轮显著修改、直接依赖的第一批关键测试与支撑脚本；
- 只把最需要收稳的文档入口链、速查表校验和证据 / 审计职责先收稳；
- 明确规定“何时从根层测试路径切到分层目录入口”；
- 明确规定“开发文档 / 证据 / 审计”三条轨道各自承载什么，不允许长期双写同一事实。

本子 plan 继续执行以下治理口径：
- 在 `SP10` 之前，前序子 plan 仍使用当前根层测试路径；`SP10` 是**第一次真正做测试路径切换**的地方；
- 切换时不能靠“旧路径 + 新路径长期双写”假装平稳，必须有明确切换点与自动化校验；
- 文档不能重新变成实现镜像；
- 同一事实不得在 `开发文档/`、`evidence/`、`audit/` 三条轨道重复粘贴成第二事实源。

## 前置条件

- `SP05` 已完成，迁移后路径稳定。
- 建议 `SP09` 已完成，页面、模板、版本化与新增专项回归口径已经稳定。
- 执行者理解：**测试目录切换延迟到本子 plan 执行；在本子 plan 真正完成切换前，既有根层测试命令仍是当前有效口径。**
- 执行者理解：`SP10` 的交接对象不再是下一份子 plan，而是后续常规迭代、后续 review 与质量门禁维护。

## 范围

### 只做什么
1. 先冻结第一批待迁测试、支撑脚本、旧命令入口与文档入口链缺口。
2. 先抽共享仓库根定位辅助，再建立分层目录与采集配置。
3. 先迁架构门禁与路径敏感门禁，再按“配置 → 主链 → 页面”顺序迁第一批直接相关回归。
4. 迁移支撑脚本，并在合适时点把质量门禁 / 文档命令切到新目录入口。
5. 把覆盖率口径接入统一门禁，但初期只覆盖本轮直达模块。
6. 收口根 README、开发文档 README、速查表、证据 README、审计 README、阶段留痕与协作入口文档。
7. 明确后续常规迭代的落位规则：新专项回归、支撑脚本、文档入口以后都按新口径继续。

### 不做什么
1. 不一次性搬迁 `tests/` 根层全部历史测试文件。
2. 不把 `开发文档/开发文档.md` 再写回实现镜像，也不在文档里粘贴大段代码快照。
3. 不回头再改主链、算法、页面协议、静态资源版本化或其他业务实现语义。
4. 不保留长期并存的“旧命令文档一份、新命令文档一份”。
5. 不为了迁测方便回退到旧源码路径、旧白名单口径或旧支撑脚本定位方式。

2. 冻结所有仍引用旧根层测试路径的命令入口：
   - 质量门禁命令
   - README / 开发文档命令

   - 速查表校验脚本入口
3. 冻结根层测试路径切换的唯一时点：

   - 根 README 缺什么

- 切换时点未写死前，不允许开始迁任何测试文件。
