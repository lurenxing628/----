# SP05：目录骨架与路径门禁

## 背景

当前仓库仍以旧路径为主：`core/services/scheduler/`、`web/routes/`、`tests/` 大量文件平铺，新目标路径尚未落地。如果不先完成目录骨架、稳定导出和路径敏感门禁修正，后续 `SP06~SP10` 会陷入“到底改旧文件还是新文件”的执行歧义。

因此 SP05 被定义为**全局硬前置**：它不是“顺手整理目录”，而是给后续所有子 plan 建立统一路径口径。它同时承接原总 plan 任务 3 中最关键的三部分：
- 目录骨架建立
- 批次 1 热点先迁
- 路径敏感门禁与 `ruff F401` 收口

更完整的路径映射、根层保留清单与延后迁移清单，统一见附录 C；SP05 正文只保留本批必做和绝不提前做的部分。本子 plan 的核心不是“移动文件”，而是**在不新增静默回退、不过度兜底、不制造空包接管中间态的前提下，稳定切换整轮整改的路径口径**。

## 前置条件

- `SP04` 已完成，请求级容器与仓储束的稳定入口已就位。
- 执行者理解本子 plan 完成前，后续子 plan 不得按迁移后路径实施。
- 执行者理解本子 plan 结束后，已迁组（`core/services/scheduler/config/`、`run/`、`summary/` 与 `web/routes/domains/scheduler/`）统一只认迁移后真实实现路径；若这些组的根层旧路径仍保留，只能是附录 C 明示的兼容薄门面，不承接新改动。
- 执行者理解延后组（`batch/dispatch/gantt/calendar`）在 `SP08/SP09` 触达前仍以根层稳定入口为准；触达子 plan 的首个提交必须先完成真实实现迁移，再允许业务改动。

## 范围

### 只做什么
1. 建立 `core/services/scheduler/{config,run,summary}/` 三个子包，并按“真实实现迁入新子包 + 同名根层兼容薄门面按三档兼容矩阵处理 + 同提交修导入”的原子规则完成批次 1 热点迁移。
2. 建立 `web/routes/domains/scheduler/`，把 scheduler 域全部路由实现文件统一迁入该域（含 `scheduler_bp.py`、`scheduler_gantt.py`、`scheduler_week_plan.py`、`scheduler_excel_batches.py`、`scheduler_excel_calendar.py`），根层 `scheduler.py` 继续保留一级门面。

### 不做什么
1. 不提前迁 `batch/dispatch/gantt/calendar/` 的全部物理文件；这些分别交给 `SP08` 或 `SP09` 触达时同步完成。本批只允许创建空包骨架，`__init__.py` 不得导入真实实现、不得导出根层旧模块、不得提前接管业务路径。
2. 不提前迁 `process/personnel/equipment/system` 域实际路由实现；本批只允许建空骨架，不允许空包先接管。
3. 不改排产主链语义、配置规则、页面协议或测试分层目录。
4. 不把目录迁移和行为改造混在一个提交里。
5. 不顺手回头改请求级装配、容器逻辑或 `ConfigService` 规则源。
6. 不允许根层 `scheduler.py` 同时从 `domains/scheduler/**` 真实现与根层兼容薄门面混导，制造双路径并行导入链。
7. 不允许把根层兼容入口做成签名探测、吞异常重试、双路径猜测或第二事实源。

### 阶段 1：先建空目录与包标记，不立即迁文件
1. 创建 `core/services/scheduler/config/`、`run/`、`summary/`。
2. 创建 `web/routes/domains/`、`web/routes/domains/scheduler/` 与 `web/routes/domains/{process,personnel,equipment,system}/`。
3. 只补 `__init__.py` 与包标记，不先堆业务逻辑；非 scheduler 四个域的 `__init__.py` 必须保持空包语义，不包含 `Import` / `ImportFrom`，也不得让根层门面提前指向空包。

**为什么先空骨架**：
- 先把物理位置建出来，后续 `git mv` 才有稳定落点；
- 也能让路径敏感门禁提前知道未来会扫到哪里。

**本阶段完成后要交代**：
- 新目录已存在；
- 还没有文件迁移，因此后续行为仍按旧路径执行；
- 下一步可以开始改导入策略。

### 阶段 2：先定导入策略、共享工具归属与兼容薄门面规则
1. 新建 `web/routes/navigation_utils.py`，承接 `_safe_next_url` 策略实现；该模块必须提供 `_safe_next_url_core(raw, url_for_func)` 作为唯一 allow/deny 策略核心，并提供 `_safe_next_url(raw)` 作为新入口，内部调用 `_safe_next_url_core(raw, url_for)`。

   - `navigation_utils.py` 禁止反向导入 `web.routes.system_utils`；旧 `system_utils._safe_next_url(raw)` 只能调用 `_safe_next_url_core(raw, url_for)`，以保留 `system_utils.url_for` monkeypatch seam。
2. 共享解析 helper `_parse_result_summary_payload` 当前已稳定落在 `web/routes/normalizers.py`；本批必须保持它作为唯一稳定入口，并让 `dashboard.py`、`system_history.py` 继续直接从 `.normalizers` 导入；禁止在完成态回退到 `scheduler_bp.py`、`domains/scheduler/**` 或根层兼容薄门面。
3. 迁入 `web/routes/domains/scheduler/**` 的文件统一遵守：
   - 同域文件之间保留相对导入（如 `from .scheduler_bp import bp`）；

   - 子包之间跨组引用统一使用 `from ..目标子包.模块 import ...` 的两级相对导入，禁止绕回根层兼容薄门面；
