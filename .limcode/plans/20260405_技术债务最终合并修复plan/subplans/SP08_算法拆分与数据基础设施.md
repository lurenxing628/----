# SP08：算法拆分与数据基础设施

## 背景

`SP07` 已把排产主链的输入、稳定调用面、编排与持久化边界收紧到稳定状态，原总 plan 的任务 6、任务 7，以及 `SP05` 延后到后续批次处理的 `gantt/` 组迁移，才终于可以在不反复被主链中间态打断的前提下推进。

但这并不意味着 `SP08` 可以“一次性大改完”。这一批同时覆盖算法公开边界、`GreedyScheduler.schedule(...)` 拆分、数据基础设施职责拆分、读模型查询迁出、行解析统一、迁移清洗下沉，以及甘特关键链和时间解析治理。如果把这些内容混在一份大提交里，执行者很容易在同一轮里同时碰到算法、事务、迁移、缓存和路径门禁，最后既难验证，也难回滚。

因此 `SP08` 虽然仍保留为一份子 plan，但**内部必须固定拆成 4 个工作区，并且每个工作区独立提交、独立回归、独立交代产物**：
1. 工作区 1：算法边界与日期解析 BUG
2. 工作区 2：`GreedyScheduler.schedule(...)` 主流程拆分
3. 工作区 3：数据基础设施热点拆分
4. 工作区 4：甘特关键链、时间解析与 `gantt/` 组归位

本子 plan 继续执行以下治理口径：
- `_parse_date()` 的非严格模式静默吐值按 BUG 修复处理；
- `calendar_engine.py:_parse_shift_start()` 不得继续静默回退 `08:00`；
- `gantt_critical_chain.py`、`gantt_service.py` 不得继续靠空结果、空值或白名单吞异常维持“表面稳定”；
- 不为拆分 `scheduler.py`、`database.py`、`backup.py` 再造第二套派工、时间解析、迁移判断或事务语义；
- 不把 4 个工作区混成一个提交，也不把路径门禁修正拖到最后一起做。

## 前置条件

- `SP07` 已完成，主链边界稳定。
- `SP05` 已完成，统一路径口径已经建立。
- 执行者理解：**工作区顺序固定为 1 → 2 → 3 → 4，每个工作区结束即形成独立提交。**
- 执行者理解：`gantt/` 组的物理迁移归本子 plan 处理；`batch/dispatch/calendar` 的剩余物理迁移继续由 `SP09` 结合页面触达范围完成。
- 执行者理解：`SP08` 不回头修改 `SP07` 已收紧的稳定调用面（`ScheduleRunInput`、`ScheduleOrchestrationOutcome`、`SummaryBuildContext`）、`persist_schedule()` 边界与 `schedule_service.py` 门面语义。

4. 不把 `batch/dispatch/calendar` 的剩余物理迁移顺手提前做完；本子 plan 只处理本批直接触达的稳定导出与 `gantt/` 组归位。
5. 不在算法层、迁移层或甘特层新增第二套时间解析函数、第二套行解析 helper、第二套迁移清洗逻辑。
6. 不改变 `transaction.py` 的嵌套保存点语义、维护窗口互斥语义或恢复链路语义。
