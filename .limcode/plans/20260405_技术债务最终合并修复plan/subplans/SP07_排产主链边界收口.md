# SP07：排产主链边界收口
## 背景

`SP05` 已把 `core/services/scheduler/{config,run,summary}/` 的真实实现路径固定下来，并把根层 `schedule_service.py` 保留为总门面。`SP07` 不再重谈路径，而是把排产主链收口成可直接实施的方案：主链顺序、owner、契约守卫、退化 owner、兼容面、summary/status 语义与最小验证口径都必须写死，避免继续依赖签名探测、静默回退、吞错继续跑或双事实源。

## 前置条件

- 后续实现统一只改迁移后真实实现路径：`core/services/scheduler/config/`、`run/`、`summary/`。
- 根层 `schedule_service.py` 仍保留为总门面，但不再新增真实业务实现。
- 根层 `config_service.py`、`schedule_input_collector.py`、`schedule_summary.py` 等若仍存在，只能是 `SP05` 登记的 compat-only 薄门面。
- `run_command.py`、`run_context.py`、`run_status_rules.py` 默认不创建。
- 当前活跃兼容面至少包括：
  - 内部兼容面：`run/schedule_orchestrator.py::_normalize_optimizer_outcome()`、`run/schedule_orchestrator.py::_merge_summary_warnings()`。
  - 公开兼容面：`summary/schedule_summary.py::build_result_summary(ctx|**kwargs)`。
  - 历史兼容点：根层 facade 返回 DTO 对 raw summary 的 shape probing；`SP07` 目标是把它收口为显式 summary contract，而不是继续扩散。

## 只做什么

1. 写死 canonical 主链顺序与 owner。
   - `schedule_service.py` 持有 `_RUN_SCHEDULE_LOCK`、`_aps_schedule_input_cache` 生命周期、strict snapshot guard，并委托 `collect_schedule_run_input()`。
   - `run/schedule_orchestrator.py::orchestrate_schedule_run()` 负责 optimizer 调用、结果校验、validated schedule payload 构建、版本号分配、`SummaryBuildContext` 装配与 summary 调用。
   - `run/schedule_persistence.py::persist_schedule()` 只负责落库、状态更新、历史留痕与操作日志留痕。
   - `summary/schedule_summary.py` / `summary/schedule_summary_freeze.py` 是 `result_status`、warning pipeline、degradation 事件与摘要序列化的 owner。
   - `schedule_service.py` 最后只负责组装 facade 返回 DTO。
2. 收口 stable types 与兼容面。
   - 稳定主线对象仍是 `ScheduleRunInput`、`ScheduleOrchestrationOutcome`、`SummaryBuildContext`。
   - `ScheduleOrchestrationOutcome` 必须携带唯一的 validated schedule payload 与 facade 需要的显式 summary contract。
   - `build_result_summary(ctx|**kwargs)` 继续保留为兼容公开面，但冻结新增 `**kwargs` caller。
3. 拆清契约守卫、退化 owner、静默兼容桥。
   - fail-fast guard：`schedule_service.py::_get_snapshot_with_strict_mode`、`run/schedule_input_contracts.py::_build_algo_operations_outcome`、`run/schedule_input_contracts.py::_build_freeze_window_seed_with_meta`。
   - 双态边界：`run/schedule_template_lookup.py::lookup_template_group_context_for_op` 在 strict 下是 guard，在 relaxed 下是退化 owner。
   - 真实退化 owner：`run/freeze_window.py`、`run/schedule_input_builder.py` 与相关 field parse / compat parse 链。
   - 必须收口的静默兼容桥：`run/schedule_optimizer_steps.py::_schedule_with_optional_strict_mode`；若未来 scheduler 不支持 `strict_mode`，只能在接入点显式适配，不允许执行时自动签名探测或 `TypeError` 后删参重跑。
4. 收口 validated payload 与持久化事实源。
   - `persist_schedule` 不再接受与 payload 平行的 `has_actionable_schedule` 判定输入。
   - “唯一事实源”改为一次性 validated schedule payload，而不是 `schedule_rows / scheduled_op_ids` 两个兄弟投影字段并列。
   - payload 至少一次性产出 `schedule_rows`、`scheduled_op_ids`、`assigned_by_op_id` 与 validation errors。
   - validated payload 必须在 `orchestrate` 阶段、版本号分配前构建。
   - 空结果不分配版本、不构建摘要、不落库；范围内坏结果行同样在此阶段失败，不允许“过滤坏行、保留好行”的部分成功。
   - `allowed_op_ids` 仍是合法范围裁剪，范围外结果行允许过滤，范围内非法结果行必须视为契约失败。
5. 修正 freeze 与 seed 的 relaxed/strict 语义。
   - `freeze_window` 的合法 `no_previous_rows` 不算 degradation。
   - 读取失败或坏行吞光才算 degradation；strict 下 fail-fast，relaxed 下允许继续但必须产出结构化证据。
   - `freeze_state=active` 仅在 `freeze_applied=true` 时成立；启用但未实际冻结时不能再混成 active。
   - `run/schedule_optimizer.py` 对坏 `seed_results`：strict 下拒绝继续优化；relaxed 下允许继续，但失败计数与样例必须进入 summary/fallback 可观测面。
6. 收口 compat-only 根层门面与附录口径。
   - 根层 wrapper 本批不删除、不新增新引用，并明确标记为 compat-only。
   - 附录 C 只登记 compat 事实与回收条件，不得在旧精确路径导入方清单未补齐前把 facade 回收矩阵当作完成态 checklist。
   - `SP05` 中 `calendar_admin.py -> ConfigService` 的未来时表述必须改成当前事实。

## 不做什么

- 不重新迁移 `config/run/summary/` 路径。
- 不回到根层兼容门面继续写真实业务实现。
- 不默认新增 `run_command.py`、`run_context.py`、`run_status_rules.py`。
- 不新增签名探测、`TypeError` 后重试、吞错继续跑或新的静默回退包装。
- 不在本批删除根层 compat-only wrapper。

## 最小交接产物

1. 主链顺序图：`schedule_service -> collect_schedule_run_input -> orchestrate_schedule_run -> persist_schedule`，其中 summary 构建发生在 `orchestrate_schedule_run` 内部，而不是 `persist_schedule` 之后。
2. 稳定调用面与兼容面清单：`ScheduleRunInput`、`ScheduleOrchestrationOutcome`、`SummaryBuildContext`，以及 `_normalize_optimizer_outcome()`、`_merge_summary_warnings()`、`build_result_summary(ctx|**kwargs)` 的处理策略。
3. 契约守卫 / 退化 owner / 静默兼容桥矩阵，并写清 strict/relaxed 裁决。
4. validated schedule payload 的边界定义：唯一事实源、版本分配前构建、空结果/坏结果阻断写入。
5. 根层 compat-only 门面与附录 C 的回收条件说明。

## 验收口径

1. 第一层：`schedule_service -> collect_schedule_run_input -> orchestrate_schedule_run -> persist_schedule` 主链不断；根层门面只委托迁移后真实实现路径。
2. 第二层：`result_status` owner、validated payload、freeze 双空语义、seed/fallback 可观测性、空结果拒绝与 compat-only 门面回收条件都与正文一致，不再依赖隐式 cache、双语义源、签名探测或静默回退。
