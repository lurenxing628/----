# SP02：质量门禁与治理台账

## 背景

SP01 解决的是“入口、依赖、文档导航与专项回归落点”，但还没有解决以下四个基础问题：

1. 谁来提供**唯一质量门禁入口**；
2. 谁来维护**唯一机器可读治理台账**；
3. `tests/test_architecture_fitness.py` 里的超长文件 / 高复杂度 / 静默回退白名单，什么时候从内联数据迁到统一台账；
4. `web/bootstrap/` 与 `web/ui_mode.py` 这类启动链热点，如何在**不提前改业务语义**的前提下纳入稳定门禁。

因此，SP02 的职责不是“清零热点代码”，而是先把**门禁入口、治理台账、受控解析口径、启动链专项回归入口、`web/bootstrap/` / `web/ui_mode.py` 的基线固化**搭好，让 SP03 以后只做真实治理，不再重复搭基础设施。

本子 plan 继续继承反静默回退、反过度兜底、反第二事实源约束，并把约束明确到可执行层面：
- 门禁不仅识别 `except Exception: pass`，还要识别“无可观测性地回退默认值”的语义，且静默回退原始条目必须按**单个 `except` 处理器**落账，不能继续沿用函数级 `count`；
- 台账必须只有**一个受控机器可读事实源**；
- 检查入口与写入入口必须分离：`scripts/run_quality_gate.py` 只读、`scripts/sync_debt_ledger.py` 只写；
- 三类主条目的人工治理字段必须通过 `scripts/sync_debt_ledger.py` 的 `set-entry-fields` 更新；`accepted_risks` 顶层对象只能通过专门的风险维护子命令更新，不能再靠直接手改结构块维持；
- 解析失败、重复结构块、缺字段、非法分类一律失败，不允许再做静默回退；
- 启动链专项回归必须继续走 `pytest` 收集，不得另写第二套执行器；
- `web/bootstrap/` 的接入不能“一刀切”并入全部旧规则，而要按与本阶段目标直接相关的三类门禁分步接入：**静默回退、文件规模、圈复杂度**；
- 三类规则的启动链接入必须使用专属扫描范围，禁止通过扩大 `CORE_DIRS`、`NO_WILDCARD_DIRS` 等全局目录常量，顺带把命名、导入边界等无计划规则扩到 `web/bootstrap/` / `web/ui_mode.py`；
- 本地执行前置检查里，“活动 APS 实例”必须通过仓库根运行时契约 / 运行时锁、`pid` 存活校验与 `/system/health` 探测联合判定，不能只看 `logs/` 文件是否存在；
- 文档一致性检查一旦升级为阻断，必须先消除当前已知阻断项，不能把脚本直接改成“永远红”的门禁摆设。

## 关键设计约束

### 1. 唯一入口约束
- 本地唯一门禁入口：`python scripts/run_quality_gate.py`
- 本地唯一台账写入入口：`python scripts/sync_debt_ledger.py`
- 托管唯一门禁入口：工作流内只执行 `python scripts/run_quality_gate.py`
- 三类治理数据唯一事实源：`开发文档/技术债务治理台账.md` 中的受控 `json` 结构块
- `scripts/run_quality_gate.py` 只读不写；`scripts/sync_debt_ledger.py` 负责唯一写入，并至少提供：
  - `refresh --mode migrate-inline-facts`：仅迁移旧内联治理事实
  - `refresh --mode scan-startup-baseline`：仅在样本点验真通过后写入 `web/bootstrap/` / `web/ui_mode.py` 启动链基线
  - `refresh --mode refresh-auto-fields`：仅在条目首次验真、自动坐标变化或显式需要刷新验证戳时刷新 `line_start` / `line_end` / `last_verified_at` 等自动字段
  - `set-entry-fields`：仅更新三类主条目的 `owner` / `batch` / `status` / `notes` / `exit_condition` 等人工治理字段
  - `upsert-risk` / `delete-risk`：专门维护 `accepted_risks`
  - `check`：只做结构、字段、`accepted_risks` 引用完整性与样本点前置条件校验，不落盘
- 受控结构块禁止直接手改；任何人工治理字段更新都必须通过 `scripts/sync_debt_ledger.py` 落盘，再由门禁负责校验结果；`set-entry-fields` 不得顺带隐式修改 `accepted_risks` 顶层对象

### 2. 唯一事实源约束
- `tests/test_architecture_fitness.py` 中当前三类内联治理数据，在 SP02 中必须迁入治理台账。
- 一旦某一类规则改为读台账，同一类数据不得继续保留第二套等价内联白名单。
- 允许保留的是“解析函数 / 校验函数 / 默认阈值常量”，不允许保留第二套治理事实数据。
- 扩展启动链接入时，禁止直接把 `web/bootstrap/` / `web/ui_mode.py` 加入 `CORE_DIRS` / `NO_WILDCARD_DIRS`；必须为文件规模、圈复杂度、静默回退三类规则单独声明扫描范围映射，确保其他既有规则仍沿原范围执行。

强制规则：
- `BEGIN` / `END` 标记各且仅各出现一次；
- `json` 必须严格可解析；
- `identity_strategy` 必填，首版固定声明“稳定键而非行号身份”；
- 必须带 `schema_version`，首版固定为 `1`；
- `silent_fallback.entries` 必须按单个 `except` 处理器记录，禁止退回到 `path:symbol -> count` 聚合口径；
- 任一必填字段缺失、字段类型错误、分类值非法、重复条目都会导致门禁失败；
- 门禁读取失败时，**禁止**回退到旧内联白名单、空白名单或“只打印 warning 继续”。

### 4. 条目字段与接受风险引用约束
- 三类主条目都必须带统一公共字段：`id`、`path`、`symbol`、`status`、`owner`、`batch`、`exit_condition`、`last_verified_at`。
- `oversize_allowlist` 条目必须额外带：`current_value`、`limit`。
- `complexity_allowlist` 条目必须额外带：`current_value`、`threshold`。
- `silent_fallback.entries` 条目必须额外带：`handler_fingerprint`、`except_ordinal`、`line_start`、`line_end`、`fallback_kind`、`source`；同一函数内多个处理器必须拆成多条原始条目，禁止再用 `count` 聚合多个命中。
- `web/ui_mode.py` 的 `silent_fallback.entries` 条目还必须带 `scope_tag`，且值只能是 `startup_guard` 或 `render_bridge`；其中 `startup_guard` 交给 SP03，`render_bridge` 只做落账并明确留给 SP09。
- `handler_fingerprint` 必须由 `path + symbol + except Exception 处理器归一化 AST 片段` 生成，用作处理器级主稳定键；`except_ordinal` 只在同一函数内出现多个结构等价处理器时用于次级消歧，不能单独承担主身份。
- `handler_fingerprint` 的归一化规则必须固定且可验证：保留 `except` 类型、是否记录日志 / 退化事件 / 状态写入、默认值回退类别、返回 / continue / cleanup 控制流与关键调用目标；忽略行号、注释、空白、字符串文案、局部变量名与不改变回退语义的机械挪动。仅当回退语义或可观测性语义变化时，才允许生成新的 `handler_fingerprint`。
- `silent_fallback.entries.id` 必须基于 `path + symbol + handler_fingerprint` 生成，不得直接把 `line_start` 或单独的 `except_ordinal` 拼入 `id` 作为主身份；`line_start` / `line_end` 仅作证据坐标。
- `scripts/sync_debt_ledger.py` 重扫时必须先按 `handler_fingerprint` 对齐旧条目；仅当同一函数内出现多个相同指纹条目时，才再结合 `except_ordinal` 做次级匹配；匹配成功后仅刷新 `line_start`、`line_end`、`last_verified_at` 等自动字段，`owner`、`batch`、`status`、`notes`、`accepted_risks` 引用等人工治理字段必须继承保留，不得因机械挪行而整体丢失。
- `id` 在整个台账范围内必须唯一；除 `id` 外，还要校验同类条目在 `(path, symbol, handler_fingerprint, except_ordinal)` 上不出现重复登记；`line_start` 仅用于辅助定位，不作为唯一性主键。
- `accepted_risks` 只能通过 `entry_ids` 引用已经存在的三类主条目；不得再次重复记录路径、函数、计数、当前值，避免形成平行事实源。
- `set-entry-fields` 只允许更新三类主条目的人工治理字段；不得创建、删除、重命名 `accepted_risks` 顶层对象。
- `accepted_risks` 的新增、更新、删减只能通过 `upsert-risk` / `delete-risk` 完成；`upsert-risk` 必须显式提交完整 `entry_ids` 集合，禁止隐藏式追加 / 删除引用。
- `check` 必须校验 `accepted_risks.entry_ids` 的引用完整性；任何引用不存在主条目的对象一律失败，必须通过 `upsert-risk` / `delete-risk` 显式修正，禁止静默删除或保留悬空引用。
- `last_verified_at` 仅在条目首次写入、`handler_fingerprint` 对齐后的 `line_start` / `line_end` 发生变化、或显式完成一次新的样本点验真且需要刷新自动字段时更新；若事实输入与自动坐标均未变化，必须保留原值，避免重复执行产生纯时间戳 diff。
- `web/ui_mode.py` 中 `scope_tag=render_bridge` 的条目，`batch` 不得标为 `SP03`，避免把渲染期职责误交给启动链专项。
- `accepted_risks` 自身必须说明：`id`、`entry_ids`、`owner`、`reason`、`review_after`、`exit_condition`。
- `scripts/sync_debt_ledger.py` 输出必须稳定：在事实输入无变化时，重复执行不得产生无意义 diff；`oversize_allowlist`、`complexity_allowlist`、`silent_fallback.entries`、`accepted_risks` 的数组排序与字段顺序都必须固定。

### 5. 原始事实与派生统计分离
- 机器可读主数据只保存**原始条目**，不把“总命中数 / 按文件分布 / 按函数分布”作为第二份机器主数据长期保存。
- 静默回退原始条目必须按单个 `except` 处理器落账，禁止再把 `path:symbol -> count` 当作长期主数据保存。
- 需要展示这些聚合视图时，可在治理台账正文或门禁输出中生成**派生摘要**；但门禁判定必须基于原始条目重新计算。

### 6. 静默回退分类约束
启动链静默回退门禁至少区分以下四类：
- `silent_swallow`：`except Exception: pass` / `...`
- `silent_default_fallback`：捕获后直接回退 `None` / `False` / `0` / 空字符串等默认值，或先写入默认值再继续后续流程，且无可观测性输出
- `observable_degrade`：先留下稳定可观测证据（日志、退化收集器、状态字段等之一），再降级返回或继续
- `cleanup_best_effort`：关闭 / 删除 / 清理 / 收尾类尽力而为逻辑

本阶段门禁目标：
- `silent_swallow`、`silent_default_fallback`：必须被识别、纳入治理并阻止未登记新增；
- `observable_degrade`、`cleanup_best_effort`：必须可被识别并进入台账或专项说明，但不能粗暴地与前两类一刀切处理。

分类判定优先级固定为：
1. **先看是否属于清理 / 删除 / 关闭 / 收尾语境**；若是，并且失败后不会改变主业务结果，只影响尽力而为收尾，则优先记为 `cleanup_best_effort`；
2. 否则，若处理器已留下稳定可观测证据（日志、退化事件、状态字段三者之一即可），再回退默认值或继续流程，则记为 `observable_degrade`；
3. 否则，若捕获后直接回退 `None` / `False` / `0` / 空字符串等默认值，或以默认值继续后续流程，则记为 `silent_default_fallback`；
4. 其余 `except Exception: pass` / `...` 记为 `silent_swallow`；
5. 同时满足多类特征时，按以上顺序择一归类，不允许同一处理器在不同批次来回换类。

### 7. 旧白名单迁移规则
- 旧超长文件白名单若当前是路径集合，则迁移为 `oversize_allowlist` 对象数组；每条必须补齐 `id`、`status`、`owner`、`batch`、`exit_condition`、`last_verified_at`。
- 旧高复杂度白名单若当前是 `path:symbol` 集合，则迁移为 `complexity_allowlist` 对象数组；`current_value` 以迁移时实测值写入。
- 旧静默回退白名单若当前是 `Counter({"path:symbol": count})`，不得原样把 `count` 搬进台账；必须先用共享扫描器在对应 `(path, symbol)` 范围内展开当前命中的单个 `except` 处理器，再逐条迁移为 `silent_fallback.entries`：
  - 每个处理器独立落 `path` / `symbol` / `handler_fingerprint` / `except_ordinal` / `line_start` / `line_end`；
  - `fallback_kind` 初始固定写为 `silent_swallow`；
  - `source` 固定写为 `migrated_from_architecture_fitness_counter`；
  - `batch` 若无更明确归属，默认先指向 `SP03`；
  - 若扫描出的处理器数量与旧 `Counter` 记录不一致，必须在步骤 3 的迁移过程中显式登记差异并中止本次迁移；登记位置沿用步骤 0 的固定留痕落盘位置，禁止静默取最大值 / 最小值继续迁移。
- 迁移顺序必须是：**先机械迁移旧事实，再补新分类条目**；禁止在同一动作里边迁移边重写统计口径，导致迁移结果不可复核。

### 8. fail-fast 约束
- 统一门禁脚本中任一子命令返回非 0 即立刻失败。
- 工作流不得使用“继续执行后统一收集失败”的隐藏容错写法。
- `ruff` 版本校验、`radon` 可导入校验、台账解析、样本点校验、文档一致性检查都属于阻断项，不允许静默跳过。
- 复杂度门禁不得依赖“缺少 `radon` 时由 `pytest.skip` 放过”的行为；统一门禁必须在进入 `pytest` 前显式确认 `radon` 可用。
- `tests/test_architecture_fitness.py` 在直接单跑时也不得继续保留 `pytest.skip` 的复杂度旁路；缺少 `radon` 时同样必须给出明确失败，避免与统一门禁形成双口径。

### 9. 启动链专项回归执行约束
启动链专项回归当前是**混合形态**：既包含 `main()` 风格文件，也包含标准 `test_*.py`。因此：
- 必须统一通过 `python -m pytest` 调用；
- 禁止在 `scripts/run_quality_gate.py` 中手写逐文件 `exec` / `subprocess` 调度逻辑去模拟收集；
- 门禁、文档、工作流三处必须共用同一条命令清单。

### 10. 本地执行前置检查约束
- 统一门禁在跑任何启动链专项回归前，必须只针对仓库根运行时目录执行“活动实例”判定。
- 判定顺序固定为：先读取仓库根运行时契约 / 运行时锁，再复用现有 `pid` + 可执行文件匹配逻辑校验进程是否仍存活，并对契约中的 `host/port` 执行 `/system/health` 探测。
- 仅当“运行时契约或运行时锁可解析”且“`pid` / 可执行文件匹配校验通过”或“健康探测成功”二者至少其一成立、且信息不矛盾时，才认定为存在活动实例并直接失败。
- 仅发现仓库根 `logs/` 契约文件、但 `pid` / 可执行文件匹配失败且健康探测也失败时，应视为陈旧痕迹：不得据此判定活动实例，也不得自动删除文件；允许打印显式提示后继续，由后续测试沿既有清理逻辑处理。
- 若契约 / 运行时锁可读，但 `pid` 存活结果与健康探测结果互相矛盾，或前置检查实现自身报错，则必须 fail-fast 并提示人工处理，禁止在不确定状态下静默继续。
