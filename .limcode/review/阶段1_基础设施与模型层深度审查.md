- 严重级别: 中

### 能力注册覆盖无告警

- ID: STAGE1-PLUGIN-001
- 严重级别: 中
- 分类: 其他
- 跟踪状态: 开放
- 相关里程碑: M2
- 说明:

  `core/plugins/registry.py:22-23` 对 capability key 直接赋值；`core/plugins/manager.py:168-171` 又只按 `after-before` 计算新增能力。结果是：如果后加载插件注册了已存在的 key，它会静默覆盖前者，但插件状态里既没有冲突告警，也不会把被覆盖 key 记为新增能力。当前仓内两个样例插件 `plugins/pandas_backend_plugin.py:50` 与 `plugins/ortools_probe_plugin.py:21` 未复现重复 key，因此本轮把影响标记为“待验证”；但从代码合同看，后续一旦发生重复声明，消费方只会拿到最后注册者。
- 建议:

  在 `PluginRegistry.register()` 中增加显式冲突策略与状态输出，把“覆盖了哪些 key”纳入插件状态。

  ### 9. 引用链与根因分析
  1. **恢复成功口径早于结构校验**
     - 入口：`web/routes/system_backup.py:241-257`
     - 关键变量：`result = mgr.restore(backup_path)`
     - 阶段1问题点：`core/infrastructure/backup.py:315-320` 在文件复制完成后立即返回 `RestoreResult(ok=True)`
     - 下游补丁：路由层必须再调 `ensure_schema()`；`tests/regression_restore_success_condition.py:80-95` 专门锁定“补校验失败时不得显示成功”
     - 根因：底层接口把“复制成功”和“恢复完成”混成一个 success 码。
  2. **插件能力静默覆盖**
     - 入口：`web/bootstrap/plugins.py:28-29`
     - 关键变量：`registry.capabilities`
     - 阶段1问题点：`core/plugins/registry.py:22-23` 直接覆盖；`core/plugins/manager.py:168-171` 只计算新增 key
     下游消费：`core/services/common/excel_backend_factory.py:24-33` 只取单个 capability key
     - 根因：注册表没有冲突策略，也没有把“覆盖发生过”纳入状态合同。
  3. **插件层反向依赖**
     - 入口：`PluginManager.load_from_base_dir()`
     - 关键变量：`cfg_repo = SystemConfigRepository(...)`
     - 问题点：`core/plugins/manager.py:10-12,95-96,147-158`
     - 影响：插件底座被迫携带上层仓库与归一化规则，分层边界倒挂。
  4. **启动留痕失败事实未外显**
     - 入口：`web/bootstrap/plugins.py:41-51`
     - 关键变量：`OperationLogger(conn0).info(...)`
     - 阶段1问题点：`core/infrastructure/logging.py:183-191` 失败时默认仅返回 `False`
     - 下游现象：调用方既不检查 `False`，又把异常整体吞掉
     - 根因：阶段1日志基础设施仍将失败视为“最佳努力”，没有形成显式调用合同。

  ### 10. 跨阶段观察

  - `core/infrastructure/database.py:106-109` 使用 `check_same_thread=False`；当前阶段只确认了连接创建合同与事务合同，尚未在真实请求并发下验证“是否真的存在跨线程共享连接”的上层使用前提，建议后续在启动/请求生命周期 review 中继续核对。
