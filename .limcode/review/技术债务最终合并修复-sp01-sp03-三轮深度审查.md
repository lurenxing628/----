## 评审发现

### 活动实例检测忽略了锁文件与契约不一致的场景

- ID: F-活动实例检测忽略了锁文件与契约不一致的场景
- 严重级别: 中
- 分类: JavaScript
- 跟踪状态: 开放
- 相关里程碑: M2
- 说明:

  在 `_assert_no_active_runtime()` 中，`pid_payload` 只取 `contract if contract is not None else lock`。当 contract 的 `pid=0`（陈旧）但 lock 中有一个**仍然存活**的不同 PID 时，且恰好健康探测也失败（比如端口错误），最终会走到 `print("陈旧运行时痕迹")` 分支放行。理论上此时仓库根可能仍有活动 APS 实例。

  实际风险：极低（需要 contract.pid=0 + lock.pid=active + health 探测失败三者同时满足），但逻辑上存在盲区。建议在 contract 和 lock 都存在时，分别检测两个 PID，或至少将 lock 的 PID 也纳入探测。
- 建议:

  在 contract 和 lock 同时存在时，取两者中 pid 更大（或任一非零）的一方作为 pid_payload，或对两个 PID 分别执行 _pid_signal 检测。
- 证据:
  - `scripts/run_quality_gate.py:157-189`
  - `scripts/run_quality_gate.py`

### set_entry_fields 函数级别不限制可修改字段

- ID: F-set-entry-fields-函数级别不限制可修改字段
- 严重级别: 中
- 分类: 可维护性
- 跟踪状态: 开放
- 相关里程碑: M2
- 说明:

  CLI 入口 `sync_debt_ledger.py` 正确限制只传入 `owner/batch/status/notes/exit_condition` 五个人工字段，但底层函数 `set_entry_fields()` 接受任意字段名的 `updates` 字典并直接 `entry[field_name] = value`。若将来有其他代码直接调用该函数，可能意外覆盖 `current_value`、`handler_fingerprint` 等自动计算字段。

  `finalize_ledger_update → validate_ledger` 能校验结构完整性，但无法阻止**合法类型但语义非法**的修改（如把 `current_value` 改成任意整数）。
- 建议:

  在函数入口处加一层 ENTRY_MANUAL_FIELDS 白名单校验：`for field_name in updates: if field_name not in ENTRY_MANUAL_FIELDS: raise QualityGateError(...)`
- 证据:
  - `tools/quality_gate_operations.py:225-247`
  - `tools/quality_gate_operations.py`

  | absent | absent | 正常放行 ✅ |
  | stale | stale | 警告放行 ✅ |

  ### 台账校验链完整性

  追踪了 `load_ledger → validate_ledger → finalize_ledger_update → save_ledger` 的完整链路：
  - `validate_ledger` 检查 schema_version、identity_strategy、主条目 id 唯一性、scope_tag 合法性、fallback_kind 合法性、accepted_risks 引用完整性。
  - `finalize_ledger_update` 语义比较（忽略 updated_at）后决定是否刷新时间戳——避免了无实质变更时的噪音更新。
  - `sort_ledger` 按路径+符号+指纹+序号四级排序，保证了输出的确定性。
  - 链路完整，无遗漏校验。

  ### 静默回退扫描分类器

  验证了 `classify_silent_fallback` 的四类分类逻辑：
  1. cleanup_best_effort：检测 try body + except body 中的清理调用
  2. observable_degrade：检测日志调用或状态写入
  3. silent_default_fallback：检测默认值返回/赋值
  4. silent_swallow：以上均无
