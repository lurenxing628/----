因此第一份报告里的“extra_state 绑定过宽”问题，**在当前工作区里也已收口**。

## 当前仍然成立的问题

### 问题一：strict 语义仍未在公开旁路入口上闭合

#### 事实
- `core/algorithms/greedy/config_adapter.py::read_critical_schedule_config()` 把空白字符串继续视为“缺失”：
  - `_is_missing(value)` 目前定义为 `value is None or (isinstance(value, str) and value.strip() == "")`
- `core/algorithms/greedy/schedule_params.py::resolve_schedule_params()` 对 `dispatch_mode` / `dispatch_rule` / `auto_assign_enabled` 仍依赖这个读取结果。
- `core/services/scheduler/config_validator.py::normalize_preset_snapshot()` 在 `strict_mode=True` 下对空白 `dispatch_mode` / `dispatch_rule` / yes/no 字段并不会拒绝，而是继续落回 `base`。
- `core/services/scheduler/config_presets.py::normalize_preset_snapshot()` 的生产包装没有把 `strict_mode=True` 传给 validator；`apply_preset()` 因而会接受坏 preset，再通过“规范化/修补”落盘。

#### 已复现实验
- 直接 dict 配置入口：`strict_mode=True` 下传入空白 `dispatch_mode` / `auto_assign_enabled`，仍然成功运行并被默认成 `batch_order` / `False`。
- preset 应用入口：存入空白 `dispatch_mode` / `auto_assign_enabled` 的 preset 后，`apply_preset()` 仍能成功应用，只是把活动方案标成 `custom`，并给出“规范化或修补”的理由。

#### 判断
这说明第二份报告的主判断是对的：**strict 语义目前主要在快照层闭合，但对外仍被支持的旁路入口没有同步收口**。如果设计目标是“显式空白不是缺失，坏值不能冒充缺省”，那当前实现还差最后一段。

### 问题二：weighted 直读对象配置时仍有静默回退缺口

#### 事实
- `core/algorithms/greedy/schedule_params.py` 的 weighted 分支在 `strategy_params is None and config is not None` 时，仍然调用：
  - `cfg_get(config, "priority_weight", 0.4)`
  - `cfg_get(config, "due_weight", 0.5)`
- `core/algorithms/greedy/config_adapter.py::cfg_get()` 继续吞掉属性访问和 `get()` 调用异常，直接回退默认值。
- 当前已有回归覆盖了关键 choice/yesno 读取失败、空白 numeric、快照 strict blank，但**没有回归锁住 weighted 的对象/getter 读取失败不可静默回退**。

#### 已复现实验
我构造了一个对象配置：关键 choice 字段走属性读取，`get()` 对任意键直接抛异常。结果 weighted 分支仍然静默得到：
- `priority_weight=0.4`
- `due_weight=0.5`

既没有 warning，也没有失败信号。

#### 判断
这部分与第一份报告的担忧一致：**虽然它不是主数据库快照链的问题，但它仍是被公开支持的配置形态，而且确实会默默改变 weighted 行为**。如果继续保留 `dict / dataclass / getter` 三种入口契约，这个缺口就应一并收掉；否则外部看到的 strict 语义仍不完整。

## 最终判断

两份报告合并后，最准确的当前结论是：

1. 第一份报告里关于快照 strict blank、UI 模式非法值告警、工艺路线 extra_state 过宽的三处问题，**在当前工作区中已经被修正**，不应再作为本轮主修目标继续扩大。
2. 第二份报告抓到的系统性问题仍然成立：**strict 语义没有在公开旁路入口上闭合**，尤其是：
   - `resolve_schedule_params()` 的直接原始配置入口；
   - `apply_preset()` 的 preset 规范化/应用入口；
   - weighted 对象/getter 配置的读取失败路径。
3. 因而当前状态仍然不适合宣称“静默回退已彻底收口”。但修复范围已经足够清楚：只要把 strict 语义继续下沉到这几个旁路入口，并补上对应回归，就可以在不扩大实现面的前提下完成真正收口。

## 修复边界建议

### 本轮必须修
1. 让 `read_critical_schedule_config()` 区分“缺失”与“显式空白”。
2. 让 `resolve_schedule_params()` 的 direct config 路径在 strict 模式下拒绝空白 `dispatch_mode` / `dispatch_rule` / `auto_assign_enabled`，而不是把它们伪装成缺省。
3. 让 weighted 的对象/getter 读取失败变为可见失败或可见告警，不再继续经 `cfg_get()` 静默吞掉。
4. 让 preset validator 与应用包装统一 strict 语义：坏 preset 要么显式拒绝，要么在入口层就明确判定为不可应用；不能先修补再假装成功应用。

### 本轮不要做
1. 不重新打开已经收口的 UI 模式与工艺路线基线逻辑。
2. 不为少量问题再抽象新的全局兼容框架、总开关或多层兜底。
3. 不把所有历史宽泛异常一次性清零，只收当前明确影响 strict 语义闭合的入口。
