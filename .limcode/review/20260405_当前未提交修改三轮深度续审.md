## 评审发现

### 严格语义仍依赖快照前置规范化

- ID: strict-config-bypass-outside-snapshot
- 严重级别: 中
- 分类: 可维护性
- 跟踪状态: 开放
- 相关里程碑: M1
- 说明:

  `build_schedule_config_snapshot()` 已在 strict 模式下拒绝空白 choice/yesno，但更下游的配置入口仍未同步。`read_critical_schedule_config()` 继续把空白字符串视为 missing，`resolve_schedule_params()` 因此会在 strict 模式下把直接传入的空白 `sort_strategy/dispatch_mode/dispatch_rule/auto_assign_enabled` 静默默认化；`GreedyScheduler` 又明确支持 dict/dataclass/getter 三种配置形态，所以这不是纯理论分支。与此同时，`config_validator.normalize_preset_snapshot()` 的 strict 语义也没有与快照层对齐：空白 dispatch 值仍会落回 base，空白 yes/no 仍被接受，而生产包装 `config_presets.normalize_preset_snapshot()` 甚至没有把 `strict_mode=True` 传入。结果是：主排产入口虽然趋于正确，但只要调用链绕过快照或走 damaged preset 应用，关键开关仍可能被静默修补，与“strict 语义闭合、无静默回退”的目标不一致。
- 建议:

  把 strict 语义下沉到所有被公开支持的配置入口：一是让 `read_critical_schedule_config()`/`resolve_schedule_params()` 区分缺失与显式空白；二是让 preset validator 的 strict 规则与快照层对齐，并决定 `apply_preset()` 是否应拒绝 damaged preset，至少补回归锁住 blank dispatch/yesno 的真实应用路径。

### M1 · 排产严格配置链与预设链路复核

- 状态: 已完成
- 记录时间: 2026-04-05T12:27:41.990Z
- 已审模块: core/services/scheduler/config_snapshot.py, core/algorithms/greedy/config_adapter.py, core/algorithms/greedy/schedule_params.py, core/algorithms/greedy/scheduler.py, core/services/scheduler/config_validator.py, core/services/scheduler/config_presets.py, core/services/scheduler/config_service.py, core/services/scheduler/schedule_optimizer.py, tests/regression_config_snapshot_strict_numeric.py, tests/regression_scheduler_strict_mode_dispatch_flags.py, tests/regression_schedule_params_strict_blank_numeric.py, tests/regression_schedule_optimizer_cfg_float_strict_blank.py, tests/regression_scheduler_apply_preset_reject_invalid_numeric.py, tests/test_greedy_scheduler_base_date.py
- 摘要:

  ### 结论
  - `build_schedule_config_snapshot()` 这一层的修复方向是对的：`_choice()` / `_yes_no()` 已能在 `strict_mode=True` 下区分缺失与显式空白，`tests/regression_config_snapshot_strict_numeric.py` 也把空白 `sort_strategy/dispatch_mode/dispatch_rule/auto_assign_enabled` 和多种数值坏值锁住了。
  - `ScheduleService -> optimize_schedule() -> GreedyScheduler.schedule()` 主链目前通常会先拿到 `ScheduleConfigSnapshot.to_dict()` 的纯快照，所以只要入口真的走 `ConfigService.get_snapshot(strict_mode=True)`，主链 strict 行为比上一轮明显收紧。

  ### 问题
  - 严格语义仍没有在“所有被支持的配置入口”上真正闭合。`read_critical_schedule_config()` 依旧把空白字符串视为 `missing`，导致 `resolve_schedule_params(strict_mode=True)` 对直接传入的原始 dict/dataclass 配置仍会把空白 `sort_strategy/dispatch_mode/dispatch_rule/auto_assign_enabled` 静默默认化；而 `GreedyScheduler` 明确宣称支持 dict/dataclass/getter 三种配置形态。
  - 预设链也没完全跟上：`config_validator.normalize_preset_snapshot()` 的 strict 分支对空白 `dispatch_mode/dispatch_rule` 仍会按 base 默认化，对空白 yes/no 仍把 `""` 当合法 false；更关键的是，生产包装 `config_presets.normalize_preset_snapshot()` 根本没有把 `strict_mode=True` 传下去。结果是 damaged preset 仍可能在应用时被兼容修补而不是显式暴露。
  - 现有新增回归覆盖了快照层和若干 helper 的孤立行为，但没有锁住“直接 config 入口 + preset 应用入口”这两个旁路，所以本轮“strict 语义彻底闭合”的目标我认为仍未完全达成。

  ### 结论判断
  - 当前修改已经把最关键的主排产入口拉回正确方向，但严格语义仍然依赖“调用方先做快照规范化”这一前提；只要走到受支持的旁路入口，静默默认化还会出现，因此我暂时不能把这一块判为完全收口。
- 结论:

  主排产入口改进成立，但 strict 语义仍依赖上游快照前置规范化；直接配置入口与预设应用入口还保留静默默认化旁路。
