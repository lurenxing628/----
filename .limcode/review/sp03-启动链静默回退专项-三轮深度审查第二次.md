### 前轮 R3-03 对 render_ui_template 的事实性判断存在错误

- ID: D1-01
- 严重级别: 高
- 分类: 其他
- 跟踪状态: 开放
- 相关里程碑: M1
- 说明:

  前轮审查 R3-03 声称：「render_ui_template() 行 380-384 会再次通过 context.setdefault(...) 注入，但它无法覆盖 env.globals 层，只能保证 render_ui_template 本身调用时的上下文安全，不能保护直接使用 render_template 的少数路由」。

  这个判断有两个事实性错误：

  **错误 1：render_ui_template 确实会写入 env.globals**
  ui_mode.py 行 410-418：
  ```python
  try:
      env.globals.setdefault("safe_url_for", safe_url_for)
      env.globals["url_for"] = template_url_for
      env.globals["get_help_card"] = get_help_card
      env.globals["get_manual_url"] = get_manual_url
      env.globals["get_full_manual_section_url"] = get_full_manual_section_url
  except Exception:
      pass
  ```
  这段代码在渲染前执行，且写入的是 env.globals（不是 context），覆盖了导入宏等间接模板路径。

  **错误 2：仓库中不存在「直接使用 render_template 的少数路由」**
  全文搜索 `from flask import.*render_template` 在 web/ 目录下结果为 0。所有路由文件、error_handlers.py 均使用 `from web.ui_mode import render_ui_template as render_template`。

  **影响链修正：**
  init_ui_mode except#4 的真实触发条件需要双重故障：
  1. 启动期 `app.jinja_env.globals[...]` 注入失败（except#4 触发）
  2. 每次渲染期 `env.globals.setdefault/[...]` 也失败（行 417-418 的 except 触发）

  第 2 条几乎不可能发生（dict.setdefault 和赋值操作几乎不会抛异常，除非 env.globals 本身是 None 或不可写），因此 except#4 的实际风险远低于前轮评估的「高风险启动守卫」。

  **修正建议：** 计划 A5 第 4 点将 except#4 列为「高风险」可以保留为审慎策略，但处置说明中应补充：render_ui_template 已有 env.globals 二次注入作为安全网，实际触发 500 需要双重故障。
- 建议:

  在计划 A5 第 4 点中补充：「注意 render_ui_template() 在渲染前会通过 env.globals 二次注入同组变量（行 410-418），因此实际触发假成功需要 init_ui_mode 启动注入与 render_ui_template 渲染注入双重失败，风险远低于无二次防御的单点故障。但出于审慎，仍建议补充可观测日志。」

### 前轮 R2-03 对 fallback_log 的分析结论正确但表述需精确化

- ID: D1-02
- 严重级别: 低
- 分类: 文档
- 跟踪状态: 已修复
- 相关里程碑: M1
- 说明:

  前轮 R2-03 说 fallback_log 在 logger=None 时「会直接 return（行 93-94），并不会写入 stderr」。

  验证 core/infrastructure/migrations/common.py 行 92-106：
  ```python
  def fallback_log(logger, level, message):
      if not logger:
          return  # ← logger=None 时确实直接返回
      method = getattr(logger, str(level or "").strip(), None)
      if callable(method):
          try:
              method(str(message))
              return
          except Exception:
              pass
      try:
          print(str(message), file=sys.stderr)  # ← logger 有值但方法调用失败时回退 stderr
      except Exception:
          pass
  ```

  结论正确：paths.py 的 runtime_base_dir() 无 logger 参数，调用 fallback_log(None, ...) 会直接返回不做任何事。但表述应更精确：fallback_log 有 stderr 回退机制，只是该机制仅在 logger 参数有效但方法调用失败时才生效，而非完全没有 stderr 回退。
- 建议:

  无需修改计划，仅记录以避免执行者对 fallback_log 能力范围的误解。

### entrypoint.py APS_PORT 解析失败与回写失败是两个不同的静默点但计划只点名了后者

- ID: D2-02
- 严重级别: 中
- 分类: 可维护性
- 跟踪状态: 开放
- 相关里程碑: M2
- 说明:

  计划对 `entrypoint.py` 单列了 4 个元失败链点和 APS_HOST/APS_PORT 环境变量**回写**失败（行 198-202）。但还有一个未被点名的语义不同的静默回退：APS_PORT **解析**失败（行 184-188）：
  ```python
  if raw_port is not None and str(raw_port).strip() != "":
      try:
          preferred_port = int(str(raw_port).strip())
      except Exception:
          preferred_port = 5000  # ← 非数字值静默回退 5000
  ```

  这与行 198-202 的环境变量回写失败是两个不同的静默点：
  - 行 184-188：读取/解析 APS_PORT 失败 → 静默回退 5000（无任何日志）
  - 行 198-202：写入 APS_HOST/APS_PORT 环境变量失败 → pass

  计划点名了后者但未点名前者。前者的风险在于：如果用户设置 `APS_PORT=abc`，应用会静默使用 5000 端口，却无任何告警提示配置有误。
- 建议:

  在计划阶段 3（B 类热点）的 `entrypoint.py` 说明中，将 APS_PORT 解析失败与 APS_PORT 回写失败分开列出，避免执行时混淆。建议对前者补充 `app.logger.warning(f"APS_PORT={raw_port} 解析失败，已回退默认端口 5000")`。
