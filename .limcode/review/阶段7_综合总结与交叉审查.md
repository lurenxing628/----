# core 目录综合审查报告

## 1. 总体结论
- 现有 `core/` 主链并非“整体不可用”，但已经形成一组跨阶段、跨层级、会相互放大的系统性风险，核心特征不是单点崩溃，而是“静默回退、伪成功、默认值吞错、降级不外显”。
- 全局最高风险不是某一个函数，而是 `strict_mode` 语义在阶段2、阶段3、阶段4之间不断漏损：配置快照、优化器配置读取、算法输入构建、算法参数解析都可能继续把坏值修补成可运行值。
- “外协周期 / 供应商默认周期 / 外部工序周期”已经形成贯穿阶段3、阶段4、阶段5、阶段6的一条默认值污染链：空白或非法输入多处被静默改写成 `1.0` 天，并最终影响排产时长。
- 冻结窗口在文义上被声明为硬约束，但在实现上存在 fail-open；摘要层又可能继续宣称 `freeze_window` 属于硬约束，形成“约束已失效、对外仍像生效”的口径冲突。
- 排产下游展示链的主风险集中在“坏时间行静默丢弃 + 前端弱兜底继续渲染”：甘特图、周计划、资源排班都可能把损坏排程包装成“当前无任务”或看似合理的默认日历。
- 阶段4的“空行动保险丝”在持久化层存在，但在真实排产主链被固定绕过；虽然“无可重排工序”和“全冻结”已有前置阻断，但在“优化器返回结果却没有任何有效排程行”的窄场景里，系统仍可能写出版本号、历史记录与摘要，却没有任何实际可执行排程行。
- 算法层仍存在两类未被真正跑通验证的高怀疑问题：坏交期会被当作“无交期”，未完工批次不会进入逾期指标。这两类问题若成立，将直接影响排序、评估、优化比较与摘要可信度。
- 阶段5的批次写回链并不缺事务，而是事务边界和默认化位置错了：改图号不强制重建工序、建批前先改全局模板、外协周期留空写成 `1.0`，都会制造“看起来成功、实际语义已漂移”的结果。
- 阶段1、阶段6共同表明：系统在运维与配置侧也有同类问题，例如恢复成功口径早于结构校验、系统配置脏值读取被静默规范化、损坏任务时间被当作从未运行。
- 当前最稳的链路主要有四条：基础设施迁移/事务主合同、阶段2公共执行器主合同、阶段5日历与资源池基础语义、阶段6维护任务 telemetry 隔离。这些稳定点可以作为治理切入点，但不足以覆盖上面的跨阶段风险。
- 综合判断：当前 `core/` 更接近“成功路径可用，但异常输入、脏数据、降级路径仍会被包装成可接受结果”的状态；因此适合按系统性根因治理，而不是继续做零散补丁。

  - 阶段3的坏交期整链、未完工批次指标盲区仍缺最小复现实跑；当前证据主要是代码链、已有单点测试与覆盖空白共同支撑。

### 第三轮：系统性根因归并与治理方向
- 最终归并出 8 个系统性根因主题：`strict_mode` 漏损、默认值滥用、fail-open 硬约束、伪成功写入、读侧静默过滤、可观测性断裂、事务边界泄漏、分层污染与重复规则。
- 治理原则不是“哪里出问题修哪里”，而是优先修共享入口与共享合同：
  1. 先收口公共数字/枚举/时间解析语义；
  2. 再收口排产主链上的 strict 与 hard constraint；
  3. 再把降级事实统一透传到摘要、历史、界面、导出；
  4. 最后再做架构拆分与长期兼容迁移。
- 本轮输出的路线图按“立即止血 / 短期修复 / 中期重构 / 长期治理”四层展开，既考虑用户可感知风险，也考虑修复杠杆。

## 5. 根因归并
### 5.1 `strict_mode` 漏损
- **涉及阶段**：阶段2、阶段3、阶段4。
- **核心文件**：`core/services/common/number_utils.py`、`core/services/scheduler/config_snapshot.py`、`core/services/scheduler/schedule_input_builder.py`、`core/services/scheduler/schedule_optimizer.py`、`core/algorithms/greedy/schedule_params.py`。
- **为什么是系统性问题**：严格模式不是单点开关，而是多层传递合同。当前配置快照、优化器配置读取、输入构建、算法参数解析都只拦一部分字段，最终把“严格”变成“部分字段严格，其余字段继续默认化”。
- **后果**：用户以为自己在做 fail-fast，系统实际却在用修补后的值继续跑完整主链。

### 5.2 默认值滥用与静默回退
- **涉及阶段**：阶段2、阶段3、阶段4、阶段5、阶段6。
- **典型表现**：空白数字变 `0/0.0`、空白外协周期变 `1.0`、空白供应商周期变 `1.0`、空白工序周期编辑变 `1.0`、空白导入字段直接补成业务默认值。
- **为什么长期存在**：实现长期偏向“不要因为脏数据中断主流程”，但没有同步设计“退化必须外显”的合同。
- **后果**：问题不会消失，只会从“抛异常”变成“结果失真”。
- **补充断点**：`schedule_template_lookup.py` 在模板或外部组缺失时直接清空组合并元数据，属于“先静默降级、再继续运行”的同型问题；一旦叠加空白 `ext_days`，还会继续滑入 `1.0` 天默认链。

### 5.3 fail-open 的硬约束与伪成功写入
- **涉及阶段**：阶段1、阶段4。
- **核心文件**：`backup.py`、`freeze_window.py`、`schedule_service.py`、`schedule_persistence.py`。
- **系统性特征**：底层先返回 success 或继续运行，再由上层补校验；一旦上层遗漏，伪成功就进入页面、历史、版本号与操作日志。排产主链虽然对“无可重排工序 / 全冻结”已有前置短路，但一旦进入持久化编排路径，保险丝参数仍被硬编码为已通过。
- **后果**：运维、审计和用户界面都可能记录一份“看起来成功、实际未完成”的留痕。

### 5.4 读侧静默过滤与弱兜底展示
- **涉及阶段**：阶段5、阶段6，侧面关联阶段2的行号丢失。
- **核心文件**：`gantt_tasks.py`、`gantt_week_plan.py`、`resource_dispatch_rows.py`、`gantt_render.js`、`operator_machine_query_service.py`、`system_config_service.py`。
- **系统性特征**：坏值不是显式暴露，而是被过滤成“无数据”、被读侧归一成“规范值”、被前端 fallback 补成“默认视觉”。
- **后果**：界面越稳定，底层坏数据越难被发现。

### 5.5 可观测性断裂
- **涉及阶段**：阶段1、阶段4、阶段5、阶段6。
- **典型断点**：日志有、摘要没有；页面有、导出没有；服务层有、页面入口没注入；警告有、硬约束列表仍旧误报。
- **为什么是系统性问题**：项目没有统一的退化状态模型，导致每一层都按自己的需要选择“记日志”“吞掉”“加 warning”或“完全忽略”。
- **后果**：排障时必须同时看日志、数据库、历史、页面和导出文件，成本极高。

### 5.6 事务边界不稳与跨域副作用泄漏
- **涉及阶段**：阶段4、阶段5。
- **核心文件**：`schedule_service.py`、`schedule_persistence.py`、`batch_service.py`、`part_service.py`。
- **表现**：主链绕过保险丝、建批前先写模板、批次头与批次工序语义脱钩。
- **后果**：即使数据库层事务是正确的，业务语义仍可能出现“局部成功 + 全局副作用”。

### 5.7 契约漂移
- **涉及阶段**：阶段1、阶段3、阶段4、阶段5。
- **典型表现**：`SkillLevel` 四值/三值漂移、冻结窗口“降级但仍是硬约束”、导出与页面对同一次查询的告警口径不同、算法评价与优化比较口径不完整。
- **补充断点**：`active_preset` 与真实生效配置脱钩，说明配置元数据本身也会漂移；用户看到的“当前预设”未必对应当前数据库里真正生效的值。
- **后果**：文档、常量、页面、摘要和数据库不再描述同一件事。

### 5.8 分层污染与重复规则
- **涉及阶段**：阶段1、阶段2、阶段4、阶段6。
- **核心文件**：`core/plugins/manager.py`、`core/services/common/excel_validators.py`、`web/routes/normalizers.py`、`core/services/scheduler/calendar_admin.py`、`core/services/personnel/operator_machine_normalizers.py`、`schedule_service.py`、`batch_service.py`。
- **表现**：底座层向上依赖仓储/服务；同类归一化规则在 common/route/service 多处复制；大文件承担输入清洗、业务判定、摘要与展示映射多重职责。
- **补充断点**：`web/routes/scheduler_excel_calendar.py` 仍旁路 `excel_import_executor.py`，说明“共享执行器合同”尚未真正收口到单入口；当前虽未出现行为漂移，但后续演进会带来双处维护风险。
- **后果**：一处修复，常常需要多点同步；边界漂移难以避免。

## 7. 架构与分层结论
- **明确存在的分层违反**：`core/plugins/manager.py` 直接依赖 `core.services.common.enum_normalizers` 与 `data.repositories.SystemConfigRepository`，底座层被上层规则反向污染。
- **职责混杂最严重的文件/目录**：
  - `core/services/scheduler/schedule_service.py`：同时承担配置读取、输入构建、冻结窗口、停机加载、资源池构建、优化调用、摘要生成、持久化编排。
  - `core/services/scheduler/batch_service.py`：同时承担批次 CRUD、模板解析补建、工序重建、跨域模板写回。
  - `core/services/common/excel_validators.py` + `web/routes/normalizers.py` + `core/services/scheduler/calendar_admin.py`：同类归一化规则散落多处。
- **前端职责越位**：`static/js/gantt_render.js` 的 `_buildFallbackCalendarDays()` 已经不仅是纯展示容错，而是在替后端定义“周末是假期、工作日 8 小时”的业务默认日历语义。
- **未确认的强循环依赖**：本轮没有证据证明存在不可打破的代码级循环依赖，但隐式跨层调用与职责倒挂已经足以造成维护困难。
- **优先拆分建议**：
  1. `schedule_service.py` 拆成“输入收口 / 优化编排 / 结果落库 / 摘要拼装”四块；
  2. `batch_service.py` 拆成“批次头服务 / 模板解析协调器 / 工序重建服务”；
  3. 统一抽取 `common` 级归一化矩阵，削减 route/service 重复逻辑；
  4. 插件层通过注入接口读取配置，不再直连仓储。

## 9. 一致性问题结论
### 9.1 默认值与兼容逻辑
- 同一类空白输入在不同层的语义不一致：
  - `parse_finite_float/int(..., allow_none=False)` 把空白改成 `0/0.0`；
  - `build_algo_operations()` 把空白工时改成 `0.0`、空白 `ext_days` 改成 `None`；
  - `external_groups.py` 与 `dispatch/sgs.py` 再把 `None` 改成 `1.0`；
  - `operation_edit_service.py` 也会把用户清空的 `ext_days` 写成 `1.0`。
- 同一套兼容策略并未形成统一矩阵，导致“哪里算错误、哪里算默认、哪里要 warning”完全依赖具体实现。

### 9.2 时间与范围口径
- 合法交期的闭区间/开区间语义已有测试，但坏交期会被统一下沉成 `None`，从而在排序、评分、指标、warm-start 里全部走“无交期”语义。
- 甘特图、周计划、资源排班对坏时间行采用一致的“直接丢弃”，但这套语义没有进入任何 contract 字段。
- 阶段2的 `row_num = idx + 2` 与后端空白行压缩一起构成导入侧的源行号漂移，说明“时间/范围/位置口径”的问题不仅存在于排产，也存在于导入审计。

### 9.3 展示与导出口径
- 页面与导出不一致最典型：资源排班页面会展示 `overdue_markers_*` 降级，导出文件完全丢失这组上下文。
- 摘要内部也存在口径冲突：冻结窗口已降级，但 `algo.hard_constraints` 仍可能包含 `freeze_window`。
- 前端 fallback 被用来维持页面可看，但没有同步向用户解释“现在看到的是推断结果而不是真实数据”。

### 9.4 读取与写入口径
- 读侧喜欢“规范化后展示”：系统配置、人机关联详情、设备/人员页面都偏向这样做。
- 写侧喜欢“补默认值继续保存”：供应商默认周期、外部工序周期、批次图号切换、模板自动补建都体现了同样偏好。
- 结果是系统长期处于“读的时候看不见问题，写的时候继续制造问题”的循环里。

## 11. 测试防线建议
### 前十个最值得补的回归场景
1. **双后端空白行行号保真**：同一工作簿分别走 `OpenpyxlBackend` 与 `PandasBackend`，断言错误行号保持原始 Excel 行号，而不是压缩后序号。
2. **strict 配置空白数字拒绝**：`priority_weight=""`、`due_weight=""`、`ready_weight=""`、`freeze_window_days=""` 在 strict 下必须报错，且 `schedule_optimizer._cfg_float()` 不得再把这些值回退为默认权重。
3. **排产输入 strict 贯穿**：`build_algo_operations()` 在 strict 下遇到空白/非法 `setup_hours`、`unit_hours`、`ext_days` 时应显式失败或留下结构化 warning，并与优化器/算法参数读取保持同口径。
4. **坏交期整链复现**：从排产入口构造非法 `due_date`，验证排序、指标、摘要、warm-start 输入是否都会外显退化，而不是静默按无交期处理。
5. **未完工批次指标语义**：两个 `failed_ops` 相同但放弃不同逾期批次的候选方案，验证优化器比较结果是否区分。
6. **冻结窗口加载失败行为**：上一版本缺失/坏时间/查询异常时，验证返回值、摘要、历史、页面是否一致显示“冻结未生效”。
7. **甘特图/周计划坏时间行过滤**：构造 `start_time/end_time` 非法的排程版本，验证接口必须返回过滤计数与退化 warning。
8. **资源排班坏时间行过滤与导出保真**：同时校验数据接口、页面 warning、导出摘要页三处都能保留过滤与超期降级信息。
9. **批次 Excel 改图号更新链**：已有批次改 `part_no` 且关闭 `auto_generate_ops` 时，应拒绝或强制重建工序。
10. **供应商页面非法默认周期**：直接走 `/process/suppliers/create` 与 `/process/suppliers/<id>/update`，验证页面提示、日志与数据库最终值一致。

### 优先新增的集成测试与边界测试
- 排产主链集成测试：`strict_mode + freeze_window + degraded summary + persistence fuse` 的组合场景。
- 导入/排产/展示一体化测试：脏模板、脏交期、脏时间行从写入到甘特图/资源排班的全链路表现。
- 运维边界测试：脏 `SystemConfig`、脏 `SystemJobState`、恢复后结构校验失败、插件能力冲突。
- 读侧脏数据可见性测试：系统页面、人机关联页面、资源调度页面在脏值存在时是否留下显式标记。

## 13. 修复优先级排序依据
- **第一优先**看“是否会在用户无感知的情况下改变真实业务语义”：例如空白数字变 `0`、空白周期变 `1.0`、坏时间行被过滤、空排程仍写历史，这些都比纯粹的代码味道更紧急。
- **第二优先**看“是否跨阶段传播”：共享解析函数、配置快照、排产入口、算法评估一旦修复，可以同时降低多个阶段的问题密度。
- **第三优先**看“是否会制造伪成功”：恢复 success 边界、冻结窗口 fail-open、warm-start hidden failure、导出丢失告警，都直接影响运维与管理判断。
- **第四优先**才看“是否属于结构债务”：插件层反向依赖、归一化重复实现、大型协调类服务文件虽然重要，但在未止住上面的语义性错误前，不宜优先于止血项。
- **证据强度也参与排序**：阶段1/2/4/5/6中有代码 + 测试源码 + 运行留痕或明确缺口支撑的问题优先级更高；阶段3中仍未实跑的问题虽然业务影响大，但实施时应先补最小复现再决定具体修法。
