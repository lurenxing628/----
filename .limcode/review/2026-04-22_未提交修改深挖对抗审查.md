## 本轮新增事实

### 1. 配置 provenance 的直接断裂点已经收敛到读链与写链，不是单个 if

事实：

1. 读链在 current_config_state 中把以下三种情况统一判成 provenance_missing：
   - active_preset 缺失
   - active_preset_reason 行缺失
   - active_preset 为空文本
2. 保存链的 hidden repair 放行条件只看 current_active_preset 是否有值，不看 active_preset_reason_missing，也不看 provenance_missing。
3. 因此 active_preset 存在但 active_preset_reason 行缺失时，页面只读态会把它视为不确定，但保存配置时仍可能把它按命名方案 hidden repair 持久化。

结论：

- 当前 blocker 不是 save_page_config 里某一处分支写错，而是 provenance completeness 在读取、保存、bootstrap、rejected path 上没有单一合同。

### 2. bootstrap 不是直接致因，但仍是根因链上的扩散点

事实：

1. bootstrap_active_provenance_if_pristine 只要看到 active_preset 行存在就直接返回。
2. 它不会主动制造 hidden_repair，但会让半残缺 provenance 长期留在库里。
3. 后续真正把这类状态重新归类并写回的是 save_page_config。

结论：

- bootstrap 不是主裂口，但如果只修 save_page_config 而不统一 bootstrap 的 completeness 判定，问题会继续残留在脏库和历史兼容场景里。

### 3. analysis 页 warning 问题已经证明是显示契约断链，不是路由漏传

事实：

1. summary display builder 已经稳定产出 warnings_preview、warning_total、warning_hidden_count。
2. analysis route 已经把 selected_summary_display 传给模板。
3. analysis 模板当前只消费 secondary degradation、parse_failed、primary_degradation、error_total，没有消费 warning。
4. 同期 week_plan 和 history 模板已经消费 warning。

结论：

- 这是 analysis 页 observability contract 缺口，不是主链 correctness 损坏。

## 根因裁决

### P1：配置 provenance completeness 没有 canonical contract

最准确的表述：

- 读取链把 active_preset_reason 行缺失视为 provenance_missing。
- 保存链 hidden repair 只要求 active_preset 非空。
- bootstrap 只要求 active_preset 行存在。
- rejected path 也会把缩略 provenance 当作 effective identity 消费。

因此同一个半残缺状态会在不同路径被同时解释为：

1. 不确定态
2. 已初始化态
3. 可沿用基线的可写态

这才是当前 blocker 的根因。

### P2：analysis 页 warning 可见性没有闭环

最准确的表述：

- 问题不在服务层、路由层，也不在 summary 生产链；问题在 analysis 页最后一跳模板消费和对应回归合同缺失。

### P2：quality gate 与 analysis 真实显示契约没有绑定

最准确的表述：

- gate 在跑，但跑的是静态 guard 清单。
- analysis 真模板 warning 可见性没有进入显式门禁，因此同类问题可以继续漏过。
