- 下一步建议: 按四个治理主题单独立项：先做调度配置单一事实源，其次拆分数据库基础设施热点，再收敛 Web 模板树与页面脚本契约，最后统一测试/证据/文档主索引并补根 README。

- 严重级别: 中
- 分类: 可维护性
- 跟踪状态: 开放
- 相关里程碑: M1
- 说明:

  项目文档、依赖声明和静态检查配置共同表明系统被明确锁定在 Win7 与 Python 3.8 约束下，同时依赖保持最小集。这有利于现场交付，但也意味着升级解释器、引入现代库特性、统一类型语法与工具链能力都会持续受限，后续每次基础设施演进都需要优先兼顾历史交付环境。

### 调度配置规则多点复制

- ID: sched-config-rule-duplication
- 严重级别: 高
- 分类: 可维护性
- 跟踪状态: 开放
- 相关里程碑: M2
- 说明:

  同一批调度配置字段的合法值、缺省值、严格模式、数值下限和兼容回退逻辑，分别散落在配置服务、配置快照、预设规范化、算法参数解析和优化运行时读取中。当前至少有 config_service、config_snapshot、config_validator、schedule_params、schedule_optimizer_steps 五处相近逻辑。这样虽然短期便于各层自保，但任何新增字段或规则收紧都需要多点同步，否则很容易出现‘页面可保存、预设可应用、算法读取却按另一套规则降级’的漂移。

## 最终结论

本次全面梳理表明，仓库并非缺少工程意识，相反已经具备分层架构、适应度门禁、回归测试、审计留痕和大量文档沉淀；真正的问题在于，这些治理措施长期与多条兼容链并行存在，导致技术债从“几个超长文件”演变成“多个事实源、多个兼容层、多个热点模块共同抬高改动成本”。

全栈最值得优先治理的 P1 技术债有四组：
1. 调度配置规则多点复制 —— 这是当前唯一明确的高优先级业务核心债，直接影响配置演进正确性；
2. 数据基础设施热点膨胀 —— `database.py` 与备份/迁移链路已成为高耦合单点；
3. Web 双模板树与页面装配策略未收敛 —— 复杂度在 route、viewmodel、模板和全局脚本之间迁移；
4. 测试与文档事实源分散 —— 回归脚本、证据、audit、综合文档和速查表形成高人工同步成本。

结论上，当前系统适合继续演进，但不适合继续以“局部补丁 + 兼容桥接 + 文档追加”作为默认模式；若不启动针对事实源收敛和热点拆分的治理，后续每一轮需求修改都会越来越依赖经验性维护。
