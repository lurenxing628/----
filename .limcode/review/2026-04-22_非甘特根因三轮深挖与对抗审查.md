## 最终裁决

### P1：version 参数合同真实分裂，且当前是行为级风险，不是单纯注释债

事实：

1. 宽松 latest-fallback 由 `web/routes/normalizers.py` 定义，并被 analysis、week-plan、reports 复用。
2. system_history 不是同一合同，它只做严格整数解析，不拒绝 0/负数。
3. resource_dispatch 也不是同一合同，它在 service 层要求严格正整数，并把 page/data/export 三个入口绑在一起。
4. analysis 与 reports 的 fallback 行为已被回归显式锁定；week-plan 的归一化版本文件名也已被锁定。

结论：

- 这不是“每个页面自己乱来”，而是当前明确存在三类 owner：
  - 路由层共享 latest-fallback
  - system_history 路由层 strict-int
  - resource_dispatch service 层 strict-positive-int
- 风险不在于“有三种行为”本身，而在于 owner 和语义命名不清，shared helper 容易继续外溢。
- 独立审核进一步收窄后，建议不是“一刀切只保留一种行为”，而是把 owner 显式化：analysis、reports、week-plan 继续走边界层宽松合同；resource_dispatch 可在边界层加统一入口，但 service 的 strict-positive 不变量必须保留。

### P1：AppError 的 HTTP 语义缺口和 route 本地吞错过多同时存在

事实：

1. 全局 `AppError` handler 目前除了 `FILE_TOO_LARGE` 外，统一返回 400。
2. 错误体系本身已经区分 `NOT_FOUND`、`PERMISSION_DENIED`、`DUPLICATE_ENTRY` 等语义。
3. 多条 route 会在本地把 `AppError` 翻译成 flash + redirect、flash + 200 降级页、或固定 400 JSON。
4. resource_dispatch 的 page/export 302 与 data 400 已被测试锁定，但这只是该路由族的 UX 契约，不足以为全局 400 压平背书。
5. 批量页面的 200 摘要回显同样是既有 UX 合同，不能粗暴删除。

结论：

- 根因不是二选一，而是“双层同时有问题”：
  - 未被 route 捕获的路径，HTTP 状态码几乎被压平成 400
  - 被 route 捕获的路径，又各自实现了不同粒度的 UX 翻译
- 修复方向不能是“全删本地 catch”或“只修全局映射”。更合理的是分层：
  - 全局层负责未捕获错误的 `ErrorCode -> HTTP` 映射
  - page/export/bulk route 只在白名单场景做本地 UX 翻译
  - data/API 尽量统一走单一 JSON 错误出口
- 独立审核补充了一点关键约束：真实业务里大量 404/403 语义是通过 `BusinessError(ErrorCode.*)` 抛出，而不是通过异常子类抛出。所以全局映射不能只按异常类做。

### P1：config provenance 的主写链 blocker 已收口，但 readonly 真相链仍有活跃 correctness 缺口

事实：

1. hidden repair 在 provenance 缺失时已被阻断，缺 `active_preset` 或缺 `active_preset_reason` 的关键回归已通过。
2. strict preset contract 的写路径护栏也已存在：`apply_preset` 缺字段拒绝、`save_preset` 先 strict 读 snapshot、rejected path 不偷写 provenance。
3. `current_config_state` 的构造仍主要信任 provenance、reason 和 degradation，不会在 readonly 链再拿当前 snapshot 与 baseline preset 做真实 diff。
4. 仓库里已有 baseline payload loader 和 diff helper，但它们主要用于 apply/bootstrap 写链，而不是 readonly 显示链。

结论：

- “hidden repair 会把缺 provenance 洗成 exact”这个旧 blocker 已被堵住，不再是当前主风险。
- 当前仍活跃的缺口，是 `provenance 完整但 baseline 已漂移` 时，`current_config_state` 仍可能报 exact。
- 独立审核进一步收窄了触发面：它更像脏库、旁路写库、升级残留、脚本/夹具篡改带来的 correctness 缺口，而不是当前配置页主写链的直接回归。
- 因此这里的修复重点不再是 save_page_config 分支，而是共享 readonly truth-chain：要把真实 diff 接回 `get_preset_display_state` / `_build_current_config_state`，并让 `_readonly_active_preset_state`、bootstrap 等消费者复用同一判定。

### P2：warning 文本主链已可见；真正未收口的是结构化 provenance 与 HTML 级守卫

事实：

1. 新摘要会把 warnings 落到顶层，analysis、week-plan、history 详情区、batches 都能显示 warning 文本。
2. analysis 的 warning 文本显示已经有真实 HTML 回归保护。
3. week-plan 与 history 当前主要锁的是 ctx / route payload，不是 rendered HTML。
4. `algo.warning_pipeline` 的结构化字段没有进入共享 display contract；只有降级事件与顶层 warnings 会进入用户可见层。
5. history 列表行只显示“告警 N 条”，不显示 preview 文本；详情区才显示 preview。
6. provenance 还有另一条独立链：config 的 `current_config_state`，它不属于 summary display contract。

结论：

- 本轮不再把“warning 文本没显示”列为主问题，因为当前代码和测试都证明它已经打通。
- 新摘要的主问题是：`warning_pipeline` 的结构化 provenance 没有稳定进入 display contract。
- week-plan/history 的主要风险是 HTML 级守卫不足，而不是当前行为已坏。
- provenance 不能再被混进这条 summary 结论里，它属于 config truth-chain 的单独问题。

## 被对抗审查收窄或推翻的点

1. “version 的最佳修复就是全部收敛到请求边界层”被收窄。
   - analysis、reports、week-plan 已基本在边界层；真正要保留双持的是 resource_dispatch：边界层可补统一入口，但 service 的 strict-positive 合同不能丢。

2. “gate/summary 的主问题是 presenter/display contract + A+C 一起改”被收窄。
   - 对 warning 来说，display contract 已经有大半，当前收益更高的是补 week-plan/history 的 rendered HTML guard。
   - 对 provenance 来说，根因不在 summary display，而在 ConfigService truth-chain。

3. “当前 provenance 缺口等价于当前配置页主写链随时会错”被收窄。
   - 现阶段更像 readonly 误报与存量脏状态识别不真，而不是当前正常 UI 写链立刻会写错。

## 不进入本轮主线的问题

1. maintenance 检测 fail-open / fail-closed 语义错位
2. error.html 相关字段重复展示
3. quality gate operations 本身
4. warning 文本丢失
5. hidden repair 在缺 provenance 时继续放行

这些点要么已被本轮证据排除，要么属于设计债/低优先级可见性问题，不应压过四条主线。

## 修复优先级建议

1. 先收口 config readonly truth-chain，堵住 exact 假阳性
2. 再收口 AppError 的全局映射，并梳理 route 本地翻译白名单
3. 同步把 version 三类合同显式化，补缺失矩阵回归
4. 最后补 warning_pipeline 的结构化 display contract 与 week-plan/history 的 rendered HTML guard

## 合入建议

当前不建议按现状合入。

阻断原因已经从早先的“hidden repair 直接漏洞”转成三条更准确的主线：

1. config readonly truth-chain 仍可能误报 exact
2. AppError 的 HTTP 语义在全局和局部两层同时失真
3. version 合同 owner 不清，shared helper 仍有外溢空间

warning 文本链路本身不再构成 blocker，但它的结构化 provenance 和 HTML 级守卫仍建议同批补齐，避免下一轮再出现“门是绿的、但页面真实显示没锁住”的情况。
