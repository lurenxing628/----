- ID: F-javascript-1

- 严重级别: 中
- 分类: 其他
- 跟踪状态: 开放

- 说明:

  ### 1. `_validated_quantity` 的逻辑不一致（中等）
  `batch_write_rules.py` 第26-30行：
  ```python
  def _validated_quantity(svc, quantity: Any, *, allow_none: bool) -> int:
      normalized = svc._normalize_int(quantity, field="数量", allow_none=allow_none)
      if normalized is None or normalized <= 0:
          raise ValidationError(""数量"必须大于 0", field="数量")
      return int(normalized)
  ```
  当 `allow_none=True` 且 `quantity` 为 `None` 时，`_normalize_int` 会返回 `None`，然后会被 `normalized is None` 捕获并抛出 `"数量"必须大于 0`。这意味着 `allow_none=True` 参数在实际执行中永远不生效——即使调用方声明允许为空，传入空值仍会报错。
  但从 `build_update_payload` 的调用来看（第143-144行），只有当 `quantity is not _MISSING` 时才调用，所以 `None` 值实际不太可能传入。不过逻辑上存在歧义。

  ### 2. `batch_excel_import` 中 `int(pr.data.get("数量"))` 可能崩溃（高）
  `batch_excel_import.py` 第83行：
  ```python
  quantity=int(pr.data.get("数量")),
  ```
  如果 `数量` 为 `None` 或非数字字符串，`int()` 会抛出 `TypeError`/`ValueError`。虽然上游的 `excel_validators` 校验器应该已经校验过，但这里属于 `_apply_row_no_tx` 内部执行，如果校验被跳过或数据被篡改，直接 `int()` 会产生难以追踪的栈溢出错误而非友好的业务异常。

  ### 3. `svc` 参数传递模式（低）
  所有拆分出的函数都将 `svc`（`BatchService` 实例）作为第一个参数传递并直接访问其内部方法（如 `svc._normalize_text`、`svc._safe_float`）。这在本质上是将私有方法暴露给外部模块，违反了封装性。更干净的做法是定义一个明确的接口/协议。
