### _shutdown 守护线程内异常无出口

- ID: V2-R3-01
- 严重级别: 中
- 分类: 可维护性
- 跟踪状态: 开放
- 相关里程碑: R3
- 说明:

  `request_runtime_server_shutdown()` 内部的 `_shutdown()` 闭包在守护线程中执行（行 180）。当 `server.shutdown()` 抛异常且 `logger is not None` 时，尝试 `logger.warning(...)`；若日志也失败，内层 `except Exception: pass` 完全吞没。当 `logger is None` 时，`server.shutdown()` 失败直接静默丢弃。

  计划 A3（补充）对 `_shutdown` 的处置说明是「低噪声可观测保留或改用共享安全日志助手」，但未特别指出 `logger is None` 时的完全无出口问题。实际上 `system_health.py` 行 53 调用时传入 `logger=current_app.logger`，因此主调用路径不会走到 `logger is None` 分支。但函数签名允许 `logger=None` 默认值，且守护线程内的异常不会传播到主线程。

  风险实际很低——`werkzeug.server.shutdown()` 几乎不会失败，且僅在停机时触发。但计划既然要求「显式决定」，应在 A3（补充）中标注这个 `logger=None` 时的无出口分支。
- 建议:

  在 A3（补充）中补充：`_shutdown` 内部闭包在 `logger is None` 时的异常完全无出口，建议补用 `fallback_log` 或 `print(file=sys.stderr)` 作为最后防线；但主调用路径已传入 logger，实际风险极低。

  1. **`_open_db` 白名单 fail-open 影响链**（独立验证上一版 R3-01）：
     `request.path` 解析失败 → 白名单判定被跳过 → 维护窗口检测对所有路径生效 → 若正在维护，`/system/health` 和 `/system/runtime/shutdown` 被 503 拦截。但 `request.path` 由 Werkzeug HTTP 解析层保证，失败概率极低。**结论：计划的 fail-open 决策正确，但应记录最坏影响链**——上一版 R3-01 仍有效。

  2. **`_maintenance_gate_response` → `render_ui_template` → `get_ui_mode` → `_read_ui_mode_from_db` 循环依赖**（独立验证上一版 R4-03）：

  3. **`init_ui_mode` except#4 globals 注入 → 模板宏链**（独立验证上一版 R4-01）：
     所有路由模块均 `import render_ui_template as render_template`，不存在直接使用原生 `render_template` 的代码。except#4 的真实传导路径是 `{% import %}` 不带 `with context` 的宏（如 `ui_macros.html` 中引用 `safe_url_for`/`get_help_card`）→ 若 globals 注入失败但 `render_ui_template` 行 412 的二次 `env.globals.setdefault(...)` 仍成功，宏链不受影响。仅在 `jinja_env.globals` 字典本身损坏时才真正传导。**结论：计划 A5 第 4 点的风险描述需修正**——上一版 R4-01 仍有效。

  4. **`_shutdown` 守护线程内异常无出口**（新发现）：
     `request_runtime_server_shutdown` 行 169-178 的 `_shutdown()` 闭包在守护线程中执行。当 `server.shutdown()` 失败时：(a) `logger` 非空则尝试 `logger.warning`，若日志也失败则内层 `except: pass` 完全吞没；(b) `logger is None` 时异常直接消失。虽然 `system_health.py` 调用时总传入 `logger=current_app.logger`，但函数签名允许 `logger=None`，且守护线程内异常不传播到主线程。这是一个微小但计划未明确标注的完全无出口点。

  5. **`validate_startup_samples` 的区间匹配逻辑**：
