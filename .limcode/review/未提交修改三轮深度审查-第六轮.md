### 配置快照哨兵同一性潜伏 BUG

- ID: sentinel-identity-broken-config-snapshot
- 严重级别: 低
- 分类: JavaScript
- 跟踪状态: 开放
- 相关里程碑: R1
- 说明:

  `config_snapshot.py` 第 80 行 `_read_repo_raw_value()` 的回退路径将 `object()` 哨兵传给 `repo.get_value(key, default=raw_missing)`，而 `ConfigRepository.get_value()` 内部 `str(val) if val is not None else default` 会把哨兵转成字符串 `<object object at 0x...>`，导致 `raw is raw_missing` 恒为 False。当前不可达（`ConfigRepository` 总有 `.get()` 方法），但如果未来有仅实现 `get_value()` 的仓库替身，会把所有缺失键误判为非法值。
- 建议:

  可在 `_read_repo_raw_value` 的回退路径中改用 `KeyError` 异常或显式类型标注 `_MISSING` 常量，而非依赖 `object()` 同一性穿透 `get_value()`。或者干脆删除这个死代码路径，因为调用方总是传入有 `.get()` 方法的仓库。
- 证据:

### R1 · 第一轮：配置快照层与排产参数链源码级逻辑验证

- 状态: 已完成
- 记录时间: 2026-04-05T12:23:13.385Z

- 摘要:

  ### 审查方法
  逐行追踪 `ConfigService.get_snapshot(strict_mode=True)` → `build_schedule_config_snapshot()` → `_read_repo_raw_value()` → `_choice()` / `_yes_no()` 的完整调用链，验证"缺失/空白/非法"三态语义是否在快照层正确闭合。然后追踪 `resolve_schedule_params()` → `_critical_config_value()` → `read_critical_schedule_config()` 的排产参数读取链。

  ### 配置快照层验证结论

  #### `_read_repo_raw_value()` 三态正确性
  1. `repo.get(key)` 返回 `None`（键不存在）→ `(True, None)` ✓
  2. `repo.get(key)` 返回记录，`config_value=""` → `(False, "")` ✓
  3. `repo.get(key)` 返回记录，`config_value="sgs"` → `(False, "sgs")` ✓
  4. 回退 `repo.get_value(key, default=raw_missing)` 路径：如果 `ConfigRepository.get_value()` 被调用，内部 `str(val)` 会把 `object()` 哨兵转为 `"<object object at 0x...>"` 字符串，导致 `raw is raw_missing` 恒为 `False`——哨兵同一性检测失效。但实际上 `ConfigRepository` **总有** `.get()` 方法，所以这个回退路径在正式运行中永远不会触达。测试替身 `_StubRepo` 的 `get_value()` 不做 `str()` 转换，所以测试中也不触发此问题。**这是一个潜伏 BUG，当前不可达，但代码设计上不严谨。**

  #### `_choice()` 严格模式闭合性
  - `missing=True` → 返回默认值，不报错 ✓
  - `missing=False, text=""` + `strict=True` → `raise ValidationError("不能为空", field=key)` ✓
  - `missing=False, text="bad"` + `strict=True` → `raise ValidationError("取值不合法", field=key)` ✓
  - `missing=False, text="bad"` + `strict=False` → 回退默认值（不报错） ✓

  完全符合"缺键可默认、显式空白/非法值严格拒绝"的设计要求。

  #### `_yes_no()` 严格模式闭合性
  同 `_choice()` 结构，严格模式下空白和非合法 yes/no 值都拒绝。非严格模式下 `to_yes_no(raw, default=normalized_default)` 会把空白字符串按 `normalize_yes_no_wide` 约定映射为 `"no"`（而非使用 `default`）。这是归一化层的固有行为，不是新引入的问题。

  #### `_get_float()` / `_get_int()` 路径
  这两个路径不使用 `_read_repo_raw_value()`，而是直接 `repo.get_value(key, default=str(default))`。缺键时返回默认值字符串，`parse_field_float` / `parse_field_int` 正常解析。严格模式下非法值（如 `"abc"`、`"0"` 但要求 `>0`）由 `parse_field_float/int` 通过 `DegradationCollector` 抛出 `ValidationError`。链路自洽。

  ### 排产参数层验证结论

  #### `read_critical_schedule_config()` 控制流
  1. `key not in _CRITICAL_SCHEDULE_KEYS` → 直接 `raise KeyError` ✓
  2. `config is None` → `missing=True` ✓
  3. 字典路径：`key not in config` → `missing=True`；`config.get(key)` 空白/None → `missing=True`；正常值 → 直接返回 ✓
  4. 属性路径：`getattr` 成功 + 空白/None → `missing=True`；正常值 → `else` 块返回 ✓
  5. `AttributeError` → 回退 `.get()` 方法 ✓
  6. 其他异常 → `error=exc` 透传 ✓

  `try/except AttributeError ... else` 控制流正确：`else` 块在 `getattr` 未抛异常时执行，`AttributeError` 时继续到 `.get()` 回退。

  #### `_critical_config_value()` 三层防线
  - `error is None` → 返回值（可能是实际值或 `missing` 时的默认值） ✓
  - `error is not None` + `strict_mode` → `raise ValidationError` ✓
  - `error is not None` + 非 strict → 计数器 + 告警 + 回退 ✓

  但注意：如果 `missing=True` 且 `error is None`，`_critical_config_value` 返回默认值，**不区分"真正缺失"和"正常读取"**。这对排产参数层没有问题，因为缺失时使用默认值是预期行为。

  #### `cfg_get()` 残留宽泛异常
  `cfg_get()` 的两处 `except Exception: pass`（line 31-32, 37-38）仅用于 `priority_weight` / `due_weight` 读取。即使读取失败返回默认值 `0.4/0.5`，下游 `_weighted_value()` → `parse_field_float()` 有独立 `DegradationCollector` 会捕获非法浮点值并在严格模式下拒绝。因此宽泛异常不影响正确性，但残留的代码味道使未来维护者容易误用。
- 结论:

  配置快照层与排产参数链的严格模式语义已正确闭合。`_choice()`/`_yes_no()` 在严格模式下正确区分"缺失→默认"与"空白/非法→拒绝"。`read_critical_schedule_config()` 的 `try/except AttributeError/else` 控制流正确。`_critical_config_value()` 三层防线完备。发现一个潜伏BUG：`_read_repo_raw_value()` 回退路径的 `object()` 哨兵会被 `ConfigRepository.get_value()` 的 `str()` 破坏同一性——但该路径在正式运行中不可达。`cfg_get()` 残留宽泛异常是低级设计债，不影响正确性。
