### 2. 已验证事实
### F-1（HIGH）`g.app_logger` 在当前请求链路中并未赋值

**事实**
- `web/bootstrap/factory.py::_open_db()` 只写入了 `g.db` 与 `g.op_logger`，没有写入 `g.app_logger`。
- 最小运行时脚本验证结果：
  - `has_db True`
  - `has_op_logger True`
  - `has_app_logger False`
  - `current_app_logger_is_none False`

**判断**
- 计划里直接写 `logger=getattr(g, "app_logger", None)` 作为落地方案，会在当前 runtime 下得到 `None`。

**建议**
- 对本次计划，优先采用**局部修复**：
  - `process_parts.py` 改为传 `current_app.logger`
  - `scheduler_excel_batches.py` 也显式传 `current_app.logger`
- 是否追加全局 `g.app_logger`，应作为独立决策，不应成为当前计划的必经前置阶段。

---

### F-2（HIGH）新增 service 日志调用必须做 `if self.logger:` guard

**事实**
- `ExternalGroupService(conn)`、`BatchService(conn)`、`PartService(conn)` 在现有测试中大量以无 logger 方式直接实例化。
- `core/services/process/part_service.py::_coerce_external_default_days()` 已有成熟 guard 模式：
  ```python
  if self.logger:
      try:
          self.logger.warning(message)
      except Exception:
          pass
  ```

**判断**
- 本轮若在 service 层直接新增 `self.logger.warning(...)` 且不做 guard，会产生直接回归风险。

**建议**
- 这应写入计划的硬要求，而不是默认写法中的隐含前提。

---

### F-3（HIGH）`ExternalGroupService._apply_separate_mode()` compatible fallback 确实无观测

**事实**
- `_apply_separate_mode()` 中 `dv is None or dv <= 0` 且 `strict_mode=False` 时，直接 `dv = 1.0` 后落库。
- 当前路径无 `logger` / `warning` / `counter`。

**判断**
- 原计划对这个核心问题的描述成立。

**建议**
- 保持兼容行为不变，但补 `logger.warning`；
- route 层显式传 logger；
- 回归测试用 mock logger 注入 service 即可，不依赖 Flask `g`。

---

### F-4（MEDIUM）batch Excel import 的 strict_mode 缺口成立，但计划漏了一个状态一致性点

**事实**
- `BatchService.import_from_preview_rows()` 没有 `strict_mode` 参数；
- `batch_excel_import.import_batches_from_preview_rows()` 没有 `strict_mode` 参数；
- `_apply_row_no_tx()` 调 `svc.create_batch_from_template_no_tx(..., rebuild_ops=True)` 未传 `strict_mode`；
- `scheduler_excel_batches.py` 当前没有 strict_mode helper / UI / hidden field。

**额外事实（本轮新增）**
- `preview_baseline` 的 extra_state 当前只包含 `auto_generate_ops`、`parts_snapshot`、`template_ops_snapshot`；**不包含 strict_mode**。
- 现有基线守卫测试 `tests/regression_excel_preview_confirm_extra_state_guard.py` 已证明该机制用于防止 preview/confirm 状态漂移。

**判断**
- 如果只加 hidden field 不把 strict_mode 纳入 baseline extra_state，preview → confirm 的“严格/兼容语义”仍可能在确认时被篡改而不触发重新预览提示。

**建议**
- 计划应把“strict_mode 纳入 `_batch_baseline_extra_state()`”列为显式动作；
- 并增加一条回归：preview 后篡改 strict_mode，confirm 必须提示重新预览。

---

### F-5（MEDIUM）`scheduler_excel_batches.py` 的 logger 传递应从“顺手修”升级为必改

**事实**
- `scheduler_excel_batches.py` 中多处 `BatchService(g.db, op_logger=...)` 没有传 logger；
- `BatchService._default_template_resolver()` 会将 `self.logger` 继续传给 `PartService`；
- 因此，Excel 批次导入自动补建模板链上的 fallback warning 是否可见，取决于 route 是否给 `BatchService` 传 logger。

**判断**
- 这不是“仅为新日志做注入”；它直接影响本次 strict_mode/compatible 可观测性修复在 Excel 导入路径上的实际落地。

**建议**
- 将 `scheduler_excel_batches.py` 的 `BatchService(..., logger=current_app.logger, op_logger=...)` 升级为本轮必改项。

---

### F-6（MEDIUM）batch Excel strict_mode 的测试目标应是“硬失败 + 原子回滚”，不是“按行计错继续”

**事实**
- `execute_preview_rows_transactional()` 默认 `continue_on_app_error=False`；
- `tests/regression_excel_failure_semantics_contracts.py` 已明确：batch import 当前合同是**硬失败**，而非 row-error continue；
- `scheduler_excel_batches.py::excel_batches_confirm()` 也没有捕获 `svc.import_from_preview_rows()` 抛出的 `AppError`。

**判断**
- 原计划中的 “导入失败/计错” 表述过于含混，容易把测试写成错误方向。

**建议**
- strict_mode 的 batch Excel regression 应断言：
  1. 导入硬失败（service 抛错或 route 返回错误页）
  2. `Batches` / `BatchOperations` 无残留
  3. 不是 `error_count+=1` 后继续提交

---

### F-7（LOW）`_build_overdue_items()` 的 `except: pass` 存在，但真实运行风险低于原计划表述

**事实**
- `_build_overdue_items()` 确有：
  ```python
  try:
      summary.warnings.append(msg)
  except Exception:
      pass
  ```
- 运行时最小脚本表明：当 `summary.warnings=()` 时，消息不会进入 summary，但仍会写 logger；当前没有 append-failure 自身的日志。
- 但真正的排产 summary 类型来自 `core.algorithms.types.ScheduleSummary`，其中 `warnings: List[str]`；greedy scheduler 构造时传入的也是 list。

**判断**
- 这是防御分支中的静默点，不是当前主链上的高频真实故障。
- 因而不宜继续标成 MEDIUM 阻塞项。

**建议**
- 可作为 Phase 3 cleanup：
  - 优先做“安全追加 + logger fallback”最小修复；
  - 若本轮时间受限，可降级到 follow-up。

---

### F-8（LOW）`_extract_freeze_warnings()` broad except 是坏味道，但真实风险更低

**事实**
- `_extract_freeze_warnings()` 当前为整段 try/except。
- 但其唯一调用点先通过 `_warning_list()` / `_merge_warning_lists()` 归一化，传入值实际上已经是 `List[str]`。
- 最小脚本验证：即使传 `[Bad(), '【冻结窗口】A']`，函数仍返回 `['【冻结窗口】A']`，因为坏对象已在 `_warning_list()` 阶段被过滤。

**判断**
- broad except 的确应清理；但“单个坏 warning 会清空整体结果”的风险，在现有调用链上几乎是死分支。

**建议**
- 将这一项降级为低优先级整理项；
- 不建议为它单独编写“坏对象丢失 freeze warnings”式 regression，因为该场景与当前调用链不一致。

---

### F-9（LOW）`_config_snapshot_dict()` 的异常分支确实无留痕，但不应在本轮做大签名改造

**事实**
- `cfg.to_dict()` 异常后直接 `pass`；
- `hasattr/getattr` 异常后直接 `continue`；
- 最小脚本验证：构造 `to_dict()` 抛异常且某属性 getter 抛异常的 cfg，对外仅得到部分 dict，看不到“不完整”标记。
- 同时，`tests/regression_schedule_summary_v11_contract.py` 与 `tests/regression_dict_cfg_contract.py` 已覆盖 `config_snapshot` 主契约；缺的是**异常路径专项覆盖**，不是“完全无测试”。

**判断**
- 计划中“应改造为 tuple 返回”过于激进。

**建议**
- 本轮最多做两种之一：
  1. 只补 logger/traceability；
  2. 在 `algo` 下新增可选 sibling 字段（例如 `config_snapshot_incomplete`），但不改 `_config_snapshot_dict()` 公开返回形态。
- 若不做该项，应在计划中明确“延后到 Phase 3 follow-up”。

---

### 3. 对计划文档的修订建议
### 必须修订
1. 将 `getattr(g, "app_logger", None)` 方案改为 **局部使用 `current_app.logger`**。
2. 在所有新增 service 日志方案中明确写出 `if self.logger:` guard。
3. 在 `scheduler_excel_batches.py` 任务中补上：
   - `BatchService(..., logger=current_app.logger, ...)`
   - strict_mode 纳入 `_batch_baseline_extra_state()`
   - 自定义模板直接补 checkbox/hidden field（或说明可局部重构，但不再假定组件复用是现状）
4. 将 batch Excel strict_mode regression 的验收语义改为：**硬失败 + 回滚原子性**。

### 应降级或收敛
1. `_build_overdue_items()` 从阻塞项降为 cleanup / 非主链风险。
2. `_extract_freeze_warnings()` 从“本轮必修 + 必补专项测试”降为低优先级整理项。
3. `_config_snapshot_dict()` 只保留“轻量增强或延期”的决策点，不在本轮默认推入结构性签名改造。

---

### 4. 总结结论
> 原计划的主方向成立：高优先级仍是 `ExternalGroupService` compatible fallback 可观测性，以及 batch Excel strict_mode 全链贯通。  
> 但计划需要三项关键收敛：**logger 注入方案改为 `current_app.logger`、service 日志 guard 明文化、strict_mode 纳入 preview_baseline extra_state。**

同时，`schedule_summary.py` 三个残留点中，只有 `_config_snapshot_dict()` 仍值得保留为 follow-up 决策点；`_build_overdue_items()` 与 `_extract_freeze_warnings()` 在当前真实调用链上的风险都比原计划表述更低，不宜继续占用与 strict_mode 同等级的实施优先级。
