### 托管依赖安装口径缺少运行时依赖

- ID: sp02-workflow-runtime-deps
- 严重级别: 高
- 分类: 测试
- 跟踪状态: 开放
- 相关里程碑: round2
- 说明:

  plan 要求 `.github/workflows/quality.yml` 在 `windows-latest` + `Python 3.8` 上执行统一门禁，但安装步骤只写了 `requirements-dev.txt`。统一门禁会运行启动链专项回归与 `tests/check_quickref_vs_routes.py`，这些检查会导入或启动应用，实际依赖 `requirements.txt` 中的运行时包。当前 `requirements-dev.txt` 只包含 `pytest`、`pytest-cov`、`radon`、`pre-commit`、`ruff`，不包含 `Flask`、`Werkzeug`、`openpyxl`、`python-dateutil`。因此在干净托管机上，工作流会在进入真正门禁逻辑前就因缺依赖失败，导致 plan 的“统一托管入口”目标无法成立。
- 建议:

  把托管安装口径改成 `python -m pip install -r requirements.txt` 再安装 `requirements-dev.txt`，或让 `requirements-dev.txt` 显式包含运行时依赖。同步更新完成判定与最小验证命令。

### 台账条目缺少稳定身份与重扫合并策略

- ID: sp02-ledger-stable-identity
- 严重级别: 中
- 分类: 可维护性
- 跟踪状态: 开放
- 相关里程碑: round3
- 说明:

  plan 已经要求 `accepted_risks` 只通过 `entry_ids` 引用原始条目，也要求 `scripts/sync_debt_ledger.py` 成为唯一写入口；但示例条目与重复校验仍明显偏向 `path + symbol + line_start` 这类行号身份。若后续 SP03 / SP09 只是插入日志、拆分函数或移动 `except` 位置，条目 `id` 与 `entry_ids` 引用就会整体漂移，脚本若没有明确的“重定位后继承人工元数据”规则，就只能二选一：要么丢失 owner / batch / status / accepted_risks，要么手工重写 JSON，破坏唯一写入口约束。这个问题不会立刻让 SP02 失败，但会让后续批次维护成本急剧上升。
- 建议:

  在 plan 中明确稳定身份规则：`id` 不得直接绑定行号；行号只作证据字段。并补充 `sync_debt_ledger.py` 的合并策略，明确自动刷新字段与人工保留字段。

### 静默回退四分类缺少判定优先级

- ID: sp02-fallback-classification-precedence
- 严重级别: 中
- 分类: 其他
- 跟踪状态: 开放
- 相关里程碑: round3
- 说明:

  plan 定义了 `silent_swallow`、`silent_default_fallback`、`observable_degrade`、`cleanup_best_effort` 四类，但没有定义冲突时的判定优先级。实际启动链代码里已经存在容易冲突的样本：例如 `_close_db` 与 `delete_stale_runtime_files()` 都属于清理语境，但处理器体本身又是典型 `except Exception: pass`。如果没有优先级表，不同实现者可能把同一处理器记成 `cleanup_best_effort` 或 `silent_swallow`，从而让基线冻结和门禁断言在后续反复抖动。

### round3 · 第三轮：台账可维护性、分类严谨性与隐藏风险审查

  - 当前版 SP02 的方向没有问题，但要真正做到“后续批次长期可维护”，还差三处关键收口：
    1. 台账条目需要稳定身份与明确合并策略；
    2. 静默回退四分类需要判定优先级；
    3. 步骤 0 的冻结事实需要明确落盘位置。

  ### 影响
  - 如果条目身份绑定到行号，SP03 / SP09 只要做了无语义的插行或挪动，`entry_ids`、`accepted_risks` 引用、人工维护的 owner / batch / status 就会整体漂移，台账会变成“每次重扫都像重建一份新账”。

  - 如果步骤 0 只在会话里“交代”而不落盘，SP03 很难复核 SP02 到底迁移了什么、文档差异当时是否真的只剩两条。

  ### 修正建议
  - 在 plan 中补一条：`id` 不得直接由行号组成；推荐使用稳定键（如 `path + symbol + except_ordinal` 或稳定片段哈希），行号只作为证据字段。
  - 明确 `scripts/sync_debt_ledger.py` 的合并模型：重扫时哪些字段自动刷新，哪些人工字段必须保留，如何在条目重定位后继承既有元数据。
  - 给四类静默回退补一张判定优先级表，至少明确 cleanup 场景与可观测降级场景的优先级。
  - 为步骤 0 指定固定落盘位置，建议写入 `开发文档/阶段留痕与验收记录.md` 或治理台账正文固定区块。
- 结论:

  第三轮结论：plan 已具备实施价值，但若不补齐“稳定身份 + 合并策略 + 分类优先级 + 冻结落盘位置”，后续维护成本会明显上升。
