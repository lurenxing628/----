### 资源时间线线性扫描导致 O(n²) 复杂度

- ID: PERF-01
- 严重级别: 高
- 分类: 性能
- 跟踪状态: 开放
- 相关里程碑: M1
- 说明:

  find_overlap_shift_end 对 segments 列表做全量线性扫描。在资源避让循环中（最多 200 次迭代），每次迭代调用 3 次（设备/人员/停机），而 occupy_resource 只做 append 不做合并——随着排产工序增多，每台设备的时间线列表持续增长。对于 500 道工序 × 5 台设备的场景，单台设备可能积累 100+ 区间段，导致每次 find_overlap_shift_end 扫描 100+ 条记录。

  整体复杂度退化为 O(ops × segments_per_resource × iterations) ≈ O(n²)。

  建议方案：
  1. occupy_resource 改为按 start 时间有序插入（用 bisect.insort）
  2. find_overlap_shift_end 用二分查找定位可能重叠的区间段
  3. 或在 timeline 上只维护 '最后结束时间'（当前已有 last_end_by_machine 但未用于避让快速路径）——大多数场景下新工序不会和历史工序重叠，只需与最近的区间比较
- 建议:

  将 timeline 改为有序列表 + 二分查找，或增加 '最后结束时间' 快速路径剪枝
- 证据:
  - `core/algorithms/greedy/downtime.py:60-77#find_overlap_shift_end`
  - `core/algorithms/greedy/downtime.py:20-28#occupy_resource`
  - `core/algorithms/greedy/scheduler.py:510-533`
  - `core/algorithms/greedy/downtime.py`
  - `core/algorithms/greedy/scheduler.py`

### str(getattr(...)) 冗长模式重复 40+ 次

- ID: ARCH-02
- 严重级别: 中
- 分类: 可维护性
- 跟踪状态: 开放
- 相关里程碑: M2
- 说明:

  整个算法层大量使用以下模式：

  ```python
  str(getattr(op, 'batch_id', '') or '').strip()
  str(getattr(op, 'machine_id', None) or '').strip()
  ```

  粗略统计出现 40+ 次。这个模式冗长且易误写。应该提取为一个工具函数：

  ```python
  def _attr_str(obj, name: str, default: str = '') -> str:
      return str(getattr(obj, name, default) or default).strip()
  ```

  这样上述 40+ 处都可以简化为：
  ```python
  bid = _attr_str(op, 'batch_id')
  machine_id = _attr_str(op, 'machine_id')
  ```

  大幅提升可读性。
- 建议:

  在 types.py 或新建 helpers.py 中定义 _attr_str() 工具函数，批量替换 40+ 处重复模式
