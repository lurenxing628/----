### task 4 遗漏 optimize_schedule 的 strict created_at 边界

- ID: task4-strict-created-at-gap-in-optimizer
- 严重级别: 高
- 分类: 其他
- 跟踪状态: 开放
- 相关里程碑: m2
- 说明:

  `optimize_schedule()` 在进入 multi-start 前就自行构造 `BatchForSort`，其本地 `_parse_datetime()` 会把坏 `created_at` 静默转成 `None`；而 `FIFOStrategy` 直接依赖 `created_at` 排序。plan 的 task 4 虽要求 `GreedyScheduler.schedule()` 对 `ready_date` / `created_at` 采用字段级 strict 包装，但没有同步要求 `optimize_schedule()` 对 `created_at` 做 strict 包装，只要求它复用共享 `parse_datetime()`。这会留下‘优化器入口静默吞掉坏 created_at，调度器入口再抛错’的裂缝，既违背“算法层日期来源只有一套”的目标，也让 strict 错误暴露时机不一致。
- 建议:

  把 task 4 明确补成：`optimize_schedule()` 对 `created_at` 也必须采用字段级 strict/non-strict 包装，错误字段名为 `created_at`；不要只在 `GreedyScheduler.schedule()` 收口。这样才能保证优化器与调度器两个算法入口的日期边界一致。

### m2 · 第二轮：统一估算器、SGS 评分与日期边界深挖

- 状态: 已完成
- 记录时间: 2026-04-07T12:32:05.696Z
- 已审模块: core/algorithms/greedy/scheduler.py, core/algorithms/greedy/auto_assign.py, core/algorithms/greedy/dispatch/sgs.py, core/algorithms/greedy/external_groups.py, core/services/scheduler/calendar_engine.py, core/services/scheduler/schedule_optimizer.py, core/algorithms/sort_strategies.py
- 摘要:

  第二轮重点核对 task 1/2/4 与现状代码的一致性。确认 plan 对内部真实执行族的识别是准确的：`GreedyScheduler._schedule_internal()` 与 `auto_assign_internal_resources()` 都是“先调到工作时刻，再同时检查设备/人员/停机三条时间线并循环推进”的同族实现；`dispatch_sgs()` 的内部评分则是另一套顺序链式近似估算，因此把它升级到与执行路径同源，方向正确。`CalendarEngine.get_efficiency()` 的 `or 1.0` 与 `_schedule_internal()` / `dispatch_sgs()` 的算法层二次 `or 1.0`、`except Exception: eff=1.0` 也确实构成当前静默回退链，plan 要求在算法层先显式收口，方向同样正确。

  但这里发现三处关键设计缺口。

  其一，task 2 对“允许降级评分键的业务不可估算分支”定义过窄。当前 `dispatch_sgs()` 对外协评分的 `ext_group_total_days` / `ext_days` 非法值，明确按业务坏数据处理成惩罚 key；真正执行时 `schedule_external()` 也只是返回 `None` 并阻断单个批次，而不是把它当程序异常终止整轮排产。如果 plan 在 task 2 只保留“缺资源、工时坏值、越窗”三类，实施者很容易把外协周期坏值误升级成未捕获异常，导致非严格模式下从“单批次失败”漂移为“整轮排产崩溃”。

  其二，task 4 对 strict `created_at` 的边界收口不完整。`optimize_schedule()` 自己就构造了一份 `BatchForSort`，并用本地 `_parse_datetime()` 静默把坏 `created_at` 转成 `None`；`FIFOStrategy` 又直接消费 `created_at`。plan 的 task 4 只明确要求 `GreedyScheduler.schedule()` 对坏 `created_at` 做字段级 strict 包装，却没有同样约束 `optimize_schedule()`。这样会留下“优化器入口静默吞掉坏 `created_at`，调度器入口再抛错”的裂缝，既不满足“算法层日期来源只有一套”的目标，也会让 strict 错误发生在较晚阶段。

  其三，task 1 的新增测试矩阵还不足以锁死统一估算器契约。现有 `auto_assign_internal_resources()` 的早停依赖两处 `earliest > best_end`，当前内部评分/执行也分别承担换型惩罚、越窗阻断等语义；但 plan 里的失败用例主要覆盖“被调用”“超过 200 片段”“效率 0/异常”“零时长”，没有直接验证 `abort_after` 的初始早停与循环内早停、`changeover_penalty` 在传/不传 `last_op_type_by_machine` 时的差异、以及 `blocked_by_window` 的边界行为。仅靠 spy 参数断言，无法防止 `estimate_internal_slot()` 形参存在但语义未生效。
- 结论:

  plan 的主方向正确，但 task 2 的不可估算分类、task 4 的 strict created_at 边界，以及 task 1 的估算器契约测试仍有缺口；若不补齐，这份 plan 不能稳妥宣称“无静默回退且边界收口完整”。

## 最终结论

这份 plan 的主骨架是成立的：主调用链判断准确，任务串行顺序合理，统一内部估算器、SGS 评分同源化、时间线有序化、日期来源收口、以及优化器轻量去重这几条主线总体都朝着“更简洁、更少重复、更少静默回退”的方向推进。

但按当前文本，它还不能直接被当作“可无风险落地”的实施依据，原因有四：
1. task 2 把可降级的业务不可估算分支写得过窄，遗漏了外协周期坏值，存在把业务坏数据误升级成整轮异常的风险。
2. task 4 没有同步收口 `optimize_schedule()` 对 `created_at` 的 strict 边界，导致优化器入口与调度器入口的日期语义仍可能分叉。
3. task 1 的估算器测试还没有锁死 `abort_after`、换型惩罚与越窗边界等核心契约，容易出现“形参接上了但语义没真正生效”的假通过。
4. task 5 的完成判定与最终验证命令没有形成闭环：既没有在最终阶段重跑大资源池 benchmark，也没有用足够宽的回归集合支撑“全链回归”说法。

因此，本 review 结论是：该 plan 可以保留整体结构并继续推进，但必须先补齐以上缺口，再进入正式实施；否则它还不足以支撑自己声明的‘无静默回退、边界完整收口、性能门槛已验证’。
