## 评审发现

### 直接装配门禁遗漏 import as 别名

- ID: gate-import-alias-blind-spot
- 严重级别: 中
- 分类: 测试
- 跟踪状态: 开放
- 相关里程碑: round2-reference-chain
- 说明:

  `scan_request_service_direct_assembly_entries()` 只会解析简单赋值别名与原始调用名，未处理 `from x import BatchService as BS` 这类导入别名。函数在 526-535 行只根据 `_call_target_name(value)` 的尾名是否以 `Service/Repository` 结尾来识别目标，而 549-553 行也只在已解析别名表中命中时替换调用名。实测用 `BS(g.db, logger=None)` 探针调用时返回空结果，说明 `tests/test_architecture_fitness.py:442-452` 新增的阻断门禁可以被导入别名绕过，当前“结果为 0”并不等价于真正无直装残留。
- 建议:

  把 `ImportFrom` / `Import` 的别名映射纳入扫描输入，至少覆盖 `from ... import X as Y` 与 `import ... as alias` 两类模式，并补一条回归用例验证别名导入不会漏检。
- 证据:
  - `tools/quality_gate_scan.py:526-535#_resolve_request_service_alias`
  - `tools/quality_gate_scan.py:549-553#scan_request_service_direct_assembly_entries`

### 预置 g.db 时工厂不会同步校验 g.services

- ID: factory-preseed-db-mixed-state
- 严重级别: 低
- 分类: 其他
- 跟踪状态: 开放
- 相关里程碑: round2-reference-chain
- 说明:

  `factory._open_db()` 只在 `"db" not in g` 时创建 `g.op_logger` 与 `g.services`。与此同时，`web/ui_mode.py:220-228` 又明确规定“只要已有 `g.db`，缺少 `g.services` 就直接抛错”。这意味着一旦未来有测试桩、扩展或自定义挂钩提前写入 `g.db`，工厂会放行该混合状态，真正的失败会推迟到路由或 `get_ui_mode()` 才暴露，定位点晚于装配入口。当前主路径与现有自定义测试工厂不会触发此问题，但它是不变量层面的缺口。
- 建议:

  在 `_open_db()` 中增加显式不变量检查：若 `g.db` 已存在但 `g.services` 缺失，则立刻抛出带上下文的实现错误；不要静默补建，以保持 fail-fast 语义。
- 证据:
  - `web/bootstrap/factory.py:295-319#create_app_core._open_db`
