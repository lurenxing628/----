# 测试先行（TDD）

## 核心观念

先写测试。看它失败。再写最小实现让它通过。

**铁律：没有先失败的测试，就不准写生产代码。**

```text
没有失败用例在前，就没有生产代码在后。
```

## 什么时候必须用

默认全部使用：

- 新功能
- BUG 修复
- 行为修改
- 重构

例外情况必须先问用户：

- 一次性原型
- 纯试验代码
- 纯配置改动
- 自动生成代码

## 如果你已经先写了代码

那就不是测试先行。

正确做法：

1. 删除那段实现
2. 回到写失败用例
3. 重新开始

规则：

- 不保留成“参考实现”
- 不一边看旧代码一边补测试
- 不允许“我先不删，等测试写完再整理”

**先写代码再补测试，不算测试先行。**

## 红 → 绿 → 重构

### 第一步：红 —— 写失败用例

一次只测一个行为。

```python
def test_rejects_empty_email():
    result = submit_form({"email": ""})
    assert result["error"] == "邮箱不能为空"
```

要求：

- 名字清楚
- 只测一个行为
- 尽量测真实行为，不要一上来就堆 mock

### 第二步：确认它真的失败

必须运行测试，亲眼看到它失败。

```bash
python -m pytest tests/path/test_xxx.py -q
```

确认三件事：

- 它确实失败了
- 失败原因是你预期的那个原因
- 不是因为拼写错误、导入错误、测试本身写坏

如果测试一开始就通过，说明你没有测到新行为；回去改测试。

### 第三步：绿 —— 写最小实现

现在才允许写代码，而且只能写**刚好够测试通过**的最小实现。

```python
def submit_form(data):
    if not data.get("email", "").strip():
        return {"error": "邮箱不能为空"}
    return {"error": None}
```

不允许：

- 顺手加额外功能
- 顺手抽象一堆配置
- 提前做以后才需要的扩展

### 第四步：确认它真的通过

再跑一次同一个测试。

```bash
python -m pytest tests/path/test_xxx.py -q
```

确认：

- 新测试通过
- 没有报错和警告
- 受影响的其他测试也通过

### 第五步：重构

只有在测试已经是绿灯时，才允许重构：

- 去重
- 改名
- 提炼辅助函数
- 让结构更清楚

重构时不能顺手改变行为。

### 第六步：重复下一轮

下一个行为，再来一轮：

- 写失败用例
- 看它失败
- 写最小实现
- 看它通过
- 需要时重构

## 好测试长什么样

| 维度 | 好 | 坏 |
|---|---|---|
| 单一性 | 一次只测一个行为 | 一个测试里塞 3 个行为 |
| 名称 | 看名字就知道系统应该怎么表现 | `test1`、`test_ok` |
| 真实度 | 优先测真实行为 | 只验证 mock 调了几次 |
| 可读性 | 一眼看懂输入、动作、断言 | 铺满无关准备代码 |

## 为什么顺序不能反过来

“先写代码，后补测试”有这些问题：

- 你会不自觉按自己已经写出的实现来设计测试
- 你不会真正验证“需求是什么”，只会验证“我写的东西是我写的东西”
- 测试第一次就通过，无法证明它真的能抓住错误
- 漏边界场景的概率很高

测试先行的价值，不是“最后有测试”，而是：

> **你亲眼看见测试先失败，再证明它被代码修好。**

## 常见借口对照表

| 借口 | 事实 |
|---|---|
| “这个太简单，不值得测” | 越简单越容易想当然。 |
| “我先实现完再补” | 那不是测试先行。 |
| “我手工验证过了” | 手工验证不稳定、不可重复。 |
| “已经写了很多了，删掉太可惜” | 留着不可信代码，后面成本更高。 |
| “我只是先探索一下” | 探索可以，但探索代码最后要丢掉。正式实现重新按测试先行开始。 |
| “测试很难写” | 很可能说明接口设计本身不清楚。 |

## 红旗信号

出现这些想法时，必须停下：

- “先把核心逻辑写了，后面再补测试”
- “我不删旧实现，拿来参考一下”
- “测试先通过也没事，说明我实现得对”
- “这次例外一下”
- “先把功能做完，再统一补测试”

**这些都说明你已经偏离测试先行。**

## BUG 修复时怎么用

修 BUG 也一样：

1. 先写一个失败用例，复现 BUG
2. 确认它失败
3. 写最小修复
4. 确认用例通过
5. 补相关回归

**没有失败用例的 BUG 修复，不算完成。**

## 回归检查清单

宣布完成前，至少核对：

- [ ] 每个新增行为都有测试
- [ ] 每个测试都先失败过
- [ ] 失败原因正确，不是测试写坏
- [ ] 实现是最小改动，不是顺手扩展
- [ ] 定向测试通过
- [ ] 相关回归通过
- [ ] 测试输出干净，无新增报错或警告

## 何时要看反模式文档

当你发现自己要大量用 mock、要写奇怪测试辅助、或很难判断测试到底在测什么时，读：

- `testing-anti-patterns.md`

## 最终要求

**生产代码的前提，是它前面已经有一个失败过的测试。**

否则，不算测试先行。
