# 测试反模式

下面这些做法会让测试“看起来存在”，但实际上不可靠。

## 1. 只测 mock，不测真实行为

坏例子：

- 只断言 mock 被调用了几次
- 不验证用户真正关心的输出

更好的做法：

- 优先断言真实返回值、真实状态变化、真实副作用

## 2. 给生产代码加只给测试用的入口

坏例子：

- 为了让测试好写，额外开放生产环境不会用到的方法

更好的做法：

- 优先通过正常公开接口测试
- 如果很难测，先想想是不是设计太耦合

## 3. 测试比实现还耦合内部细节

坏例子：

- 断言内部私有步骤的调用顺序
- 代码一重构，行为没变，测试先全炸

更好的做法：

- 断言外部可观察行为

## 4. 测试准备代码过大，掩盖意图

坏例子：

- 测一个小行为，要铺几十行前置数据

更好的做法：

- 抽小型构造辅助
- 让每个测试只保留和当前行为直接相关的数据

## 5. 没亲眼看见红灯

坏例子：

- 测试一写好就通过
- 你也不回头确认它是否真的抓得住错误

更好的做法：

- 必须先看见它失败，再写实现

## 最终要求

**测试要证明行为，不是证明你会写测试。**
