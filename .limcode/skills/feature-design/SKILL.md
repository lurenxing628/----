# 三轮对抗式功能设计

> **定位**：用于"用户想加新功能，但还没想清楚怎么做"的场景。
> 核心原则：**先设计、再压测、最后换人审，确认没问题了才动手写代码**。
> 方法论：**三轮对抗**——Round 1 需求分析 + 设计探索、Round 2 设计压力测试、Round 3 独立评审。

## 与其他 Skill 的区别

| Skill | 触发场景 | 核心动作 |
|-------|---------|---------|
| `deep-investigate` | 怀疑有 bug 但不确定 | 三轮对抗：假设验证 → 收敛质疑 → 换人审 |
| `aps-deep-review` | 改动前/合并前 | 引用链追踪 + 边界检查 + 回归风险 |
| `aps-post-change-check` | 改完代码后 | 跑 lint + 架构合规 + 联动提醒 |
| **`feature-design`（本 Skill）** | **想加新功能** | **三轮对抗：需求拆解 → 设计压测 → 独立评审** |

## 触发词

用户提到以下关键词时触发：

> 复杂功能设计 / 跨层方案设计 / 需要独立审核员的设计 / 需要三轮对抗式设计 / 数据结构或接口联动较重的新功能 / 设计评审

## 子代理兼容门禁

涉及独立审核员、继承主模型、兼容回退时，统一遵循：

- `.limcode/skills/_shared/subagent-compat.md`

本技能自己的特例只有一条：Round 3 的独立审核员优先使用 LimCode 原生已注册的独立审核子代理；如果当前宿主尚未注册原生子代理，则以 `.limcode/agents/independent-auditor.md` 作为角色正文源文件。

## 双轨验证原则（全流程生效）

**脚本和推理，缺一不可。** 此原则贯穿三轮始终。

| 只靠推理 | 只靠脚本 | 双轨 |
|---------|---------|------|
| 容易漏掉运行时才暴露的问题 | 脚本只测它测到的，测不到的更危险 | 互相补位 |

**执行规则**：

1. 能推理的先推理，得出预期结论。
2. 推理不确定的（如"这个 SQL 查询性能够不够"、"这个字段类型兼不兼容"），写最小脚本验证。
3. 脚本通过后，**不要停下**，继续追问：
   - "脚本覆盖了现有数据的兼容场景吗？"
   - "脚本覆盖了边界值（空值、零值、超长输入）吗？"
   - "有没有脚本测不到但逻辑上可能出问题的场景？"
4. 对脚本测不到的边界，用逻辑推理补充风险说明。

## 工作流（给 Agent 的执行指引）

### Round 0: 接收需求，建立 Todo

1. 把用户描述的功能需求记录下来。
2. 区分用户**显式说的**需求和 Agent **推测的**需求（避免过度设计）。
3. 创建 Todo，包含以下条目：
   - [ ] Round 1: 需求分析 + 设计探索
   - [ ] Round 2: 设计压力测试 + 方案比较
   - [ ] Round 3: 独立设计评审——启动 independent-auditor
   - [ ] 汇总: 三轮裁决——合并分歧 + 最终 design 结论
   - [ ] 子代理门禁：确认当前工具接口是否支持项目自定义审核员；若不支持，改走共享兼容回退路径

---

### Round 1: 需求分析 + 设计探索（主 Agent）

**目标**：搞清楚功能本质、影响范围、可能的实现方案。

**Step 1.1: 需求拆解**

- 用一句大白话说清楚"这个功能本质是干什么的"。
- 拆解为可执行的子任务。
- 明确标注：
  - **用户说的**：用户明确要求的部分
  - **我推测的**：Agent 认为需要但用户没提的部分（标注为待确认）

**Step 1.2: 影响域评估**

对接 `change-scope-protocol.mdc`，对每个子任务回答：

| 检查项 | 内容 |
|--------|------|
| 涉及哪一层？ | Route / Service / Repository / Schema / Template / 前端 JS |
| 需要改哪些文件？ | 列出具体路径 |
| Schema 变更？ | 如果是，自动标记为"高成本"（需 migration + 文档） |
| 枚举变更？ | 如果是，自动标记为"需全局搜索" |
| 新建文件？ | 如果是，检查命名是否符合现有模式（`xxx_service.py`、`xxx_repo.py`、`xxx_pages.py`） |
| 新增依赖？ | 如果是，标记并说明理由 |

**Step 1.3: 读现有代码**

对涉及的每个模块，读代码了解当前实现：
- 相关的 Service / Repository 有哪些现有方法可复用
- 相关的 Model 有哪些字段
- 相关的 Route / Template 的现有结构
- 对每个关键文件只读一次，读后直接分析

**Step 1.4: 设计方案（至少 2 个，大白话四问）**

每个方案必须回答以下四个问题：

```markdown
### 方案 X：{{一句话名字}}

**要做什么？**
> 用一句话说清功能本质。
> 例："给排产结果加一个'导出 Excel'按钮，点了之后把甘特图数据导出成表格。"

**怎么做？**
> 用一两句话说清方案思路，不涉及代码细节。
> 例："在排产页面加个按钮，点击后调新增的 /scheduler/export 接口，
>      后端用 openpyxl 生成 Excel 返回给前端下载。"

**成本是什么？**
> 改几个文件、需不需要改数据库结构、需不需要更新文档。
> 例："新增 1 个 Route 函数、1 个 Service 方法、改 1 个模板。
>      不改数据库。需要更新面板与接口清单。"

**会引起什么后果？**
> 对现有功能的影响、风险、最坏情况。
> 例："好处：用户可以离线查看排产结果。
>      风险：如果排产数据量很大，生成 Excel 可能慢。
>      最坏情况：超大数据量下可能超时——可以加个进度提示。"
```

**Step 1.5: 自评**

```markdown
### 自评：Round 1 可能遗漏了什么
- {{诚实列出自己不确定的地方}}
- {{标注哪些设计决策还需要 Round 2 压测}}
```

**输出后等用户说"继续"才进入 Round 2。**

---

### Round 2: 设计压力测试（主 Agent）

**目标**：对 Round 1 的设计方案做全方位压测，找出设计盲区。

**Step 2.1: 逐条质疑设计决策**

对 Round 1 的每个设计决策逐条追问：
- "这个 Service 方法签名设计合理吗？调用方好不好用？"
- "这个 Schema 设计会不会以后要改？有没有预留余地？"
- "这个 Route 的参数校验够不够？恶意输入怎么办？"
- "这个方案有没有比它更简单的替代方案？"

**Step 2.2: 边界情况清单**

逐一检查：

| 边界情况 | 检查点 |
|---------|--------|
| 空值 / 零值 | 所有输入参数逐一假设为 None / 0 / ""，代码能否处理 |
| 超长输入 | 字符串字段有没有长度限制 |
| 并发场景 | 快速双击提交会不会重复创建 |
| 数据库为空 | 新功能在空数据库下能否正常运行 |
| 老数据兼容 | 已有数据能不能正常跑新功能（尤其 Schema 变更时） |
| 权限 / 状态前置 | 新功能对数据状态有没有前置要求（如"只有已排产的才能导出"） |

**Step 2.3: 架构兼容性检查**

对照 `architecture-invariants.mdc`：
- 是否符合四层架构（Route → Service → Repository → DB）
- Route 有没有直接操作数据库
- Service 有没有导入 `flask.request`
- 新文件的命名是否符合现有模式
- 是否与现有 Service / Repository 的职责边界冲突
- 是否引入循环依赖

**Step 2.4: 联动更新预判**

对照 `change-scope-protocol.mdc`，预判实现后需要联动更新的内容：

```markdown
### 联动更新预判

| 变更类型 | 触发联动 | 需要更新 |
|---------|---------|---------|
| 新增 Route | 是 | 面板与接口清单 |
| Schema 变更 | 是 | migration 脚本 + 开发文档数据模型 + 速查表字段字典 |
| 新增枚举 | 是 | enums.py + 速查表枚举清单 + 全局搜索引用点 |
| Service 签名变更 | 是 | 搜索所有调用方确认兼容 |
| Excel 模板变更 | 是 | 速查表 Excel 模板清单 |
| flash 文案 | 是 | 面板与接口清单用户可见文案 |
| 无以上变更 | 否 | — |
```

**Step 2.5: 双轨验证**

对推理不确定的点（如 SQL 性能、字段类型兼容性），写最小脚本验证。

**Step 2.6: 修正方案 + 推荐选择**

- 根据压测结果修正设计方案
- 标注哪些设计决策被修正、为什么
- 输出修正后的方案对比表：

```markdown
| 对比项 | 方案 A | 方案 B |
|--------|--------|--------|
| 一句话描述 | {{大白话}} | {{大白话}} |
| 要改几个文件 | N | N |
| 需不需要改数据库 | 是/否 | 是/否 |
| 能不能满足需求 | 能/部分 | 能/部分 |
| 会不会影响现有功能 | 不会/极小概率 | 不会/极小概率 |
| 改完出了问题怎么回退 | {{大白话}} | {{大白话}} |
| 后续维护麻烦吗 | {{大白话}} | {{大白话}} |
```

```markdown
**我的建议**：选方案 X。
> 用一句大白话说理由。
```

**输出后等用户说"继续"才进入 Round 3。**

---

### Round 3: 独立设计评审（子代理 — independent-auditor）

**目标**：由独立的审核员（Subagent），在全新的上下文中，对设计方案做对抗性评审。

#### 为什么需要换人审

Round 1-2 都是同一个 Agent 在同一个上下文窗口中完成的。即使 Round 2 做了"压力测试"，同一个 Agent 天然倾向于维护自己设计的方案（设计者偏差）。Round 3 启动一个独立 Subagent，它：
- 看不到主 Agent 在 Round 1-2 的设计推理过程
- 只看到打包给它的"设计方案"（当作待审声明）
- 独立读代码验证方案可行性，天然无偏差

#### 主 Agent 的工作

1. **打包待审材料**：将 Round 1-2 的设计方案整理为结构化文本，每条设计决策标注为"未经独立验证的设计声明"。
2. **调用 independent-auditor**：

```
优先路径：LimCode 原生已注册的独立审核子代理；未注册时回退到 `.limcode/agents/independent-auditor.md` 角色正文源文件
兼容回退：统一遵循 `.limcode/skills/_shared/subagent-compat.md`

传入内容：
  1. 用户的原始需求描述
  2. Round 1 的设计方案（逐条，标注为"待审设计决策"）
  3. Round 2 的压测结果和修正后的推荐方案
  4. 联动更新预判清单
```

#### 主 Agent 收到评审报告后

**Step 3.1: 汇总三轮结果**

```markdown
## 三轮汇总

### 一致的结论（三轮都认同）
- {{列出}}

### 分歧点（Round 3 审核员提出异议）
| # | Round 1-2 设计决策 | Round 3 判定 | 分歧原因 |
|---|-------------------|-------------|---------|

### Round 3 新发现（前两轮遗漏）
- {{列出}}
```

**Step 3.2: 裁决分歧**

对每个分歧点，主 Agent 必须：
- 重新读代码验证（不能只凭记忆）
- 给出裁决：采纳 Round 1-2 / 采纳 Round 3 / 需要更多证据
- 附上裁决证据

**Step 3.3: 输出最终 design 结论（大白话格式）**

```markdown
## 1. 这个功能要做什么？
- 你要的是：{{用一句大白话复述用户需求}}
- 拆解为 N 个子任务：{{列出}}
- 哪些是你说的，哪些是我建议加的：{{区分标注}}

## 2. 推荐怎么做？
- 推荐方案：{{名字}}
- 一句话描述：{{大白话}}
- 独立审核员是否认同：{{是 / 否，理由}}

## 3. 要改哪些东西？
| 文件 | 改什么（一句话） | 涉及哪层 | 是新建还是修改 |
|------|---------------|---------|-------------|

## 4. 成本
- 改动文件数：N
- Schema 变更：是/否（如是，需 migration + 文档更新）
- 文档联动：{{列出需要更新的文档}}
- 预估工作量：S/M/L

## 5. 风险和后果
- 对现有功能的影响：{{大白话}}
- 最坏情况：{{大白话}}
- 回退方案：{{大白话}}

## 6. 实现顺序建议
1. {{先做什么}}
2. {{再做什么}}
3. {{然后做什么}}
4. {{最后做什么}}

## 7. 验证计划
- {{大白话说怎么验证功能正确}}

## 8. 遗留风险
- {{大白话说已知局限}}
```

**Step 3.4: 落 design 文档并移交**

- 将最终 design 写入 `.limcode/design/YYYY-MM-DD_<主题>_design.md`
- 请用户先审 design 文档
- 用户确认后，下一步只能进入 `writing-plans`

**改动前门禁**：design 文档写完并经用户确认后，才能进入 `writing-plans`；不能在本技能里直接跳到实现。

## 逃逸条件（简单功能可简化流程）

如果功能满足以下**所有条件**，可以跳过 Round 3（不启动独立审核员），直接在 Round 2 结束后输出最终 design 结论并写入 design 文档：

- 只涉及 1-2 个文件
- 不涉及 Schema 变更
- 不涉及枚举变更
- 不新增依赖
- 不改现有 Service/Repository 的公开方法签名

跳过 Round 3 时必须明确告知用户："这个功能比较简单，我跳过了独立评审环节。如果你希望更谨慎，可以说'继续 Round 3'。"

---

## 执行纪律

1. **不要一上来就写代码**。三轮全部完成（或走了逃逸条件）、最终 design 输出并经用户确认后，才允许进入 `writing-plans`。
2. **Round 之间必须等用户说"继续"**才进入下一轮。不得自动跳轮。
3. **变更边界严格**。实现时只改计划中列出的文件，不顺手修无关问题。发现的无关问题以文字列出，不修改。
4. **读写纪律**。对单个文件只读一次、只写一次。禁止对同一文件反复 grep、反复读取、反复修改。
5. **证据优先**。每个设计决策必须附带证据（现有代码路径、架构规则引用）。不确定的地方写验证脚本。
6. **Round 3 优先使用 LimCode 原生已注册的独立审核子代理**；若当前宿主尚未注册，则以 `.limcode/agents/independent-auditor.md` 作为角色正文源文件，其余兼容要求统一遵循 `.limcode/skills/_shared/subagent-compat.md`。
7. **分歧不得静默忽略**。如果 Round 3 和 Round 1-2 有分歧，主 Agent 必须明确说明分歧点和裁决理由。
8. **联动更新不能漏**。进入实施后，必须按 Round 2 的联动更新预判清单逐项完成。
9. **全程中文简体回复**。

## 约束

- 如果用户明确说"不用设计了直接做"，尊重用户意愿，但提醒风险。
- 如果功能过于复杂（涉及 10+ 文件或 3+ 层），建议用户拆分为多个子功能分别设计。
- Subagent 不支持嵌套（Round 3 的审核员不能再启动 subagent）。
