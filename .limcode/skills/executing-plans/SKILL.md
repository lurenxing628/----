# 按 plan 执行

## 目标

读取 plan、先审后做、逐项执行、逐项验证，最后进入收尾。

调用本技能后，先告诉用户：

> 我正在使用 `executing-plans` 来执行这份 plan。

## 重要说明

如果当前环境支持子代理，并且任务之间大体独立，**优先使用**：

- `subagent-driven-development`

因为它能做到：

- 每个任务独立上下文
- 任务间自动 review
- 实施与审核分离

只有在下面情况才优先使用本技能：

- 当前环境不适合子代理
- 用户要求就在当前会话直接做
- 任务之间高度耦合，不适合拆给独立子代理

## 执行流程

### 第 1 步：读取并审 plan

先做这几件事：

1. 完整读取 plan 文件
2. 批判性审阅 plan
3. 找出任何会阻碍实施的问题
4. 没问题再建待办并开始执行

审 plan 时重点看：

- 文件路径是否准确
- 任务顺序是否合理
- 有没有信息缺口
- 有没有明显占位语
- 有没有和 design 或用户要求冲突的地方
- 有没有超出当前会话约束的步骤

如果发现关键问题，必须先向用户提出，不能硬做。

## 第 2 步：确认是否需要隔离工作区

如果还不在隔离工作区里，并且用户没有明确允许直接在当前分支上做，实现前优先考虑：

- `using-git-worktrees`

规则：

- 没明确许可，不要默认在 `main` / `master` 上直接实现
- 如果用户明确说就在当前分支改，可以继续

## 第 3 步：逐项执行任务

对 plan 里的每个任务，按下面顺序：

1. 标记任务为进行中
2. 严格按 plan 里的步骤执行
3. 运行 plan 指定验证
4. 记录结果
5. 标记任务完成

### 执行中的硬规则

- 不跳过验证
- 不把多个步骤揉成一步
- 不静默扩大范围
- 不因为“看起来简单”就省掉测试
- plan 写着要测试先行，就调用 `test-driven-development`
- plan 写着要 review，就调用 `requesting-code-review`

## 第 4 步：遇到阻塞时立刻停下

出现下面任一情况，立即暂停并向用户说明：

- 缺依赖
- 测试持续失败
- plan 指令不清楚
- 必要上下文缺失
- 需要扩范围才能继续
- 验证结果和预期冲突

**不要用猜测顶过去。**

## 第 5 步：全部完成后的收口

当所有任务都完成并通过验证后，按顺序收口：

1. 必要时再补一轮 `requesting-code-review`
2. 如果改过代码，建议跑 `aps-post-change-check`
3. 调用 `finishing-a-development-branch`

调用收尾技能前先告诉用户：

> plan 已执行完，我接下来会使用 `finishing-a-development-branch` 来完成这次工作。

## 什么时候回到前面重审

如果出现这些情况，要回到“读 plan”阶段：

- 用户修改了 plan
- design 变了
- 发现 plan 的主路径不成立
- 实施中暴露出 plan 没覆盖的关键前提

## 核心原则

- 先审 plan，再执行
- plan 怎么写，就怎么做
- 不跳验证
- 被阻塞就停
- 不硬推
- 收尾前要先确认实现真的成立

## 与其他技能的关系

- `using-git-worktrees`：实现前隔离工作区
- `writing-plans`：本技能执行它产出的 plan
- `test-driven-development`：实现每一步时遵守测试先行
- `requesting-code-review`：任务间 / 阶段后做 review
- `finishing-a-development-branch`：执行完成后的收尾

## 红旗信号

下面这些念头出现时，说明你已经偏了：

- “这一步先不验了”
- “先把后面几步一起做掉”
- “plan 写得不清楚，我自己理解一下大概做”
- “先修出结果再补解释”
- “这点范围扩展不用告诉用户”

**一旦出现这些想法，立即停下，回到 plan 或向用户确认。**
