# 参考：事无巨细的核对清单与回填规则（APS）

> 目标：当代码已经发生大量修改时，能够**按“修改文件 → 影响面 → 事实来源 → 回填章节 → 验证方式”**把变更完整回写到 `开发文档/`，并确保每一章都符合现状。

## 0. 范围与红线

### 0.1 只允许修改的目标文档

- `开发文档/开发文档.md`
- `开发文档/系统速查表.md`
- `开发文档/实现计划表.md`
- `开发文档/阶段留痕与验收记录.md`
- `开发文档/面板与接口清单.md`

### 0.2 忽略清单（不读不改）

- `开发文档/V1.2/**`
- `开发文档/拆分高耦合文件_2026-01-29.md`
- `开发文档/拆分高耦合文件_P1P2_实施策划.md`
- `开发文档/拆分高耦合文件.md`
- `开发文档/自检报告_2026-01-26.md`

### 0.3 回填默认策略（避免跑偏）

- **事实优先级**：`schema.sql` / `web/routes/*` / `templates/*` / `core/*`  > 文档
- **默认不改代码**：发现文档与实现冲突，先以实现为准回填文档；若怀疑实现是 bug，记录“未决问题/风险”，不要擅自改代码。
- **逐文件覆盖**：凡是 `git status` 里出现的变更文件（含新增文件）都必须：
  - 被阅读/理解（至少知道变更意图）
  - 在最终输出中出现一条“影响与回填”记录（可为“无文档影响”）

## 1. 事实来源（从哪里拿“现状”）

### 1.1 路由/接口/页面事实

- **蓝图注册与前缀**：`app.py`
- **路由实现**：`web/routes/*.py`
- **模板（经典）**：`templates/**/*.html`
- **模板（现代 v2）**：`web_new_test/templates/**/*.html`
- **前端 JS/CSS**：`static/js/*.js`、`static/css/*.css`
- **UI 模式与安全链接**：`web/ui_mode.py` 及相关路由（常见：`/system/ui-mode`）

抽取清单（用于回填“面板与接口清单”“速查表”）：
- URL（最终访问路径）
- endpoint 名称与所属 blueprint
- HTTP method（GET/POST）
- query 参数名（含默认行为，例如未提供/空值的语义）
- form 字段名（**字段名不可随意改，文档必须与实现一致**）
- 用户可见文案：按钮文字、确认弹窗文字、空数据提示、flash 成功/失败文案
- 模板映射：v1/v2 文件路径、渲染键（若存在）

### 1.2 数据库事实

- **表结构/字段/索引/外键/唯一性**：`schema.sql`（优先级最高）
- **模型定义**：`core/models/*`
- **仓储层（SQL 与字段映射）**：`data/repositories/*`

抽取清单（用于回填“开发文档/数据模型”“速查表/字段字典”）：
- 新增/删除/变更的表
- 字段名、类型声明、默认值、NOT NULL、UNIQUE、FK、索引
- 枚举值口径（例如 `day_type` 统一为 `workday/holiday`）
- 迁移/兼容策略（如旧值 `weekend` 视为 `holiday`，是否落库归一）

### 1.3 排产/日历/配置事实

- **服务层**：`core/services/scheduler/*`
- **算法层**：`core/algorithms/*`
- **与日历相关**：`core/services/scheduler/calendar_service.py`（以及拆分出的 admin/engine 类，如存在）
- **配置键与默认值**：`core/services/scheduler/config_service.py` + `schema.sql` 的 `ScheduleConfig`

抽取清单（用于回填“开发文档/排产算法”“速查表/策略配置与约束”）：
- ScheduleConfig 的 key 列表、默认值、是否暴露到 UI
- 排产入口参数：`start_dt`、`end_date`、`enforce_ready` 等
- 关键行为：齐套约束语义、排产窗口语义、冻结窗口、SGS 派工、自动资源分配等
- 日历覆盖语义：全局 WorkCalendar 与 OperatorCalendar 的优先级与回退规则
- 落库与留痕：写入 `Schedule` / `ScheduleHistory` / `OperationLogs` 的字段与时机

### 1.4 Excel 模板与导入导出事实

- **模板目录（交付物）**：`templates_excel/`
- **Excel 路由**：`web/routes/*excel*.py`
- **通用导入组件**：`templates/components/excel_import.html`（以及 `static/js/excel_handler.js`）
- **后端实现**：`core/services/common/*`（`excel_service.py`、`openpyxl_backend.py`、`excel_audit.py`、`excel_templates.py`）

抽取清单（用于回填“速查表/模板清单”“开发文档/Excel 导入导出”）：
- 每个模块的 Excel 页面 URL 与模板文件名
- 导入模式（append/overwrite/replace）的语义与限制
- “预览有 ERROR 行则整批拒绝导入”的规则是否覆盖所有导入点
- 预览行号规则、中文错误提示规则
- 留痕 detail 的固定键名（必须英文键名、结构稳定）

## 2. 修改文件 → 文档回填：映射规则（核心）

> 先从 `git status` / `git diff` 列出变更文件，然后按路径归类。每个文件至少落一条映射。

### 2.1 路由类变更（`web/routes/**`）

必须核对并可能回填到：
- `开发文档/面板与接口清单.md`：页面/接口、参数、flash 文案、模板路径、UI 模式差异
- `开发文档/系统速查表.md`：HTTP 接口清单（至少新增/删除/变更的接口要同步）
- 视情况：`开发文档/开发文档.md`（若新增了系统级能力/约束，如齐套约束开关、UI 模式等）

验证方式：
- 用搜索定位 `@bp.route(...)`、函数名、methods
- 对照模板的 `form action` 与字段名
- 对照 `flash()` 文案（成功/失败/警告）

### 2.2 模板类变更（`templates/**`、`web_new_test/templates/**`、`static/**`）

必须核对并可能回填到：
- `开发文档/面板与接口清单.md`：用户可见文案（按钮/提示/确认/空数据）、页面结构、入口导航
- `开发文档/系统速查表.md`：若模板变更影响“操作方式/关键规则”的简述（例如新增筛选参数、默认行为）

验证方式：
- 从模板中抽取文案与关键表单字段
- 若存在 UI 模式：确认 v1/v2 模板映射规则与 `safe_url_for` 的降级语义

### 2.3 数据库变更（`schema.sql`、`core/models/**`、`data/repositories/**`）

必须核对并可能回填到：
- `开发文档/开发文档.md`：数据模型章节（表/字段/枚举/约束/预留字段）
- `开发文档/系统速查表.md`：字段字典与枚举值、关键表用途
- `开发文档/实现计划表.md`：若新增/调整里程碑或新增模块步骤（例如新增 OperatorCalendar）

验证方式：
- `schema.sql` 为准核对：字段名、索引、FK/UNIQUE
- repo 的 SQL 与返回字段是否一致（避免文档写了字段但 repo 根本不读/不写）

### 2.4 排产/算法/日历变更（`core/services/scheduler/**`、`core/algorithms/**`）

必须核对并可能回填到：
- `开发文档/开发文档.md`：排产算法、日历约束、增强开关、策略键（10.x / 6.6 / 11.x / 12.x 等相关章节）
- `开发文档/系统速查表.md`：ScheduleConfig 键、排产入口参数、甘特/周计划数据口径
- `开发文档/面板与接口清单.md`：排产页面参数、默认值、提示文案、失败行为

验证方式：
- 追踪“入口参数 → service → algorithm → persistence/logging”的全链路
- 对照测试与 evidence 中的报告（如存在）确认行为与文案

### 2.5 测试与证据变更（`tests/**`、`evidence/**`）

必须核对并可能回填到：
- `开发文档/阶段留痕与验收记录.md`：新增回归脚本、证据路径、待提交项清单
- `开发文档/开发文档.md`：若测试暴露“口径统一/兼容迁移”规则（例如 day_type 归一）

验证方式：
- 从测试名/断言提炼“规则口径”（例如“个人日历覆盖全局假期仍可排产”）
- evidence 报告中的关键结论是否需要同步到文档

## 3. 按文档的逐章节核对清单

> 建议顺序：先“最容易误导用户操作”的接口/文案，再到数据模型与算法描述。

### 3.1 `开发文档/面板与接口清单.md`（最高优先级）

逐章核对点（每个模块都要过一遍）：
- **Blueprint 与前缀**：是否与 `app.py` 注册一致
- **全局导航**：`templates/base.html` 的入口与文案是否一致
- **每个页面**：
  - 页面 URL、endpoint、路由文件
  - 模板路径（v1/v2 双栈时要写清楚：渲染键与实际文件）
  - GET query 参数：参数名、默认值、空值语义（例如 `status=` 表示全部）
  - POST form 字段：字段名、是否多值、placeholder/允许格式
  - confirm 弹窗文案（如有）
  - flash 文案（成功/失败/警告/未选中/参数缺失等）
- **唯一 JSON API**（若仍保持设计）：例如甘特数据 `GET /scheduler/gantt/data` 的返回结构与前端错误文案
- **UI 模式**：是否仍存在“经典/现代”切换；`safe_url_for` 降级说明是否匹配现状

常见漏项提醒：
- 路由被拆分（文件变化），但 URL/endpoint 不变：文档要解释“入口文件 + 子模块”结构，避免读者找错文件。
- 模板文案变了但文档没跟：尤其是“错误提示/确认弹窗/空数据提示”。
- 新增了 query 参数（如日期范围、版本、筛选）但文档遗漏。

### 3.2 `开发文档/系统速查表.md`（高频查阅）

逐章核对点：
- **全局硬约束**：Win7/离线/openpyxl-only/并发锁不实现/备份策略/留痕要求
- **目录与关键文件**：关键入口/服务/算法/路由拆分是否更新（新增文件必须补齐）
- **术语表/枚举值**：与 `core/models/enums.py`（如有）与 DB 字段口径一致
- **HTTP 接口清单**：
  - 至少同步新增/删除/改名接口
  - 关键参数（尤其排产、日历、Excel 导入确认）要写明默认值与限制
- **Excel 模板清单**：
  - `templates_excel/` 是否存在对应文件
  - 模板字段是否与导入/导出实现一致
- **留痕字段规范**：`OperationLogs.detail` 的固定键是否与实现一致
- **ScheduleConfig 键**：新增键、默认值、是否暴露 UI、是否参与算法
- **DB 字段字典**：字段名/索引/外键/预留表与现状一致（以 `schema.sql` 为准）

常见漏项提醒：
- “口径统一/兼容迁移”规则（如 `day_type`）在速查表里也要写清楚。
- 新增 Excel 模板但没同步清单（或模板文件名不一致）。

### 3.3 `开发文档/开发文档.md`（综合规范与原始方案留档）

逐章核对点（按“改动最常触发的章节”优先）：
- **运行环境/依赖锁定/不做清单**：Win7/离线/openpyxl-only/并发删除/打包版本
- **数据模型**：表结构、字段、枚举、预留字段、唯一性策略
- **排产算法与增强开关**：
  - 多策略排序/派工/自动分配/改进模式/OR-Tools/冻结窗口等开关是否已落地
  - 齐套约束、排产窗口、日历覆盖等关键行为是否与实现一致
- **Excel 导入导出**：预览/差异/导入严格策略/留痕键名
- **日志/留痕/事务/备份**：实现点是否仍在同一文件/同一语义
- **UI 模式与 safe_url_for**：若仍存在双栈模板，必须与现状一致

常见漏项提醒：
- 文档内“原始代码保留”片段：若实现已移动/拆分，需要同步文件路径与注释（避免读者按旧路径找不到）。
- 文档内跨文件链接（`(开发文档.md#...)`）如标题变化要更新锚点。

### 3.4 `开发文档/实现计划表.md`（里程碑与步骤对齐）

逐章核对点：
- 若系统已落地新增能力（例如：人员专属日历、齐套约束开关、UI 模式、批量停机计划等），确保计划表中：
  - 步骤/里程碑描述与现状一致
  - 引用的开发文档章节仍存在且锚点正确
  - “阻断验收”描述与当前验收脚本/证据路径一致

### 3.5 `开发文档/阶段留痕与验收记录.md`（证据与回归脚本索引）

逐章核对点：
- evidence 路径是否存在且命名一致
- 新增回归测试脚本是否加入建议执行清单
- “待提交/待测试”标记是否需要更新（例如版本号从 `（待提交）` 变为真实 commit）
- 重要规则变更是否补充了可复现的验证方式（SQL/页面路径/脚本命令）

## 4. 最终输出质量门槛（过线才算完成）

- **覆盖率**：所有变更文件都在“逐文件检查结果”里出现；没有“只写总结不逐文件”的情况。
- **双写一致**：接口/参数/默认值的变更，至少同步到：
  - `面板与接口清单.md`（面向 UI/验收）
  - `系统速查表.md`（面向开发/运维查阅）
- **DB 一致**：凡是涉及表/字段/枚举的回填，必须以 `schema.sql` 为准自证。
- **文案一致**：涉及用户提示的变更，必须能在模板或路由里找到相同/等价中文文案。
- **不触碰忽略文档**：忽略清单里的文件不读不改。
