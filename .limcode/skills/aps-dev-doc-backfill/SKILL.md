# APS 开发文档回填

## 适用场景（触发词）

- 回填开发文档 / 更新开发文档 / 对齐开发文档 / 文档同步
- 接口清单更新 / 面板清单更新
- 速查表更新 / 文档与现状不一致

## 硬性约束（必须遵守）

- **只回填文档**：默认只编辑 `开发文档/` 里的目标文档；代码/SQL/模板/测试只作为事实来源读取。若发现实现疑似 bug，仅记录为“待确认”，不要擅自改代码。
- **忽略清单（不读不改）**：
  - `开发文档/V1.2/**`
  - `开发文档/拆分高耦合文件_2026-01-29.md`
  - `开发文档/拆分高耦合文件_P1P2_实施策划.md`
  - `开发文档/拆分高耦合文件.md`
  - `开发文档/自检报告_2026-01-26.md`
- **事无巨细**：对每一个“被修改的文件”必须检查，并输出“它改变了什么 → 影响哪些文档章节 → 如何验证”。
- **以现状为准**：当文档与实现冲突时，默认以“当前仓库实现”为事实来源（`schema.sql`、`web/routes/*`、`templates/*`、`core/*`）。
- **中文一致性**：用户可见文案（flash/页面提示/错误信息）在文档中必须保持中文，且表达与页面一致。
- **路径风格**：文档与回填说明中的路径统一用正斜杠（例如 `web/routes/scheduler_batches.py`）。

## 目标文档（允许修改）

- `开发文档/开发文档.md`
- `开发文档/系统速查表.md`
- `开发文档/实现计划表.md`
- `开发文档/阶段留痕与验收记录.md`
- `开发文档/面板与接口清单.md`

## 执行流程（推荐固定步骤）

### Step 0：建立“变更文件清单”（强制）

1. 读取 `git status`，获取所有 modified/untracked 文件。
2. 读取 `git diff`（必要时 `--staged`），获取变更内容与范围。
3. 生成一份表格：`文件 → 变更摘要（1~2 行）→ 变更类型（DB/路由/模板/算法/配置/测试/其他）`。

> 要求：**每个变更文件都必须在后续“回填映射表”中出现**（哪怕结论是“对文档无影响”）。

### Step 1：并行 subagent 扫描事实（固定 4 路）

优先策略：

- 当前环境支持子代理 → 按 4 路并行执行
- 当前环境只支持少量子代理 → 先并行最重的 2 路，其余顺序补扫
- 当前环境不支持子代理 → 主代理按同样的 4 类事实**顺序执行**，不得因为没有子代理就跳过任何一类

无论走哪条路径，输出都必须包含“文件路径 + 关键事实 + 证据定位（函数/路由/表/字段/模板）”。

1. **变更面扫描**：从 `git diff` 归类模块、行为变化、用户可感知变化（尤其是：参数、默认值、错误提示、策略开关）。
2. **接口与页面事实**：扫描 `app.py`、`web/routes/*.py`、`templates/`、`web_new_test/templates/`，抽取：URL、endpoint、方法、query/form 字段、flash 文案、模板映射（含 UI 模式与 `safe_url_for`）。
3. **数据库事实**：扫描 `schema.sql`、`core/models/*`、`data/repositories/*`，抽取：表/字段/索引/枚举值口径/预留字段与说明。
4. **排产与日历事实**：扫描 `core/services/scheduler/*`、`core/algorithms/*`，抽取：ScheduleConfig 键、默认值、排产窗口、齐套约束、日历覆盖语义、落库/留痕点。

### Step 2：生成“回填映射表”（强制）

把 Step0 的每个变更文件映射到目标文档与章节（至少一条映射或注明“无影响”）。

输出格式（示例）：

- `web/routes/scheduler_run.py`
  - **影响**：`开发文档/面板与接口清单.md`（排产调度 → 执行排产/模拟排产参数与文案）
  - **影响**：`开发文档/系统速查表.md`（HTTP 接口清单 / 排产参数默认值）
  - **验证**：对照路由函数、模板按钮文案、`flash()` 文案是否一致

### Step 3：逐文档逐章节核对与回填（强制）

对每个目标文档：先抽取所有标题（`# / ## / ###`），逐节核对“是否仍符合现状”。

回填优先级（从最容易造成误导的开始）：

1. **接口/页面/参数/文案**（`面板与接口清单.md`）
2. **速查表：关键约束、关键文件、枚举、Excel 模板、ScheduleConfig 键、DB 字段字典**（`系统速查表.md`）
3. **综合开发文档：架构/数据模型/算法/日历/留痕/事务/并发删除/备份/交付**（`开发文档.md`）
4. **实现计划表：步骤与现实能力对齐**（`实现计划表.md`）
5. **阶段留痕与验收记录：证据路径/回归脚本/待提交项**（`阶段留痕与验收记录.md`）

### Step 4：回填后自检（强制）

- **一致性**：文档中的 URL/参数/默认值/枚举/表字段必须能在代码与 `schema.sql` 中找到对应事实来源。
- **章节完整性**：任何章节若已过期必须更新；若已不再适用必须明确标注“已废弃/仅留档”。
- **可追溯**：关键规则（如齐套约束、日历覆盖、路由拆分）必须能追溯到具体文件。

### Step 5：最终输出（固定模板）

按以下模板输出（回填完成后在对话里给用户）：

```markdown
## 回填摘要

### 修改了哪些文档
- 开发文档/面板与接口清单.md：更新了 [...]（原因：...；证据：web/routes/...）
- 开发文档/系统速查表.md：更新了 [...]（原因：...；证据：schema.sql / core/...）
- 开发文档/开发文档.md：更新了 [...]（原因：...；证据：...）
- 开发文档/实现计划表.md：更新了 [...]（原因：...）
- 开发文档/阶段留痕与验收记录.md：更新了 [...]（原因：...）

### 逐文件检查结果（按 git 变更文件）
- <fileA>：做了什么 → 影响章节 → 已回填/无影响 → 验证方式
- <fileB>：...

### 未决问题 / 风险（需要你确认）
- ...
```
