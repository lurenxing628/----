# 写可执行的 plan

## 目标

把已经批准的 design，写成一个**上下文极少的人也能照着执行**的 plan。

假设执行者：

- 会写代码，但不了解这个仓库
- 不熟悉业务背景
- 测试意识一般
- 容易过度实现
- 容易在没验证时就宣布完成

所以你的 plan 必须：

- 把每一步写清楚
- 把文件路径写准确
- 把验证方式写准确
- 尽量把“猜”降到最低

## 开始时先声明

调用本技能后，先告诉用户：

> 我正在使用 `writing-plans` 来写这次实现的 plan。

## plan 存放位置

- 默认目录：`.limcode/plans/`
- 默认命名：`YYYY-MM-DD_<主题>.plan.md`

如果用户已有更具体命名要求，按用户要求。

## 使用前提

只有在满足下面条件时，才进入本技能：

- design 已经存在
- 用户已经认可 design
- 当前还没有开始正式实现

如果 design 仍然含糊，先退回 `brainstorming`。

## 范围检查

如果一份 design 涵盖多个互相独立的子系统，就不该只写一个大 plan。

应该：

- 先建议拆成多个 plan
- 每个 plan 单独可验证、可交付、可 review

## 先定文件结构，再拆任务

在写任务前，先把要改的文件和职责映射出来。

至少回答：

- 会新建哪些文件？
- 会改哪些现有文件？
- 每个文件负责什么？
- 哪些文件必须一起改？
- 有没有文件已经过大，应该顺势拆分？

要求：

- 单个文件职责要清楚
- 变化频繁、一起修改的内容尽量靠近
- 跟随现有代码风格，不无故重塑仓库结构

## 任务颗粒度

**每一步只做一个动作，目标是 2~5 分钟完成。**

好的颗粒度示例：

- 写一个失败用例
- 运行该用例，确认它因为缺功能而失败
- 写最小实现让该用例通过
- 重新运行该用例
- 运行受影响的定向回归
- 更新文档或提示文案
- 如用户要求提交，再准备提交

差的颗粒度示例：

- “实现整个模块”
- “把前后端都改好”
- “写完后记得测一下”

## plan 头部模板

每份 plan 必须以这个结构开头：

```markdown
# <功能名称> 实施 plan

> **执行方式**：优先使用 `subagent-driven-development`；如果当前环境不适合子代理或用户要求当前会话直接执行，则使用 `executing-plans`。

**目标**：一句话说明最终要完成什么。

**总体做法**：2~3 句说明方案主线。

**涉及技术 / 模块**：列出关键目录、模块、库。

---
```

## 任务模板

每个任务至少包含下面这些信息：

````markdown
### 任务 N：<一句话名字>

**目标**
- 这一任务完成后，系统多了什么能力，或消除了什么风险。

**文件**
- 新建：`exact/path/to/new_file.py`
- 修改：`exact/path/to/existing_file.py`
- 测试：`tests/exact/path/to/test_file.py`
- 联动：`开发文档/...`（如需要）

- [ ] **步骤 1：写失败用例**

```python
def test_xxx():
    ...
```

- [ ] **步骤 2：运行用例，确认失败原因正确**

运行：`python -m pytest tests/xxx.py -q`
预期：因为缺少功能或当前行为错误而失败，而不是因为拼写错误或测试本身写坏。

- [ ] **步骤 3：写最小实现**

说明改哪个文件、改什么点。
如果这一步逻辑不显然，就给出关键代码片段。

- [ ] **步骤 4：重新运行定向验证**

运行：`python -m pytest tests/xxx.py -q`
预期：通过。

- [ ] **步骤 5：运行受影响回归**

运行：`python -m pytest tests/yyy.py tests/zzz.py -q`
预期：通过。

- [ ] **步骤 6：联动更新 / 收尾**

如果改了接口、文案、模板、枚举、导入导出，写清需要联动更新什么。
````

## 绝不允许出现的占位语

下面这些都属于 **plan 失败**，不能写：

- `TODO`
- `TBD`
- “后面再补”
- “视情况处理”
- “自行实现异常处理”
- “写一些测试”
- “类似任务 3”
- “按项目规范处理”但不写具体处理点

如果一步涉及代码而你完全不写关键修改点，执行者就会猜；让执行者猜，就是 plan 不合格。

## 必须写清的内容

每个任务都要尽量写清：

- 准确文件路径
- 改动边界
- 关键数据流或函数签名变化
- 要补哪些测试
- 运行哪些命令验证
- 预期结果是什么
- 是否需要联动文档

## 与测试先行的关系

写 plan 时，默认执行者会使用 `test-driven-development`。

因此计划中的实现步骤，应尽量写成：

1. 先写失败用例
2. 确认失败
3. 写最小实现
4. 确认通过
5. 运行相关回归

## 与变更范围协议的关系

写 plan 时，必须对照 `.limcode/rules/change-scope-protocol.mdc`：

- 有没有跨层改动
- 有没有 schema 变化
- 有没有接口签名变化
- 有没有模板 / 文案 / Excel 模板联动

如果会触发这些联动，就写进 plan，不要等实现时再想起来。

## 自审清单

plan 写完后，至少自己过这三遍：

### 1. 覆盖检查

对着 design 看：

- design 里的每条要求，有没有对应任务？
- 有没有 design 说了但 plan 漏掉的内容？

### 2. 占位语检查

全文搜索：

- `TODO`
- `TBD`
- “适当处理”
- “自行判断”
- “类似前面”

发现就改掉。

### 3. 一致性检查

看后面任务里出现的：

- 函数名
- 类型名
- 配置键
- 枚举值
- 文件路径

是否和前文一致。名字前后不一致，就是 plan 有 BUG。

## 写完后的移交流程

plan 写完并保存后，给用户两个执行选项：

```markdown
plan 已完成并保存。接下来有两个执行方式：

1. `subagent-driven-development`（推荐）—— 每个任务由独立子代理执行，并在任务间做两阶段 review
2. `executing-plans` —— 在当前会话按 plan 逐项执行

你希望用哪种方式？
```

规则：

- 如果当前环境支持子代理并且任务可拆分，优先推荐 `subagent-driven-development`
- 如果当前环境不适合子代理，或用户明确要求当前会话直接做，使用 `executing-plans`

## 核心原则

- 先 design，后 plan
- 每步只做一件事
- 尽量把猜测空间清零
- 文件路径必须准确
- 验证命令必须明确
- 占位语零容忍
- plan 必须能被执行，而不只是“看起来像 plan”

## 何时升级到项目专用技能

如果这个 plan 涉及明显的项目专项，例如：

- 复杂深度排查后的修复
- 大范围文档回填
- 大型跨层审查

可以在 plan 中显式写出后续要串联的项目技能，例如：

- `deep-investigate`
- `aps-dev-doc-backfill`
- `aps-deep-review`
- `aps-post-change-check`

但它们是补充，不替代本技能的主职责。
