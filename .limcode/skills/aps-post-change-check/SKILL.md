# APS 改动后一键检查

## Quick start

- 运行：`python .limcode/skills/aps-post-change-check/scripts/post_change_check.py`
- 或直接告诉 Agent："检查一下"

## 适用场景（触发词）

- 用户提到：**检查一下 / 查一下 / 改完了 / 验一下 / 对话后检查 / 质量检查 / 收尾检查 / 快检**
- 每次完成一个功能改动后建议运行；如果用户明确要“代码审查”，优先进入 `requesting-code-review` 或 `aps-deep-review`

## 工作流（给 Agent 的执行指引）

### 1) 变更范围审查

- 运行 `git status --short` 和 `git diff --name-only`，列出所有已改动文件。
- 与用户确认：是否有"不该改的文件被改了"。

### 2) Ruff Lint 检查

- 对所有已改动的 `.py` 文件运行 `ruff check`。
- 如果有新增 lint 问题，列出并建议修复。
- 不要求修复历史遗留问题，只关注本次新增。
- 若本机未安装 `ruff`，脚本会显示 **SKIP** 并在总结里给出“检查被跳过”的警告（不因缺工具直接阻断）。

### 3) 架构分层合规快检

- 运行脚本：`python .limcode/skills/aps-post-change-check/scripts/post_change_check.py`
- 脚本会自动检查已改动文件是否存在分层违反。
- 若文件读取失败，脚本会显示 **FAIL**。读取都没完成时，不能把快检结果当成“可提交”证据。

### 4) 代码质量检查

脚本自动对改动文件执行：
- **裸字符串枚举检查**：是否使用 `== "pending"`（以及 `in/not in (...)` 集合校验）而非 `BatchStatus.PENDING` 等枚举（集合校验默认按 INFO 提示）
- **future annotations 检查**：核心目录 Python 文件是否有 `from __future__ import annotations`

### 5) 圈复杂度检查（改动文件）

脚本使用 Radon 对改动文件执行圈复杂度分析：
- 阈值：单函数复杂度 **>15** 即报警（项目门禁；比 Radon 的 C 上限 20 更严格）
- 只检查本次改动的文件，不检查全项目
- 如有高复杂度函数，建议拆分（提取子函数、策略模式、查表替代 if-else）
- 若本机未安装 `radon`，脚本会显示 **FAIL**。复杂度检查没执行时，不能把快检结果当成“可提交”证据。

### 6) 联动更新提醒

根据改动文件类型，按 `.limcode/rules/change-scope-protocol.mdc` 的"联动更新检查表"提醒用户是否需要同步更新（Route→Template/面板清单、Schema→开发文档/速查表、枚举→全局搜索等）。脚本会自动检测改动类型并生成对应提醒。

### 7) 输出结论

```markdown
## 改动后检查结论

### 变更范围
- 改动文件数：N
- 文件列表：（列出）
- 范围评估：正常 / ⚠ 有未预期文件

### Lint 检查
- 结果：PASS / FAIL
- 新增问题：N 个（列出）

### 架构合规
- 结果：PASS / FAIL
- 违反项：（列出）

### 代码质量
- 裸字符串枚举：N 项
- 缺少 future annotations：N 项

### 圈复杂度（改动文件）
- 结果：PASS / FAIL
- 超标函数：N 个（列出函数名和复杂度等级）

### 联动更新提醒
- [ ] 需要更新 XXX（原因：改了 YYY）
- [x] 无需额外更新

### 建议下一步
- 提交 / 需先修复 / 需运行文档回填
```

## 深度审查升级提示

当检测到以下情况时，应主动建议用户运行 `深度 review`（`aps-deep-review` skill）：

- 变更涉及 **Service 公开方法的签名变更**（参数增删、返回类型变化）
- 变更涉及 **schema.sql 或 migration**
- 变更涉及 **3 个以上文件的跨模块改动**
- 变更涉及 **core/models/enums.py**（枚举变更影响全链路）

提示话术：
> 本次改动涉及 Service 公开方法签名变更，建议运行"深度 review"追踪引用链，确认所有调用者已适配。

### 8) 审阅报告归档（audit/，LLM）

当本次检查出现 **FAIL**、或出现 **PASS（有警告）**、或用户要求“留痕/审计/后续改造决策”时，Agent 需要基于本次检查输出与 evidence 生成一份“取舍型审阅报告”，并归档到仓库根目录一级目录 `audit/`，用于后续审计与排期。

- **目录**：`audit/YYYY-MM/`
- **文件命名**：`YYYYMMDD_HHMM_post_change_check_review.md`

**强制规则（防止幻觉/跑偏）**：

- 只基于“脚本输出 + git diff/文件清单 + 现有 evidence 文件（若有）”做判断；每条结论必须能对应到证据（路径/片段）。
- 报告中必须分离：**事实**（脚本输出/证据）/ **判断**（LLM 归类与优先级）/ **建议**（是否改、怎么改、成本）。
- 只输出审阅与建议，不自动修改代码（除非用户后续明确要求）。

**审阅报告模板（建议）**：

```markdown
# 改动后检查审阅报告（LLM）

- 时间：YYYY-MM-DD HH:MM
- 触发原因：FAIL / PASS（有警告） / 用户要求审计
- 证据：
  - （粘贴关键脚本输出片段）
  - `git status --short`（摘要）
  - `git diff --name-only`（文件列表）

## 事实摘要（来自证据）
- ...

## 结论与取舍（LLM）

### 必须改（P0/P1）
- [ ] <问题>（原因/风险）| 建议：<最短修复路径> | 成本：S/M/L

### 建议改（P2）
- [ ] ...

### 不建议改 / 暂缓
- [x] <问题>（原因：误报/ROI 低/需要更大重构才有收益）

## 成本与风险评估
- 预估总成本：S/M/L
- 回归风险：低/中/高
- 推荐策略：先修 P0 → 再清 P1 → 最后处理 P2（或纳入重构批次）
```

## 约束

- 只做检查，不做任何修改。
- 如果发现问题，列出来让用户决定，不要自动修复。
