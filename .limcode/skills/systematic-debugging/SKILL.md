# 系统化排查

## 核心观念

乱试补丁只会浪费时间，还会制造新问题。

**铁律：没有根因调查，就不能提修复。**

```text
先查根因，再提修复。
```

如果你还没完成“根因调查”，那你就还没到可以提修复方案的时候。

## 何时使用

下面这些情况都要用：

- 测试失败
- 线上 BUG
- 页面行为异常
- 数据不一致
- 构建失败
- 集成失败
- 性能明显异常
- “我怀疑这里有问题，但还没证据”

尤其在这些时候更不能跳过：

- 时间紧
- 看起来像个小问题
- 已经试过两个补丁但还没好
- 你自己也没完全看懂问题

## 与 `deep-investigate` 的关系

- `systematic-debugging`：通用根因排查技能，适合绝大多数技术问题
- `deep-investigate`：更重的三轮对抗式排查，适合复杂、跨模块、需要独立审核的场景

如果用户明确要求：

- “先别改代码，先查清楚”
- “帮我独立验证一下结论”
- “来一轮更深的排查”

优先升级到 `deep-investigate`。

## 四阶段流程

必须一阶段一阶段完成，不能跳。

---

## 第一阶段：根因调查

### 1. 仔细读错误信息

不要直接略过：

- 报错文本
- 栈信息
- 文件路径
- 行号
- 错误码
- 警告信息

很多问题的线索就藏在报错原文里。

### 2. 稳定复现

先搞清楚：

- 能不能稳定复现？
- 精确步骤是什么？
- 每次都会发生吗？
- 只有某些输入、某些环境、某些数据才会发生吗？

如果连复现条件都没搞清楚，就不能靠猜。

### 3. 查近期变化

检查：

- 最近改过哪些相关文件
- 最近提交了什么
- 新增了哪些依赖、配置、默认值
- 开发环境和运行环境是不是不一致

### 4. 跨组件系统必须加证据采集

如果问题跨越多层或多组件，例如：

- Route → Service → Repository → DB
- 页面 → 接口 → 调度服务
- 导入预览 → 确认写入 → 汇总展示

在提修复前，必须先加最小证据采集：

- 每层输入是什么
- 每层输出是什么
- 关键配置值是否一致
- 状态在哪一层开始偏了

**先用证据定位断点，再谈修复。**

### 5. 倒着追值

当报错出现在很深的位置时，不要盯着出错点补丁。

要问：

- 这个坏值从哪里来的？
- 上一层是谁传进来的？
- 再往上一层是谁决定的？
- 最初在哪里偏掉了？

辅助文档：

- `root-cause-tracing.md`

## 第二阶段：模式对比

在修之前，先看“正确的例子”。

### 1. 找同仓库里正常工作的相似代码

问题不只是“哪里坏了”，还要看：

- 同类功能里，哪里是好的？
- 它和坏的这份实现相比，差在哪？

### 2. 完整读参考实现

如果你在套某个模式：

- 不要只看两行
- 不要凭印象改
- 要把参考实现完整读完

### 3. 列出全部差异

把工作正常版本和异常版本的差异列出来：

- 参数
- 默认值
- 类型
- 调用顺序
- 状态前置
- 数据结构
- 环境依赖

不要自己裁定“这点差异应该无关”。

### 4. 看依赖前提

很多问题不是代码本身，而是前提不满足：

- 依赖数据为空
- 配置缺失
- 枚举值不对
- 生命周期顺序不对
- 外部状态不一致

---

## 第三阶段：假设与验证

像做实验一样排查。

### 1. 一次只立一个假设

明确写出来：

> 我认为根因是 X，因为证据 Y 和 Z 指向它。

不要同时提三个猜测。

### 2. 最小验证

只做足够验证这个假设的最小改动或最小脚本。

- 一次只测一个变量
- 不把多个修复堆在一起
- 不做“顺便重构”

### 3. 验证后必须回头判断

- 假设成立 → 进入第四阶段
- 假设不成立 → 回到第一阶段，用新证据重建假设

### 4. 不懂就承认不懂

如果你发现：

- 没看懂这个模块
- 不确定这个字段语义
- 不知道这个默认值是谁决定的

要直接说清楚，继续补证据，不要装懂。

---

## 第四阶段：实施修复

### 1. 先写失败用例

在修之前，先留下能证明 BUG 存在的失败用例。

- 能写自动化测试就写自动化测试
- 自动化不方便时，写最小验证脚本
- 没有失败用例，不算进入修复阶段

写测试时要调用：

- `test-driven-development`

### 2. 只修根因，不修表象

- 一次只做一个修复
- 只修已确认的根因
- 不做“既然来都来了”的顺手优化

### 3. 修完就验证

至少验证：

- 失败用例现在通过
- 受影响的相关回归没有坏
- 用户看到的问题真的消失了

### 4. 修了还不对，就停下

如果修复不生效：

- 不要立刻叠第二个补丁
- 回到第一阶段，结合新证据重查

### 5. 连续三次修复失败，要质疑架构

如果已经出现：

- 修了 A，又暴露 B
- 修了 B，又暴露 C
- 每次都得大范围补洞
- 修这里伤那里

这往往不是“再补一个点”能解决的，而是结构本身有问题。

这时要停下来，和用户讨论：

- 是不是设计模式本身就错了？
- 是不是状态边界不清？
- 是不是应该做结构收口，而不是继续打补丁？

## 红旗信号

如果你脑子里冒出下面这些话，说明你已经开始偏了：

- “我先试一个小补丁看看”
- “大概率就是这个”
- “先改了再补验证”
- “这个应该很简单”
- “先加几个改动一起试”
- “不用写测试，我手动看下就知道”
- “我觉得差不多了”
- “已经试两次了，再来一次应该能好”

一旦出现这些念头，立刻退回第一阶段。

## 用户提示你跑偏的信号

如果用户说：

- “你是不是还没确认？”
- “先别猜”
- “证据呢？”
- “这只是你的推测吧？”
- “先查清楚再改”

说明你已经开始跳过流程了，必须退回根因调查。

## 常见借口对照表

| 借口 | 事实 |
|---|---|
| “这很简单，不需要流程” | 简单问题也有根因。 |
| “时间紧，先修” | 乱修更慢。 |
| “先补一下，回头再查” | 第一个补丁会决定后面全都乱掉。 |
| “我先加几个修复一起试” | 这样你永远不知道到底哪个才有效。 |
| “我已经大概知道了” | 大概知道 ≠ 有证据。 |
| “我手动看过了，不用测试” | 手动观察不能代替可重复验证。 |

## 快速清单

| 阶段 | 关键动作 | 达成标准 |
|---|---|---|
| 根因调查 | 读报错、复现、查变化、加证据 | 搞清楚问题发生在哪、为什么发生 |
| 模式对比 | 找正确示例、比差异 | 知道异常实现和正常实现差在哪 |
| 假设验证 | 一次一个假设、最小验证 | 明确锁定根因或推翻假设 |
| 实施修复 | 先失败用例，再最小修复，再回归 | 问题消失且无明显回归 |
