# 合同边界解耦

## 目标

把“这个文件太大”改成“这个模块同时保护了哪些合同”。解耦不是把函数搬到更多文件里，而是把输入、状态、依赖、输出、诊断、测试这些边界重新命名，让每个模块只负责一类变化。

## 先判断问题类型

出现这些信号时，优先按合同边界解耦处理：

- 一个入口同时做配置适配、输入清洗、执行编排、状态统计、公开输出组装。
- 同名变量在不同层有不同含义，例如 raw input 和 typed object 混用。
- 测试只能 patch 模块全局对象、时间、随机数、外部客户端，说明依赖入口不清楚。
- 内部诊断字段被页面、历史记录、API 响应或导出功能间接消费。
- helper 被直接调用时绕过了主链路校验、默认值、降级记录或 strict 错误顺序。
- review finding 集中出现在 size guard、QualityGate、兼容别名、summary projection、direct-call contract。

如果只是普通重复代码或局部函数太长，不要套用完整流程，先做小范围重构。

## 阶段 1：画出合同地图

改代码前，先列出当前模块保护的合同。

至少追这几类链路：

- 入口合同：公开函数、路由、任务入口、CLI、兼容旧路径、返回类型。
- 输入合同：用户输入、配置快照、raw dict、DTO、数据库行、外部 API payload。
- 转换合同：raw 到 typed 的强校验位置，默认值补齐位置，strict 首错顺序。
- 执行合同：核心算法、外部服务、重试、超时、随机数、时钟、并发锁。
- 状态合同：best/current/attempts/trace/stats/cache/progress 等运行态变量。
- 输出合同：API 响应、页面字段、历史落库、导出文件、日志、diagnostics。
- 测试合同：哪些行为必须进 regression，哪些高风险测试必须进 QualityGate。

输出一张简表：合同名、当前代码位置、消费者、失败表现、是否公开、是否需要保持兼容。

## 阶段 2：深挖根因

不要只问“怎么拆文件”，要问“为什么这些职责会挤在一起”。

常见深层根因：

- 多个层各有事实源，例如页面配置、服务配置、核心算法配置互相重叠。
- raw 数据和 typed 数据没有明确边界，导致坏输入被下游静默清理。
- 搜索、执行或导入流程有隐含状态模型，但状态散落在多个 helper 里。
- 内部诊断被前端或持久化间接依赖，实际已经变成公共合同。
- 没有显式依赖入口，测试只能 monkeypatch 全局变量。
- 质量门禁只跑旧测试，新合同测试没有绑定到必跑集合。

如果用户要求多个 subagent，把任务按链路拆开：配置/输入、执行状态、依赖接缝、输出摘要、门禁测试。主代理负责合并结论和去重，不把当前最阻塞的判断丢给子代理等待。

## 阶段 3：先锁行为

生产代码前先补合同测试。测试要证明“不变”和“新边界不会被绕开”。

优先覆盖：

- 坏配置、缺字段、坏输入混合时，错误顺序保持现状。
- raw 输入只能在指定边界强校验，typed 对象由核心层按自己的规则处理。
- 公开入口签名、返回形状、兼容旧路径保持不变，除非 plan 明确要改。
- fake clock、fake rng、fake client、fake executor 能通过显式 runtime seam 控制。
- 公开输出字段走 allowlist，内部诊断进入明确 diagnostics 或 operation log。
- size guard、分页、截断、导出上限真的限制最终结果，不只是打标。
- direct-call helper 也必须走完整归一化，不允许“看起来像对象”就短路。
- 新增高风险合同测试登记进 QualityGate 或项目等价门禁。

## 阶段 4：按边界拆模块

拆分时按“变化原因”命名模块，不按原文件里的函数顺序命名。

常见拆法：

- `*_config`：只解析当前层自己的配置合同，不抢核心层事实源。
- `*_contracts` 或 `*_input`：只负责 raw 输入到 typed 对象的强校验转换。
- `*_runtime`：集中放 clock、rng、factory、client、hook、executor 等可替换依赖。
- `*_state`：集中管理运行状态和对外统计形状。
- `*_execution` 或 `*_search`：放核心执行循环、搜索循环、重试循环。
- `*_public_summary` 或 `*_projection`：把内部结果投影成公开输出，明确 allowlist。
- `*_diagnostics`：承接内部排查信息，并纳入持久化、大小和隐私规则。

主入口拆完后只做编排：拿输入、调用边界模块、执行核心流程、组装返回。不要让核心算法或底层模块反向依赖服务层。

## 阶段 5：处理复审问题

review finding 不要当成孤立 bug，要反推是哪条合同没有写死。

处理方式：

- 先复现：补一个最小失败测试，让问题从“可能”变成“会失败”。
- 再定性：它破坏的是输入、状态、依赖、输出、持久化、门禁里的哪条合同。
- 再修根因：收紧边界或移动职责，不用宽泛 fallback、吞异常、静默默认值盖住。
- 再扩验证：跑失败测试、相邻 regression、门禁 registry、lint 和 diff check。
- 最后复审：用对抗视角重新看是否还有绕过路径。

典型例子：如果截断标记为 true 但最终 JSON 仍超限，根因不是“少删一个字段”，而是 size guard 没有对最终输出负责。

## 阶段 6：收口

完成前确认这些问题都能回答：

- 每个新模块只负责哪一类合同。
- 哪些行为被测试证明保持不变。
- 哪些 raw/typed 边界、public/internal 边界、runtime dependency 边界被写死。
- 哪些新增测试已经进入门禁。
- 是否还有未跟踪源码或测试落在 intended boundary 外。

推荐验证：

- 目标 regression 和新增合同测试。
- 与输出、持久化、分析页、导出、质量门禁相关的相邻测试。
- `ruff check` 或项目等价 lint。
- `git diff --check`。
- 项目级 post-change check。

如果证据不足，就明确说证据不足；不要把“局部测试能跑”包装成“整体已经安全”。

## 反模式

避免这些做法：

- 只移动函数，不移动合同和测试。
- 为了兼容加入宽泛 `getattr`、空 dict、静默降级、吞异常。
- 让内部诊断混进公开字段，或让公开字段没有 allowlist。
- 标记 truncated 但不保证最终大小真的受控。
- 新增测试不进门禁，导致合同只在本地存在。
- 直接调用 helper 时能绕过主链路校验。
- 用子代理堆数量替代清晰的链路拆分和主代理综合判断。
