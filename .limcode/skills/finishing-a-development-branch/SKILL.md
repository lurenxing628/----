# 完成开发分支

## 核心观念

**先验证，再给选项；用户选完，再执行；最后再清理。**

调用本技能后，先告诉用户：

> 我正在使用 `finishing-a-development-branch` 来完成这次工作。

## 第 1 步：先验证测试

在给任何收尾选项之前，先验证测试。

优先顺序：

1. 先跑 plan 中明确要求的最终验证
2. 再跑本次改动最关键的回归组合
3. 如用户要求或项目约定需要，再跑更广的检查

对 APS 仓库，常见命令会是：

```bash
python -m pytest tests/xxx.py tests/yyy.py -q
```

如果测试失败，必须停下，并告诉用户：

```text
当前还有测试失败，不能进入合并 / 推送 / 丢弃流程。
需要先处理这些失败：
- ...
```

## 第 2 步：确认基线分支

优先用 git 判断当前分支通常从哪里切出：

```bash
git merge-base HEAD main
git merge-base HEAD master
```

如果结果不清楚，直接问用户：

> 这次工作默认回到哪个基线分支？是 `main`、`master`，还是别的分支？

## 第 3 步：只给这 4 个选项

向用户呈现**准确且固定**的 4 个选项，不要自己加第五个：

```text
实现已经完成。你希望怎么处理？

1. 本地合并回 <基线分支>
2. 推送并创建 Pull Request
3. 保留当前分支，稍后我再处理
4. 丢弃这次工作

请选择一个编号。
```

不要把解释写成长段，保持清楚直接。

## 第 4 步：执行用户选择

### 选项 1：本地合并

```bash
git checkout <基线分支>
git pull
git merge <功能分支>
```

合并后，必须再做一次验证。

验证通过后：

```bash
git branch -d <功能分支>
```

然后进入“工作树清理”。

### 选项 2：推送并创建 Pull Request

先确认分支已准备好，再执行：

```bash
git push -u origin <功能分支>
```

如果环境安装了 `gh` 并且用户希望直接建 Pull Request，再执行：

```bash
gh pr create --title "<标题>" --body "<摘要与验证说明>"
```

说明：

- 默认**保留分支和工作树**，因为 Pull Request 期间通常还要继续改
- 如果用户明确要求“推完就清理工作树”，再执行清理

### 选项 3：保留当前分支

直接告诉用户：

- 分支名
- 当前工作树路径
- 当前状态（是否干净、是否还有未提交改动）

**不要清理工作树。**

### 选项 4：丢弃这次工作

必须先做强确认。

提示词：

```text
这会永久删除当前分支及其提交，并可能移除对应工作树。
如果你确认要丢弃，请明确回复：discard
```

只有用户明确回复 `discard`，才能继续：

```bash
git checkout <基线分支>
git branch -D <功能分支>
```

然后进入“工作树清理”。

## 第 5 步：工作树清理

如果这次工作使用了 git 工作树，再决定是否清理。

### 默认清理策略

- 选项 1：清理
- 选项 2：默认保留；用户明确要求时再清理
- 选项 3：保留
- 选项 4：清理

清理前先确认当前分支对应的工作树路径：

```bash
git worktree list
```

再执行：

```bash
git worktree remove <工作树路径>
```

## 常见错误

### 错误 1：跳过测试验证

后果：

- 合并了坏代码
- 建了会挂的 Pull Request

### 错误 2：不给固定选项，问得太散

后果：

- 用户得重新组织自己的决策
- 收尾流程变得含糊

### 错误 3：没确认就删分支

后果：

- 工作不可恢复

### 错误 4：Push 以后立刻清工作树

后果：

- Pull Request 期间要改东西时又得重新找回环境

## 与其他技能的关系

- `using-git-worktrees`：本技能会清理它创建的工作树
- `git-commit-safe-workflow`：如果推送前还需要补提交或 amend，先切过去处理
- `requesting-code-review`：正式收尾前最好已经做过必要 review

## 红旗信号

- 测试还没过，就开始问合并方式
- 用户没确认，就直接删分支
- 合并后不再验证一次
- Push 后默认强删工作树
- 用户明明要保留现场，你却帮他清掉了

## 最终要求

**没有测试通过，就没有收尾选项。**

先验证，再选路，再执行。
