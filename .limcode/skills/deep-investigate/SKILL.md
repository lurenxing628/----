# 三轮对抗式排查

> **定位**：比 `aps-deep-review`（代码审查）更聚焦——专门用于"用户怀疑有 bug / 一致性问题，但不确定是否属实"的场景。
> 核心原则：**先证伪、再定位、最后才出方案**。不接到问题就改代码。
> 方法论：**三轮对抗**——Round 1 发散、Round 2 收敛、Round 3 换人审。

## 与其他 Skill 的区别

| Skill | 触发场景 | 核心动作 |
|-------|---------|---------|
| `aps-post-change-check` | 改完代码后 | 跑 lint + 架构合规 + 联动提醒 |
| `aps-deep-review` | 改动前/合并前 | 引用链追踪 + 边界检查 + 回归风险 |
| **`deep-investigate`（本 Skill）** | **怀疑有 bug 但不确定** | **三轮对抗：发散 → 收敛 → 独立审核员换人审** |

## 触发词

用户提到以下关键词时触发：

> 深度排查 / 查清楚再改 / 这个问题属实吗 / 一致性问题 / 跨进程不一致 / 帮我验证一下这个假设 / 写 A 库打 B 服务 / 三轮排查 / 对抗审查 / 独立验证

- Round 3 优先使用 LimCode 原生已注册的独立审核子代理；未注册时回退到 `.limcode/agents/independent-auditor.md` 角色正文源文件

## 双轨验证原则（全流程生效）

**脚本和推理，缺一不可。** 此原则贯穿三轮始终。

| 只靠推理 | 只靠脚本 | 双轨 |
|---------|---------|------|
| 容易漏掉运行时才暴露的问题 | 脚本只测它测到的，测不到的更危险 | 互相补位 |

**执行规则**：

1. 能推理的先推理，得出预期结论。
2. 写最小脚本验证推理结论是否正确。
3. 脚本通过后，**不要停下**，继续追问：
   - "脚本覆盖了所有启动方式吗？（IDE / 脚本 / 打包 / 测试）"
   - "脚本覆盖了跨进程场景吗？"
   - "有没有脚本测不到但逻辑上可能出问题的边界？"
4. 对脚本测不到的边界，用逻辑推理补充风险说明。

**反面教材**：
- 跑了脚本就收工，不追问覆盖率 → 遗漏风险
- 推理了半天不跑脚本，"应该没问题" → 运行时打脸
- 脚本报错就认定有问题，不推理脚本本身是否写对了 → 误判

## 工作流（给 Agent 的执行指引）

### Round 0: 接收假设，建立 Todo

1. 把用户描述的问题记录为**待验证假设**（不是已确认事实）。
2. 创建 Todo，包含以下条目：
   - [ ] Round 1: 发散发现——初步调查 + 发现清单
   - [ ] Round 2: 收敛验证——逐条质疑 + 方案比较
   - [ ] Round 3: 独立对抗审——启动 independent-auditor 换人审
   - [ ] 汇总: 三轮裁决——合并分歧 + 最终结论
   - [ ] 子代理门禁：确认哪些步骤使用项目调查代理，哪些步骤需要兼容回退

3. 涉及子代理兼容与回退时，统一按 `.limcode/skills/_shared/subagent-compat.md` 执行；不能静默降级。

---

### Round 1: 发散发现（默认主 Agent）

> 默认由主 Agent 执行。只有在确实需要并行且通过共享子代理门禁时，才把 Round 1 分发给调查代理。优先使用 LimCode 原生已注册的调查子代理；若当前宿主尚未注册，则以 `.limcode/agents/deep-investigate-worker.md` 作为角色正文源文件，其余兼容回退统一遵循 `.limcode/skills/_shared/subagent-compat.md`。

**目标**：用证据尽量找全问题，不急着下结论。

**Step 1.1: 识别关键变量**

识别假设中的关键变量（如：DB 路径、端口号、枚举值、配置项）。

**Step 1.2: 追踪决策链**

对每个关键变量：
- 在源码中，这个值从哪里来？（硬编码 / 环境变量 / 配置文件 / 运行时参数）
- 在不同启动方式下，这个值是否一致？（IDE 启动 / 脚本启动 / 打包启动 / 测试启动）
- 跨进程场景下，这个值是否可能不一致？（调用方进程 vs 被调用方进程）

画出决策链（文字即可）：
```
变量 X 的值由以下链路决定：
环境变量 FOO → config.py 读取 → ServiceA 构造函数 → RepositoryB 连接 → 实际操作
              ↑ 分叉点          ↑ 断裂点
```

**Step 1.3: 初步同类扫描**

用 `rg`（ripgrep）全局搜索问题模式，覆盖：
- 主代码（`core/`、`web/`、`data/`）
- 脚本（`.limcode/skills/`、`start.bat`、`*.ps1`）
- 测试（`tests/`）
- 文档/workflow（`.limcode/`）

**Step 1.4: 双轨验证**

按"双轨验证原则"执行：推理 + 脚本 + 追问覆盖率。

**Step 1.5: 输出发现清单**

```markdown
## Round 1 发现清单

### 假设验证
- 假设：{{用户原始描述}}
- 初步判断：属实 / 部分属实 / 不属实 / 待进一步验证
- 证据：{{文件路径:行号，关键片段}}

### 决策链
- {{关键变量的完整链路}}
- 分叉点：{{列出}}
- 断裂点：{{列出}}

### 同类问题（初步）
| 文件 | 问题 | 严重度 | 备注 |
|------|------|--------|------|

### 初步方案方向
- 方向 A：{{一句话}}
- 方向 B：{{一句话}}
- 方向 C：{{一句话}}

### 自评：Round 1 可能遗漏了什么
- {{诚实列出自己不确定的地方}}
```

**关键约束**：
- 不要假设"当前 shell 的环境变量"等于"另一个进程的实际配置"。
- 不要假设"代码里写了默认值"就等于"运行时一定用的是默认值"。
- 不要假设"文件存在"就等于"文件被正确加载了"。
- 对每个关键文件只读一次，读后直接分析。

**输出后等用户说"继续"才进入 Round 2。**

---

### Round 2: 收敛验证（默认主 Agent）

> Round 2 与 Round 1 相同：默认由主 Agent 执行。只有在共享子代理门禁已经通过时，才允许并行委派；优先使用 LimCode 原生已注册的调查子代理，未注册时回退到 `.limcode/agents/deep-investigate-worker.md` 角色正文源文件。

**目标**：对 Round 1 的每条发现逐一质疑，补遗漏、修正错误、给出可决策的方案。

**Step 2.1: 逐条质疑 Round 1**

对 Round 1 的每条发现，逐条追问：
- "这条结论的证据充分吗？有没有反例？"
- "这个引用链有没有漏掉的分支？"
- "同类排查有没有漏网的？"
- "初步方案方向有没有遗漏的替代方案？"

**Step 2.2: 双轨补充验证**

对 Round 1 中标注为"待进一步验证"或"不确定"的地方：
- 推理能解决的先推理
- 推理不确定的写最小脚本验证
- 验证后继续追问覆盖率

**Step 2.3: 修正发现清单**

明确标注哪些发现被修正、哪些被确认、哪些新增。

**Step 2.4: 方案比较（大白话四问）**

给出至少 3 个修复方向。

**通用方向模板**（根据具体问题选择适用的）：

| 方向 | 适用场景 |
|------|---------|
| **A. 维持现状** | 问题影响小，修复成本高 |
| **B. 最小补丁** | 只修当前问题点，不改架构 |
| **C. 运行时契约** | 新增文件/接口暴露关键信息，供跨进程对齐 |
| **D. 防御性拒绝** | 无法确认一致性时拒绝执行，报错退出 |
| **E. 显式参数** | 强制由调用方传入关键值，消除隐式依赖 |
| **F. 集中配置** | 把分散的硬编码收敛到单一配置源 |

#### 大白话表达规则（Round 2-3 强制生效）

向用户展示方案和结论时，**必须用大白话**。技术细节放在折叠块或附录中，正文只说人话。

**每个方案必须回答这四个问题**：

```markdown
### 方案 X：{{一句话名字}}

**问题出在哪？**
> 用一两句日常口语解释问题的本质。
> 例："脚本写死了数据库地址，但你的服务可能用的是另一个数据库，
>      导致脚本往 A 库写数据、服务从 B 库读数据，两边对不上。"

**怎么解决？**
> 用一两句话说清楚方案思路，不涉及代码细节。
> 例："让脚本启动时先去问一下正在运行的服务'你用的是哪个数据库'，
>      拿到真实地址后再操作。"

**成本是什么？**
> 要改几个文件、大概多少行、需不需要改数据库结构、会不会影响现有功能。
> 例："改 2 个脚本文件，加起来约 20 行。不改数据库，不影响现有功能。
>      回退的话 git revert 一下就行。"

**会引起什么后果？**
> 改了之后有什么好处、有什么潜在风险、最坏情况是什么。
> 例："好处：以后不管数据库路径怎么配，脚本都能自动对齐。
>      风险：如果服务没启动，脚本会报错退出
>     （但这其实是正确行为，总比写错库好）。"
```

**方案对比表**（在所有方案列完之后，附一个汇总表）：

```markdown
| 对比项 | 方案 A | 方案 B | 方案 C |
|--------|--------|--------|--------|
| 一句话描述 | 不改，接受现状 | 只打个补丁 | 加个运行时对齐机制 |
| 要改几个文件 | 0 | 1 | 3 |
| 能不能根治 | 不能 | 治标不治本 | 能 |
| 会不会影响现有功能 | 不会 | 不会 | 极小概率 |
| 改完出了问题怎么回退 | 不需要 | git revert | git revert + 删 1 个文件 |
| 后续维护麻烦吗 | 不麻烦（但问题还在） | 不麻烦 | 稍微多一点点 |
```

**最后给一句推荐**：

```markdown
**我的建议**：选方案 X。
> 用一句大白话说理由。
> 例："性价比最高——改动最小，但能根治问题，而且出了事随时能回退。"
```

**输出后等用户说"继续"才进入 Round 3。**

---

### Round 3: 独立对抗审（子代理 — independent-auditor）

**目标**：由一个独立的审核员（子代理），在全新的上下文中，对 Round 1-2 的结论做对抗性验证。

#### 为什么需要换人审

Round 1-2 都是同一个 Agent 在同一个上下文窗口中完成的。即使 Round 2 要求"质疑 Round 1"，同一个 Agent 天然倾向于确认自己之前的结论（确认偏差）。Round 3 启动一个独立 Subagent，它：
- 看不到主 Agent 在 Round 1-2 的推理过程
- 只看到打包给它的"结论"（当作未验证声明）
- 独立读代码验证，天然无偏差

#### 主 Agent 的工作

1. **打包待审材料**：将 Round 1-2 的所有结论整理为结构化文本，每条标注为"未经独立验证的声明"。
2. **调用 independent-auditor**：

```
优先路径：LimCode 原生已注册的独立审核子代理；未注册时回退到 `.limcode/agents/independent-auditor.md` 角色正文源文件
兼容回退：统一遵循 `.limcode/skills/_shared/subagent-compat.md`
传入内容：
  1. 用户的原始问题描述
  2. Round 1-2 的所有结论（逐条，标注为"待审声明"）
  3. Round 2 的推荐方案及理由
```

#### 主 Agent 收到审核报告后

**Step 3.1: 汇总三轮结果**

```markdown
## 三轮汇总

### 一致的结论（三轮都认同）
- {{列出}}

### 分歧点（Round 3 审核员提出异议）
| # | Round 1-2 结论 | Round 3 判定 | 分歧原因 |
|---|---------------|-------------|---------|

### Round 3 新发现（前两轮遗漏）
- {{列出}}
```

**Step 3.2: 裁决分歧**

对每个分歧点，主 Agent 必须：
- 重新读代码验证（不能只凭记忆）
- 给出裁决：采纳 Round 1-2 / 采纳 Round 3 / 需要更多证据
- 附上裁决证据

**Step 3.3: 输出最终结论并落 review（大白话格式）**

```markdown
## 1. 这个问题是真的吗？
- 你说的是：{{用一句大白话复述用户的原始描述}}
- 我查下来：{{属实 / 部分属实 / 不属实}}
- 证据：{{用大白话说在哪个文件的哪一行看到了什么，证明了什么}}
- 独立审核员意见：{{一致 / 有补充 / 有异议}}

## 2. 根因是什么？
> 用大白话解释"为什么会出这个问题"。
> 然后附上技术层面的引用链：
- 决策链：{{环境变量 → config → Service → Repository → 实际操作}}
- 在哪一步断了/分叉了：{{列出}}

## 3. 还有没有类似的问题？
| 文件 | 问题（大白话） | 严重吗 |
|------|---------------|--------|

## 4. 方案对比
（Round 2 的完整方案对比，经 Round 3 审核后如有修正则标注）

## 5. 我的建议
- 推荐：方案 X
- 一句话理由：{{大白话}}
- 独立审核员是否认同：{{是 / 否，理由}}

## 6. 要改哪些东西
- 文件清单和每个文件改什么（一句话）：
- 会不会改数据库结构：是/否
- 需不需要同步更新文档：是/否
- 风险：{{大白话说最坏情况}}
- 回退方案：{{大白话说怎么撤回}}

## 7. 改完怎么验证
- {{大白话说测试步骤}}

## 8. 还有什么隐患
- {{大白话说已知局限}}
```

**Step 3.4: 落排查结论文档**

- 将最终排查结论写入 `.limcode/review/YYYY-MM-DD_<主题>_review.md`
- 如果用户需要“取舍型审阅报告 / 审计归档 / 排期建议”，再额外写入 `audit/YYYY-MM/`

**改动前门禁**：最终结论与 review 文档输出后，**必须等用户确认**才能开始修改代码。

## 执行纪律

1. **不要一上来就改代码**。三轮全部完成、最终结论输出并经用户确认后，才进入修改阶段。
2. **Round 之间必须等用户说"继续"**才进入下一轮。不得自动跳轮。
3. **变更边界严格**。只改与问题直接相关的文件，不顺手修无关问题。发现的无关问题以文字列出，不修改。
4. **读写纪律**。对单个文件只读一次、只写一次。禁止对同一文件反复 grep、反复读取、反复修改。
5. **证据优先**。每个结论必须附带证据（文件路径 + 行号或命令输出）。不确定的地方写验证脚本。
6. **Round 3 必须优先使用 LimCode 原生已注册的独立审核子代理**；若当前宿主尚未注册，则以 `.limcode/agents/independent-auditor.md` 作为角色正文源文件，兼容回退统一遵循 `.limcode/skills/_shared/subagent-compat.md`。
7. **分歧不得静默忽略**。如果 Round 3 和 Round 1-2 有分歧，主 Agent 必须明确说明分歧点和裁决理由。
8. **全程中文简体回复**。
9. **并行前先过子代理门禁**。当用户要求多个子代理并行排查时，先确认共享兼容门禁是否满足；不满足就说明限制，不能静默降级。

## 约束

- 本 Skill 只做调查和出计划，不自动修改代码。
- 如果 Round 1 结论已经很明确（假设完全不属实），直接告知用户，可跳过 Round 2-3。
- 如果问题过于复杂（涉及 10+ 文件），建议用户拆分为多次排查。
- Subagent 不支持嵌套（Round 3 的审核员不能再启动 subagent）。
