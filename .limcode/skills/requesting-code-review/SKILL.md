# 发起代码审查

## 核心观念

越早 review，越少返工。

代码审查不是“最后走个形式”，而是：

- 防止实现偏离 design / plan
- 防止任务间错误累积
- 防止把问题带到下一阶段或主分支

## 什么时候必须做

### 必做

- `subagent-driven-development` 的**每个任务**完成后
- 一个重要功能完成后
- 合并前

### 强烈建议做

- 修完复杂 BUG 后
- 做完一轮重构后
- 改动跨 3 个以上文件时
- 改了 Service 签名、枚举、Schema、导入导出链路时

## 审查路径

优先顺序：

1. **优先使用 LimCode 原生已注册的审查子代理**（项目模板与安装口径见 `.limcode/subagents/README.md`）
2. 如果当前接口还没有注册原生子代理：
   - 使用当前环境可用的通用子代理
   - 把 `.limcode/agents/code-reviewer.md` 作为角色正文源文件，完整内联其审查标准与输出格式
3. 如果当前环境不支持子代理：
   - 主代理按同样清单做内联 review
   - 但必须明确告诉用户：这是降级路径，质量不如独立审查

## review 落盘约定

默认情况下：

- 普通任务级 review 可以只在对话中输出
- 但只要满足下面任一条件，就建议落到 `.limcode/review/`：
  - 跨 3 个以上文件
  - 涉及公开 Service 签名、Schema、枚举、导入导出链路
  - 用户要求留痕
  - review 发现 `Critical` 或 `Important` 问题

默认命名：`.limcode/review/YYYY-MM-DD_<主题>_review.md`

## 审查前准备材料

至少准备这些信息：

- 本次实现了什么
- 它本来应该做什么（来自 design / plan / 用户要求）
- 本次变更文件清单
- 关键提交范围
- 已跑过哪些测试
- 哪些地方你自己最没把握

如果能拿到提交范围，建议准备：

```bash
git rev-parse HEAD
git rev-parse HEAD~1
```

如果需要更准确基线，可以用主分支或任务起点提交作为 `BASE_SHA`。

## 审查步骤

### 1. 先明确审查目标

至少说清楚：

- **实现内容**：这次到底改了什么
- **验收依据**：design / plan / 用户要求是什么
- **审查范围**：哪些文件、哪些提交

### 2. 先做“是否按要求实现”审查

先看：

- 有没有漏需求
- 有没有做多了
- 有没有跟 plan 冲突
- 有没有悄悄扩大范围

这一步先于代码风格和重构建议。

### 3. 再做代码质量审查

再看：

- 是否遵守分层边界
- 是否有明显回归风险
- 是否有糟糕测试
- 是否有命名、复杂度、错误处理问题
- 是否有未收口的文档联动点

### 4. 处理反馈

- **关键问题**：必须先修
- **重要问题**：原则上修完再继续
- **次要问题**：可记录后处理，但不能假装没看见

如果你认为审查意见不对，可以反驳，但必须给技术理由和证据。

## 推荐输出格式

```markdown
## review 结果

### 优点
- ...

### 问题
| 严重度 | 位置 | 问题 | 为什么重要 | 建议 |
|---|---|---|---|---|

### 结论
- 可以继续 / 先修后继续 / 建议升级到深度审查
```

如果需要留痕，按同样结构写入 `.limcode/review/`。

只有在用户明确要求“取舍型审计 / 后续排期建议 / 审计归档”时，才额外写入 `audit/YYYY-MM/`；
普通代码审查默认归档到 `.limcode/review/`，不要一上来就写进 `audit/`。

## 触发深度审查的升级条件

如果本次改动满足任一条件，应该主动建议升级到 `aps-deep-review`：

- 改了公开 Service 方法签名
- 改了 `schema.sql` 或迁移脚本
- 改动跨 3 个以上模块
- 改了核心枚举或统一契约
- 你怀疑调用链还有没覆盖的地方

提示话术可以用：

> 这次改动已经超出普通任务级 review 的舒适区，建议补一次 `aps-deep-review`，确认调用链和影响面都收干净。

## 与其他技能的关系

- `test-driven-development`：先保证改动有测试，再来 review
- `subagent-driven-development`：每个任务后都要走一轮 review
- `executing-plans`：至少每个阶段结束要走 review
- `finishing-a-development-branch`：最终收尾前应完成必要 review

## 红旗信号

下面这些做法不能接受：

- 因为“改动很小”就跳过 review
- review 发现关键问题还继续推进
- 把 review 当成纯样式检查
- 没对照 design / plan 就直接说“代码看起来不错”
- 用户问风险时，只回一句“应该没问题”

## 最终要求

**先核对“做得对不对”，再核对“写得好不好”。**

顺序反了，review 就失焦了。
