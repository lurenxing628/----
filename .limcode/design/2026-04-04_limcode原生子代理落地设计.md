# 背景

当前仓库里的 `.limcode/agents/*.md` 更接近“角色提示词正文”或 Cursor 风格遗留物，而不是 LimCode 宿主原生会自动扫描的子代理注册目录。

根据 Lim-Code 源码，可确认两件关键事实：

1. **技能** 由 `backend/modules/skills/SkillsManager.ts` 扫描，优先目录是：
   - `<workspace>/.limcode/skills`
   - `<workspace>/.agents/skills`
2. **子代理** 不是从 `.limcode/agents/` 目录扫描，而是由 `backend/tools/subagents/*` + `backend/modules/settings/*` 共同驱动：
   - 原生工具名：`subagents`
   - 注册表：`backend/tools/subagents/registry.ts`
   - 默认执行器：`backend/tools/subagents/executor.ts`
   - 配置入口：`limcode.toolsConfig.subagents`
   - 工作区级可落点：`.vscode/settings.json` 中的 `"limcode.toolsConfig"`

因此，当前仓库需要把“角色定义”和“宿主原生子代理注册”明确拆开。

# 目标

1. 让本仓库对子代理的设计**对齐 LimCode 原生模型**。
2. 保留现有 `.limcode/agents/*.md` 中已经整理好的角色正文，不浪费已有工作。
3. 明确区分：
   - 角色正文源文件
   - 宿主原生子代理配置
   - 通用回退路径
4. 给出可以直接落在仓库里的**项目级模板与安装口径**。

# 非目标

1. 不修改 Lim-Code 上游源码。
2. 不假装 `.limcode/agents/` 会被 LimCode 自动扫描注册。
3. 不在不知道用户真实 `channelId` 的前提下，强行生成可立即运行的启用态子代理。

# 设计原则

## 1. 原生机制优先

只要宿主是 LimCode，就优先使用它原生的：

- `subagents` 工具
- `SubAgentRegistry`
- `limcode.toolsConfig.subagents`

而不是继续把 `.limcode/agents/*.md` 宣称为“已注册子代理”。

## 2. 角色正文与注册配置分层

- `.limcode/agents/*.md`：保留为**角色正文源文件 / 提示词模板源**
- `.limcode/subagents/*.jsonc`：保存**原生子代理配置模板**
- `.vscode/settings.json`：项目级实际注入口（如果用户愿意启用）

## 3. 渠道配置延迟绑定

LimCode 原生子代理配置必须包含：

- `channel.channelId`
- 可选 `channel.modelId`

这两个值是用户环境相关信息，仓库里不能伪造。

因此项目内应提供：

- 可复制的模板
- 明确的占位说明
- 可能的话，提供生成脚本或安装说明

但默认不应把未知占位值标成可直接运行的启用态配置。

# 目标结构

## 一、角色正文层

继续保留：

- `.limcode/agents/deep-investigate-worker.md`
- `.limcode/agents/independent-auditor.md`
- `.limcode/agents/aps_doc_scan.md`

但文档定位改成：

> 这些文件是角色正文源文件，不等于 LimCode 已原生注册的子代理。

## 二、原生子代理模板层

新增：

- `.limcode/subagents/README.md`
- `.limcode/subagents/limcode-subagents.template.jsonc`

职责：

- 说明 LimCode 原生子代理的真实配置位置
- 给出首版四件套和专项扩展件
- 给出 `limcode.toolsConfig.subagents` 的模板
- 明确 `channelId` 需要用户绑定

## 三、工作区注入层

项目级原生注入位置为：

- `.vscode/settings.json`

写法为：

```jsonc
{
  "limcode.toolsConfig": {
    "subagents": {
      "maxConcurrentAgents": 1,
      "agents": [
        {
          "type": "aps-plan-implementer",
          "name": "APS 实施子代理",
          "description": "按单任务边界做最小实现、验证与自查",
          "systemPrompt": "...",
          "channel": {
            "channelId": "<由用户填写>"
          },
          "tools": {
            "mode": "whitelist",
            "list": ["..."]
          },
          "maxIterations": 12,
          "maxRuntime": 600,
          "enabled": false
        }
      ]
    }
  }
}
```

默认保持 `enabled: false`，直到用户填入真实渠道并确认启用。

# 首版子代理集合

## 首版通用四件套

1. `aps-plan-implementer`
2. `aps-spec-reviewer`
3. `aps-code-quality-reviewer`

### 设计理由

这是最小可闭环的一条链：

- 一个负责实现
- 两个负责任务级两阶段审查
- 一个负责阶段末整体验收

足以支撑：

- `subagent-driven-development`
- `requesting-code-review`
- 大多数实现型流程

## 第二批专项件

1. `aps-deep-investigate-worker`
2. `aps-independent-auditor`
3. `aps-doc-scan`

### 设计理由

这些角色适合复杂排查、对抗审和大文档扫描，不应在首版就和通用四件套混为一谈。

## 并发上限

项目默认建议：

- `maxConcurrentAgents = 1`

理由：

- 同一分支上的多个实现型子代理并发写代码风险高
- 先把单链路跑顺，比追求并发更重要
- 真要并发时，再由用户按任务性质提高上限

# 与现有技能的关系

## `subagent-driven-development`

应改成优先依赖：

1. LimCode 原生已注册子代理
2. 若未注册，则回退到角色正文源文件 + 宿主通用子代理
3. 若仍不可用，则主代理降级执行

## `feature-design` / `deep-investigate`

它们当前引用 `.limcode/agents/*.md` 的地方，应解释为：

- 默认角色正文源文件
- 不是 LimCode 自动发现目录
- 若用户已按模板注册原生子代理，则优先调用原生子代理

# 迁移口径

## 当前需要修正的表述

需要避免继续使用以下不准确口径：

- “`.limcode/agents/` 是 LimCode 原生子代理定义目录”
- “宿主会直接从 `.limcode/agents/` 自动注册子代理”

## 替代表述

统一改为：

- `.limcode/agents/`：角色正文源文件 / 兼容回退提示词来源
- `limcode.toolsConfig.subagents`：LimCode 原生子代理注册配置
- `.vscode/settings.json`：项目级原生注入位置

# 落地步骤

1. 新增 `.limcode/subagents/README.md`
2. 新增 `limcode.toolsConfig.subagents` 模板文件
3. 改写 `.limcode/agents/README.md`，明确其不是宿主原生注册目录
4. 改写共享子代理门禁与核心技能说明，区分“原生已注册”与“角色正文回退”
5. 视情况把 `.vscode/settings.json` 补成带模板说明的项目级 LimCode 设置入口

# 风险与回退

## 风险

1. 用户没有真实 `channelId`，导致模板无法直接启用
2. 过早把角色正文和原生注册混为一谈，会继续误导后续维护
3. 若把多个实现型子代理默认并发打开，容易造成同分支写冲突

## 回退

如果用户暂时不想启用 LimCode 原生子代理，也可以只保留：

- 角色正文源文件
- 共享兼容门禁
- 主代理降级执行路径

这样不会影响现有技能继续工作，只是少了原生子代理自动执行能力。

# 最终结论

本仓库后续对子代理的正确设计应是：

- **技能** 继续走 `.limcode/skills/`
- **角色正文源文件** 保留在 `.limcode/agents/`
- **LimCode 原生子代理注册** 明确走 `limcode.toolsConfig.subagents`
- **项目级落点** 优先用 `.vscode/settings.json`

也就是说，后续要做的不是“继续把 `.limcode/agents` 当成原生子代理目录”，而是“围绕 LimCode 原生 `subagents` 机制，把现有角色正文重新组织成可注册、可回退、可维护的项目方案”。
